# Redis Cluster

> 代码：`code/http-redis-cluster/src/main.rs`　运行：  
> `cargo run -p http-redis-cluster`（需要提前准备 Redis Cluster）

[《接入 Redis》](redis.md) 已经讲过单机连接、常用命令和错误映射。本课只补集群：
数据为什么被拆开、客户端怎样找到节点、哪些命令会突然报 `CROSSSLOT`，以及故障时
为什么不能无脑重试。

示例中的地址、key 和节点名全部是本机教学占位符，可以直接替换成自己的测试环境。

----

# 先分清三种方案

Redis 常见高可用方案不是一回事：

| 方案 | 数据放在哪里 | 解决什么 |
| --- | --- | --- |
| 单机 | 一台主节点 | 最简单，容量和故障域都在一台机器 |
| Sentinel | 一主多从，完整数据每台都有 | 主从切换，不做数据分片 |
| Cluster | 数据按槽位分散到多个主节点 | 分片扩容，同时支持主从故障转移 |

如果数据量能放进一台机器，只是担心主节点故障，Sentinel 可能已经够用。
Cluster 的核心价值是 **分片**：不同 key 可以落在不同主节点，因此总容量和吞吐量
不再完全受一台机器限制。

代价也很明确：

- 客户端必须理解槽位和重定向；
- 多 key 操作受到同槽限制；
- 扩缩容会发生槽位迁移；
- 热点 key 仍可能压垮单个节点。

“节点变多”不等于所有命令自动线性扩容。先理解路由规则，才能正确设计 key。

----

# 槽位决定节点

Redis Cluster 固定有 **16384 个 hash slot**，编号是 `0..16383`。
每个 key 会先算出槽位：

```text
slot = CRC16(key) mod 16384
```

集群再维护一张“槽位属于哪个主节点”的表：

```text
0..5460       → 节点 A
5461..10922   → 节点 B
10923..16383  → 节点 C
```

上面的区间只是示意。真实分配会随建群、扩容和迁移变化。

一次 `GET user:42` 的过程是：

1. 客户端对 `user:42` 计算槽位；
2. 查询本地槽位表，找到负责节点；
3. 把命令发到该节点；
4. 如果拓扑已变化，根据重定向更新路由并重试。

所以 Cluster 客户端不是普通 TCP 连接加一个地址数组。它内部要维护多个节点连接、
槽位拓扑和重定向逻辑。

> 🔩 底层视角：槽位在 key 和物理节点之间加了一层间接映射。迁移时只需把一段槽位
> 改派给新节点，不必改变业务 key，也不要求客户端提前知道每个 key 的物理位置。

----

# seed 只是入口

示例从环境变量读取多个初始节点：

```rust
let seeds = vec![
    "redis://127.0.0.1:7000",
    "redis://127.0.0.1:7001",
    "redis://127.0.0.1:7002",
];
let client = ClusterClient::new(seeds)?;
let mut connection = client.get_async_connection().await?;
```

这些地址叫 **seed node**，只是让客户端第一次进入集群的“敲门地址”。
客户端连上其中任意可用节点后，会读取完整拓扑。

几个容易误解的点：

- 不需要把集群所有节点都永久写进配置；
- 但只写一个 seed，会让启动依赖那个节点当时可用；
- seed 必须属于同一个集群，认证信息也应一致；
- `ClusterClient::new` 主要检查地址，真正联网发生在
  `get_async_connection().await`。

示例允许使用逗号分隔的环境变量：

```bash
REDIS_CLUSTER_NODES=redis://127.0.0.1:7000,redis://127.0.0.1:7001
```

不要把密码直接写进源码、教程或日志。生产中应通过密钥管理或受控配置注入，
日志只记录脱敏后的节点标识。

# 客户端自动路由

redis-rs 的异步集群连接在
[官方 cluster_async 文档](https://docs.rs/redis/latest/redis/cluster_async/) 中定义。
启用的 feature 是：

```toml
redis = { version = "1", features = ["tokio-comp", "cluster-async"] }
```

连接建立后，普通命令仍使用熟悉的 `AsyncCommands`：

```rust
let visits: i64 = connection.incr("tutorial:visits", 1).await?;
```

客户端计算 key 的槽位，并把命令送到对应节点。异步 `ClusterConnection`
可以 clone、跨任务复用，通常不需要再套一层普通连接池。

不过“可并发复用”不代表可以无限堆请求。某个节点变慢时，大量请求可能在节点连接前排队。
应用层仍要设置：

- 单次命令总超时；
- 业务并发上限；
- 有界重试次数；
- 失败率、排队时间和节点延迟监控。

容量保护要放在故障发生前，不能等请求已经堆满内存才开始限流。

----

# MOVED 是永久改道

如果客户端把命令发给了错误节点，节点可能返回：

```text
MOVED 1234 127.0.0.1:7002
```

含义是：“槽位 1234 现在归另一个节点，请以后发过去。”

合格的 Cluster 客户端会：

1. 解析 `MOVED`；
2. 把当前命令改发到目标节点；
3. 刷新或修正本地槽位表；
4. 后续同槽命令直接走新节点。

业务代码通常不需要手写这套流程。但监控中如果持续出现大量 `MOVED`，说明客户端拓扑
长期过期、代理配置异常，或集群正在频繁变更，不能只把它当普通成功请求忽略。

----

# ASK 是临时绕行

槽位迁移过程中，源节点可能返回：

```text
ASK 1234 127.0.0.1:7003
```

`ASK` 和 `MOVED` 的区别：

| 重定向 | 含义 | 客户端动作 |
| --- | --- | --- |
| `MOVED` | 槽位归属已改变 | 更新长期槽位表 |
| `ASK` | 迁移中，本次先去目标节点 | 只临时改发这次命令 |

处理 `ASK` 时，客户端会先向目标节点发送 `ASKING`，再发送原命令。
不要看到一个 `ASK` 就永久改写槽位表，否则迁移尚未完成时会制造更多错误。

这些细节正是应使用成熟 Cluster 客户端，而不是自己按地址轮询节点的原因。

----

# 多 key 必须同槽

单机 Redis 可以直接：

```text
MGET user:42:profile user:42:settings
```

Cluster 中，如果两个 key 落在不同槽位，一条命令无法同时交给两个主节点，会得到：

```text
CROSSSLOT Keys in request don't hash to the same slot
```

受影响的不只是 `MGET`，还包括：

- 同时操作多个 key 的命令；
- `MULTI/EXEC` 事务中的多个 key；
- Lua 脚本声明的多个 `KEYS`；
- 要求同一节点原子完成的业务操作。

处理方法不是捕获错误后重试，因为同样的 key 永远还是不同槽。要么拆成多条独立命令，
接受它们不再原子；要么在 key 设计阶段让相关 key 进入同一槽。

----

# hash tag 控制同槽

Redis 只使用 key 中第一组有效花括号内容计算槽位：

```text
tutorial:user:{42}:profile
tutorial:user:{42}:settings
```

两者都按 `42` 计算，因此一定同槽。花括号中的部分叫 **hash tag**。

示例可以安全执行：

```rust
let values: Vec<Option<String>> = redis::cmd("MGET")
    .arg([profile_key, settings_key])
    .query_async(&mut connection)
    .await?;
```

设计 hash tag 时要同时考虑“原子性”和“分布”：

- `user:{42}:profile`：同一用户相关数据聚在一起，用户之间仍能分散；
- `{all-users}:42`：所有用户都进同一槽，制造巨大热点；
- 空花括号或位置混乱：可能与团队预期不一致。

好的 tag 通常是天然业务分区键，例如租户 ID、用户 ID 或订单 ID。
不要为了让某次 `MGET` 成功，把全系统 key 都塞进一个槽。

----

# pipeline 不等于事务

pipeline 的作用是减少网络往返：一次发送多条命令，再批量读取响应。
它不自动提供原子性，也不能突破跨槽限制。

同槽 pipeline 可以由一个节点处理；跨槽命令是否能被客户端拆分、如何返回部分失败，
取决于具体 API。保姆级原则是：

1. 需要原子性：相关 key 设计成同槽，再使用事务或 Lua；
2. 只为提速：按槽位分组后分别 pipeline；
3. 能接受独立成功失败：直接并发单 key 命令，并明确收集每个结果。

不要把“一批发送”误认为“要么全部成功，要么全部失败”。

----

# 热点不会自动消失

Cluster 只能把 **不同 key** 分散。一个每秒访问几十万次的热点 key 仍只属于一个槽，
最终仍由一个主节点负责。

常见热点来源：

- 全局计数器；
- 所有实例争用的一把锁；
- 全站共享的大对象；
- hash tag 设计过宽，把大量 key 固定到同槽。

可选治理方式：

- 按时间、租户或随机桶拆分计数，读取时聚合；
- 本地缓存只读热点，并设置可接受的一致性窗口；
- 把大对象拆成按需读取的小块；
- 重新设计 tag，让业务分区均匀分布。

拆分前必须先确认业务是否要求强原子性。为了均匀而破坏正确性，同样是事故。

----

# 重试先看幂等

连接在“命令已发出、响应未收到”时断开，客户端无法确定服务端是否已经执行。

对 `GET` 重试通常安全；对 `INCR` 重试可能多加一次；对队列弹出、扣减余额等操作，
重复执行后果更严重。

因此重试策略要按命令语义分类：

| 操作 | 重试判断 |
| --- | --- |
| 纯读取 | 通常可在总预算内有限重试 |
| `SET key 固定值` | 多数场景幂等，但仍要看过期时间等附加效果 |
| `INCR`、`LPUSH` | 非幂等，不能因连接错误直接假设未执行 |
| Lua/事务 | 根据脚本业务语义判断，不能只看 Redis 命令名 |

客户端为了处理 `MOVED/ASK` 的协议级重定向会做必要重发；业务层额外重试仍应有
总时限、次数上限和退避，不能层层各重试三次，最后放大成几十次请求。

----

# 读副本有代价

默认从主节点读取最容易理解，因为刚写入的数据能按主节点顺序被观察。
如果开启副本读，可以分担只读压力，但复制通常是异步的：

1. 写入主节点成功；
2. 客户端立刻从副本读取；
3. 副本尚未追上；
4. 读到旧值或未找到。

这叫复制延迟。是否允许副本读，要由业务一致性要求决定，而不是只看吞吐量。
登录态、锁、刚写后读等场景尤其要谨慎；报表、趋势等可容忍短暂陈旧的读取更适合。

----

# 启动与运行策略

集群暂时不可用时，服务有三种常见选择：

- **硬依赖**：连接失败就拒绝启动，交给部署系统重试；
- **延迟就绪**：进程启动，但 readiness 保持失败，连接恢复后再接流量；
- **软依赖**：保留本地旧数据或关闭非核心功能。

不要只实现“启动时连接一次”。运行过程中还会遇到节点重启、故障转移、槽位迁移和网络抖动。
健康检查也不能只看 TCP 能否连通，至少要观察真实命令成功率和延迟。

建议监控：

- 各命令耗时和超时率；
- 按错误类别统计的失败数；
- `MOVED/ASK/CROSSSLOT` 数量；
- 各节点连接状态和重连次数；
- 热 key、慢命令和大 key；
- 槽位覆盖是否完整。

# 常见错误还原

## 把节点当负载均衡

随机挑节点发送命令会持续收到重定向。应让 Cluster 客户端按槽位路由。

## 所有 key 用同一 tag

跨槽错误消失了，但所有流量集中到一个槽，集群退化成单节点热点。

## CROSSSLOT 后重试

这是 key 设计问题，不是临时网络问题。相同命令重试不会改变槽位。

## 连接错误就重试写

响应丢失不等于命令未执行。先判断幂等性，再决定是否重试。

## 把 seed 当完整拓扑

seed 只负责首次发现。节点扩缩容后，客户端应通过集群协议刷新拓扑。

# 动手实验

1. 用两个没有 hash tag 的 key 执行 `MGET`，观察 `CROSSSLOT`；
2. 把两个 key 改成共同的 `{42}`，用 `CLUSTER KEYSLOT key` 比较槽位；
3. 停掉一个 seed，但保留另一个可用 seed，观察客户端是否仍能发现集群；
4. 迁移一个实验槽位，观察期间的 `ASK` 和完成后的 `MOVED`；
5. 把所有测试 key 固定为同一 tag，观察它们是否全部落到一个节点。

----

# 三句话带走

1. Redis Cluster 把 key 映射到 16384 个槽位，客户端按槽位路由到负责节点。
2. `MOVED` 是永久改道，`ASK` 是迁移期临时绕行；成熟客户端会处理协议细节。
3. 多 key 原子操作必须同槽，hash tag 能控制同槽，但滥用会制造单节点热点。

# 附：本课生词表

- **Redis Cluster** —— Redis 自带的分片与故障转移方案。
- **hash slot** —— key 到物理节点之间的 16384 个逻辑槽位。
- **seed node** —— 客户端首次进入集群、发现完整拓扑的初始节点。
- **拓扑（topology）** —— 节点、角色和槽位归属的整体关系。
- **`ClusterClient`** —— redis-rs 创建集群连接的客户端对象。
- **`ClusterConnection`** —— 维护节点连接并按槽位路由的异步连接句柄。
- **`MOVED`** —— 槽位归属已经改变的永久重定向。
- **`ASK` / `ASKING`** —— 槽位迁移期间的一次性重定向协议。
- **`CROSSSLOT`** —— 一条要求同节点执行的命令包含不同槽位 key。
- **hash tag** —— key 花括号中的槽位计算部分，用于让相关 key 同槽。
- **热点 key** —— 请求高度集中、足以成为单节点瓶颈的 key。
- **幂等（idempotent）** —— 重复执行多次与执行一次具有相同业务结果。
- **复制延迟** —— 主节点写入后，副本尚未同步到最新状态的时间窗口。
