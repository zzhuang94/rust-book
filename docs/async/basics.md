# 线程贵在傻等

> 先建立矛盾：线程是昂贵资源，却大量浪费在「等待」上。

**线程就是雇人干活。** 开一个线程，操作系统就派一个「工人」给你：工人有自己的工位（栈内存，默认约 8MB），由操作系统统一排班（调度）。

线程干重活（算数）没问题，但服务器的日常大部分是 **等**：等网络包、等数据库回复、等磁盘。一个典型 Web 请求，线程 99% 的时间在干等。

同时服务 1 万个连接会怎样：

- 雇 1 万个工人 → 每人一个 8MB 工位，内存爆炸；
- 操作系统排班要「上下文切换」，工人太多，光排班就把 CPU 耗光；
- 最气人的是：这 1 万个工人 99% 的时间都坐着干等。

**核心矛盾：线程是昂贵资源，却大量浪费在「等待」上。**

----

# 交单走人的思路

> async 把「等待」从工人身上剥离出来。

> 餐厅服务员（线程）给 3 号桌点完菜，把单子交给厨房， **不站在厨房门口等菜**，而是转身去服务 7 号桌。菜好了厨房按铃，服务员再回来端菜。

对应到代码，`.await` 就是那个「交单走人」的点。于是 4 个服务员（线程）就能伺候 1 万桌（连接）——因为没人傻等，人人都在干活。

----

# async 是菜谱

> Rust 这里有个和多数语言不同的点，必须先立住：**`async fn` 不是「启动一个任务」，而是「写好一份菜谱」。**

```rust
let fut = say("A", 100);   // 什么都没发生！只是把菜谱拿在手里
fut.await;                 // 现在才开始照着做
```

`async fn` 的返回值叫 **Future**。它本质是一个 **状态机结构体**：编译器把函数体按 `.await` 切成几段，  
「做到第几步、局部变量是什么」全部打包存在这个结构体里——进度不存在线程的 8MB 栈上，而是存在这个几十字节的小结构体里。

菜谱很「懒」：没人做它就永远只是纸。谁来做？运行时（Tokio， [《Tokio 运行时》](tokio.md)）。

----

# await 问一句夹书签

> 一句话：**`.await` = 「这步要等 → 我把进度夹个书签、让出线程；门铃响了，再从书签处接着干。」**

拆成三步看：

**第一步：先问一句「好了吗」（poll 一次）。** 如果碰巧已经好了（`Ready`）——比如数据早就到了—— **根本不暂停**，  
拿着结果直接往下走，一纳秒都不浪费。`.await` 不是「必定切换」，而是「需要等才切换」。

**第二步：没好（`Pending`）→ 做两件事再走人。**

- **留电话**：把「叫醒方式」（`Waker`）登记给事件源——定时器登记在运行时的时间表上，网络 IO 登记给操作系统的看门人（epoll）；
- **让位**：把线程还给运行时。注意这个「让位」便宜得惊人——就是一次普通的函数返回，不进内核。线程转身就去推进别的任务了。

**第三步：事件发生 → 门铃响 → 从书签处继续。** 数据到了/时间到了，事件源按铃（`waker.wake()`），  
运行时把这个任务放回就绪队列，某个空闲线程捡起来，从上次的书签处（状态机的当前状态）接着执行。

书签是什么？就是上一节说的状态机：编译器按 `.await` 把函数切成 N 段，书签 = 「我在第几段 + 还活着的局部变量」。  
所以暂停一个任务不需要保存 8MB 的栈和一堆寄存器，只要这个小结构体安安静静躺在那儿。

# 三个新手误区

> 每个都有人踩，先打预防针。

**误区一：「`.await` 会让它并发跑起来。」** 不会。`.await` 是 **等**，不是「扔到后台」。  
两个 `.await` 挨着写就是串行（等完第一个才开始第二个）。想并发：`join!`（后面讲）或 `tokio::spawn`（[《Tokio 运行时》](tokio.md)）。

**误区二：「`.await` 会阻塞线程。」** 恰恰相反，`.await` 让出线程。真正阻塞线程的是 `std::thread::sleep`、  
同步 IO 这类「钉死工人」的操作——它们才是 async 里的毒药。

**误区三：「不 `.await` 它也会自己跑。」** 不会。菜谱不做不出菜。调用 async 函数后「没反应」，几乎总是忘了 `.await`。

# 线程与 async 对照

> 一张表收尾大白话部分。

| 概念 | 线程世界 | async 世界 |
| --- | --- | --- |
| 干活单位 | 线程（OS 管理） | 任务 task（运行时管理） |
| 一个的成本 | ~8MB 栈 + 内核对象 | 几十~几百字节的状态机 |
| 等待时 | 线程阻塞，白占资源 | 让出线程，线程去干别的 |
| 暂停时保存什么 | 整个栈 + 寄存器快照 | 状态机里的几个字段 |
| 谁来唤醒 | OS 调度器 | Waker 按铃 → 运行时重新排队 |
| 切换成本 | 进内核，微秒级 | 函数返回，纳秒级 |

**一句话总结**：线程是「每桌雇一个专属服务员」；async 是「几个服务员靠登记本伺候全场」。Rust 负责把代码编译成可暂停的菜谱（Future），  
`.await` 是「交单走人」的动作，运行时（下一课的 Tokio）负责当厨房。

最后提醒：async 只优化「等得多」的场景（网络 IO）。纯计算的活（转码、算哈希）工人一直在干活没在等，async 帮不上忙，老老实实用线程。

# 打破 Go 的直觉

> 下面进入严谨部分。先从 Go 的直觉出发，再打破它。

Go 里函数天生「同步、会阻塞」，想并发就 `go` 一下：

```go
func say(name string, delayMs int) string {
    time.Sleep(time.Duration(delayMs) * time.Millisecond)
    return "你好, " + name
}

go say("A", 100)
go say("B", 100)
```

Go 的心智模型：函数被调用 → **立即执行**；想并发 → `go` 一下，运行时替你调度。

Rust 的心智模型完全不同，这是入门第一道坎：

> **调用一个 async 函数，什么都不会立即发生。它只是给你一个「待办事项」。这个待办事项要交给「运行时」驱动，才会真正执行。**

# 返回状态机不是结果

> `async fn` 的返回值不是结果，而是一台编译器生成的状态机。

```rust
async fn say(name: &str, delay_ms: u64) -> String {
    sleep(Duration::from_millis(delay_ms)).await;
    format!("你好, {name}")
}
```

关键认知：

- 签名写着 `-> String`，但 `say(...)` 的 **真实返回类型** 是 `impl Future<Output = String>`；
- 编译器把 async 函数体 **改写成一个实现了 `Future` trait 的状态机**。

「状态机」这个词本身是什么意思（红绿灯的例子、手工改写演示、为什么暂停/恢复因此变成零成本）， [《协程与状态机》](../os/coroutine-state.md) 有保姆级解释，  
第一次见这个词请先去读它。

`Future` trait 长这样（简化）：

```rust
trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context) -> Poll<Self::Output>;
}

enum Poll<T> {
    Ready(T),   // 完成了，这是结果
    Pending,    // 还没好，稍后再来问
}
```

- 运行时会 **反复调用 `poll`** 来「推一下」这台状态机；
- 每次 poll 要么 `Ready(结果)`（干完了），要么 `Pending`（还在等）。

把 `say` 想象成这台状态机：

```
状态0: 还没开始
   ↓ 第一次 poll
状态1: 等 sleep —— 每次 poll 检查定时器到了没
   ↓ 定时器就绪后的某次 poll
状态2: 算出 format!(...)，返回 Ready("你好, ...")
```

> **本质**：`async/await` 是语法糖。编译器把顺序代码翻译成「可暂停、可从暂停点恢复」的状态机；暂停点就是每个 `.await`。

对照 Go：goroutine 是运行时层面的「绿色线程」，有自己的栈，被 GMP 抢占式调度；Rust 不给任务分配栈，  
把「暂停/恢复」编译进状态机对象——叫 **无栈协程**。好处是每个 Future 极轻（就是个 enum，几十字节）；  
代价是心智负担从运行时挪到编译期（你会看到 Future、Pin 这些词）。

> 🔬 **底层视角**：一段代码「执行到哪了」，本质是 CPU 寄存器的当前值；OS 线程能暂停/恢复，靠内核把这些寄存器存进线程上下文、  
> 恢复时再装回去（见 [《协程与状态机》](../os/coroutine-state.md)）。goroutine 的「有自己的栈」是同一思路的用户态复刻：  
> Go 给每个 goroutine 配一份小栈 + 寄存器快照，切换时就换这两样。而 Rust 的状态机干脆 **不存寄存器也不留栈** ——编译器提前算好「每个暂停点需要保留哪些变量」，  
> 把它们直接变成 enum 的字段；恢复执行 = 普通地调一次 poll + match 到对应状态，连「快照」这道工序都省了。  
> 这就是三种方案的体积差（OS 线程 ~8MB / goroutine ~2KB / Future 几十字节）的真正来源：  
> 保存的东西一个比一个少。

# 惰性构造不等于执行

> Future 只是一台还没人 poll 的状态机——构造它，一行业务代码都不会跑。

```rust
let fut = say("惰性演示1000", 1000); // ← 一行业务代码都没跑！
logln!("future 已创建，但函数体还没有执行");
let msg = fut.await;                 // ← 这里才开始执行
```

跑一下就能验证：「future 已创建」 **先** 打印，say 内部日志后出现。

> ⚠️ **新手头号大坑**：调用了 async 函数却「没反应」——几乎总是因为 **没 `.await`**（也没交给 spawn）。  
> 编译器会警告 `unused implementer of Future that must be used`，  
> 看到它就知道漏了 await。

对照 Go：`say(x)` 立即执行、`go say(x)` 立即调度—— **Go 没有「惰性」概念**，这是最需要重建的直觉。

为什么 Rust 设计成惰性？因为惰性让「组合」与「取消」廉价且安全：

- 多个 Future 可以拼成一个大 Future（`join!`）再整体交给运行时，中间零调度开销；
- 没被 poll 过的 Future 直接 drop 就等于「取消」，不会有半执行的副作用（[《Tokio 运行时》](tokio.md) 的 `select!` 靠这点工作）。

# await 严谨版

> 大白话版在前面（问一句、夹书签、等门铃），这里是它的严谨版。

`.await` 展开后的语义是个循环：

```rust
sleep(dur).await;
// 概念上 ≈：
loop {
    match sleep_future.poll(cx) {
        Poll::Ready(v) => break v,            // 好了，继续往下
        Poll::Pending  => yield_to_runtime(), // 没好，让出执行权
    }
}
```

**要点一：`.await` 让出的是「执行权」，不是阻塞线程。** 等待期间， **同一个 OS 线程** 可以去推进成千上万个别的任务；  
这就是异步能用少量线程扛海量并发的根本原因。🔬 底层视角：「让出」这个动作便宜得惊人——poll 返回 Pending 就是 **一次普通的函数返回**，  
工作线程接着调用下一个任务的 poll，全程不进内核。对比 OS 线程切换（保存/恢复寄存器、进内核、缓存变冷，  
微秒级），这里是纳秒级。同一个词「切换」，成本差了三个数量级——这就是异步扛并发的底气。

**要点二：等好了，谁来叫醒？—— `Waker`。**

- poll 时运行时塞给 Future 一个 `Waker`（藏在 `Context` 里）；
- Future 返回 `Pending` 前，把 Waker 交给事件源（定时器、epoll…）；
- 事件真正发生时，事件源调用 `waker.wake()`，运行时把该任务重新排队、再来 poll。
- 你几乎永远不用手写 Waker，但知道有「叫醒」机制，异步就不是黑魔法；
- 🔬 那「事件源」是谁在盯着？定时器是运行时内部的一张时间表；网络 IO 则注册进操作系统的 **多路复用器**（Linux 的 epoll）——Tokio 会去问内核「这批 socket 哪些就绪了」，  
  拿到答案就 wake 对应任务。完整链条：网卡收到数据 → 内核标记 socket 就绪 → epoll 返回 → Tokio 调 waker → 任务回到就绪队列 → 某个工作线程 poll 它。

对照 Go：Go 里 Sleep/IO 阻塞时，运行时自动挂起 goroutine、线程去跑别人；Rust 的 `.await` + Waker 效果相同，  
只是「挂起/唤醒」显式可见。

> 🚨 **由此推出一条铁律**：绝不在 async 函数里做「阻塞线程」的事——`std::thread::sleep`、同步文件 IO、长时间纯计算。
>
> - 异步任务共享少量线程，一个任务卡线程 = 连累整线程上所有任务；
> - 需要阻塞/重计算 → `tokio::spawn_blocking`（[《Tokio 运行时》](tokio.md)）；
> - Go 不太有这问题（调度器会在阻塞系统调用时补线程），这是两者运行时设计的重要差异。
> - 场景(5) 的 `say_blocking` 就是这条铁律的现场演示。「阻塞」在操作系统层面的确切含义——线程被内核挂起、  
> 期间它背着的一切停摆——见 [《协程与状态机》](../os/coroutine-state.md)。

# 顺序与并发

> 顺序 `.await` 是串行、`join!` 是并发，耗时差在哪，看时间线。

```rust
// 顺序（场景(2)）：A 完全结束才开始 B，总耗时 ≈ 200ms
let a = say("A", 100).await;
let b = say("B", 100).await;

// 并发（场景(3)）：join! 同时推进，总耗时 ≈ 100ms
let (a, b) = tokio::join!(say("A", 100), say("B", 100));
```

```
顺序:
  A: [====100ms====]
  B:                [====100ms====]
                                   ^ 200ms 完成

并发:
  A: [====100ms====]
  B: [====100ms====]
                    ^ 100ms 完成
```

`join!` 的原理：把几个 Future 组合成 **一个** 大 Future；一次 poll 里挨个 poll 每个子 Future，  
谁 Pending 就先放着；全部 Ready 才整体 Ready。场景(4) 进一步验证：三个不同延时并发跑，  
总耗时 = **最慢的那个**。

**并发 vs 并行辨析（重要）**：

- **并发（concurrency）**：多个任务「同时进行中」，可能靠一个线程快速切换。`join!` 属于此；
- **并行（parallelism）**：多个任务真的同时各占一个 CPU 核。要 `tokio::spawn` 到多线程运行时才谈得上（[《Tokio 运行时》](tokio.md)）；
- Go 对照：`join!` ≈ 两个 goroutine + `WaitGroup.Wait()`，但 Go 的 goroutine 可能真并行，  
  `join!` 的子 Future 在同一任务内交替；
- 🔬 用 [《协程与状态机》](../os/coroutine-state.md) 的话说：`join!` 只提供「并发的 **结构**」——它整体是一个任务，  
  同一时刻只被一个工作线程 poll， **永远吃不到第二个 CPU 核**。想吃多核（并行的 **资源**），得 spawn 成独立任务，  
  调度器才有机会把它们放到不同线程/不同核上（[《Tokio 运行时》](tokio.md)）。

# 三句话带走

1. **`async fn` 返回惰性的状态机（Future）**：调用 ≠ 执行，必须 `.await` 或交给 spawn。
2. **`.await` = poll 子任务；没好就让出执行权，好了被 Waker 叫醒** ——让出线程，而非阻塞线程。
3. **顺序 `.await` 串行（耗时相加）；`join!` 并发（耗时取最大）**；要多核并行用 `spawn`（下一课）。

**遗留问题**：谁在不停 poll、谁响应 wake？—— 运行时。答案就是 **Tokio**， [《Tokio 运行时》](tokio.md) 登场。

# 附：本课生词表

> 通用语法（宏、unwrap、format!、`{name}` 内插…）见 [《Rust 语法底座》](../start/syntax-primer.md)，  
> 此处只列本课新出现的。

- **`async fn`** ——函数声明前加 `async`，效果有二：(1) 函数体里允许用 `.await`；  
  (2) 返回类型从 `T` 变成 `impl Future<Output = T>`（惰性状态机）。
- **`.await`** ——只能出现在 async 上下文里的后缀操作符；语义是「驱动这个 Future 到完成拿结果；  
  没就绪就让出线程」。为什么是后缀？链式调用好看：`client.get(url).send().await?.json().await?` 从左到右一路读。
- **`impl Future<Output = String>`** ——读作「某个实现了 Future、产出 String 的类型」；  
  `impl Trait` 见 [《类型系统与 trait》](../lang/types-traits.md)，`Output` 是关联类型。
- **`#[tokio::main]`** ——属性宏。贴在 `async fn main` 上，展开后：创建 Tokio 运行时 → 把 main 函数体当顶层 Future 交给它驱动；  
  没有它，`async fn main` 直接编译报错。
- **`tokio::time::sleep`** ——Tokio 的 **异步** 睡眠（让出线程）；区分 `std::thread::sleep`（同步、  
  卡死线程，async 里的毒药）。
- **`Duration::from_millis(100)`** ——`Duration` 是标准库时长类型 ≈ Go 的 `time.Duration`；  
  `from_millis` 是关联函数（`::` 调用、无需实例）；同族 `from_secs`、`from_micros`。
- **`Instant::now()` / `t.elapsed()`** ——单调时钟打点/求耗时 ≈ Go 的 `time.Now()` + `time.Since(t)`；  
  测耗时用它（不受系统改时间影响）。
- **`tokio::join!(a, b, ...)`** ——宏：并发推进多个 Future， **全部完成** 后按传入顺序返回元组；  
  谁都不会被丢弃/取消（对比 `select!`）；不开新任务/线程。
- **`async move { ... }`（async 块）** ——把一段代码变成 Future（同样惰性）；  
  `move` 把用到的外部变量所有权搬进来；块要活得比当前作用域久（如 spawn 出去）就必须 move。
- **`String::from("...")`** ——从字面量造拥有所有权的 String；场景(6) 刻意用它演示 move 进 async 块。
