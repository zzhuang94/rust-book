# 需求与 Go 参照

> 「定期更新内存数据 + 高并发读取」拆成一个写者、很多读者。先看 Go 版。

- **一个写者**：每隔一段时间，整体刷新一份内存数据；
- **很多读者**：高频、并发地读取这份数据。

Go 里你会很自然地写：

```go
var mu sync.RWMutex
var data Data

// 写者
go func() {
    for range time.Tick(period) {
        fresh := recompute()
        mu.Lock(); data = fresh; mu.Unlock()
    }
}()

// 读者（每个请求）
mu.RLock(); snapshot := data; mu.RUnlock()
```

这段 Go 代码能跑，但 **正确性靠自觉**：哪个读者忘了 `RLock`，就是一个数据竞争；只有 `go run -race` 才可能抓到，  
线上表现为偶发诡异 bug。Rust 的同款思路一模一样，但「谁能碰数据、碰时有没有锁」是 **编译器强制检查** 的。

> 🔬 **底层视角：竞争到底从哪来**（见 [《操作系统基础》](../concurrency/os-basics.md)）：  
> 同进程的线程共享同一个地址空间——`data` 就是内存里的一个地址，每个线程都能直接读写它。两个核心同时执行「读旧值 → 加一 → 写回」，  
> 就会互相覆盖（各自读到 5、各自写回 6，一次更新凭空消失）；更隐蔽的是每个核心还有自己的缓存，你这个核写的值，  
> 别的核什么时候能「看见」都成问题。本课的锁、原子指令、Send/Sync，全是在这层硬件现实之上重建秩序的工具。

# 为什么不能直接共享

> Go 里把变量丢给几个 goroutine 就完了。Rust 里同样的事，两条路都被编译器堵死。

**路一：借用。** 把 `&config` 传给任务？

```
error[E0373]: async block may outlive the current function,
              but it borrows `config`, which is owned by the current function
```

原因：`tokio::spawn` 的任务是 `'static` 的，可能比当前函数活得久；借了栈上变量，函数返回后引用就悬垂了——编译器不允许。

**路二：移动。** `async move` 把 `config` 搬进任务？

```
error[E0382]: use of moved value: `config`
```

原因：所有权只有一份，搬给任务 A 之后，任务 B（和 main 自己）就没得用了。

**结论**：要让 N 个任务共享一份数据，需要第三条路——让「所有权」本身可以被多方持有。这就是 `Arc`。

# Arc 解决共享

> `Arc<T>` = 原子引用计数智能指针，先解决「多方持有同一份数据」。

```rust
let config = Arc::new(vec![...]);     // 数据放到堆上，挂一个计数器（=1）
let handle = Arc::clone(&config);     // 计数 +1，得到新句柄（不复制数据！）
drop(handle);                         // 计数 -1
// 计数归零的那一刻，堆上数据被释放
```

场景(1) 用 `Arc::strong_count()` 让你亲眼看到计数变化：

```
初始引用计数 = 1
spawn 后引用计数 = 4     （1 个 main + 3 个任务）
任务全部结束后引用计数 = 1
```

对照 Go：Go 里多个 goroutine 拿同一个指针，回收交给 GC；Rust 没有 GC，`Arc` 的引用计数就是 **手动的、确定性的 GC** ——最后一个句柄消失的瞬间释放，不用等垃圾回收扫描。

注意场景(1) 里任务们只是 **读** config。只读共享是最轻松的情形：`Arc<T>` 允许多任务同时 `&` 读，  
不需要任何锁。但先别急着往下——有个更基础的问题：**编译器凭什么判定「Arc 跨任务是安全的、别的类型不行」？  
** 这就是下一节的 Send/Sync。

# Send 与 Sync 体检

> 这一节是全课的理解地基。后面所有「编译器拦你/放行你」的现象，根子都在这里。

## 先认识 marker trait

[《Rust 语法底座》](../start/syntax-primer.md) 讲过 trait ≈ Go 的 interface，  
通常带方法。但有一小类 trait **没有任何方法**，叫 **marker trait（标记特征）** ——它们的唯一作用是给类型 **贴标签**：

```rust
pub unsafe auto trait Send {}   // 标准库里的定义（简化）：空的！
pub unsafe auto trait Sync {}
```

- 标签本身什么都不做；
- 但泛型约束可以 **检查标签**（`fn f<T: Send>(...)`），没标签的类型过不了编译；
- 你最早学过的 `Copy` 也是 marker trait，同一家族。

所以 Send/Sync 不是什么运行时机制，就是两张「体检合格证」，编译期查验。

## Send 能不能搬到别的线程

直观问句：**「这个值被搬到另一个线程去使用、去销毁，会不会出乱子？」**

- 会出乱子 → 没有 Send 标签；不会 → 有。
- 绝大多数类型不会出乱子：`i32`、`String`、`Vec<T>`……它们的数据自己管自己，搬到哪个线程都一样。
- **经典反例：`Rc<T>`**。它的引用计数是 **普通整数**，两个线程同时 `clone()`（计数 +1）会互相覆盖，  
  导致少计 → 数据被提前释放 → 悬垂指针。所以编译器 **故意不给** Rc 发 Send 标签。
- `Arc<T>` 的计数是 **原子操作**，怎么并发都不会加坏 → 有标签。

这就回答了 Rc vs Arc 的本质区别：不是功能不同，是 **一个通得过体检、一个通不过**。

## Sync 能不能被同时引用

直观问句：**「多个线程同时拿着 `&T`（只读引用）通过它做事，会不会出乱子？」**

- `i32`、`String`：多线程同时读，没事 → Sync；
- **反例：`RefCell<T>`**（单线程的运行时借用检查容器）：它的借用计数器不是线程安全的，多线程同时通过 `&RefCell` 去 borrow 会把计数弄坏 → 不发 Sync 标签；
- **有意思的正例：`Mutex<T>`**。它内部明明是可变的，凭什么 Sync？——因为所有访问都要先过锁，同一时刻只有一人能碰数据。  
  **锁的本质就是「把不能共享的修改，变成能共享的修改」**，Sync 标签就是这个承诺的凭证。

严格定义是「`T: Sync` 当且仅当 `&T: Send`」，初学记住直观问句就够了。

## 编译器自动推导且会传染

你 **从来不需要手写** `impl Send for X`（手动实现是 unsafe 的，留给库作者）。规则很简单：

> 结构体/枚举的 **所有字段** 都是 Send，它就自动是 Send；Sync 同理。

推论：这个属性是 **递归、会传染的** ——你的结构体里塞了一个 `Rc` 字段 → 整个结构体失去 Send → 捕获它的 async 块失去 Send → `tokio::spawn` 拒收，  
报错一路指回那个 Rc；换成 `Arc` → 标签自动恢复。这意味着「这个类型能不能跨线程」是从每个字段 **算出来** 的客观事实，  
而不是程序员的口头承诺。

## 谁在查体检表

[《Tokio 运行时》](tokio.md) 见过 spawn 的签名：

```rust
pub fn spawn<F>(future: F) -> JoinHandle<F::Output>
where F: Future + Send + 'static, ...
//                ^^^^ 就是在这里查表
```

编译器在 **每个调用点** 核对标签，于是：任务里用了 Rc → `error[E0277]: Rc<...> cannot be sent between threads safely`；  
任务持着 std 锁的 guard 跨 `.await`（guard 没有 Send 标签）→ 同样的报错；全部通过 → 放行，  
且 **保证运行期不可能出现数据竞争**。

> 🔬 为什么 async 任务也要过「跨线程」体检？因为工作窃取是动真格的：任务在 `.await` 处暂停后，可能被 **另一个 OS 线程**（大概率还是另一个 CPU 核）捡起来继续跑。  
> 对硬件来说，这就是货真价实的跨线程/跨核数据迁移——所以当年为线程设计的 Send/Sync 体检，原封不动适用于任务。

## 体检器与速查表

场景(2) 用两个空泛型函数当「体检器」，能编译 = 有标签：

```rust
fn assert_send<T: Send>(_v: &T) {}
fn assert_sync<T: Sync>(_v: &T) {}

assert_send(&Arc::new(42));   // ✅ 通过
// assert_send(&Rc::new(42)); // ❌ 放开即报错 E0277
```

常见类型体检速查表：

| 类型 | Send | Sync | 一句话原因 |
| --- | --- | --- | --- |
| `i32` / `String` / `Vec<T:Send>` | ✅ | ✅ | 数据自管自己，无共享内部状态 |
| `Arc<T>`（T: Send+Sync） | ✅ | ✅ | 计数是原子的 |
| `Rc<T>` | ❌ | ❌ | 计数是普通整数，并发会加坏 |
| `Mutex<T>`（T: Send） | ✅ | ✅ | 访问都要过锁 |
| `RefCell<T>` | ✅ | ❌ | 借用计数非线程安全，不能共享引用 |
| std 锁的 guard（MutexGuard 等） | ❌ | —— | 锁定状态绑定在拿锁的线程上，不许带走 |

对照 Go：Go 里 **任何值都能被任何 goroutine 访问**，编译器不做任何跨线程检查；数据竞争靠 `go run -race` 在 **运行时抽查** ——测试没跑到的路径就是漏网之鱼。  
Rust 把同一问题变成 **类型问题**：标签不对，编译不过，漏网率为零。这就是「无畏并发」的确切含义—— **不是不会死锁**（死锁还是可能的，  
场景(8) 会死给你看），而是 **编译过了就没有数据竞争**。

# 想改数据靠内部可变

> `Arc<T>` 只给只读访问。想改，需要「内部可变性」容器。

直觉写法，编译失败：

```rust
let shared = Arc::new(vec![1, 2, 3]);
shared.push(4);
// error[E0596]: cannot borrow data in an `Arc` as mutable
```

为什么？回忆借用规则：**要么多个只读引用（`&T`），要么一个可变引用（`&mut T`），不能同时**。`Arc` 的本职是让很多人同时持有 → 人人只读 → 谁都不许改。

想改，就需要一类特殊容器：**哪怕通过只读引用（`&self`），也能安全地修改内部数据** ——术语叫「内部可变性（interior mutability）」。  
它们的共同套路是把「检查/排队」从编译期挪到运行期：

| 容器 | 安全手段 | 线程 | 本课场景 |
| --- | --- | --- | --- |
| `Cell` / `RefCell` | 运行时借用检查 | 单线程（不 Sync） | 不用（列出防混淆） |
| `Mutex<T>` | 互斥锁排队 | 多线程 | (4) |
| `RwLock<T>` | 读共享/写独占 | 多线程 | (5) |
| `AtomicU64` 等 | 硬件原子指令 | 多线程 | (6) |

于是多任务「共享 + 可改」的标准形态就是两层组合：

> **`Arc<锁<T>>`：`Arc` 管共享所有权，锁管修改安全。**

# Mutex 最简单的锁

> 互斥锁：数据装在锁里面，不 lock 拿不到。

经典题目：4 个任务并发给同一个计数器各 +1000。

```rust
let counter = Arc::new(Mutex::new(0u64));
// 每个任务里：
*counter.lock().unwrap() += 1;
```

**和 Go 最大的不同：数据装在锁里面。**

```go
// Go：锁和数据是两个变量，配对全靠自觉
var mu sync.Mutex
var n int
mu.Lock(); n++; mu.Unlock()   // 谁忘了 Lock 也能编译，埋雷
```

```rust
// Rust：Mutex<T> 把 T 装在里面，不 lock 拿不到数据
let counter = Mutex::new(0u64);
*counter.lock().unwrap() += 1;   // 不存在"绕过锁"的语法
```

**「忘了加锁」在 Rust 里不是 bug，是编译错误。**

**guard 与 RAII：为什么不需要 defer Unlock。** `lock()` 返回的不是数据，是一个 **guard（「通行证」）**：  
持有它期间独占数据；用起来像 `&mut T`； **它被 drop 的那一刻 = 解锁**。

| 步骤 | 发生什么 |
| --- | --- |
| `counter.lock()` | 排队拿锁，返回 `Result<Guard, _>` |
| `.unwrap()` | 解出 guard（Err = 锁中毒，见下） |
| `*guard += 1` | 解引用，修改数据 |
| 语句结束 | guard（临时值）drop → **自动解锁** |

这套「资源跟着值的生命周期走」叫 **RAII**。所以 Rust 不需要 `defer mu.Unlock()`。

**锁中毒（那个 unwrap 在解什么）**：若某任务持锁期间 panic，锁被标记「中毒（poisoned）」，  
之后 lock 得到 `Err`；设计意图是 panic 可能让数据停在半更新的不一致状态。常见处理：`.unwrap()`（中毒就崩）或 `.unwrap_or_else(|e| e.into_inner())`（确认无碍继续用）。  
Go 的 Mutex 没有中毒概念：持锁 panic 后锁烂在那里，别人永久阻塞。

> 🔬 **底层视角：lock() 那一瞬间**：没人抢时，加锁就是 **一条原子指令**（CAS：「如果锁状态是 0 就改成 1」），  
> 纳秒级、不进内核——所以短临界区的 std 锁几乎免费。有人抢时才走慢路径：通过 futex 之类的系统调用请内核把线程改成阻塞态，  
> 锁释放时再由内核唤醒排队者。「快路径留在用户态、慢路径才进内核」是现代锁的通用设计。

# RwLock 读多写少

> 读写锁：读锁可多持、写锁独占。这是本项目的内核。

Mutex 连 **读** 也互斥：100 个读者也得排队。我们的场景读远多于写，太浪费。RwLock 区分两种锁：

| 方法 | 锁类型 | 语义 |
| --- | --- | --- |
| `.read()` | 读锁 | 任意多个读者 **同时** 持有，互不阻塞 |
| `.write()` | 写锁 | 独占，写时挡住所有人 |

≈ Go 的 `sync.RWMutex`（`RLock`/`Lock`）。

**写者套路：锁外算好，锁内整体替换。**

```rust
// ★ 重活（构造新数据）在锁外做完……
let fresh = (version, vec![version as i64; 3]);
// ……进锁只做一次赋值，guard 语句末尾即 drop = 解锁
*data.write().unwrap() = fresh;
```

写锁只占「一次赋值」的时间，读者几乎不被挡。 **整体替换** 而非逐字段修改，是高并发写的标准姿势（[《ArcSwap 无锁读》](../http/arcswap.md) 把它推到极致）。

**读者套路：进锁就 clone，尽快放锁。**

```rust
let (ver, items) = {
    let guard = data.read().unwrap(); // 读锁（读者间互不阻塞）
    (guard.0, guard.1.clone())        // 尽快复制出需要的数据
};                                     // ← guard 在此 drop，解锁
sleep(Duration::from_millis(40)).await;  // 此刻未持锁
```

那个 `{ }` 作用域块保证锁在后续 `.await` **之前** 释放。为什么「之前」二字性命攸关 → 见后面「别把 std 锁跨 await」。

# 原子量连锁都不用

> 单个整数的加减，硬件本身有原子指令，不需要锁的排队开销。

```rust
let counter = Arc::new(AtomicU64::new(0));
counter.fetch_add(1, Ordering::Relaxed);   // 原子 +1，无锁
let n = counter.load(Ordering::Relaxed);   // 原子读
```

- `fetch_add` 是「读-加-写」三合一的 **原子** 操作——拆成 `load` 再 `store` 就不原子了（动手实验会让你亲手制造竞态）；
- HTTP 服务统计读取次数用的就是它。

`Ordering`（内存序）怎么选：`Relaxed` 只保证本操作原子、不约束周围指令（独立计数器够用）；`SeqCst` 最严格全局顺序（拿不准就用它）。  
对照 Go 的 `atomic.Int64`：Go 把内存序定死了（相当于最严格档），Rust 让你选，是为极致性能场景留的口子。

> 🔬 **底层视角**：`fetch_add` 会编译成一条带 lock 前缀的 CPU 指令（x86 上是 `lock xadd`）——「原子」是硬件在缓存一致性协议层面把那个缓存行锁住一瞬间，  
> 不是软件模拟出来的。而 `Ordering` 管的是另一件事：编译器和 CPU 为了性能都会 **重排** 指令顺序，  
> 内存序就是你告诉它们「这个原子操作前后，哪些重排不许做」。独立计数器谁先谁后无所谓 → Relaxed 随它排。

场景(6) 的实证：4 个任务各 +1000，结果 **精确 4000**。（普通 u64 裸加？Rust 直接不给编译；Go 能编译、结果随机少。）

# std 锁 vs tokio 锁

> 这是异步 Rust 里最容易选错的一对工具，值得单独一节掰开揉碎。

## 什么时候必须持锁跨 await

前面全用 std 锁，因为 **临界区都不含 `.await`**。但有的需求就是要「持锁期间做异步操作」：独占一条连接（发请求 → `await` 等响应 → 处理，  
全程不能让别人插队）；往共享日志里写「开始/结束」两条 **配对** 记录，中间隔着一次异步 IO。

这时用 `tokio::sync::Mutex`：

```rust
let journal = Arc::new(tokio::sync::Mutex::new(Vec::new()));
let mut guard = journal.lock().await;   // 注意：lock 是异步的，.await 排队
guard.push("开始");
do_async_io().await;                    // ★ 持锁跨 await：tokio 锁允许
guard.push("结束");
// guard drop → 解锁
```

场景(7) 的输出证明了两件事：两个任务的「开始/结束」 **各自相邻、不交错**；任务 2 干等了任务 1 整整 300ms（**异步锁不是免费的**，  
临界区照样要尽量短）。

## 本质区别是等锁方式

两把锁最根本的差异不在 API 长相，而在 **「锁被别人占着时，你怎么等」**：

- **std 锁**：`lock()` 拿不到锁时， **把当前 OS 线程整个挂起**（阻塞态：不占核心、不干活，  
  等内核叫醒）。这个工作线程彻底停摆——它身上排队的其他异步任务全部躺枪；
- **tokio 锁**：`lock().await` 拿不到锁时，返回 Pending， **只挂起当前任务** ——线程立刻让出去跑别的任务；  
  锁释放时，通过 Waker 唤醒排队的任务。

画成时间线（假设 2 个工作线程、任务 A 持锁较久）：

```
用 std 锁（有竞争时）:
  线程1: [任务A 持锁工作中..........................]
  线程2: [任务B lock() → 线程整个卡住 zzzzzzzzzzzz]   ← 线程2上排队的任务C/D/E 全部停摆
  （2 个工作线程瘫了 1 个，服务吞吐直接砍半）

用 tokio 锁（有竞争时）:
  线程1: [任务A 持锁工作中..........................]
  线程2: [任务B lock().await→Pending让出][跑任务C][跑任务D][任务B被唤醒继续]
  （线程2 照常干活，只有任务B 自己在排队）
```

**那为什么还敢在 async 代码里用 std 锁？** 因为本课前几个场景的临界区都极短：竞争窗口小到「真的撞上并阻塞一瞬」的代价，  
比 tokio 锁每次加锁都要维护等待队列 + Waker 的固定开销还便宜。**tokio 官方文档的建议就是：  
临界区短且不含 await，优先用 std 锁。**

## 全维度对比表

| 维度 | `std::sync::Mutex/RwLock` | `tokio::sync::Mutex/RwLock` |
| --- | --- | --- |
| 加锁调用 | `lock().unwrap()`（同步） | `lock().await`（异步） |
| 等锁时发生什么 | **阻塞整个工作线程**（OS 挂起） | 只挂起当前任务，线程照常跑别人 |
| guard 跨 `.await` | ❌ 编译不过（guard 无 Send 标签） | ✅ 允许（guard 有 Send 标签） |
| 锁中毒 | 有（lock 返回 Result） | 无（lock 不返回 Result） |
| 公平性 | 不保证（由 OS 决定） | **FIFO 公平**（按 lock 调用顺序排队） |
| 等锁能否被取消 | 不能（线程卡住就是卡住） | 能：放进 `select!`/`timeout` 可中途放弃 |
| 无竞争时的开销 | 极低（约等于一次原子 CAS） | 较高（任务上下文 + 队列维护） |
| 实现层 | 包装 OS 原语（futex/SRWLock 等） | 纯用户态（原子状态 + 等待队列 + Waker） |
| 依赖 | 标准库，同步代码也能用 | 需要 tokio 运行时 |

## std guard 为何不是 Send

这不是随意规定，把体检逻辑走一遍就明白了：

- std 锁底层是 OS 同步原语，很多平台（如 pthread）规定：**哪个线程加的锁，必须由哪个线程解**。  
  guard 里因此隐含着「归属线程」这个事实——把它搬到别的线程去 drop（= 在别的线程解锁）是未定义行为；
- 而 `.await` 之后，任务可能被工作窃取 **搬到另一个工作线程** 恢复执行。guard 若还活着，恢复后它就在「别的线程」了；
- 所以编译器不给 std guard 发 Send 标签，spawn 的体检就会精确拦下「guard 跨 await」这一危险动作——**!Send 不是缺陷，  
  恰好是那道保险**；
- tokio 锁是纯用户态实现，不关心「谁在哪个线程 drop guard」——所以它的 guard 可以安全跨线程，发 Send 标签。

## 优先重构掉跨 await

遇到「我好像需要持锁跨 await」时，先别急着换 tokio 锁——大多数情况可以 **重构成不跨**：

- **读**：进锁把需要的数据 clone 出来，放锁，再拿着副本去 await（场景(5) 读者的标准姿势）；
- **写**：先 await 把结果算好，再进锁一次性写回（场景(5) 写者的「锁外算好、锁内替换」）；
- 只有「锁保护的就是那个异步资源本身」（独占连接、必须原子完成「读 → 异步操作 → 写」序列）才真的绕不开，这才是 tokio 锁的主场；
- 更进一步：如果一把锁竞争激烈成了瓶颈，考虑换架构——数据交给一个任务独管、别人用 mpsc 发消息请求它（Actor 模式），锁就整个消失了。

## 选型清单

1. 临界区不含 await？→ **std 锁**（首选，覆盖 99% 场景）；
2. 含 await 且重构不掉？→ **tokio 锁**；
3. 还需要「等锁可超时/可取消」或「严格排队公平」？→ **tokio 锁**（顺带的收益）；
4. 锁本身成了瓶颈？→ 换设计：拆更细的锁 / 原子量 / ArcSwap / 消息传递。

两个对称的常见误区：❌「异步代码就该统一用 tokio 锁」（临界区不含 await 时 std 锁更快）；❌「async 里绝对禁止 std 锁」（短临界区的 std 锁是官方推荐的正确用法）；  
真正禁止的只有一件事：**持着 std guard 跨 `.await`**。

# 别把 std 锁跨 await

> ⚠️ 铁律。std 锁的 guard 绝不能跨 `.await` 持有——编译器也会拦你。

**错误** 示范（场景(9) 注释里的写法）：

```rust
let guard = data.read().unwrap();
sleep(Duration::from_millis(10)).await;   // ❌ 攥着 std 锁 .await
logln!("{}", guard.0);
```

为什么危险（回忆 `.await` 会让出线程去跑别的任务）：

1. **锁被长时间占用**：你挂起的整段时间锁都没放，别的读者/写者全堵住，吞吐塌陷；
2. **极易死锁**：你持读锁挂起，让出的线程上正好排着要写锁的任务，而它又在等你放锁……单线程运行时下直接死锁；
3. **编译器会拦**：std 锁的 guard **没有 Send 标签**，guard 活着跨越 `.await`，  
   整个任务就不满足 spawn 的 Send 约束：

```
error: future cannot be sent between threads safely
   note: `std::sync::RwLockReadGuard<'_, ...>` is not `Send`
```

有了前面的背景，这条报错现在应该能逐字读懂了。两条出路：**a) 首选** ——像场景(5) 那样用 `{ }` 在 await 前放锁（99% 够用）；  
**b) 真的要跨** ——场景(7) 的 tokio 异步锁。

对照 Go：持锁跨阻塞调用「只是慢」，不报错不崩——隐患留到线上。Rust 用类型系统把它变成编译错误。

# 死锁编译器不管

> Rust 编译器防的是数据竞争，不防死锁。死锁是逻辑问题，得靠纪律预防。

经典配方：

```
任务甲：锁 A → 睡 50ms → 想锁 B（B 在乙手里，等）
任务乙：锁 B → 睡 50ms → 想锁 A（A 在甲手里，等）
        ↑ 互相攥着对方要的东西，谁也不放手 —— 永远僵住
```

场景(8) 用 `timeout(1s, ...)` 兜底，让你能 **看到** 死锁而程序不挂死：

```
[...] 任务甲：锁住了 A，准备去锁 B
[...] 任务乙：锁住了 B，准备去锁 A
（1 秒沉默……两个"拿到了"的日志都没出现）
[...] ★ 1 秒超时：死锁确认！
```

两个必须知道的事实：**Rust 编译器不防死锁**（它防的是数据竞争）； **Go 的 `-race` 同样查不出死锁**（Go 运行时只在「所有 goroutine 全部睡死」时才 panic，  
部分死锁照样潜伏）。

清场的小彩蛋：演示里用 `h1.abort()` 终止僵死的任务——任务的 future 被 drop，它持有的 guard 一并 drop，  
**锁自动释放**。又是那两条原理的合奏：[《async 基础》](basics.md) 「丢弃 future = 取消」+ 本课的 RAII。

**预防死锁的四板斧**：

| 手段 | 做法 | 场景(8) 用的 |
| --- | --- | --- |
| **统一加锁顺序** | 要拿多把锁时，全项目约定固定顺序（先 A 后 B），环路不可能形成 | ✅ 修复版 |
| **缩小临界区** | 一次只持一把锁、拿到就快用快放，尽量别嵌套 | —— |
| **try_lock / 超时** | 拿不到就放弃重来，不死等（`try_lock()` / `timeout(d, lock())`） | 观察用了 timeout |
| **用消息代替共享** | 数据归一个任务独管，别人发消息请求它改（Actor 模式） | —— |

对照 Go：四板斧一模一样——这是并发编程的通识，不是 Rust 特产。

# 选型决策树

> 本课总结成一棵树。

```
要在多个任务间共享数据？
├── 只读             → Arc<T>                          （场景(1)）
├── 只是整数计数/标志  → Arc<AtomicU64 / AtomicBool>     （场景(6)）
├── 要读要写
│   ├── 读多写少      → Arc<RwLock<T>>                  （场景(5)）
│   │     └── 读 QPS 极高、整份替换 → ArcSwap（ArcSwap 一课）
│   └── 读写均衡/简单 → Arc<Mutex<T>>                   （场景(4)）
│         └── 临界区必须含 .await → tokio::sync::Mutex  （场景(7)）
└── 其实不必共享      → 消息传递 mpsc（Tokio 一课）/ Actor
```

（无论选哪条，能不能跨任务由 Send/Sync 体检把关——这些组合全都天生合格，这正是选它们的原因。）

# 动手实验

1. **亲手触发三种编译错误**：a) 把场景(1) 的 `Arc::clone` 那行删掉、直接在任务里用外层 `config`（体会 E0373/E0382）；  
   b) 放开场景(2) 注释里的 `assert_send(&rc)`（体会 E0277 "cannot be sent between threads safely"）；  
   c) 按场景(9) 注释改出「持 std guard 跨 await」（体会 guard not Send）。逐字读报错，  
   再改回来；
2. **体会「传染性」**：定义 `struct Config { name: String, cache: std::rc::Rc<Vec<u8>> }`，  
   对它调用 `assert_send`——报错会指出是 cache 字段的 Rc 拖累了整个结构体；把 Rc 换成 Arc，  
   立刻通过；
3. **看并发读不互斥**：场景(5) 读者临界区里加 `logln!("读者 {id} 拿到读锁")` + `std::thread::sleep(5ms)`，  
   多个读者时间戳几乎相同；把 `read()` 改成 `write()`，变成一个接一个；
4. **感受整体替换的价值**：把场景(5) 写者改成「持写锁逐个 push 并每次 sleep(30ms)」（锁内做重活），  
   读者日志明显停顿；改回锁外算好，停顿消失；
5. **亲手制造竞态**：把场景(6) 的 `fetch_add` 拆成 `let v = c.load(...); c.store(v + 1, ...)`（三步非原子），  
   跑几遍——结果大概率 < 4000 且每次不同；
6. **感受异步锁的排队代价**：场景(7) 把任务数从 2 改成 5，总耗时 ≈ 5 × 300ms（完全串行）；
7. **改死锁参数**：把场景(8) 甲乙的 `sleep(50ms)` 都删掉再多跑几遍——死锁还必然发生吗？（不必然：  
   没有睡眠时，先跑的任务可能一口气拿到两把锁。死锁是 **时序敏感** 的，这正是它难查的原因。）
8. **体验「等锁可超时」**：把场景(7) 第二个任务的 `journal.lock().await` 改成 `timeout(Duration::from_millis(100), journal.lock()).await`——任务 1 持锁 300ms，  
   任务 2 等 100ms 就放弃，打印「等锁超时」。

# 三句话带走

1. **共享靠 `Arc`，能不能跨线程由 Send/Sync 两张「体检表」在编译期把关**（自动推导、会传染、Rc/std-guard 不合格）；  
   修改靠内部可变性容器：`Arc<Mutex<T>>` / `Arc<RwLock<T>>` / `Arc<AtomicU64>` 按需选，  
   数据装在锁 **里面**，忘加锁 = 编译错误。
2. **写者「锁外算好、锁内整体替换」，读者「进锁 clone、尽快放锁」**；std 锁绝不跨 `.await`（guard 无 Send 标签，  
   编译器会拦），真要跨用 tokio 异步锁——但锁内 await 的排队代价看得见。
3. **编译器防的是数据竞争，不防死锁**：多把锁统一加锁顺序、缩小临界区、try_lock/超时兜底、或干脆用消息传递代替共享。

# 附：本课生词表

> 通用语法见 [《Rust 语法底座》](../start/syntax-primer.md)；spawn/move/timeout 见 [《Tokio 运行时》](tokio.md)、  
> [《超时、限流与任务组》](task-control.md) 生词表。

- **`Arc<T>` / `Arc::new(v)` / `Arc::strong_count(&x)`** ——原子引用计数智能指针：  
  `new` 放堆上挂计数器（=1）；`strong_count` 读当前计数；计数归零瞬间释放，确定性回收不等 GC。
- **为什么写 `Arc::clone(&data)` 而不是 `data.clone()`** ——功能一样（计数 +1）；  
  社区惯例用前者，一眼看出「这是廉价句柄克隆，不是深拷贝」。
- **marker trait（标记特征）** ——没有任何方法的空 trait，纯粹给类型贴标签供泛型约束检查；`Send`/`Sync`/`Copy` 都是。
- **`Send` / `Sync`** ——`Send`：所有权可安全搬到另一线程（Rc ❌）；`Sync`：可被多线程同时 `&` 引用（RefCell ❌；  
  Mutex ✅）；编译器按字段 **自动推导**、会传染；手动 impl 是 unsafe。
- **`fn assert_send<T: Send>(_: &T) {}`（体检器技巧）** ——空泛型函数当编译期断言：  
  能编译 = 类型有该标签；实际项目用它「钉死」某类型必须线程安全。
- **内部可变性（interior mutability）** ——一类容器：通过只读引用（`&self`）也能安全修改内部；  
  手段是把检查/排队挪到运行期；`Arc<锁<T>>` 两层组合 = 共享 + 可改的标准形态。
- **`Mutex<T>` / `.lock()`** ——互斥锁，数据装在锁里面；`lock()` 返回 `Result<Guard, _>`（Err = 锁中毒）；  
  guard 是 RAII 通行证，drop 即解锁。
- **`RwLock<T>` / `.read()` / `.write()`** ——读写锁：读锁可多持、写锁独占 ≈ `sync.RWMutex`；  
  `*data.write().unwrap() = fresh;` 一行完成「加锁-整体替换-解锁」。
- **`{ ... }` 作用域块控制解锁时机** ——块是表达式：guard 块内创建、块结束即 drop（解锁）；  
  块的值（clone 出来的数据）带出去慢慢用。
- **`AtomicU64` / `fetch_add` / `load` / `Ordering`** ——原子整数 ≈ `atomic.Int64`，  
  无锁；`fetch_add` 是原子「读-加-写」三合一；`Relaxed` 独立计数够用；不需要 `mut`。
- **`tokio::sync::Mutex` / `.lock().await`** ——异步锁：等锁时只挂起当前 **任务**（std 锁卡住整个 **线程**）；  
  guard 有 Send 标签， **允许跨 await**；无锁中毒； **FIFO 公平**；等锁 **可取消**；  
  代价是比 std 锁慢。
- **`handle.abort()`** ——强制取消一个 spawn 出去的任务：future 被 drop，它持有的 guard 一并 drop → 锁被释放；  
  对照 Go：goroutine **无法** 被外部强杀，只能靠 ctx 自觉退出。
- **`Rc<T>` vs `Arc<T>`** ——`Rc` 非原子计数更快但无 Send/Sync（只能单线程）；  
  跨任务/线程一律 `Arc`，用错了编译器会拦。
