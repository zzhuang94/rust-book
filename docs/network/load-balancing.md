# 为什么需要负载均衡

> 代码：`code/network-load-balancing/`　运行：
> `cargo run -p network-load-balancing`

前面几课讲的都是“一个客户端连一个服务端”。真实系统里，一个服务
往往要扛住远超单机能力的流量，答案不是把这台机器换得无限大，
而是**多开几台一样的机器，前面架一个入口，帮客户端挑一台去处理**。
这个入口就是**负载均衡（Load Balancer，缩写 LB）**。本课不讲某个
具体产品（Nginx/HAProxy/云 LB）的配置项，只讲“LB 到底在干什么、
为什么会有那些经典的坑”。

如果你还没读过 [《TCP 该知道的现象》](tcp.md)，建议先读一遍——本课
会反复用到“连接”“keepalive”这些概念。

----

# 为什么要加一层

> 单台机器扛不住，最直接的想法是“再加几台”，但客户端总得先知道
> 该连哪一台。

假设你有 3 台完全一样的后端服务，各自监听自己的端口。客户端要么：

- 自己维护一份后端列表，自己挑一台连——这意味着**每个客户端都要
  懂“怎么挑”**，一旦某台后端下线，所有客户端都要同步更新列表；
- 或者只知道**一个固定入口地址**，把“该连哪台后端”这个决定，
  交给入口自己去做。

第二种就是负载均衡的核心价值：**把“选哪台后端”的知识，从每个
客户端里挪到一个集中的地方**。客户端永远只连那一个入口地址，
后端增加、减少、重启，客户端完全不需要知道，也不需要改代码。

对应 Go：

```go
// 没有 LB：客户端自己挑
backends := []string{"127.0.0.1:9001", "127.0.0.1:9002"}
conn, err := net.Dial("tcp", backends[rand.Intn(len(backends))])

// 有 LB：客户端只认一个地址，挑后端的逻辑挪到 LB 里
conn, err := net.Dial("tcp", "127.0.0.1:8080")
```

对应 Rust：

```rust
// 没有 LB
let backends = ["127.0.0.1:9001", "127.0.0.1:9002"];
let stream = TcpStream::connect(backends[idx]).await?;

// 有 LB：客户端代码完全不变，挑后端逻辑在 LB 进程里
let stream = TcpStream::connect("127.0.0.1:8080").await?;
```

两段代码的差异只在于：**“挑后端”这行代码，到底写在客户端里，
还是写在一个独立的中间进程里**。本课的示例代码就是自己动手写一个
这个“独立中间进程”，看清楚它内部到底做了什么。

# 四层还是七层转发

> LB 也分“看多深”——只看 IP/端口转发字节流，还是连 HTTP 内容
> 都要解析。

回顾 [《分层》](layers.md) 讲过的四层模型，LB 按“工作在哪一层”
分成两大类：

| 类型 | 工作层次 | 能看到什么 | 典型场景 |
| --- | --- | --- | --- |
| 四层 LB | 传输层（TCP/UDP） | 只看得到 IP、端口，看不到内容 | 转发速度快，通用性强 |
| 七层 LB | 应用层（HTTP） | 能看到 URL、header、body | 按路径路由、按 header 分流 |

**四层 LB**（比如最简单的 TCP 转发）只把字节原样搬到后端，
它连“这是不是 HTTP 请求”都不知道——对它来说数据只是不透明的
字节流。**七层 LB** 会先把 HTTP 报文解析出来，才能做出“`/api/v1`
走 A 组后端，`/api/v2` 走 B 组后端”这类基于内容的决策。

代价也很直观：七层 LB 要解析协议，比四层多一层开销；但换来的是
更灵活的路由能力。本课的示例代码是一个**四层 LB**——只转发字节，
不关心里面是什么协议，这样才能用一份代码演示“选后端 + 转发”
这件事的骨架，不被具体协议细节绕进去。

> 🔬 底层视角：四层 LB 在 Linux 上常见的高性能实现是 `IPVS`（LVS），
> 工作在内核里，转发一个包的开销接近纯路由转发；七层 LB（如
> Nginx）跑在用户态，要经过完整的 TCP 收发 + HTTP 解析才能转发，
> 单机吞吐天然比四层低一个量级。这也是为什么大厂架构里常见
> “四层 LB 顶在最前面分流，七层 LB 在后面做精细路由”的两级结构。

# 轮询与最少连接

> 选后端的算法五花八门，但入门先掌握两个最常见、最直观的。

**轮询（Round Robin）**：按顺序依次把新连接分给下一台后端，
分完一圈从头再来。实现极其简单——维护一个索引，每次用完加一、
越界归零：

```rust
struct RoundRobin {
    backends: Vec<SocketAddr>,
    next: usize,
}

impl RoundRobin {
    fn pick(&mut self) -> SocketAddr {
        let addr = self.backends[self.next];
        self.next = (self.next + 1) % self.backends.len();
        addr
    }
}
```

对应 Go 里几乎一模一样的写法：

```go
type RoundRobin struct {
    backends []string
    next     int
}

func (r *RoundRobin) Pick() string {
    addr := r.backends[r.next]
    r.next = (r.next + 1) % len(r.backends)
    return addr
}
```

轮询的隐藏假设是**每个请求的处理成本差不多**。如果后端 A 处理的
请求都很轻、后端 B 都很重，轮询会让 B 越堆越忙——这时就需要
**最少连接（Least Connections）**：每次挑“当前正在处理的连接数
最少”的那台，而不是死板按顺序轮。它需要 LB 自己维护一份
“每台后端当前有多少活跃连接”的计数，复杂度比轮询高一档，
但能更公平地应对请求成本不均的场景。

本课的示例代码用的是轮询——足够讲清楚“LB 内部要维护状态、
按状态做决策”这件事的骨架，最少连接留给你在动手实验里自己扩展。

# 健康检查踢掉坏节点

> 光会“选”还不够。如果选中的那台后端已经挂了，LB 必须能察觉，
> 别再往死人堆里派活。

**健康检查（health check）**是 LB 定期主动探测每台后端是否还
活着的机制，常见两种方式：

- **被动健康检查**：LB 转发请求时，如果连接失败或超时，就临时
  把这台后端标记为“不健康”，一段时间内不再选它；
- **主动健康检查**：LB 起一个独立的定时任务，周期性地主动连一次
  （或发一个专门的 `/health` 请求），根据结果更新健康状态。

两种通常会一起用：主动检查负责“定期巡检”，被动检查负责
“遇到问题立刻反应”，别等到下一次巡检周期才发现。

对应 Go 的一个极简主动检查骨架：

```go
func healthCheck(addr string) bool {
    conn, err := net.DialTimeout("tcp", addr, time.Second)
    if err != nil {
        return false
    }
    conn.Close()
    return true
}
```

对应 Rust：

```rust
async fn health_check(addr: SocketAddr) -> bool {
    let res = timeout(Duration::from_secs(1), TcpStream::connect(addr)).await;
    matches!(res, Ok(Ok(_)))
}
```

**健康检查最容易踩的坑是“探测和真实流量走的是同一条能力，
却验证的是不同的东西”**——比如只探测“端口能不能连上”，但
后端进程可能端口开着、数据库连接却早已耗尽，这种“半死不活”
的状态最难被简单探测发现，往往需要业务自己在 `/health` 接口里
补充关键依赖的检查。

# 会话粘滞是双刃剑

> 有些业务要求“同一个用户的请求，每次都落到同一台后端”，
> 这个需求叫会话粘滞（session affinity / sticky session）。

如果每台后端各自在内存里存了这个用户的登录状态、购物车之类的
数据，轮询把同一用户的连续请求分到不同后端，用户会发现自己的
状态“凭空消失”。**会话粘滞**就是让 LB 记住“这个用户上次去了
哪台后端”，之后尽量继续分给同一台，常见实现方式：

- 按客户端 IP 做哈希，同一个 IP 总是落到同一台后端；
- 读取一个专门的 cookie（比如 LB 自己写的 `X-LB-Session`），
  按 cookie 里的标识固定路由。

**会话粘滞的代价是它直接违背了负载均衡“均衡”这两个字的初心**：
一旦某个用户的流量特别大，粘在它身上的那台后端会持续过载，
而其他后端可能很闲，LB 却不能把这部分流量挪走。更彻底的解法
是**把状态从后端进程里搬出去**——存到 Redis 之类的共享存储
（参考 [《接入 Redis》](../http/redis.md)），这样任何一台后端
都能处理任何用户的请求，粘滞就成了不必要的复杂度。

> 🔬 底层视角：按 IP 哈希做粘滞，在“客户端在 NAT 后面、大量用户
> 共享同一个公网 IP”的场景下会失真——这些用户会被粘到同一台
> 后端，造成隐藏的不均衡。下一课 [《代理与 NAT》](proxy-nat.md)
> 会详细讲这个“LB/NAT 之后看到的地址不是真实客户端”的问题。

# LB 后面看不到真实 IP

> 从程序员视角最容易忽略、却最容易在排障时踩坑的一点：**后端
> 收到连接后，`peer_addr()` 拿到的不是客户端的真实地址，是 LB
> 自己的地址**。

四层 LB 如果做的是“转发字节”而不是“原地址透传”，它会自己
主动向后端发起一个新连接，把客户端的数据搬进这个新连接里发过去
——这意味着**后端服务器的 `accept()` 返回的连接，对端地址是
LB 机器的地址，不是客户端浏览器的地址**：

```rust
let (stream, peer) = listener.accept().await?;
// peer 在有 LB 的场景下，通常是 LB 自己的 IP，不是真实客户端 IP！
```

如果后端代码里有“按客户端 IP 限流”“按 IP 记录访问日志”之类的
逻辑，直接用 `peer_addr()` 在 LB 后面基本是错的——所有请求
看起来都来自同一个 IP（LB 的 IP）。真正的客户端 IP 需要 LB
在转发时额外“标注”进去，七层 LB 常见做法是加一个
`X-Forwarded-For` 请求头，这部分留给下一课
[《代理与 NAT》](proxy-nat.md) 详细展开。

对应 Go 里一个常见的踩坑现场：

```go
func handler(w http.ResponseWriter, r *http.Request) {
    // 在 LB/代理后面，这里拿到的往往是 LB 的地址，不是真实客户端
    clientIP := r.RemoteAddr
}
```

**记住这一条直觉：只要架构图里客户端和你的进程之间多了一个
盒子（LB、代理、NAT 网关），你在代码里直接读到的“对端地址”
就要打一个问号，先确认它到底是谁的地址。**

# 一张简易示意图

把前面几节串起来，一次带 LB 的请求大致是这样走的：

```
客户端                    LB（本课示例：轮询 + 健康检查）
   │                              │
   │  连接 127.0.0.1:8080         │
   ├─────────────────────────────▶│
   │                              │ 1. 看一眼健康后端列表
   │                              │ 2. 轮询挑一个：后端 A
   │                              │ 3. 自己连一条新连接到 A
   │                              ├───────────────▶ 后端 A（127.0.0.1:任意端口）
   │                              │◀───────────────┤
   │◀─────────────────────────────┤ 4. 双向转发数据
   │        （回显内容）            │
```

本课的示例代码（`code/network-load-balancing/`）就是照这张图
搭的最小骨架：一个进程内启动两个后端 `TcpListener`（绑定 `:0`，
让系统分配端口），一个前端按轮询把客户端连接转发到其中一个
后端，并在日志里打印“这次选了哪台后端”。

# 一致性哈希减少重新分配

> 按 IP 哈希做会话粘滞有个隐藏问题：**后端数量一旦变化
> （扩容、缩容、某台挂了），几乎所有客户端的哈希结果都会
> 变到不同的后端**。一致性哈希就是为了缓解这个问题设计的。

最朴素的哈希分配是 `hash(key) % 后端数量`——直观，但后端数量
从 3 变成 4 时，`% 4` 和 `% 3` 的结果几乎完全不同，等于几乎
所有客户端都要被重新分配到新的后端，之前保存在旧后端上的
状态（如果做了会话粘滞）会大面积失效。

**一致性哈希（consistent hashing）**的思路是：把每台后端映射
到一个抽象的“环”上的多个点位，每个请求的 key 也映射到环上的
一个点，沿着环顺时针找到的第一台后端就是它该去的地方。这样
增删一台后端，**只影响环上这台后端附近的那一小段范围**，
其余大部分客户端的分配结果完全不受影响：

```
朴素哈希 % N：       后端数量变化 → 几乎所有 key 的分配结果都变
一致性哈希（环形）：  后端数量变化 → 只影响环上相邻的一小段范围
```

业务程序员不需要自己从零实现一致性哈希（Go/Rust 生态都有现成的
库），但理解这个取舍能帮你判断什么时候该考虑它：**如果你的
后端本身有状态（缓存类服务尤其常见），后端数量又会经常变化，
一致性哈希能显著减少“缓存全部失效”这类连锁反应**；如果后端
完全无状态，用哪种哈希方式差别不大，轮询往往已经足够。

# 优雅关闭要等连接排空

> 重启或缩容一台后端时，直接把进程杀掉，正在处理的请求会被
> 硬生生打断——这是生产环境里非常常见、但容易被忽视的一类
> 用户可感知故障。

一个后端要下线（部署新版本、缩容），完整流程应该是：

1. **先让 LB/服务发现停止把新连接分给这台后端**——但不动它
   已经在处理的连接；
2. **等这台后端当前正在处理的请求全部处理完**（这个等待过程
   常被称为**连接排空（connection draining）**）；
3. **确认没有活跃连接了，再真正关闭进程**。

如果跳过第 2 步直接杀进程，正在这台后端上处理的请求会因为
连接被强制中断而失败，这类失败通常发生在“部署”这个本该是
无损的操作期间，用户体验上很容易被理解成“系统不稳定”，
但根因其实是发布流程没有做好排空。

对应 Go 里 `http.Server` 自带的优雅关闭支持：

```go
srv := &http.Server{Addr: ":8080"}
go srv.ListenAndServe()

// 收到关闭信号后：
ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
defer cancel()
srv.Shutdown(ctx) // 拒绝新连接，等待现有请求处理完，最多等 30 秒
```

Rust 生态里，`axum`/`hyper` 也提供了类似的优雅关闭钩子
（`with_graceful_shutdown`），思路完全一致：**先停止接受新
连接，再等现有连接自然结束，超时后才强制关闭**。这一步和
LB 侧的健康检查要配合起来——后端进入排空状态时，应该主动让
自己的健康检查探测“变得不健康”，这样 LB 会提前不再往它派发
新连接，排空过程能更快收尾。

# 常见误区

> 负载均衡相关最容易被想当然的几个说法，先纠正过来。

- **误区一：负载均衡等于“流量绝对均匀分配”。** 轮询只保证
  “分配顺序均匀”，不保证“每台后端实际负载均匀”——如果请求
  成本差异很大，轮询下某些后端实际负载可能明显更高，这也是
  “最少连接”算法存在的原因，选算法要匹配你的流量特征；
- **误区二：只要配了健康检查，故障就能被立刻发现。** 健康检查
  有检测周期，从后端真正出问题到被标记为不健康之间，必然存在
  一段“已经不健康但还在被派发流量”的窗口——窗口大小取决于
  检查周期和判定失败次数的配置，不是零延迟的；
- **误区三：LB 只是“转发数据”，本身不会成为瓶颈。** LB 本身
  也是一个需要处理所有流量的进程，七层 LB 尤其如此（要解析
  HTTP），LB 自己的性能、连接数上限、CPU 也是系统整体容量规划
  里必须考虑的一环，不能假设它有无限的处理能力；
- **误区四：会话粘滞和无状态设计是互斥的，用了粘滞就不用管
  状态管理了。** 会话粘滞只是缓解问题的权宜之计，真正干净的
  解法始终是把状态从后端进程搬出去（存到共享存储），粘滞更适合
  当作过渡方案，而不是长期架构选择。

# 排错对照

> 负载均衡场景下的问题，往往表现在“客户端体验”和“单台后端”
> 两个层面，按下面的表格分别定位。

| 现象 | 大概率原因 | 排查方向 |
| --- | --- | --- |
| 某台后端明显比其他后端负载高很多 | 用了轮询但请求成本不均，或者会话粘滞导致某类用户集中到这台 | 检查负载分配算法是否匹配实际流量特征，考虑最少连接或调整粘滞策略 |
| 部署新版本时，用户偶发看到请求失败 | 发布流程没有做连接排空，直接杀掉了还在处理请求的旧进程 | 检查发布脚本是否调用了优雅关闭，是否给了足够的排空等待时间 |
| 后端日志里所有请求看起来来自同一个 IP | 业务代码在读传输层 `peer_addr()`，而不是 LB 传递的真实客户端 IP | 回顾“LB 后面看不到真实 IP”一节，改用 `X-Forwarded-For` |
| 扩容后一段时间内命中率大幅下降（缓存类服务） | 用了朴素哈希 `% N`，后端数量变化导致几乎全部 key 重新分配 | 评估是否需要换成一致性哈希 |
| 某台后端已经挂了，但 LB 还在往它转发请求 | 健康检查周期设置太长，或探测的内容和真实故障不匹配 | 缩短健康检查周期，检查探测逻辑是否覆盖了真正会出问题的依赖 |

# 动手实验

1. 运行 `cargo run -p network-load-balancing`，连续发起多个连接，
   观察日志里“轮询选中后端 0 / 1”是否严格交替；
2. 把其中一个后端改成故意 `sleep` 很久再回复，观察轮询是否仍然
   “不管死活”地把下一个请求派给它——这正是没做健康检查的后果；
3. 尝试自己加一个 `healthy: Vec<bool>` 状态，跳过被标记为不健康
   的后端，体会“选后端”和“判断健康”是两件独立的事；
4. 把轮询改成随手写的“最少连接”：维护每台后端的活跃连接数，
   每次选数值最小的一台，对比两种算法在“成本不均”场景下的差异。

# 三句话带走

1. 负载均衡把“该连哪台后端”的决定，从每个客户端搬到一个集中
   的入口进程，后端增减对客户端完全透明。
2. 四层 LB 只转发字节看不到协议内容、七层 LB 能解析 HTTP 做
   精细路由；轮询、最少连接是最基础的两种选后端算法。
3. 会话粘滞和健康检查各自有代价——粘滞牺牲均衡性，健康检查
   要小心“探测通过但业务已经半死”；LB 后面看到的对端地址
   往往不是真实客户端 IP。

# 附：本课生词表

- **负载均衡（Load Balancer，LB）** —— 位于客户端和多台后端
  之间，负责把请求分摊到不同后端的中间层。
- **四层 LB** —— 工作在传输层，只转发字节，不解析应用层协议。
- **七层 LB** —— 工作在应用层，能解析 HTTP 内容做基于内容的路由。
- **轮询（Round Robin）** —— 按顺序依次分配请求给每台后端的算法。
- **最少连接（Least Connections）** —— 优先分配给当前活跃连接数
  最少的后端的算法。
- **健康检查（health check）** —— 主动或被动探测后端是否可用的
  机制，分主动探测和被动标记两种。
- **会话粘滞（session affinity）** —— 让同一用户的请求尽量落到
  同一台后端的策略，常按 IP 哈希或 cookie 实现。
- **`X-Forwarded-For`** —— HTTP 头，用于在经过代理/LB 后传递
  真实客户端 IP，详见下一课。
- **一致性哈希（consistent hashing）** —— 把后端和请求 key 映射
  到同一个环上分配的算法，后端数量变化时只影响环上邻近的一小段。
- **连接排空（connection draining）** —— 后端下线前，先停止接受
  新连接、等现有连接自然处理完，再真正关闭进程的过程。
- **优雅关闭（graceful shutdown）** —— 进程收到关闭信号后，
  完成连接排空再退出，而不是立即强制终止的关闭方式。
