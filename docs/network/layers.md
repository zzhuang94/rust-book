# 分层是为了解耦

> 代码：`code/network-layers/`　运行：`cargo run -p network-layers`

你在 Gin 里写过 `r.GET("/user", handler)`，在 axum 里写过
`Router::new().route("/user", get(handler))`。这一行代码背后，数据其实要经过好几层  
才能从对方的网卡走到你的 `handler` 里。本课不讲“背下 OSI 七层名词”，只讲一件事：  
**分层到底解决了什么问题，业务程序员真正碰到的是哪几层**。

如果你还不确定“阻塞”“socket”“fd”这些词，先读
[《计算机组成入门》](../os/computer-basics.md)。本章之后依次是
[《寻址》](addressing.md)、[《socket 编程模型》](socket.md)、
[《TCP 该知道的现象》](tcp.md)、[《UDP 与双栈 socket》](udp-sockets.md)，
建议按顺序读。

# 七层还是四层

> 教科书讲 OSI 七层，真实世界的协议栈按 TCP/IP 四层实现。两套名词要对得上，
> 但不用逐字背。

OSI 是一套“理论参考模型”，TCP/IP 是“现实里真正跑的模型”。对应关系：

| OSI 七层 | TCP/IP 四层 | 你会遇到的东西 |
| --- | --- | --- |
| 应用层 | 应用层 | HTTP、DNS、gRPC、你写的业务协议 |
| 表示层 | 应用层 | TLS 加密、JSON/Protobuf 编码 |
| 会话层 | 应用层 | 基本被应用层协议自己吸收（如 HTTP 的 Cookie/Session） |
| 传输层 | 传输层 | TCP、UDP |
| 网络层 | 网络层 | IP、路由 |
| 数据链路层 | 网络接口层 | 以太网、Wi-Fi、MAC 地址 |
| 物理层 | 网络接口层 | 网线、光纤、无线电波 |

真相是：OSI 的“表示层”“会话层”在实际协议栈里基本没有独立的实现，
它们的职责被应用层协议（HTTP、TLS）自己吞并了。**四层模型才是你机器上
真正跑着的代码结构**，OSI 更多是用来交流概念的词汇表。

> 🔬 底层视角：Linux 内核网络栈的代码目录基本就是按四层组织的——
> `net/ethernet`（网络接口层）、`net/ipv4`（网络层）、
> `net/ipv4/tcp.c`（传输层）。你的应用进程只通过系统调用碰到最上面
> 传输层暴露出来的 socket API，网络层和网络接口层完全由内核代劳。

# 你只碰这三层

> 业务代码几乎不会直接操作网络接口层，实际接触的是应用层、传输层、网络层。

写业务代码时，你的手基本停在这三层：

- **应用层**：你写的协议解析——HTTP 路由、JSON 解析、业务字段校验；
- **传输层**：你选 TCP 还是 UDP，是否设置超时、`Nagle` 算法（后面 [《TCP》](tcp.md) 细讲）；
- **网络层**：你偶尔会看到 IP 地址，但几乎不会自己拼 IP 包头。

网络接口层（网卡驱动、MAC 地址、以太网帧）完全由操作系统内核和网卡硬件负责，
业务程序员一辈子可能都不会直接碰它。这不是偷懒，而是分层设计的目的：
**每层只需要相信“下一层能把我的数据完整送到对面的同一层”，不用关心
下面具体怎么实现**。

对应 Go：

```go
conn, err := net.Dial("tcp", "example.com:80")
// 你完全不知道、也不需要知道这条连接走的是网线还是 Wi-Fi
```

对应 Rust：

```rust
let stream = std::net::TcpStream::connect("example.com:80")?;
// 同样看不到、也不需要看到网络接口层
```

两段代码里，`net.Dial` 和 `TcpStream::connect` 都只操作到“传输层”这一级——
你给它一个地址和端口，剩下的封装、路由、网卡发送全部被藏起来了。

# 各层各管一段

> IP、TCP、HTTP 分别负责什么，用一句话说清楚，别的细节以后再补。

三个你真正会打交道的协议，各自的职责边界：

- **IP（网络层）**：只负责“把一个包从 A 机器送到 B 机器”，不保证到达、不保证顺序、
  不管里面装的是什么；
- **TCP（传输层）**：在 IP 之上加“连接、顺序、重传、流量控制”，
  让你看到的是一条可靠的字节流（细节见 [《TCP》](tcp.md)）；
- **HTTP（应用层）**：在 TCP 字节流之上，规定“怎么把请求和响应写成文本”——
  方法、路径、header、body，纯粹是字符串格式的约定（已经在
  [《从零手写 HTTP》](../http/http-from-scratch.md) 里逐字节拆过一次）。

一个直观的比方：**IP 是快递公司只管把箱子送到门口；TCP 是快递员按顺序敲门、
丢了会重投；HTTP 是箱子里那张写着格式的订单纸**。你写业务代码时基本只填订单，
但知道下面还有两层在替你干活，遇到问题才不会瞎猜。

对应 Go 里“同一件事、不同层次”的两种写法：

```go
// 应用层视角：只关心 HTTP 语义
resp, err := http.Get("http://example.com/")

// 传输层视角：自己管理连接，自己拼 HTTP 文本
conn, err := net.Dial("tcp", "example.com:80")
conn.Write([]byte("GET / HTTP/1.1\r\nHost: example.com\r\n\r\n"))
```

对应 Rust：

```rust
// 应用层视角（依赖 reqwest，内部走 hyper + tokio）
let body = reqwest::get("http://example.com/").await?.text().await?;

// 传输层视角：自己拿到 TcpStream，自己写字节
let mut stream = std::net::TcpStream::connect("example.com:80")?;
stream.write_all(b"GET / HTTP/1.1\r\nHost: example.com\r\n\r\n")?;
```

`http.Get` / `reqwest::get` 都是“应用层”调用，内部自己拨了一次 TCP 连接、
自己拼了 HTTP 报文；`net.Dial` / `TcpStream::connect` 是“传输层”调用，
把 HTTP 报文怎么拼交还给你自己。**同一件事，看你站在哪一层写代码**。

# 封装与解封装

> 数据从应用层往下走，每一层都往外面裹一层“信封”；到对方那边再从外往里一层层拆开。

这就是分层协议真正的运作方式，术语叫 **封装（encapsulation）** 和
**解封装（decapsulation）**：

```
发送方（往下走，层层裹）           接收方（往上走，层层拆）
┌──────────────┐                   ┌──────────────┐
│  HTTP 报文    │                   │  HTTP 报文    │
├──────────────┤   加 TCP 头        ├──────────────┤   剥 TCP 头
│ TCP[HTTP]    │  ───────────────▶ │ TCP[HTTP]    │ ───────────────▶
├──────────────┤   加 IP 头         ├──────────────┤   剥 IP 头
│ IP[TCP[..]]  │  ───────────────▶ │ IP[TCP[..]]  │ ───────────────▶
├──────────────┤   加以太网帧头      ├──────────────┤   剥以太网帧头
│ 以太网帧      │  ───────────────▶ │ 以太网帧      │ ───────────────▶
└──────────────┘        网线         └──────────────┘
```

关键认知：**每一层只认得自己那一层的“信封”，完全不关心里面装的内容**。
IP 层只看 IP 头里的目标地址，TCP 头以及里面的 HTTP 文本对它来说都是不透明的
一段字节；HTTP 解析器也完全不知道自己是通过 TCP 还是别的传输方式送过来的。

这正是“分层解耦”的价值：你今天把 HTTP 换成 gRPC、把 TCP 换成 QUIC，
只要接口协议不变，其他层完全不用动。反过来，**如果某一层出了问题
（比如丢包），你也只能在那一层或它上面感知到，看不到具体是哪根网线、
哪个路由器出的问题**——这也是排查网络问题总是从上层现象往下层原因反推的原因。

> 🔬 底层视角：以太网帧还有一个“最大传输单元”限制（MTU，常见 1500 字节）。
> 如果 IP 层要发的数据超过这个限制，会被拆成多个以太网帧分别发送，
> 到对方那边再拼回一个完整的 IP 包——这是网络接口层自己的事，
> 应用层和传输层完全感觉不到。

# 一次请求的旅程

> 把前面几节串起来：一次 `http.Get` 从你的代码出发，到对方网卡送达，再一路返回。

假设你在本机（`127.0.0.1`）访问一个跑在 `example.com` 的服务：

```
客户端进程                                    服务端进程
   │ http.Get("http://example.com/")              │
   ▼                                               ▲
应用层：拼 HTTP 请求文本                    应用层：解析 HTTP 请求文本
   │                                               ▲
   ▼                                               │
传输层：TCP 三次握手，把 HTTP 文本切成        传输层：把收到的字节按 TCP 顺序
        字节流一段段发送、等待确认                   拼回完整字节流，交给上层
   │                                               ▲
   ▼                                               │
网络层：给每个 TCP 分段裹上 IP 头，           网络层：拆 IP 头，确认是发给
        写上目标 IP，交给路由                       自己的，把数据交给传输层
   │                                               ▲
   ▼                                               │
网络接口层：裹以太网帧头，                    网络接口层：从网卡收到电信号，
            通过网卡把信号发到网线上                拆出以太网帧交给网络层
   │                                               ▲
   └──────────── 经过若干路由器转发 ────────────────┘
```

几个容易被忽略、但对写业务代码很重要的现象：

- 你写的 `http.Get` 只对应图里最上面那一小格，剩下所有箭头都是标准库和内核代劳的；
- “请求”和“响应”是两次对称的旅程——响应从服务端应用层出发，一路封装下去，
  再到客户端一路解封装上来；
- 路由器只理解到网络层（IP），它们不会、也不需要拆开 TCP 头去看里面的 HTTP 内容
  （除非是专门做深度包检测的设备）。

# handler 之下发生了什么

> 把“分层”落回你最熟悉的场景——写一个 HTTP handler 时，键盘敲下的每一行代码
> 对应哪一层。

Gin 里最常见的一行：

```go
func handler(c *gin.Context) {
    c.JSON(http.StatusOK, gin.H{"msg": "ok"})
}
```

Rust 的 axum 版本几乎是逐行对照：

```rust
async fn handler() -> Json<serde_json::Value> {
    Json(serde_json::json!({ "msg": "ok" }))
}
```

这一行看起来只是“返回一个 JSON”，但它触发的下层动作是：

1. **应用层**：框架把 `gin.H{...}` / `serde_json::json!` 序列化成 JSON 文本，
   拼上 `HTTP/1.1 200 OK`、`Content-Type`、`Content-Length` 等响应头；
2. **传输层**：拼好的响应文本被切成若干段，通过已经建立好的 TCP 连接依次
   `write` 出去，等待对方 TCP 确认收到；
3. **网络层**：每个 TCP 段被裹上 IP 头，按目标 IP 查路由表决定从哪个网卡送出；
4. **网络接口层**：内核网卡驱动把数据交给网卡硬件，转换成电信号或无线电波
   发送出去。

你在 `handler` 里看到的只有第 1 步；第 2~4 步全部是标准库、内核、
网卡驱动分层接力完成的，你既看不到也不需要看到。**这正是分层的价值：
你只需要对自己那一层的“信封”负责，写业务代码时才不用同时操心
TCP 重传和网卡驱动**。

# 动手实验

1. 在终端执行 `curl -v http://example.com/`，观察 `-v` 打出的 `>` `<` 行——
   那正是应用层看到的 HTTP 报文，和图里“一次请求的旅程”对上号；
2. 执行 `ping example.com`（Windows/Linux 通用），观察它只依赖网络层（ICMP），
   完全不涉及 TCP、HTTP，体会“分层”让 `ping` 可以只测最底下几层是否通畅；
3. 用 `netstat -an`（Windows）或 `ss -tn`（Linux）看当前机器上建立的 TCP 连接，
   注意每条连接只显示“IP:端口”，看不到里面跑的是 HTTP 还是别的协议——
   这就是传输层不关心应用层内容的证据；
4. 把本课开头的 Go/Rust 两段代码分别换成“应用层调用”和“传输层手写”，
   对比抓包工具（如 Wireshark）里看到的 TCP 载荷是否完全一样。

# 三句话带走

1. OSI 七层是交流用的参考词汇，真实协议栈按 TCP/IP 四层实现，
   业务程序员实际只碰应用层、传输层、网络层。
2. 分层的核心机制是封装/解封装：每层只认自己的“信封”，完全不关心
   里面装的内容，这才让协议可以独立演化。
3. 你在 handler 里写的一行返回，背后是应用层拼报文、传输层切段发送、
   网络层查路由、网络接口层过网卡的连续接力。

# 附：本课生词表

- **OSI 七层** —— 理论参考模型：物理、数据链路、网络、传输、会话、表示、应用层。
- **TCP/IP 四层** —— 真实协议栈实现：网络接口层、网络层、传输层、应用层。
- **封装（encapsulation）** —— 数据从上层往下层传递时，每层加上自己的头部信息。
- **解封装（decapsulation）** —— 数据从下层往上层传递时，每层剥掉对应的头部信息。
- **MTU** —— 一次网络接口层能发送的最大数据长度，超出会被拆成多帧。
- **路由（routing）** —— 网络层根据目标 IP 决定数据包该往哪个方向转发。
- **深度包检测** —— 网络设备越过网络层，去解析上层协议内容的行为，非默认能力。
