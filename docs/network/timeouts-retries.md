# 超时和重试不是小事

> 代码：`code/network-timeouts-retries/`　运行：  
> `cargo run -p network-timeouts-retries`  

[《TCP 该知道的现象》](tcp.md) 的“超时不止一种”一节已经提过  
连接超时和读写超时的区别，[《reqwest 与上游容错》](../http/reqwest-resilience.md)  
也讲过重试要有节制。这一课把这两个话题单独拎出来，从“网络小白”  
的视角完整讲一遍——因为这两件事**几乎是所有网络故障排查的  
起点**，也是业务代码里最容易被想当然写错的地方。  

本课不引入新的网络协议知识，全部建立在前面几课的地基上：  
如果“连接”“读写”“重传”这些词还不熟，建议先回顾  
[《TCP》](tcp.md) 和 [《socket 编程模型》](socket.md)。

# 超时至少分三种

> 新手最常犯的错误，是把“超时”当成一个笼统的旋钮去调，  
> 实际上至少要分清三种，分别在不同阶段起作用。  

- **连接超时（connect timeout）**：从发起连接到真正建立成功  
  （[《TCP》](tcp.md) 讲的三次握手完成）最多等多久。如果目标  
  机器完全不可达（网络层面走不通），操作系统不会立刻告诉你  
  失败，而是要一直等到这个超时才报错；  
- **读写超时（read/write timeout）**：连接已经建立之后，  
  某一次具体的读或写操作最多等多久。连接建立成功不代表对方  
  会立刻回应——它可能在处理别的事情，也可能已经悄悄挂掉但你  
  还没发现；  
- **整体超时（overall/request timeout）**：一次完整业务操作  
  （可能包含连接、发送请求、等待响应、读取响应体好几个步骤）  
  从开始到结束，总共最多能花多久。  

三者的关系可以理解成“整体超时是一个大预算，连接超时和读写超时  
是这个预算里某个具体步骤单独设的小上限”。一个常见的误区是  
**只设了其中一种就以为够了**：  

- 只设连接超时、不设读写超时的服务，一旦对方连上之后就不再  
  说话（或者说得很慢），会在读操作上无限等待，连接慢慢堆积，  
  最终耗尽资源——这正是 [《TCP》](tcp.md) “超时不止一种”一节  
  强调过的坑；  
- 只设整体超时、不设连接超时，虽然能兜底不会永远卡住，但没办法  
  在“连接阶段”和“处理阶段”之间分别分配预算，一旦整体超时快到  
  了，你也不知道时间到底花在哪一步。  

Rust 标准库需要手动分别设置连接超时和读写超时：  

```rust
use std::net::{SocketAddr, TcpStream};
use std::time::Duration;

let addr: SocketAddr = "127.0.0.1:8080".parse()?;
// 连接超时：三次握手最多等 3 秒
let stream = TcpStream::connect_timeout(&addr, Duration::from_secs(3))?;
// 读写超时：单次 read/write 最多等 5 秒
stream.set_read_timeout(Some(Duration::from_secs(5)))?;
stream.set_write_timeout(Some(Duration::from_secs(5)))?;
```

Tokio 里更常见的做法是用 `tokio::time::timeout` 把每一步单独  
包一层，语义更直观：  

```rust
use tokio::time::{timeout, Duration};

let stream = timeout(
    Duration::from_secs(3),
    tokio::net::TcpStream::connect("127.0.0.1:8080"),
).await.context("连接超时")??;

let n = timeout(Duration::from_secs(5), stream.read(&mut buf))
    .await
    .context("读超时")??;
```

对应 Go（`net.DialTimeout` 管连接，`conn.SetReadDeadline` 管  
读写，两者要分别设置）：  

```go
conn, err := net.DialTimeout("tcp", "127.0.0.1:8080", 3*time.Second)
conn.SetReadDeadline(time.Now().Add(5 * time.Second))
n, err := conn.Read(buf)
```

如果你用的是 HTTP 客户端（比如 `reqwest`），大多数库会同时暴露  
“连接超时”和“整体超时”两个选项，把读写超时的细节包在整体超时  
背后：  

```rust
let client = reqwest::Client::builder()
    .connect_timeout(Duration::from_millis(300))  // 连接阶段
    .timeout(Duration::from_secs(1))               // 从发起到读完响应体
    .build()?;
```

[《reqwest 与上游容错》](../http/reqwest-resilience.md) 的  
“两种超时不同”一节详细讲过这一对参数，这里只是把它放回  
“超时至少分三种”这张更完整的地图里。

# 幂等才能盲目重试

> 超时之后要不要重试，第一个该问的问题不是“重试几次”，  
> 而是“这个操作重试安全吗”。  

超时并不代表对方一定没收到、也没处理这个请求——**客户端放弃  
等待，只说明客户端这一端不想再等了，请求可能已经到达对方，  
甚至已经被处理并产生了副作用**，只是响应还没送回来（或者响应  
送回来的路上又超时了）。这是理解“重试为什么危险”的第一个关键  
认知。  

**幂等（idempotent）** 是判断“能不能安全重试”的核心概念：一个  
操作执行一次和执行多次，最终效果完全一样，就叫幂等。用几个  
常见业务场景对比：  

| 操作 | 是否幂等 | 重试是否安全 |
| --- | --- | --- |
| 查询一条订单（`GET`） | 幂等 | 安全，重试几次结果都一样 |
| 把某个状态设为“已完成”（`PUT`） | 幂等 | 安全，设成同一个值多次没差别 |
| 扣一次库存、下一次订单（`POST`） | **不幂等** | **危险**，重试可能导致扣两次库存 |
| 给余额加 10 元 | **不幂等** | **危险**，重试可能变成加 20 元 |

HTTP 方法本身有一份“语义上应该幂等”的约定（[《HTTP 是文本协议》](http-protocol.md)  
提到过方法的含义）：  

- `GET`、`HEAD`、`PUT`、`DELETE` **语义上应该幂等**——多次执行  
  和一次执行效果一样；  
- `POST`、`PATCH` **语义上通常不幂等**——每次调用可能都在“新增”  
  或“叠加”一些东西。  

但要提醒一句：**这只是约定，不是强制保证**。如果后端 `PUT`  
接口的实现里悄悄写了“每次调用都往日志表插一条记录”，它就不再  
是真正幂等的，即使语义上你以为它是。判断能不能重试的最终依据，  
是接口的**真实实现行为**，不是方法名字。  

不幂等操作也不是完全不能重试，业界常见的解法是**幂等键  
（idempotency key）**：客户端为每次“业务意义上唯一”的操作生成  
一个唯一 ID，随请求一起传给服务端；服务端收到请求后，先检查这个  
ID 是否已经处理过——处理过就直接返回上次的结果，不再重复执行  
业务逻辑。这样即使客户端因为超时重试了好几次，同一个幂等键对应  
的操作也只会真正生效一次。

# 指数退避加抖动

> 决定重试之后，紧接着要回答的问题是“间隔多久重试”——  
> 立刻重试往往会让本来就不稳定的服务雪上加霜。  

最朴素但危险的写法，是失败后立刻重试、循环个几次：  

```rust
// 反面例子：没有任何间隔，故障时会瞬间打出去一堆重复请求
for _ in 0..3 {
    if let Ok(resp) = try_request().await {
        return Ok(resp);
    }
}
```

如果失败的原因是“对方暂时扛不住”（比如短暂过载、正在重启），  
立刻重试只会让本来就吃紧的对方，瞬间收到更多请求，情况变得更糟——  
这在分布式系统里是一种真实存在、有专门名字的现象，叫**重试风暴**  
（retry storm）。  

**指数退避（exponential backoff）** 的思路是：每次重试间隔的等待  
时间，按指数增长——比如第一次失败后等 100ms，第二次等 200ms，  
第三次等 400ms，让重试的压力随时间递减而不是持续轰炸：  

```rust
use tokio::time::{sleep, Duration};

let mut delay = Duration::from_millis(100);
for attempt in 1..=5 {
    match try_request().await {
        Ok(resp) => return Ok(resp),
        Err(e) if attempt == 5 => return Err(e), // 到达最大次数，不再重试
        Err(_) => {
            sleep(delay).await;
            delay *= 2; // 100ms → 200ms → 400ms → 800ms → 1.6s
        }
    }
}
```

单纯的指数退避还有一个问题：如果同时有一大批客户端在同一时刻  
遭遇同样的失败（比如某个下游服务突然抖动一下），它们会在完全  
相同的时间点集体重试，形成一次“惊群”（thundering herd）——  
这批重试请求还是会同时砸到刚恢复一点点的服务上。**抖动  
（jitter）** 就是在退避时间里加一点随机量，把这批重试的时间点  
稍微打散：  

```rust
use rand::Rng;

let base_delay = Duration::from_millis(100) * 2u32.pow(attempt - 1);
let jitter_ms = rand::thread_rng().gen_range(0..base_delay.as_millis() as u64 / 2);
let delay = base_delay + Duration::from_millis(jitter_ms);
sleep(delay).await;
```

对应 Go 的思路完全一样，社区里常用 `time.Sleep` 手写，或者用  
`github.com/cenkalti/backoff` 这类现成库：  

```go
delay := 100 * time.Millisecond
for attempt := 1; attempt <= 5; attempt++ {
    if resp, err := tryRequest(); err == nil {
        return resp, nil
    }
    jitter := time.Duration(rand.Int63n(int64(delay) / 2))
    time.Sleep(delay + jitter)
    delay *= 2
}
```

一份完整可用的重试策略，通常要同时满足这几点（回顾  
[《reqwest 与上游容错》](../http/reqwest-resilience.md) “重试要  
有节制”一节）：只重试临时性错误、指数退避、加抖动、限制最大  
次数或总时间、每次尝试和整体操作都受超时控制——**少任何一条，  
都可能把“临时故障”放大成“连锁故障”**。

# 连接池复用

> 重试和超时的成本，很大一部分取决于“每次尝试要不要重新建连接”——  
> 连接池能把这个成本压到最低。  

[《TCP》](tcp.md) 讲过三次握手，[《TLS 与 HTTPS》](tls.md) 讲过  
TLS 握手，两者都意味着“每建立一条新连接，都要付出额外的网络  
往返时间”。如果你的重试逻辑每次都重新建一条连接，那么重试的  
代价会比想象中高得多——不仅要重试业务请求本身，还要重新扛一次  
握手开销。  

**连接池（connection pool）** 的思路是：预先建立并保留一批连接，  
业务代码需要发请求时，从池子里借一条已经建好的连接来用，用完  
归还回池子，而不是每次都新建、用完就扔。这样重试时，大概率能  
复用一条已经握过手的连接，省掉重复建连的成本。  

[《reqwest 与上游容错》](../http/reqwest-resilience.md) 已经强调  
过这一点：`reqwest::Client` 内部自带连接池，应该在进程启动时  
创建一次、全局复用，而不是每次请求都 `Client::new()`：  

```rust
// 全局只建一次，之后所有请求（包括重试）共享这个连接池
let client = reqwest::Client::builder()
    .connect_timeout(Duration::from_millis(300))
    .timeout(Duration::from_secs(1))
    .build()?;
```

对应 Go：  

```go
// http.Client 内部的 Transport 自带连接池，同样应该全局复用
client := &http.Client{Timeout: time.Second}
```

一个容易被忽略的连接池细节：**池子里的连接也可能已经悄悄失效**  
（比如对端主动关闭、或者中间的网络设备把这条连接的记录清掉了，  
回顾 [《TCP》](tcp.md) “keepalive 探活”一节）。成熟的连接池实现  
通常会在借出连接前做一次轻量检查，或者在真正使用时如果发现连接  
已经断开，就自动换一条新连接重试一次——这也是为什么“重试”和  
“连接池”这两个话题总是被放在一起讨论：**很多“需要重试”的失败，  
根源就是拿到了一条已经过期的池中连接**。

# Go context 与 Rust tokio::time::timeout 对照

> 两边解决的是同一个问题——“这个操作不能无限等下去”——  
> 但实现思路有一个关键差异。  

Go 用 `context.Context` 承载超时和取消信息，需要**被调用的每一层  
函数自己检查 `ctx`**，才能真正响应超时：  

```go
ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
defer cancel()

req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
resp, err := client.Do(req)
// err 里会包含 ctx 超时导致的取消信息
```

Rust 里 `tokio::time::timeout` 走的是完全不同的路子——**调用方  
单方面给任意一个 future 包一层时限，不需要被调用的代码配合**，  
到点直接把里面的 future 整个丢弃（drop）：  

```rust
use tokio::time::{timeout, Duration};

match timeout(Duration::from_secs(3), client.get(url).send()).await {
    Ok(Ok(resp)) => { /* 正常拿到响应 */ }
    Ok(Err(e)) => { /* 请求本身失败（不是超时） */ }
    Err(_) => { /* 超时：里面的 future 已经被丢弃、取消 */ }
}
```

[《三类控制手段》](../async/task-control.md) 里已经讲过这个差异：  
**Go 的取消要靠被调用方层层检查 `ctx`，Rust 的取消是调用方单方面  
说了算，配不配合都会被 drop**。放到网络场景里，这个差异会体现在：  

- Go 里如果某个下游库内部的循环忘了检查 `ctx.Done()`，`context`  
  超时了但那个循环仍然继续跑，直到它自己碰到下一个检查点；  
- Rust 里 `timeout()` 到点后，无论内部的 future 在做什么，  
  都会被强制丢弃——这通常更“干脆”，但也要求写代码的人清楚  
  “哪些操作被取消一半是安全的”，比如一次没写完的数据库事务  
  被半路取消，需要底层库自己保证不会留下损坏的状态。  

从整个调用链的角度看，两边有一个共同的最佳实践：**超时预算应该  
从最外层往内层逐级分配**，而不是每一层都各自设一个独立的大超时。  
比如整个用户请求只给了 800ms 处理，那么它内部调用下游服务时，  
就不应该再给下游一个独立的 3 秒超时——外层早就已经超时放弃了，  
内层还在傻等没有意义。

# 业务最常踩的坑清单

> 汇总本课和前面几课提到过的、真实业务里反复出现的超时/重试坑，  
> 排障时可以按这张清单逐条对照。  

- **只设了连接超时，没设读写超时**——对方连上之后不说话，  
  连接无限期挂着，慢慢堆积到资源耗尽；  
- **超时时间设得比下游服务实际响应时间还短**——看起来“老是超时”，  
  其实只是预算给得不够，需要先搞清楚正常响应大概要多久；  
- **对非幂等操作无脑重试**——扣款、下单类接口叠加执行，  
  产生重复副作用，事后很难对账排查；  
- **重试没有指数退避、没有抖动**——故障期间一堆客户端同时反复  
  重试，把还没完全恢复的服务再打垮一次；  
- **重试次数或总时间没有上限**——一个永远连不上的地址，客户端  
  卡在“无限重试”里，表现上像是“程序卡死了”；  
- **每次请求都新建连接（包括每次重试）**——握手成本被反复叠加，  
  超时预算大量浪费在建连接上，而不是真正的业务处理；  
- **定时任务的重试和下一轮调度重叠**——上一轮还没重试完，  
  下一轮定时又开始了，同时跑的请求数越滚越多；  
- **把“超时”和“对方明确拒绝”混为一谈**——[《TCP》](tcp.md)  
  讲过，连接被拒绝（`ConnectionRefused`）通常说明对方机器可达  
  但没有进程在监听，这和网络不通导致的超时是完全不同的故障，  
  排查方向也不同（详见下一课 [《网络排障工具箱》](debug-tools.md)）。

# 动手实验

1. 写一个只设置读写超时、不设置连接超时的客户端，连一个确定不通  
   的地址，观察它卡住的时间和设置的读写超时数值是否吻合（提示：  
   连接阶段根本没走到读写这一步）；  
2. 用本课的指数退避代码，把一个总是失败的请求跑满 5 次重试，  
   打印每次重试前实际等待的时间，确认间隔确实在成倍增长；  
3. 给同一段重试代码加上随机抖动，多跑几次观察每次的具体等待时间  
   是否有波动，而不是每次都完全相同；  
4. 用 [《三类控制手段》](../async/task-control.md) 的 `timeout()`  
   包一个故意执行很久的异步任务，验证超时之后任务内部的代码  
   确实不再继续打印日志（被丢弃了）；  
5. 找一个非幂等的接口示例（比如“余额加 N”），设计一个幂等键  
   方案，让同一个幂等键无论重试多少次，余额只被加一次。

# 三句话带走

1. 连接超时、读写超时、整体超时是三件不同的事，缺一种都会留下  
   隐患；超时代表“我不等了”，不代表“对方没收到、没处理”。  
2. 只有幂等操作才能安全地盲目重试；非幂等操作要靠幂等键或业务  
   层面的去重来保证“重试多次也只生效一次”。  
3. 重试必须配合指数退避加抖动、限制次数和总时间，并尽量复用  
   连接池里已经握好手的连接；预算要从调用链最外层往内层逐级分配。

# 附：本课生词表

- **连接超时** —— 建立连接（TCP 握手，或加上 TLS 握手）最多  
  等待的时间。  
- **读写超时** —— 连接建立后，单次读或写操作最多等待的时间。  
- **整体超时** —— 一次完整业务操作从开始到结束的总时间上限。  
- **幂等（idempotent）** —— 一个操作执行一次和执行多次，最终  
  效果完全一样。  
- **幂等键（idempotency key）** —— 客户端为一次业务操作生成的  
  唯一标识，服务端据此避免重复执行。  
- **重试风暴（retry storm）** —— 大量客户端同时无间隔重试，  
  把本就不稳定的服务打垮的现象。  
- **指数退避（exponential backoff）** —— 每次重试间隔按指数  
  增长，逐步降低对故障服务的压力。  
- **抖动（jitter）** —— 在退避时间里加入随机量，避免大量客户端  
  同一时刻集体重试。  
- **连接池（connection pool）** —— 预先建立并复用的一批连接，  
  避免每次请求都重新握手。  
- **`context.Context`（Go）** —— 携带超时/取消信息，需要被调用  
  代码层层检查才能生效。  
- **`tokio::time::timeout`（Rust）** —— 给任意 future 包一层  
  时限，到点直接丢弃内部 future，无需对方配合。
