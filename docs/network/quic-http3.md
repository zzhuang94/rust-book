# HTTP2 卡在队头阻塞

> 代码：`code/network-quic-http3/`　运行：`cargo run -p network-quic-http3`

前面 [《TCP 该知道的现象》](tcp.md) 讲过 TCP 是“可靠、按顺序”的  
字节流。这一课要讲一个反直觉的事实：**正是“按顺序”这个优点，  
在 HTTP/2 这种一条连接跑多个请求的场景下，反而变成了一个大坑**。  
QUIC 和 HTTP/3 的出现，很大程度上就是为了解决这个坑。本课定位  
是**建立直觉**，不深挖拥塞控制、丢包重传算法这些数学细节。

如果你还没读过 [《TLS 是 HTTP 的外壳》](tls.md)，建议先读一遍——  
本课会用到“握手”这个概念，QUIC 的握手和 TLS 握手关系很密切。

----

# HTTP2 为什么会头阻塞

> HTTP/1.1 每个请求占一条 TCP 连接太浪费，HTTP/2 把多个请求  
> “塞”进同一条 TCP 连接里并发跑，副作用是丢一个包能拖累所有请求。

HTTP/1.1 时代，浏览器为了并发加载多个资源，会同时开好几条  
TCP 连接（常见上限 6 条），每条连接顺序处理请求。HTTP/2 换了  
思路：**一条 TCP 连接里，用“流（stream）”把多个请求/响应  
交错在一起并发传输**，省掉了反复建连的开销。

问题出在 TCP 自己的“按顺序交付”承诺上。TCP 保证的是**整条  
连接**的字节按顺序到达——如果第 3 个 TCP 段在网络中丢了，  
即使第 4、5、6 个段已经先到了，操作系统的 TCP 接收缓冲区也  
**必须攥着它们不交给上层**，直到第 3 段重传成功、缺口补上：

```
TCP 段：[1][2][3 丢了][4][5][6]
应用层能读到的：只有 [1][2]，[4][5][6] 全部被卡住等 [3] 重传
```

放到 HTTP/2 的场景里：哪怕丢的那个包只属于“流 A”这一个请求，  
同一条 TCP 连接上跑的“流 B”“流 C”这些完全无关的请求，也会  
被 TCP 层的重传等待一起拖住——**这就是队头阻塞（head-of-line  
blocking）**：队伍最前面堵住了，后面全部人跟着等，即使他们  
本来可以先走。

对应 Go/Rust 都一样：这个问题发生在**内核 TCP 层**，应用层  
框架（`net/http`、`hyper`）拿到的已经是被卡住的字节流，  
业务代码完全无法绕开，只能等 TCP 自己重传补齐。

> 🔬 底层视角：这里的“队头阻塞”专指**传输层**丢包造成的阻塞，  
> 和 HTTP/1.1 时代“同一条连接内多个请求必须严格顺序处理”的  
> **应用层队头阻塞**是两个层次的问题——HTTP/2 的多路复用刚好  
> 解决了后者，却暴露出前者。

# QUIC 换成 UDP 打底

> 既然问题根源是“TCP 的按顺序交付是整条连接级别的”，QUIC 的  
> 解法很直接：**别用 TCP，自己在 UDP 上重新实现一套“分流的  
> 可靠传输”**。

UDP 本身不保证顺序、不保证到达（[《UDP 与双栈 socket》]  
(udp-sockets.md) 讲过），QUIC 在 UDP 之上自己实现了可靠传输，  
但关键设计是：**可靠性和顺序保证是按“流”分别维护的，不是  
整条连接绑在一起**。

```
QUIC 连接（跑在一个 UDP socket 上）
 ├── 流 A：自己的顺序保证，丢包只影响流 A
 ├── 流 B：自己的顺序保证，丢包只影响流 B
 └── 流 C：自己的顺序保证，丢包只影响流 C
```

丢一个包，QUIC 能识别出“这个包属于流 A”，只让流 A 等待重传，  
流 B、流 C 完全不受影响——这正是对症解决了上一节的队头阻塞。

QUIC 还顺手解决了几个 TCP 的老问题：

- **内置 TLS**：QUIC 把加密握手直接设计进协议本身，不像  
  TCP+TLS 是“先握手建连，再握手加密”两轮往返，QUIC 常见只需  
  一轮往返（甚至 0 轮，如果之前连过）就能开始发数据；
- **连接迁移**：TCP 连接由“源 IP、源端口、目标 IP、目标端口”  
  四元组确定，换网络（WiFi 切 4G）四元组变了连接就断；QUIC  
  用一个独立的连接 ID 标识连接，换网络后连接可以继续用。

对应 Rust 生态，`quinn` 是最常用的 QUIC 实现：

```rust
let endpoint = quinn::Endpoint::client("0.0.0.0:0".parse()?)?;
let conn = endpoint.connect(server_addr, "localhost")?.await?;
let (mut send, mut recv) = conn.open_bi().await?;
```

对应 Go 标准库暂无内置 QUIC，常用第三方库 `quic-go`：

```go
conn, err := quic.DialAddr(ctx, "127.0.0.1:4433", tlsConf, nil)
stream, err := conn.OpenStreamSync(ctx)
```

两边的编程模型都是“先建一个 QUIC 连接，再在连接上开多个流”，  
和 HTTP/2 的“流”概念是同一套心智模型，只是 QUIC 把它做到了  
传输层，彻底摆脱了 TCP 的整连接级排队。

# HTTP3 长什么样

> HTTP/3 = HTTP 语义（方法、header、状态码不变）+ QUIC 传输，  
> 业务视角基本不用改什么。

HTTP/3 相比 HTTP/2，**请求长什么样、状态码什么意思、header  
怎么用完全没变**，唯一的区别是底下换了传输协议：

| | HTTP/1.1 | HTTP/2 | HTTP/3 |
| --- | --- | --- | --- |
| 传输层 | TCP | TCP | QUIC（基于 UDP） |
| 多路复用 | 无（要开多条连接） | 有，但受 TCP 队头阻塞 | 有，流级别独立 |
| 加密 | 可选 TLS | 事实上必须 TLS | 内置在 QUIC 里 |
| 握手往返 | TCP + TLS 两轮 | TCP + TLS 两轮 | 通常一轮 |

对业务代码来说，好消息是：**用 `reqwest` 这类客户端库发请求，  
换成 HTTP/3 通常只是打开一个 feature 开关，业务逻辑完全不变**  
——因为 HTTP 语义没变，变的只是库内部怎么把数据送出去。

```rust
// 概念示意：具体 crate/feature 名称以其官方文档为准
let client = reqwest::Client::builder().http3_prior_knowledge().build()?;
```

**这正是分层设计的又一次体现**（回顾 [《分层》](layers.md)）：  
应用层协议（HTTP 语义）和传输层实现（TCP/QUIC）解耦，换底层  
传输方式，上层业务代码几乎不用跟着改。

# 你什么时候会碰到它

> 不是所有场景都需要关心 QUIC/HTTP3——大部分内部服务之间的  
> 调用，TCP 仍然是又稳又简单的默认选项。

比较容易碰到 QUIC/HTTP3 的场景：

- **公网、面向大量终端用户的服务**：浏览器、手机 App 访问的  
  公网 CDN、网关，用户网络质量参差不齐（弱网、频繁切换网络），  
  QUIC 的连接迁移、抗丢包能力收益明显；
- **视频/直播/实时通信**：这类场景对“队头阻塞”特别敏感，  
  丢一个包不该拖累其他还在正常传输的流；
- 你在浏览器开发者工具的网络面板里，看到协议列写着 `h3`——  
  说明这次请求走的就是 HTTP/3。

不太需要关心的场景：

- **内网服务之间的调用**：延迟低、丢包率低、网络稳定，TCP  
  的队头阻塞在这种环境下基本不构成实际问题；
- 你在写的是一个内部 RPC/微服务，网络环境完全自己可控，  
  引入 QUIC 带来的额外复杂度往往大于收益。

**一句话判断标准：如果你的服务要直接面对不可控的公网终端  
用户，值得关注；如果只是内网服务间调用，TCP 通常够用。**

# 拥塞控制不深挖

> QUIC 内部也有一套类似 TCP 的拥塞控制算法，本课只提一句，  
> 不展开数学细节。

TCP 和 QUIC 都需要解决“发送方一次该发多少数据，才不会把网络  
挤爆、也不会浪费带宽”这个问题，这套机制叫**拥塞控制**（常见  
算法名字如 `Cubic`、`BBR`）。这是一个非常专精的话题，本课刻意  
不深入——业务程序员需要的直觉只有一句：

**拥塞控制是传输层自己的事，应用层代码基本感知不到它的存在，  
只会在网络变差时，透过“吞吐下降”“延迟升高”这类现象间接感受到  
它在起作用。** 如果你的服务对这类细节有强诉求（比如自建 CDN、  
音视频传输优化），那是一个专门的深水区课题，值得单独系统学习，  
不在本课范围内。

# 动手实验（教学程序）

> `quinn` 在部分 Windows 环境下编译依赖较重，本课示例默认走  
> **教学程序**：用一个纯 UDP 的小程序，对比 TCP 的“粘包”故事，  
> 建立“无连接、消息天然带边界”的直觉；如果你的环境能顺利编译  
> `quinn`，示例代码里也带了一个可选的最小 QUIC echo，供你打开。

1. 运行 `cargo run -p network-quic-http3`，观察程序演示的  
   “UDP 多条独立消息”和 [《TCP》](tcp.md) 课里“粘包”现象的对比；
2. 阅读示例代码里大段注释，对照本课讲的“QUIC 按流隔离丢包”  
   的直觉，想一想如果这是 QUIC，丢了一条消息会不会拖累别的消息；
3. 如果你的工具链能顺利编译 `quinn`（在 `Cargo.toml` 里打开  
   `quic-demo` feature），运行一次真正的 QUIC 最小 echo，观察  
   日志里的连接建立、流打开、收发过程；
4. 用浏览器打开开发者工具的网络面板，访问一个支持 HTTP/3 的  
   公开站点，观察协议列是否出现 `h3`。

# 三句话带走

1. HTTP/2 的多路复用解决了应用层队头阻塞，但底下的 TCP 仍是  
   整条连接级别按顺序交付，一丢包就拖累所有并发的流。
2. QUIC 基于 UDP 自己实现按流隔离的可靠传输，丢包只影响那一个  
   流，顺带内置了 TLS 和连接迁移。
3. HTTP/3 只是把 HTTP 语义搭在 QUIC 上，业务代码基本不用改；  
   公网面向终端用户的服务收益明显，内网服务间调用通常用不上。

# 附：本课生词表

- **队头阻塞（head-of-line blocking）** —— 队伍最前面卡住，  
  导致后面本可以先处理的东西也被迫等待的现象。
- **QUIC** —— 基于 UDP 实现的传输协议，按流隔离可靠性与顺序，  
  内置加密，支持连接迁移。
- **HTTP/3** —— 运行在 QUIC 之上的 HTTP，语义与 HTTP/1.1、  
  HTTP/2 相同，只是传输层换成了 QUIC。
- **连接迁移（connection migration）** —— 网络环境变化（换  
  WiFi/4G）后，连接凭独立的连接 ID 继续存活，不像 TCP 靠四元组。
- **拥塞控制（congestion control）** —— 传输层控制发送速率、  
  避免网络拥塞的机制，TCP 和 QUIC 都有各自的实现。
- **`quinn`** —— Rust 生态常用的 QUIC 协议实现 crate。
