# HTTP 是文本协议

> 代码：`code/network-http-protocol/`　运行：
> `cargo run -p network-http-protocol`

前面几课讲的分层、寻址、socket、TCP，都是“怎么把字节可靠地送到对方”。
这一课往上走一层：字节送到之后，**双方约定的格式**是什么。这个格式
就是 HTTP——一份规定“怎么写字符串”的协议，程序员一辈子都在用它，
但很少有人从头把它读一遍。

如果 TCP 的握手、粘包、超时还不熟，建议先读 [《TCP 该知道的现象》](tcp.md)，
本课会假设你已经知道“连接”“字节流”这两个词是什么意思。

本课的边界要提前说清楚：**这里只讲 HTTP 协议本身**——报文长什么样、
状态码什么意思、连接怎么复用。**怎么用 Rust 写一个真正的 HTTP 服务、
怎么用 axum 搭工程化项目，不在这一课**，那是 [《从零手写 HTTP》](../http/http-from-scratch.md)
和 [《axum 入门》](../http/axum.md) 两课的事——本课读完，建议直接接着读它们。

# 先用 curl 看一眼真身

> 别急着看理论，先用工具把 HTTP 报文的真实样子打出来，后面每一节
> 讲的东西都能在这段输出里找到对应位置。

`curl` 有个 `-v`（verbose，详细）参数，会把它发出去的请求和收到的响应，
**逐行原样打印出来**——这正是你想要的“协议原文”：

```bash
curl -v http://localhost:8080/
```

输出大致是这样（`>` 开头是发出去的请求，`<` 开头是收到的响应）：

```
> GET / HTTP/1.1
> Host: localhost:8080
> User-Agent: curl/8.5.0
> Accept: */*
>
< HTTP/1.1 200 OK
< Content-Type: text/plain; charset=utf-8
< Content-Length: 5
<
hello
```

**这就是 HTTP 的全部本质**：客户端往 TCP 连接里写了几行文本（请求），
服务端往同一条连接里写了几行文本（响应）。没有二进制魔法、没有加密
（这是明文 HTTP，加密版见 [《TLS 与 HTTPS》](tls.md)），就是格式固定的字符串。

接下来每一节都会回头指着这段输出讲“这一行到底是什么”。

----

# 请求报文的四个部分

> 一份 HTTP 请求，从上到下永远是这四块：请求行、请求头、一个空行、
> 可选的 body。顺序不能变，空行也不能少。

拿上面 curl 打出来的请求举例，逐块拆开：

```
GET / HTTP/1.1              ← ① 请求行
Host: localhost:8080        ← ② 请求头（可以有很多行）
User-Agent: curl/8.5.0      ←    每行一个"键: 值"
Accept: */*                 ←
                             ← ③ 空行：告诉对方"头写完了"
（这里放 body，GET 通常没有） ← ④ body（可选）
```

逐块解释，尽量说人话：

- **① 请求行**：三个东西用空格隔开——**方法**（`GET`）、**路径**
  （`/`）、**协议版本**（`HTTP/1.1`）。方法就是“我想干什么”，
  常见的有 `GET`（我要拿数据）、`POST`（我要提交数据）、
  `PUT`/`DELETE`（改/删），路径就是“我要哪个资源”；
- **② 请求头（header）**：一堆“键: 值”，每行一条，用来传各种元信息——
  `Host` 是“我要访问的域名”，`User-Agent` 是“我是什么客户端”，
  想传多少条头就写多少行；
- **③ 空行**：这是**最容易被忽略、但最关键**的一行。HTTP 规定
  “连续两个换行（空行）代表头部结束了”，对方靠这一行才知道
  “从这里往下不再是头，是 body 了”——没有这一行，对方根本不知道
  头什么时候写完；
- **④ body（消息体）**：请求真正携带的数据，比如 `POST` 提交的
  表单或 JSON。`GET` 请求语义上是“只读查询”，通常不带 body。

每一行的结尾用的是 `\r\n`（回车+换行两个字符），不是 Unix 常见的
单个 `\n`——这是 HTTP 协议自己规定的行分隔符，[《从零手写 HTTP》](../http/http-from-scratch.md)
手写解析时会具体碰到这个细节，这里先记住这个事实就够。

# 响应报文和请求对称

> 服务端回你的响应，结构和请求几乎一样，只是第一行换了个名字。

```
HTTP/1.1 200 OK                              ← ① 状态行
Content-Type: text/plain; charset=utf-8      ← ② 响应头
Content-Length: 5                            ←
                                              ← ③ 空行
hello                                         ← ④ body
```

对比一下：请求的第一行叫“请求行”（方法+路径+版本），响应的第一行
叫**状态行**（版本+状态码+原因短语），其余三块（头、空行、body）
两边结构完全一样。这不是巧合——HTTP 本来就是“一问一答”，
问和答共用同一套“头+空行+body”的格式，唯一不同的只是第一行
在说“我要什么”还是“结果如何”。

----

# 状态码怎么记

> 不用背每一个状态码，记住“百位数字代表的大类”，剩下的碰到再查。

状态码是三位数字，**百位数字**决定了这条响应属于哪一大类：

| 百位 | 大类 | 一句话直觉 |
| --- | --- | --- |
| 1xx | 信息性 | 业务代码基本碰不到，先跳过 |
| 2xx | 成功 | 你要的事情办成了 |
| 3xx | 重定向 | 去别的地方找你要的东西 |
| 4xx | 客户端错误 | **你**（发请求的这一方）传错了 |
| 5xx | 服务端错误 | **对方**（服务器）自己出问题了 |

日常业务里高频遇到的具体状态码：

| 状态码 | 含义 | 常见场景 |
| --- | --- | --- |
| `200 OK` | 成功 | 请求正常处理完，body 里是结果 |
| `201 Created` | 创建成功 | `POST` 成功新建了一个资源 |
| `204 No Content` | 成功但没内容 | 处理成功，但没有东西可返回 |
| `301`/`302` | 永久/临时重定向 | 让浏览器换个地址再请求一次 |
| `400 Bad Request` | 请求格式不对 | 参数缺失、JSON 格式错误 |
| `401 Unauthorized` | 没登录/没鉴权 | 缺少或错误的身份凭证 |
| `403 Forbidden` | 有身份但没权限 | 知道你是谁，但不让你做这件事 |
| `404 Not Found` | 资源不存在 | 路径写错，或数据确实没有 |
| `429 Too Many Requests` | 请求太频繁 | 触发了限流 |
| `500 Internal Server Error` | 服务端内部错误 | 服务端代码抛了异常/panic |
| `502`/`503`/`504` | 网关/服务不可用/网关超时 | 上游服务挂了、过载或响应太慢 |

一个非常实用的排查直觉：**看到 4xx，先怀疑自己发的请求；
看到 5xx，先怀疑对方服务出了问题**——这条经验能帮你在排查线上
问题时，第一时间把“方向”定对，不至于对着一个 401 去翻服务端日志。

> 🔬 底层视角：状态码本身只是响应第一行里的一个数字字符串，
> HTTP 服务器（或框架）负责把它和一段固定的“原因短语”（比如
> `200` 配 `OK`）拼在一起写出去，但**客户端真正依赖判断的是数字，
> 原因短语只是给人看的注释**，浏览器和大多数客户端库根本不解析它。

# Content-Length 划出 body 边界

> 回顾 [《TCP》](tcp.md) 讲过的核心事实：TCP 只是字节流，没有消息边界。
> HTTP 靠 `Content-Length` 这个头，自己在字节流里划出“body 到哪结束”。

`Content-Length` 的值是 **body 的字节数**（不是字符数——一个汉字
UTF-8 编码通常占 3 个字节）。对方读 HTTP 报文时的逻辑是：

1. 一直读到空行，确认头部读完；
2. 从头里找到 `Content-Length` 的值，假设是 `100`；
3. 接着往后再读满 `100` 个字节，这些就是完整的 body，不多不少。

这正是 [《TCP》](tcp.md) “字节流没有消息边界”一节里说的
“长度前缀”方案在 HTTP 里的具体应用——`Content-Length`
就是这份协议自己实现的长度前缀，只不过它写在头部而不是二进制
前几个字节里。如果服务端算错了这个数字（比如少算了几个字节），
对方会一直卡在“还没读够”上等待，直到超时——这是一个真实存在、
调试起来很折磨人的 bug 来源。

body 大小不确定、需要一边生成一边发送的场景（比如流式响应），
HTTP 还提供了另一种方案叫**分块传输编码**（`Transfer-Encoding: chunked`），
用一系列“先写长度、再写这一块内容”的小段落代替一次性写总长度——
本课不展开细节，只需要知道它是 `Content-Length` 之外的另一条路，
两者不会同时出现在同一个响应里。

# Keep-Alive 连接复用

> 每次请求都重新三次握手一条新连接，代价不小；HTTP/1.1 默认让同一条
> TCP 连接被多次请求复用。

早期 HTTP/1.0 的默认行为是“一个请求用一条连接，用完就关”——每次
请求都要重新走一遍 [《TCP》](tcp.md) 讲过的三次握手，代价不小。
HTTP/1.1 把这个默认行为改成了 **Keep-Alive**（连接保活）：
一条 TCP 连接完成一次请求响应后，**默认不关闭**，可以接着发下一个
请求，省掉重复握手的开销。

curl 的输出里能看到这个头（有些服务器会显式写出来，有些默认开启
但不显式写）：

```
< Connection: keep-alive
```

如果服务端明确不想复用这条连接（比如响应完就要关闭），会写：

```
< Connection: close
```

回顾 [《从零手写 HTTP》](../http/http-from-scratch.md) 里最朴素的手写版本，
它每次响应完都写了 `Connection: close`——这不是疏忽，而是故意
避开 Keep-Alive 的复杂度，先把协议本身讲清楚。真正的 HTTP 服务器
（比如 hyper）会正确处理 Keep-Alive：同一条连接上，服务端读完一个
完整请求（靠 `Content-Length` 或空 body 判断边界）就处理并回应，
然后**不关闭连接**，回到“继续读下一个请求”的状态，循环下去，
直到某一方主动关闭或者空闲太久被超时踢掉。

> 🔬 底层视角：Keep-Alive 复用的是同一条 TCP 连接，但请求和响应
> 在这条连接里仍然是“一来一回、严格排队”的——上一个请求没处理完，
> 下一个请求的响应不能提前到达。这个限制叫**队头阻塞**（head-of-line
> blocking），是 HTTP/1.1 一直没解决好的问题，下一节的 HTTP/2
> 正是为了解决它。

----

# HTTP/1.1 与 HTTP/2 的直觉差别

> 不需要懂 HTTP/2 的二进制帧格式，只需要记住它解决了什么问题：
> 一句话——**多路复用**。

HTTP/1.1 的 Keep-Alive 只解决了“连接可以复用”，没解决“一条连接
上的多个请求必须排队”——上一节说的队头阻塞。**HTTP/2 的核心改进
就是让多个请求/响应可以在同一条连接上交错并发传输**，不用排队等
上一个完全处理完。这就是**多路复用（multiplexing）**：多条“逻辑
上独立的对话”共享同一条物理连接，谁的数据先准备好就先发一部分，
不用死等前面的人说完。

日常业务视角下，HTTP/2 带来的直观好处是：浏览器不再需要为了绕开
队头阻塞而对同一个域名开好几条并发连接（老式浏览器优化手段），
一条连接就能同时跑很多个请求，减少了握手和端口消耗。

至于协议版本具体怎么协商（比如 TLS 握手时的 ALPN）、二进制帧格式
细节，属于更底层的实现话题，本课不展开——业务程序员只需要知道
“HTTP/2 不是 HTTP/1.1 的简单加密版，它连传输方式本身都变了”，
以及“这个升级通常是框架/服务器帮你做的，业务代码基本不需要感知”。

# Header 名字不区分大小写

> 新手容易忽略的一个细节：`Content-Type`、`content-type`、
> `CONTENT-TYPE` 在 HTTP 里是**同一个** header，大小写不影响
> 匹配。

HTTP 规范规定 header 名字大小写不敏感，这意味着服务端读某个
header 时，不应该用完全匹配大小写的字符串比较去查找：

```rust
// 不严谨的写法：只能匹配到完全一样大小写的写法
if headers.get("Content-Type").is_some() { /* ... */ }

// 更稳妥：先统一转成小写再比较，或者用专门处理这件事的库类型
```

成熟的 HTTP 库（`hyper`/axum 的 `HeaderMap`，Go 的
`http.Header`）都已经在内部处理好了大小写不敏感的查找，
业务代码通常不需要自己写这段逻辑，但**理解这个规则**，能帮你
看懂为什么不同客户端发出来的请求，header 名字大小写风格
可能完全不同，却都能被服务端正确识别——这不是巧合，是协议
本身的约定。

另外一个容易被忽略的点：**同一个 header 名字可以出现多次**，
比如 `Set-Cookie` 经常在一个响应里出现好几行，各自设置不同的
cookie；服务端/客户端库通常会把这类可重复的 header 收集成
一个列表，而不是像普通 header 那样只保留“最后一个”或“第一个”，
这一点在自己解析原始报文时容易漏掉。

# 常见误区

> 这一节收集从“看得懂协议”到“真的写对代码”之间最容易踩的坑。

- **误区一：`GET` 请求不能带 body，协议层面禁止。** HTTP 规范
  没有明确禁止 `GET` 带 body，但**语义上强烈不建议**——很多
  中间代理、缓存、库默认会忽略或丢弃 `GET` 请求的 body，业务
  依赖“`GET` 带 body”这个行为很容易在换了某个中间组件之后
  悄悄失效，正确做法是需要传参数就用查询字符串，需要传大量
  结构化数据就换 `POST`；
- **误区二：状态码 `200` 就代表业务一定成功了。** 状态码只是
  HTTP 协议层面“这次请求—响应有没有正常完成”的判断，很多
  系统会在 `200 OK` 的响应体里再放一层业务自己的错误码
  （比如 `{"code": 40001, "msg": "余额不足"}`），协议层成功
  不等于业务层成功，两层错误码不能混为一谈；
- **误区三：`Content-Length` 是可选的，写不写都行。** 除了用
  分块传输编码（`Transfer-Encoding: chunked`）替代的场景，
  普通响应缺少 `Content-Length` 会导致对方不知道 body 到哪结束，
  只能靠连接关闭来判断（这也是 HTTP/1.0 时代常见的做法），
  在 Keep-Alive 连接上这样做基本行不通；
- **误区四：Header 和 URL 查询参数是完全一样的东西，随便选一个
  传参数就行。** 两者生命周期和可见性不同——URL 会被完整记录在
  各种访问日志、浏览器历史、代理日志里，敏感信息（如密码、
  token）不应该放进 URL 查询参数，更适合放进 header 或 body。

# 排错对照

> HTTP 层面的问题，多数能直接从 `curl -v` 的输出里读出线索，
> 按下面的表格把现象对上大概率原因。

| 现象 | 大概率原因 | 排查方向 |
| --- | --- | --- |
| 响应卡住不完整，最后超时或连接被重置 | `Content-Length` 算错了（比实际 body 更大），对方一直等没等到的字节 | 检查服务端计算 body 长度的逻辑，确认字节数而不是字符数 |
| 同样的请求，有些客户端正常、有些报 400 | 请求 header 大小写或格式被某个客户端库处理成了服务端没预期的形式 | 用 `curl -v` 对比不同客户端实际发出的原始请求内容 |
| 一直以为长连接生效了，但每次请求都重新握手 | 服务端或中间代理返回了 `Connection: close`，没有开启 Keep-Alive | 检查响应头里的 `Connection` 字段实际取值 |
| 浏览器网络面板显示走的是 `h2`，自己的客户端库却报协议错误 | 客户端库不支持 HTTP/2，或者服务端强制要求 HTTP/2（部分场景下 TLS 之上会协商到 HTTP/2） | 检查客户端库的 HTTP 版本支持情况，必要时显式指定使用 HTTP/1.1 |
| 请求体是 JSON，但服务端解析失败 | `Content-Type` 没写对（比如写成了 `text/plain`），或者字符编码不一致 | 检查请求头 `Content-Type` 是否为 `application/json`，确认编码是 UTF-8 |

# 动手实验

1. 找一个本机能访问的服务（比如后续课程里跑起来的任意一个 axum
   示例），用 `curl -v` 打一次请求，把打印出来的每一行对照本课
   四个部分（请求行/头/空行/body），确认自己能认全；
2. 用 `curl -v -X POST -d "name=abc"` 发一个带 body 的 `POST`
   请求，观察 curl 自动帮你加上了哪个头（提示：和 body 长度有关）；
3. 找一个会返回 404 的路径请求一次，观察状态行里的数字和后面
   跟着的原因短语；
4. 连续用同一个 `curl` 命令重复请求同一个服务两次，观察响应头里
   `Connection` 的值，判断这个服务端有没有开启 Keep-Alive；
5. 如果装了 Wireshark 或 tcpdump（详见 [《网络排障工具箱》](debug-tools.md)），
   抓一次 HTTP 请求的包，肉眼确认协议原文和 curl -v 打出来的一致。

# 三句话带走

1. HTTP 就是跑在 TCP 字节流上的纯文本协议：请求是“请求行+头+空行
   +body”，响应是“状态行+头+空行+body”，结构完全对称。
2. 状态码记百位数字的大类就够（2xx 成功、4xx 是你的问题、5xx 是
   对方的问题）；`Content-Length` 是靠字节数在字节流里划出 body
   边界的“协议自带长度前缀”。
3. Keep-Alive 让多个请求复用同一条连接，但 HTTP/1.1 里仍然要排队；
   HTTP/2 用多路复用解决了这个排队问题，这些通常由框架代劳，
   业务代码基本不用手动处理。

# 附：本课生词表

- **请求行** —— HTTP 请求的第一行，格式是“方法 路径 协议版本”。
- **状态行** —— HTTP 响应的第一行，格式是“协议版本 状态码 原因短语”。
- **请求头 / 响应头（header）** —— 一系列“键: 值”组成的元信息，
  每行一条。
- **空行** —— 头部和 body 之间的分隔行，标志“头部到这里结束”。
- **body（消息体）** —— 请求或响应真正携带的数据内容，可选。
- **状态码** —— 三位数字，百位代表大类（2xx 成功、4xx/5xx 出错等）。
- **`Content-Length`** —— 响应头之一，记录 body 的字节数，供对方
  判断读到哪里算完整。
- **`Transfer-Encoding: chunked`** —— 分块传输编码，body 大小不
  确定时代替 `Content-Length` 的另一种边界划分方式。
- **Keep-Alive（连接保活）** —— HTTP/1.1 默认让同一条 TCP 连接
  被多次请求复用，省去重复三次握手。
- **队头阻塞（head-of-line blocking）** —— 同一条连接上，前一个
  请求没处理完，后一个请求的响应也发不出来的现象。
- **多路复用（multiplexing）** —— HTTP/2 的核心改进：多个请求/
  响应交错共享同一条连接，不用严格排队。
- **Header 大小写不敏感** —— HTTP header 名字不区分大小写，
  `Content-Type` 与 `content-type` 指的是同一个字段。
- **业务错误码** —— 应用自己在响应体里定义的错误标识，独立于
  HTTP 协议本身的状态码，两者是两层不同的判断。
