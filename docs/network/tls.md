# TLS 是 HTTP 的外壳

> 代码：`code/network-tls/`　运行：`cargo run -p network-tls`

上一课 [《HTTP 是文本协议》](http-protocol.md) 里，curl 打出来的每一行
请求响应都是**明文**——谁能看到这条 TCP 连接上的数据，就能看到
完整的请求内容，包括路径、header，甚至密码字段。这一课讲的 TLS，
就是专门用来堵住这个漏洞的一层。

本课定位是“建立直觉，看懂现象”，**不会展开椭圆曲线、RSA 之类的
密码学数学细节**——那些是专门的密码学课题，业务程序员真正需要的，
是知道“握手在干什么、证书在证明什么、自签证书为什么会报错”这几件
和日常调试直接相关的事。

----

# 为什么必须多这一层

> 明文 HTTP 在公共网络上传输，路径上的任何一个环节都能看到、
> 甚至篡改你的数据。

一条 HTTP 请求从你的电脑出发，中间可能经过家里的路由器、
运营商的网络设备、若干个中转路由器，才能到达目标服务器——回顾
[《分层》](layers.md) 讲过的“封装与解封装”，**每一跳路由器都要拆开
网络层的包头才能决定往哪转发，如果传输层以上的数据是明文，
这些设备（或者架在它们旁边窃听的人）也能顺手看到里面的内容**。

明文传输至少暴露三类风险：

- **窃听（eavesdropping）**：任何能看到这条连接流量的人，
  都能读到请求里的账号、密码、业务数据；
- **篡改（tampering）**：中间人可以悄悄修改传输中的内容，
  而两端完全无法察觉——比如把下载的文件偷偷换成带毒的版本；
- **伪装（impersonation）**：你以为在跟真正的服务器说话，
  实际上在跟一个冒充者说话，它转手把你的数据原样（或篡改后）
  转发给真服务器，你却毫无察觉——这就是常说的“中间人攻击”。

**TLS（Transport Layer Security，传输层安全）** 就是在 TCP 之上、
应用层协议之下加的一层，同时解决这三个问题：加密（防窃听）、
完整性校验（防篡改）、身份认证（防伪装）。它并不是 HTTP 专属的——
任何跑在 TCP 上的协议都可以套一层 TLS，只不过“HTTP 套了 TLS”
这个组合特别常见，专门有了个名字。

----

# HTTPS 就是 HTTP 套了一层壳

> 记住这一个等式，剩下的都是细节：**HTTPS = HTTP + TLS**。

HTTPS 不是另一套报文格式，请求行、头、空行、body 那一整套
[《HTTP 是文本协议》](http-protocol.md) 讲的东西，**完全原样保留**。
唯一的区别是：这些文本内容不再直接写进 TCP 连接，而是先经过
TLS 这一层加密，加密后的密文才被写进 TCP 连接；对方收到密文后，
先用 TLS 解密还原出原始的 HTTP 文本，再按上一课的方式解析。

用分层的视角描述这个关系：

```
应用层：HTTP 报文（请求行/头/空行/body，和明文时完全一样）
──────────────────────────────────────────────
新增一层：TLS（负责加密、校验、认证这段数据）
──────────────────────────────────────────────
传输层：TCP（负责可靠字节流，和之前完全一样）
```

也正因为这样，[《TCP 该知道的现象》](tcp.md) 讲的三次握手、
粘包、超时这些现象，**在 HTTPS 场景下全部照常发生**——TLS 只是
在 TCP 连接建立之后、应用数据发送之前，多插入了一次自己的
“握手”，用来协商加密参数。下一节具体讲这次额外的握手在做什么。

URL 上的区别只是外表：`http://` 换成 `https://`，端口默认从
`80` 换成 `443`，这两个只是约定俗成的默认值，本质区别在于连接
建立之后走不走 TLS 这层。

# 证书与 CA 的直觉

> TLS 要解决“身份认证”，靠的是证书；证书为什么能被信任，
> 靠的是一条“谁给谁签名担保”的信任链。

假设你访问 `example.com`，你怎么确认对方真的是 `example.com`，
而不是冒充者？TLS 的答案是：服务器出示一份**证书**，里面写着
“我是 `example.com`，这是我的公钥”，并且这份证书**由一个大家都
信任的第三方签了名**——这个第三方叫 **CA（证书颁发机构，
Certificate Authority）**。

信任链大致是这样传递的：

```
根 CA（预装在操作系统/浏览器里，天生被信任）
  │ 签发证书给
  ▼
中间 CA
  │ 签发证书给
  ▼
example.com 的证书（服务器实际出示的这一份）
```

你的操作系统或浏览器里预置了一批“天生信任”的根 CA 公钥。当服务器
出示证书时，客户端会顺着签名链一层层往上验证：这份证书是不是
由某个中间 CA 签的？那个中间 CA 又是不是由某个根 CA 签的？
一路验证到根 CA，且根 CA 在自己的信任列表里——**验证通过，
才认为“这确实是 example.com，不是冒充者”**。

这个信任链模型解释了几个常见现象：

- 为什么证书“过期了”会报错——证书上写了有效期，过期后即使签名
  链本身没问题，客户端也会拒绝信任；
- 为什么换了新证书需要“重新签发”而不是自己随便改改——证书内容
  一旦被篡改，签名就验证不通过，这是密码学签名的性质保证的；
- 为什么“自己签给自己用”的证书浏览器会报警——下一节详细展开。

> 🔬 底层视角：证书本质上是一份包含“域名、公钥、有效期、签发者”
> 等字段的结构化数据，外加签发者用自己的私钥对这些字段计算出的
> 一个签名。验证证书就是用签发者的公钥去验证这个签名是否吻合——
> 只要签发者的私钥没有泄露，攻击者伪造不出能通过验证的签名。

# 握手在程序员眼里发生了什么

> 不需要记密码学算法名字，只需要记住握手最终产出了什么、
> 解决了什么问题。

TLS 握手发生在 TCP 三次握手**之后**、应用数据发送**之前**，
大致做了这几件事（按发生顺序，细节因协议版本略有差异）：

```
客户端                                    服务端
  │──── 1. "我能支持这些加密方式" ─────────▶│
  │                                        │
  │◀──── 2. "选定一种方式 + 我的证书" ───────│
  │                                        │
  │   3. 验证证书（顺着签名链验到信任的 CA）  │
  │                                        │
  │──── 4. 协商生成一份双方都知道、          │
  │        但外人不知道的"会话密钥" ────────▶│
  │                                        │
  │◀════ 5. 后续所有数据用会话密钥加密 ════▶│
```

程序员真正需要建立的直觉：

1. **第 2 步的证书验证，对应上一节的信任链**——验证失败（证书
   过期、域名不匹配、签发者不受信任），握手在这里就会失败，
   连接根本建立不起来，应用层代码完全看不到任何 HTTP 报文；
2. **第 4 步协商出的是一份对称密钥**，只在这一次连接里有效——
   握手本身用的是相对昂贵的非对称加密只为了安全地交换这份密钥，
   之后大量的应用数据加密改用便宜得多的对称加密，这是一个性能
   上的权衡设计；
3. **握手是有额外网络往返（round trip）成本的**——这意味着建立
   一条 HTTPS 连接比建立一条明文 HTTP 连接慢，这也是为什么
   [《HTTP 是文本协议》](http-protocol.md) 讲的 Keep-Alive
   连接复用在 HTTPS 场景下价值更高：握手的成本只需要付一次，
   之后的请求都能蹭这条已经握好手的连接。

对应到 Rust 生态，握手这一层完全由 TLS 库替你完成，你的代码
基本感觉不到它：

```rust
// reqwest 内部会自动完成 TLS 握手，业务代码不用手写任何握手逻辑
let body = reqwest::get("https://localhost:8443/").await?.text().await?;
```

对应 Go：

```go
// net/http 的 http.Get 对 https:// 的处理是完全一致的，
// crypto/tls 在标准库内部自动接管握手
resp, err := http.Get("https://localhost:8443/")
```

两边的共同点是：**只要 URL 是 `https://`，握手这一整套流程都是
标准库/框架自动做的**，业务代码唯一可能手动碰到的，是握手失败
之后暴露出来的错误信息。

# 自签证书的坑

> 开发环境最常踩的坑，几乎全都和“证书不是正规 CA 签的”有关。

正规 CA 签发证书通常要走域名验证流程，开发环境很多时候没有
真实域名（比如只是本机 `localhost` 或内网测试环境），这时候
常见做法是**自己签一份证书给自己用**，叫**自签证书**（self-signed
certificate）——用自己的私钥给自己签名，没有任何 CA 参与。

自签证书能正常完成 TLS 握手的加密和完整性保护，**但没法通过
信任链验证**——因为它的“签发者”就是它自己，不在任何操作系统
预置的信任列表里。客户端（浏览器、`curl`、`reqwest`）默认都会
拒绝这样的证书，常见报错关键字类似：

```
certificate verify failed
self-signed certificate
NET::ERR_CERT_AUTHORITY_INVALID
```

开发时常见的几种应对方式，按推荐程度排列：

1. **把自签证书的公钥手动加入本机信任列表**（或者用工具生成一份
   受本机信任的开发证书）——最安全，代价是每台开发机要配置一次；
2. **在客户端代码里显式关闭证书校验**，仅限本机开发环境使用，
   例如 `curl -k`（`--insecure`）或者给 HTTP 客户端库传一个
   “跳过校验”的选项——**这个选项绝对不能带进生产配置**，一旦
   生产环境也关闭校验，中间人攻击的防线就形同虚设；
3. **本地开发直接用明文 HTTP**，把 TLS 相关的问题完全挪到部署
   阶段用正规证书解决——很多团队的默认做法，简单但要记得部署前
   补上真正的证书配置。

> 🔬 底层视角：curl 遇到自签证书报错时，报错信息里通常会直接写出
> 是在“证书链”哪一步验证失败——多花一分钟读完整段报错文字，
> 往往比直接加 `-k` 跳过更有价值，能帮你确认这到底是“证书确实
> 没配对”还是“系统时间不对导致有效期判断出错”这类更隐蔽的问题。

**不要把真实的证书私钥或证书内容贴到聊天记录、issue、日志里**——
证书本身（公钥部分）泄露问题不大，但配套的**私钥**一旦泄露，
任何人都能伪造该证书对应服务器的身份，风险等同于泄露了密码。

# Rust 与 Go 的 TLS 生态对照

> 两边的选择都不止一个库，这里只建立“大概有哪几种”的概念，
> 具体项目该用哪个，看项目本身的依赖场景。

Go 的 TLS 支持内建在标准库 `crypto/tls` 里，`net/http` 默认
就用它，几乎不需要额外选型：

```go
// http.Get 对 https:// URL 自动走 crypto/tls，业务代码无感
resp, err := http.Get("https://localhost:8443/")
```

Rust 生态里 TLS 不是标准库自带的，常见有两个方向：

| 方向 | 代表库 | 一句话特点 |
| --- | --- | --- |
| 纯 Rust 实现 | `rustls` | 不依赖系统 OpenSSL，编译部署更省心 |
| 绑定系统库 | `native-tls` | 底层调用系统自带的 TLS 实现（如 OpenSSL） |

`reqwest` 这类 HTTP 客户端库，会把这两种实现都做成**可选特性
（feature）**，由使用者在 `Cargo.toml` 里选择：

```toml
# 选纯 Rust 的 rustls 实现
reqwest = { version = "0.12", features = ["rustls-tls"] }
# 或者选绑定系统 TLS 库的实现
reqwest = { version = "0.12", features = ["native-tls"] }
```

[《reqwest 与上游容错》](../http/reqwest-resilience.md) 那一课的
示例只请求本机明文 `http://`，所以特意关闭了默认特性、没有引入
任何 TLS 实现——这提醒了一个容易被忽略的事实：**引入 TLS 库
本身是要花编译时间和二进制体积成本的，只在真的需要 `https://`
时才开启对应特性**。

一张对照小结：

| 关注点 | Go | Rust |
| --- | --- | --- |
| TLS 实现来源 | 标准库内建 `crypto/tls` | 第三方库（`rustls`/`native-tls`），要显式选 |
| `net/http` 默认行为 | 自动支持 `https://` | `reqwest` 要开对应 feature 才支持 |
| 跳过证书校验 | `tls.Config{InsecureSkipVerify: true}` | 客户端库对应的 `danger_accept_invalid_certs` 之类选项 |
| 自定义信任的 CA | `tls.Config.RootCAs` | 客户端 builder 上的 `add_root_certificate` 之类方法 |

两边的共同底线是同一句话：**“跳过证书校验”的选项只应该出现在
开发/测试代码里，绝不能出现在生产环境的默认配置中**。

# ALPN 协商用哪个应用层协议

> TLS 握手不只是协商加密参数，还顺带解决了一个问题：
> **同一个 443 端口，客户端和服务端怎么确定接下来用 HTTP/1.1
> 还是 HTTP/2 说话？**答案是 ALPN。

**ALPN（Application-Layer Protocol Negotiation，应用层协议
协商）**是 TLS 握手过程中的一个扩展字段：客户端在
`ClientHello` 里列出自己支持的应用层协议（比如
`["h2", "http/1.1"]`），服务端从列表里选一个自己也支持的，
写进握手响应里确认下来。握手完成后，双方从第一个字节开始就
按协商好的协议交换数据，不需要再额外猜测或试探。

这解决的问题是：**在 TLS 加密开始之前，应用层协议还没确定**，
但 HTTP/1.1 和 HTTP/2 的报文格式完全不同（HTTP/2 是二进制帧），
如果没有 ALPN，双方只能先按某个协议“猜着发”，猜错了再重新
握手，代价很高。ALPN 把这个决策提前到了 TLS 握手阶段一次性
搞定。

对应 Rust（`reqwest`/`hyper` 内部依赖 TLS 库自动处理 ALPN，
业务代码通常不需要手动指定）：

```rust
// reqwest 默认会在支持的情况下自动通过 ALPN 协商到 HTTP/2
let client = reqwest::Client::builder().build()?;
```

对应 Go：

```go
// http.Transport 同样依赖 crypto/tls 的 ALPN 协商，业务无感
resp, err := http.Get("https://example.com/")
```

这也解释了 [《HTTP2 卡在队头阻塞》](quic-http3.md) 提到的
一个细节：同一个服务完全可能因为不同客户端支持的协议列表不同，
被 ALPN 协商到不同的 HTTP 版本——这不是 bug，是协议设计上
故意留出的灵活性。

# 常见误区

> TLS/HTTPS 相关最容易被搞混的几个说法，排障前先对照检查。

- **误区一：HTTPS 只加密了 body，header 和 URL 路径是明文的。**
  实际上除了上一节讲过的 SNI 字段（在握手阶段就是明文），
  TLS 加密覆盖的是**握手完成之后的全部应用层数据**，包括请求
  行、所有 header 和 body——中间人能看到你访问的域名（靠 SNI），
  但看不到具体访问的路径和内容；
- **误区二：证书快过期了，只要服务还在跑就不用管。**
  证书过期后客户端会直接拒绝信任、握手失败，服务进程本身可能
  完全正常运行，但所有 HTTPS 请求会突然全部失败——这是一种
  “服务看起来活着，但完全不可用”的典型故障，证书有效期需要
  提前监控和续期，不能等到过期当天才发现；
- **误区三：只要用了 TLS，就不需要再关心应用层的身份认证。**
  TLS 解决的是“你确实在和你以为在对话的服务器说话”，不解决
  “这个用户是谁、有没有权限做这件事”——身份认证（登录、token、
  session）是应用层自己的职责，TLS 提供的是更底层的传输安全；
- **误区四：内网服务之间不需要 TLS，反正外面进不来。**
  内网并不天然等同于“绝对可信”——同一个内网里的其他服务、
  被入侵的机器，都可能具备窃听内网流量的能力，很多安全要求
  较高的场景（金融、医疗）即使在内网也要求服务间通信走 TLS
  （常见做法是 mTLS，双向验证身份），是否需要取决于你的安全
  合规要求，不能一概而论。

# 排错对照

> HTTPS 连接失败时，`curl -v` 的报错信息通常已经指出了具体
> 卡在哪一步，按下面的表格定位问题。

| 现象 | 大概率原因 | 排查方向 |
| --- | --- | --- |
| `certificate has expired` | 证书已过期 | 检查证书有效期，重新签发/续期 |
| `self-signed certificate` / `unable to get local issuer certificate` | 自签证书，或缺少中间 CA 证书，客户端信任链验证失败 | 检查证书链是否完整，是否需要手动信任或换正规签发的证书 |
| `certificate verify failed: hostname mismatch` | 证书上的域名和实际访问的域名不一致 | 检查证书的 CN/SAN 字段是否覆盖了当前访问的域名 |
| TCP 连接建立成功，之后长时间无响应，最终超时 | 握手过程中一方卡住（较少见，可能是加密套件不匹配） | 用 `curl -v` 观察握手具体卡在哪个阶段，检查双方支持的 TLS 版本/加密套件是否有交集 |
| 生产环境正常，本机调试环境报证书错误 | 调试环境用了自签证书或走了不同的证书配置 | 回顾“自签证书的坑”一节，确认调试环境的证书处理方式 |
| 同一个域名，有些客户端连 HTTP/2 有些连 HTTP/1.1 | ALPN 根据各自客户端支持的协议列表协商出了不同结果 | 这是正常现象，不是故障，确认业务逻辑没有假设“一定是某个版本” |

# 动手实验

1. 用 `curl -v https://` 加上一个你确定支持 HTTPS 的地址（比如
   `example.com`），观察 `-v` 输出里除了 HTTP 头之外，还多打印
   出的握手相关信息（证书主题、有效期等）；
2. 对比同一台服务分别用 `http://` 和 `https://` 访问（如果服务
   同时支持两者），感受第一次建立连接时 HTTPS 明显更慢的现象，
   体会“握手多一轮网络往返”的直觉；
3. 用 `curl -k` 访问一个自签证书的地址，对比不加 `-k` 时的报错，
   读懂报错信息里具体指出的验证失败原因；
4. 如果本地有 Wireshark（见 [《网络排障工具箱》](debug-tools.md)），
   抓一次 HTTPS 连接的包，确认握手阶段的报文是明文可读的（比如
   能看到域名），但握手完成后的应用数据是完全看不懂的密文——
   这正是“身份协商阶段部分信息是明文，业务数据全程加密”的证据。

----

# 三句话带走

1. HTTPS 就是 HTTP 套了一层 TLS：报文格式完全不变，只是数据
   在写入 TCP 之前先加密、读出之后先解密，同时解决窃听、篡改、
   伪装三类风险。
2. 证书的身份认证靠一条“谁给谁签名担保”的信任链，最终追溯到
   操作系统/浏览器预置的根 CA；自签证书没有这条链，客户端默认
   会拒绝信任，报错信息通常会指出具体在哪一步验证失败。
3. Rust 需要显式选择 `rustls` 或 `native-tls` 之类的 TLS 实现，
   Go 的 `crypto/tls` 是标准库内建；不管哪种语言，跳过证书校验
   的选项都只能留在开发环境，绝不能带进生产配置。

# 附：本课生词表

- **TLS（传输层安全）** —— 在 TCP 之上、应用层之下加的一层，
  提供加密、完整性校验、身份认证。
- **HTTPS** —— HTTP 套了一层 TLS 后的组合名字，报文格式不变。
- **证书（certificate）** —— 一份声明“我是谁、这是我的公钥”并
  被签名担保的数据。
- **CA（证书颁发机构）** —— 负责给证书签名担保的受信任第三方。
- **信任链** —— 从服务器证书一路验证签发者，直到某个操作系统
  预置的根 CA 为止的验证过程。
- **自签证书** —— 自己给自己签名的证书，没有 CA 参与，默认不被
  客户端信任。
- **中间人攻击** —— 攻击者冒充服务器与客户端通信，两端都以为在
  跟对方直接对话。
- **会话密钥** —— TLS 握手协商出的、仅本次连接使用的对称加密
  密钥，用于加密之后的应用数据。
- **`rustls` / `native-tls`** —— Rust 生态两种常见 TLS 实现方向：
  纯 Rust 实现，或绑定系统自带的 TLS 库。
- **`crypto/tls`** —— Go 标准库内建的 TLS 实现，`net/http` 默认
  使用它支持 `https://`。
- **ALPN** —— TLS 握手阶段协商接下来使用哪个应用层协议（如
  HTTP/1.1 还是 HTTP/2）的扩展机制。
- **mTLS（双向 TLS）** —— 客户端和服务端互相验证对方证书的
  TLS 模式，常用于内网服务间的高安全要求场景。
