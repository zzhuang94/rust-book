# 地址决定连去哪

> 代码：`code/network-addressing/`　运行：`cargo run -p network-addressing`

上一课 [《分层》](layers.md) 说网络层负责“把包送到对方机器”，
但它怎么知道“对方机器”是谁？靠的就是这一课的主角：**IP 地址和端口**。
这一课把 IPv4/IPv6、端口、几个长得像却完全不同的地址（`0.0.0.0`、
`127.0.0.1`、局域网 IP）、DNS 解析，一次性讲透，顺便列出新手最常踩的坑。

----

# IPv4 长什么样

> 四段十进制数字，每段 0~255，这是你天天见到但可能没细想过的格式。

`192.168.1.10` 这种写法叫 **点分十进制**，本质是一个 32 位（4 字节）的整数，
拆成 4 段、每段一个字节（0~255）方便人读。32 位意味着 IPv4 理论上限是
2 的 32 次方，约 43 亿个地址——这个数字在互联网发展初期看起来够用，
现在早就不够分了，这正是 IPv6 存在的原因。

Rust 里对应的类型：

```rust
use std::net::Ipv4Addr;

let addr: Ipv4Addr = "192.168.1.10".parse()?;
// 或者直接用四个字节构造，和点分十进制一一对应
let addr2 = Ipv4Addr::new(192, 168, 1, 10);
```

对应 Go：

```go
addr := net.ParseIP("192.168.1.10")
// Go 的 net.IP 底层是 []byte，同样是四个字节（IPv4 场景）
```

两边都把“看起来像字符串”的地址解析成了固定长度的字节序列——
这一点在后面判断“两个地址是否相等”时很关键：**应该比较字节，
不要比较原始字符串**（`"192.168.001.010"` 和 `"192.168.1.10"`
字符串不同，但字节相同）。

----

# IPv6 为什么长这样

> 8 组十六进制数字，用冒号分隔，还允许用 `::` 省略连续的零段。

`2001:db8::1` 是典型的 IPv6 地址，完整展开是
`2001:0db8:0000:0000:0000:0000:0000:0001`——8 组，每组 16 位，
一共 128 位。128 位地址空间大到几乎不用担心用尽（约 3.4×10^38 个）。

三条读 IPv6 地址的规则：

- `::` 只能在一个地址里出现一次，代表“中间省略的连续零组”；
- 回环地址（相当于 IPv4 的 `127.0.0.1`）是 `::1`；
- 任意地址（相当于 IPv4 的 `0.0.0.0`）是 `::`。

Rust 对应：

```rust
use std::net::Ipv6Addr;

let loopback: Ipv6Addr = "::1".parse()?;
let any: Ipv6Addr = "::".parse()?;
```

对应 Go：

```go
loopback := net.ParseIP("::1")
any := net.ParseIP("::")
```

带端口书写 IPv6 时必须加方括号，因为地址本身就含冒号，
不加括号会和“地址:端口”的分隔符冲突：`[::1]:8080`，
这一点在 [《UDP 与双栈 socket》](udp-sockets.md) 里已经用到过。

> 🔬 底层视角：IPv4 和 IPv6 是两套完全独立的协议，不能直接互通——
> 一台只支持 IPv6 的机器连不上纯 IPv4 的服务，反之亦然。
> “双栈”（同时支持两种）是过渡期的工程手段，不是协议本身互通了。

----

# 端口区分进程

> IP 地址找到机器，端口号找到机器上的哪个进程在监听。

端口是一个 16 位整数，范围 0~65535，分三段惯例：

| 范围 | 惯例 | 例子 |
| --- | --- | --- |
| 0~1023 | 知名端口，多数系统要特权才能监听 | 80（HTTP）、443（HTTPS）、22（SSH） |
| 1024~49151 | 注册端口，可自由使用 | 你的业务服务常用这一段 |
| 49152~65535 | 临时端口，操作系统自动分配 | 客户端 `connect` 时的本地端口 |

一个容易忽略的点：**客户端发起连接时也占用一个本地端口**，
只是通常由操作系统自动挑一个空闲的临时端口，你的代码看不到、
也不用关心它是多少。真正决定“连到哪个服务”的是目标地址的端口，
不是你本地的端口。

Rust 里绑定固定端口和让系统自动分配端口的写法：

```rust
use std::net::TcpListener;

let fixed = TcpListener::bind("127.0.0.1:8080")?;
let random = TcpListener::bind("127.0.0.1:0")?; // 端口 0 = 让系统挑一个空闲端口
let real_port = random.local_addr()?.port();
```

对应 Go：

```go
fixed, err := net.Listen("tcp", "127.0.0.1:8080")
random, err := net.Listen("tcp", "127.0.0.1:0")
realPort := random.Addr().(*net.TCPAddr).Port
```

端口 0 这个技巧在写测试时特别有用——[《UDP 与双栈 socket》](udp-sockets.md)
的“端口 0 特别适合自动化测试”一节已经用过一次，这里再强调：
**多个测试并行跑时，谁都不用抢一个写死的端口号**。

# 0.0.0.0 和 127.0.0.1 不是一回事

> 新手最容易搞混的三个地址：`0.0.0.0`、`127.0.0.1`、局域网 IP，
> 它们在“绑定”和“连接”两种场景下的含义完全不同。

先分清两种动作：

- **绑定（bind）**：服务端“我要监听哪个地址”；
- **连接（connect）**：客户端“我要连去哪个地址”。

三个地址在这两种场景下的行为：

| 地址 | 作为 bind 目标 | 作为 connect 目标 |
| --- | --- | --- |
| `0.0.0.0` | 监听本机**所有**网卡（本机网卡、局域网网卡） | 无意义，不能连接到“所有地址” |
| `127.0.0.1` | 只监听本机回环接口，外机器连不进来 | 只能连到本机上的服务 |
| 局域网 IP（如 `192.168.1.5`） | 只监听这一个网卡 | 连到局域网内这台具体机器 |

最常见的坑：**开发时习惯把服务绑定在 `127.0.0.1`，部署到容器或云主机后
外部访问不了**——因为 `127.0.0.1` 只接受“来自本机自己”的连接，
容器外部、甚至容器编排系统的健康检查探针，都不算“本机自己”。
生产环境的服务端一般应该绑定 `0.0.0.0`（或具体的局域网 IP），
只有本地调试、且明确不希望外部访问时才用 `127.0.0.1`。

Rust 示例（服务端场景，对比两种绑定）：

```rust
// 只有本机能连，容器外部连不进来
let dev = TcpListener::bind("127.0.0.1:8080")?;

// 本机所有网卡都能连，生产常用
let prod = TcpListener::bind("0.0.0.0:8080")?;
```

对应 Go：

```go
dev, _ := net.Listen("tcp", "127.0.0.1:8080")
prod, _ := net.Listen("tcp", "0.0.0.0:8080")
```

> 🔬 底层视角：`0.0.0.0` 在内核里不是一个真实网卡的地址，
> 而是“通配”标记——内核收到发往本机任意网卡地址的数据包时，
> 都会尝试匹配到绑定了 `0.0.0.0` 的那个 socket。真正的数据包
> 到达时，源地址和目标地址仍然是具体的 IP，`0.0.0.0` 只出现在
> 绑定阶段。

# 域名怎么变成 IP

> 浏览器地址栏里的 `example.com`，最终必须变成一个 IP 地址才能真正建立连接。

这一步叫 **DNS 解析（域名解析）**。两种最常见的记录类型：

- **A 记录**：域名 → IPv4 地址；
- **AAAA 记录**：域名 → IPv6 地址（多一个 A 是为了兼容历史命名习惯）。

同一个域名可以同时有 A 记录和 AAAA 记录，客户端拿到两种地址后，
会按系统策略选一个先尝试连接（现代系统通常优先尝试 IPv6，
失败后再退回 IPv4，这个策略常被称为 Happy Eyeballs）。

Rust 里最常用的是把域名交给标准库，让它连同 DNS 解析一起完成：

```rust
use std::net::TcpStream;

// "example.com:80" 这种带域名的地址，标准库会自动做 DNS 解析
let stream = TcpStream::connect("example.com:80")?;
```

如果只是想拿到解析结果、不建立连接：

```rust
use std::net::ToSocketAddrs;

for addr in "example.com:80".to_socket_addrs()? {
    println!("解析到: {addr}");
}
```

对应 Go：

```go
addrs, err := net.LookupHost("example.com")
// 或者直接 net.Dial，内部自动做解析
conn, err := net.Dial("tcp", "example.com:80")
```

操作系统层面的解析顺序（Windows / Linux 都类似）：先查本机的
`hosts` 文件（Windows 在
`C:\Windows\System32\drivers\etc\hosts`，Linux 在 `/etc/hosts`），
命中就直接用；没命中再去问配置好的 DNS 服务器。**这就是为什么
本地改 `hosts` 文件能让某个域名指到别的 IP，常用于开发调试。**

# 常见连接坑

> 这一节专门收集“代码没写错，但连不上”的典型场景，出问题先对照排查。

- **连错了地址类型**：把服务端 `bind` 用的 `0.0.0.0` 误当成客户端
  `connect` 的目标——`0.0.0.0` 不是一个可以连接的具体地址，
  客户端应该用具体的 IP（`127.0.0.1` 或局域网 IP）；
- **IPv6 优先导致连不上**：域名同时有 A 和 AAAA 记录，系统优先尝试
  IPv6，但如果 IPv6 路由不通（常见于家庭网络、部分云环境），
  会有明显的连接延迟甚至超时，看起来像是“服务挂了”；
- **容器里的 `localhost` 不是宿主机**：容器有自己独立的网络命名空间，
  容器里访问 `localhost`/`127.0.0.1` 指向的是容器自己，不是宿主机，
  也不是宿主机上跑的其他容器——想访问宿主机服务，要用容器网络配置的
  网关地址或专门的主机别名（如 Docker Desktop 的
  `host.docker.internal`）；
- **端口没监听但进程在跑**：进程活着不代表端口在监听，可能绑定的是
  另一个地址、端口写错了，或者启动时端口被占用而悄悄失败——
  用 `netstat -an`（Windows）/ `ss -tlnp`（Linux）确认端口和地址
  是否和预期一致；
- **DNS 缓存**：改了 DNS 记录后本机还在用旧地址，是本机或系统的
  DNS 缓存没过期，不是配置没生效，等 TTL 过期或手动清缓存即可。

# 动手实验

1. 执行 `ipconfig`（Windows）或 `ip addr`（Linux），找到自己机器的
   局域网 IP，分别尝试绑定 `127.0.0.1` 和这个局域网 IP 启动一个
   简单服务，用另一台机器验证只有绑局域网 IP 才能连进来；
2. 执行 `nslookup example.com`（或 `dig example.com`），观察它同时
   返回的 A 记录，尝试加上 `nslookup -type=AAAA example.com`
   看有没有 AAAA 记录；
3. 在本机 `hosts` 文件里给 `example.com` 加一行指向 `127.0.0.1`，
   再 `ping example.com`，观察解析结果被本机配置覆盖；
4. 如果本机装了容器环境，在容器里执行 `curl 127.0.0.1:某端口`，
   对比同一个端口在宿主机上是否能访问，体会“容器里的 localhost
   是另一个世界”。

# 三句话带走

1. IPv4 是 32 位地址（点分十进制），IPv6 是 128 位地址（冒号十六进制），
   两者协议不互通，双栈只是过渡期的工程手段。
2. `0.0.0.0`（监听所有网卡）、`127.0.0.1`（只本机）、局域网 IP
   （只这一个网卡）在 bind 和 connect 两种场景下含义不同，别混用。
3. 域名要先经 DNS 解析成 IP 才能连接；本机 `hosts` 文件优先于远程
   DNS 服务器，容器里的 `localhost` 不等于宿主机的 `localhost`。

# 附：本课生词表

- **点分十进制** —— IPv4 地址的书写形式，四段十进制数字用点分隔。
- **A 记录 / AAAA 记录** —— DNS 里域名到 IPv4 / IPv6 地址的映射记录。
- **回环地址（loopback）** —— 只指向本机自己的地址，IPv4 是 `127.0.0.1`，
  IPv6 是 `::1`。
- **任意地址（any address）** —— 绑定时表示“本机所有网卡”，IPv4 是
  `0.0.0.0`，IPv6 是 `::`。
- **临时端口（ephemeral port）** —— 客户端连接时由操作系统自动分配的
  本地端口，通常在 49152~65535。
- **Happy Eyeballs** —— 域名同时有 IPv4/IPv6 地址时，客户端并行或
  优先尝试其中一种、快速回退到另一种的连接策略。
- **`hosts` 文件** —— 本机静态域名映射表，优先级高于远程 DNS 查询。
- **网络命名空间** —— 容器隔离网络环境的机制，容器内的 `localhost`
  指向容器自己。
