# 代理与 NAT 改了什么

> 代码：`code/network-proxy-nat/`　运行：`cargo run -p network-proxy-nat`

上一课讲的负载均衡，本质上是“代理”的一种特殊形态：客户端连的
不是真正处理请求的机器，中间有个盒子帮忙转了一手。这一课把
**代理（proxy）**和**NAT** 这两个经常被搞混、但成因完全不同的
东西分开讲清楚，重点回答一个程序员最常踩的坑：**“我在代码里
读到的对端地址，到底是不是真的？”**

如果还没读上一课，建议先看 [《负载均衡》](load-balancing.md)——
本课会延续它结尾提到的“LB 后面看不到真实 IP”这个问题往下讲。

----

# 正向代理帮客户端

> 正向代理站在**客户端这一侧**，替客户端去访问外面的世界。

最熟悉的例子就是公司内网访问外网时用的 HTTP 代理：你的浏览器
配置了一个代理地址，所有请求先发给代理，代理再替你去连真正的
目标网站，拿到响应后转交给你。**从目标网站的角度看，它只看到
代理的地址在访问它，完全不知道背后还有一个真实的你**。

```
你的浏览器 ──▶ 正向代理（替你出面） ──▶ example.com
```

对应 Go 里配置一个正向代理的写法：

```go
proxyURL, _ := url.Parse("http://127.0.0.1:8081")
client := &http.Client{
    Transport: &http.Transport{Proxy: http.ProxyURL(proxyURL)},
}
resp, err := client.Get("http://example.com/")
```

对应 Rust（`reqwest`）：

```rust
let client = reqwest::Client::builder()
    .proxy(reqwest::Proxy::http("http://127.0.0.1:8081")?)
    .build()?;
let resp = client.get("http://example.com/").send().await?;
```

正向代理常见的用途：企业网关做统一出口审计、翻墙工具、给爬虫
换出口 IP。**关键认知：正向代理是“客户端主动选择使用”的，
客户端知道自己在用代理，目标服务器不知道背后的真实客户端。**

----

# 反向代理帮服务端

> 反向代理站在**服务端这一侧**，替服务端接客户端的请求。

反向代理和正向代理刚好镜像：客户端根本不知道、也不需要知道
自己面对的是一个代理，它以为自己直接在跟真正的服务器说话，
实际上请求先落在反向代理上，代理再决定转给后面哪台真正的
服务器处理。上一课的负载均衡就是反向代理最常见的形态之一。

```
浏览器 ──▶ 反向代理（客户端以为这就是服务器） ──▶ 真正的后端
```

对应 Go：客户端代码完全看不出差异，还是普通的一次 HTTP 请求：

```go
resp, err := http.Get("http://127.0.0.1:8080/")
// 8080 这个地址背后是反向代理还是真实服务器，客户端代码毫无区别
```

对应 Rust 同理：

```rust
let body = reqwest::get("http://127.0.0.1:8080/").await?.text().await?;
```

正向/反向代理的区别，用一句话记：**正向代理是“客户端的代言人”，
反向代理是“服务端的门神”**。两者在网络包层面做的事情很像——
都是接一条连接、转发到另一条连接——区别主要在“它替谁说话、
谁知道它的存在”。

# NAT 让对端地址变身

> NAT 不是代理，但效果上也会让“服务端看到的对端地址”和
> “客户端真实地址”不一致——原理完全不同，值得分开理解。

**NAT（Network Address Translation，网络地址转换）**发生在
网络层，通常在路由器/网关上完成：家里的路由器把你电脑的私有
地址（比如 `192.0.2.10`，这类内网地址不会出现在公网上）
转换成路由器自己的公网地址，再把包发出去；回包到达路由器后，
再反向转换回来送给你的电脑。

```
你的电脑（私有地址） ──▶ 路由器（做 NAT，换成公网地址） ──▶ 公网服务器
```

对公网服务器来说，**它收到的连接，源地址是路由器的公网地址，
不是你电脑的私有地址**——这一点和反向代理造成的现象很像（后端
看到的对端地址是代理/LB 的地址），但 NAT 是网络层的地址转换，
代理是应用层/传输层主动转发，两者机制完全不同，只是表现类似。

> 🔬 底层视角：NAT 网关内部维护一张“转换表”，记录
> “私有地址:端口 ↔ 公网地址:端口”的映射，每条 TCP/UDP 连接
> 占一条表项。这也是为什么 NAT 后面的设备如果长时间不发数据，
> 网关可能会把这条映射悄悄清掉——[《TCP》](tcp.md) 讲的
> keepalive 探活，很大一部分就是为了应对这种情况。

一个常见的连锁反应：如果你的服务同时部署在 NAT 后面，又在
后端记录“来源 IP”，你会发现大量不同的真实用户，在你的日志里
显示成了同一个 IP（NAT 网关的公网 IP）——这和上一课“会话粘滞
按 IP 哈希会失真”是同一个根因。

# X-Forwarded-For 传真实 IP

> 既然代理/NAT 会把真实客户端地址“抹掉”，业界的补救办法是
> **在应用层协议里手动把它带上**。

七层代理（反向代理、LB）在转发 HTTP 请求给后端之前，通常会在
请求头里加上（或追加）一行：

```
X-Forwarded-For: 203.0.113.5
```

`203.0.113.5` 就是代理在自己那一层看到的“对端地址”——也就是
真实客户端的地址（假设代理和客户端之间没有再套一层代理/NAT）。
后端拿到请求后，不再看传输层的 `peer_addr()`，而是去读这个头：

```go
clientIP := r.Header.Get("X-Forwarded-For")
```

```rust
let client_ip = headers.get("x-forwarded-for");
```

如果请求经过了**多级代理**，这个头会变成逗号分隔的一串地址，
**最左边是最初的客户端，后面依次是每一级代理自己看到的上一跳
地址**：

```
X-Forwarded-For: 203.0.113.5, 198.51.100.9
```

读这个头时的正确做法是：只信任你自己架构里**最后一级、你完全
控制的代理**追加的那一段，从右往左数到你信任的代理数量为止。

----

# 伪造头是常见风险

> `X-Forwarded-For` 本质上只是一个普通的 HTTP 请求头，**任何
> 客户端都可以自己手写这个头，伪造成任意 IP**。

如果你的后端没有代理在前面，却直接信任客户端发来的
`X-Forwarded-For` 做限流、做白名单，攻击者只需要一行 curl：

```bash
curl -H "X-Forwarded-For: 127.0.0.1" http://127.0.0.1:8080/admin
```

就能伪装成“本机地址”绕过基于 IP 的访问控制。**正确的信任
边界是：只信任你自己架构里、经过身份确认的那一级代理写的值，
永远不要直接信任客户端自己带过来的这个头**。常见做法：

- 让最外层的反向代理**覆盖**（而不是追加）这个头，避免客户端
  自己伪造的值混进链条；
- 后端只解析头里“从右数第 N 段”，`N` 等于你架构里可信代理的
  层数，这个数字要写死在配置里，不能凭头里的内容自己判断。

**一句话记住：这个头能提供“便利的线索”，但没有任何密码学
签名保护它，安全边界永远要建立在你自己控制的那一层。**

# 本地用反向代理思路

> 不需要真的装一个 Nginx 才能理解反向代理——本课的示例代码
> 自己动手写了一个最简版本。

一个反向代理最核心的骨架，剥掉所有配置项之后，其实就三步：

1. 在前端端口 `accept` 一个客户端连接；
2. 主动向后端服务发起一个新连接；
3. 把两条连接的数据**双向搬运**——客户端发来的搬给后端，
   后端回的搬给客户端。

Rust 里第三步有一个现成的工具，`tokio::io::copy`，只搬一个方向；
两个方向各起一个搬运，用 `tokio::try_join!` 让它们同时跑：

```rust
let (mut client_r, mut client_w) = client_stream.into_split();
let (mut backend_r, mut backend_w) = backend_stream.into_split();

let client_to_backend = tokio::io::copy(&mut client_r, &mut backend_w);
let backend_to_client = tokio::io::copy(&mut backend_r, &mut client_w);

tokio::try_join!(client_to_backend, backend_to_client)?;
```

对应 Go 里几乎是同一个思路，用两个 goroutine 各跑一个方向：

```go
go io.Copy(backendConn, clientConn)
io.Copy(clientConn, backendConn)
```

本课示例代码（`code/network-proxy-nat/`）就是照这个骨架写的：
前端监听 `127.0.0.1:0`，`accept` 到客户端后连一次后端 echo 服务，
双向转发，并在日志里打印**“前端看到的客户端地址”**和
**“连到后端时，代理自己在本地用的地址”**——这一组对照正好
演示了本课反复强调的直觉：**经过代理之后，后端看到的“客户端”
地址其实是代理自己的地址，不是真实客户端**。

# SNI 让代理不解密也能分流

> 反向代理面对 HTTPS 流量时有个特殊本领：**不用先解密，
> 也能看到客户端想访问的域名**，靠的是 TLS 握手里一个专门的字段。

回顾 [《TLS 是 HTTP 的外壳》](tls.md) 讲的握手过程：客户端发出的
第一个握手包（`ClientHello`）里，有一个叫**SNI（Server Name
Indication，服务器名称指示）**的扩展字段，明文写着“我要访问的
域名是什么”——这个字段必须是明文，因为服务端要先知道该用哪张
证书回应，才能开始加密协商。

这给反向代理带来一个很实用的能力：**代理可以只看 `ClientHello`
里的 SNI 字段，就决定把这条连接转发给后面哪台真正的服务器**，
完全不需要先解密整条 TLS 连接、看到内容之后再决定路由。这种
做法常被称为 **SNI 路由** 或 **TLS passthrough**：

```
客户端 ClientHello（明文 SNI: api.example.com）
        │
        ▼
   反向代理：只读 SNI，不解密
        │
        ├─ SNI 是 api.example.com → 转发给后端 A（保持整条连接加密）
        └─ SNI 是 admin.example.com → 转发给后端 B
```

这和上一课 [《HTTP 是文本协议》](http-protocol.md) 讲的“七层代理
解析明文 HTTP header 做路由”是同一个思路，只是 SNI 路由发生在
**TLS 握手阶段**，代理完全不用先解密，天然适合“代理只想转发、
不想（或不能）持有后端证书私钥”的场景。

> 🔬 底层视角：正因为 SNI 是明文的，网络路径上的任何设备（包括
> 运营商）都能看到你访问的域名，即使后续内容是加密的——这也是
> 为什么后来出现了加密 SNI（ECH，Encrypted Client Hello）这类
> 补丁方案，本课不展开，只需要知道“域名本身不加密”是当前主流
> TLS 握手的现实。

# NAT 之后为什么难被连入

> 前面讲的是 NAT 怎么让“出站”连接正常工作，这一节讲一个新手
> 常忽略的反向问题：**NAT 后面的设备，天生很难被外面主动连进来**。

回顾“NAT 让对端地址变身”一节的转换表：NAT 网关只有在**内网设备
先主动发出去一个包**之后，才会临时建立一条“私有地址:端口 ↔
公网地址:端口”的映射。如果没有人先发起连接，公网上没有任何设备
知道该往哪个私有地址转发——这意味着**两个都在各自 NAT 后面的
设备，没法直接互相发起连接**，这正是很多点对点（P2P）应用、
游戏联机、内网穿透工具要解决的核心难题。

常见的绕过办法，按复杂度从低到高：

- **公网中转服务器**：两端都主动连到一台公网服务器上，数据全部
  经过它转发——本课前面讲的反向代理本质上就是这类角色，简单可靠，
  代价是所有流量要绕一圈，服务器带宽成本随连接数线性增长；
- **打洞（NAT hole punching）**：借助一个公网服务器帮两端交换
  彼此的公网地址和端口，再让两端同时主动向对方的地址发送数据——
  利用的正是“只要我先发了包，我这边的 NAT 就会打开一条临时映射”
  这个机制，双方几乎同时“各自打开一个洞”，之后就能直接互通，
  不再需要服务器转发数据（只在建立阶段起中介作用）；
- 打洞在一部分 NAT 类型上不生效（比如某些运营商用的**对称型
  NAT**，给同一个内网地址访问不同公网目标时会分配不同的公网
  端口，猜不出对方实际会用哪个端口），这种情况只能退回中转方案。

业务程序员不需要自己实现打洞逻辑，但理解这个背景，能帮你看懂
一些底层库的复杂性从何而来——比如很多实时通信 SDK 内部同时
维护“优先尝试直连打洞，失败退回中转服务器”的双重路径，这不是
过度设计，是应对 NAT 现实的必要复杂度。

# 常见误区

> 这一节收集代理/NAT 相关最容易搞混的几个说法。

- **误区一：反向代理和负载均衡是两个不同的东西。** 上一课已经
  提过，负载均衡是反向代理的一种常见形态，不是并列的两个概念——
  区分它们没有意义，重点是“这个组件替谁说话、怎么选后端”；
- **误区二：只要用了 HTTPS，代理就看不到我访问的域名。**
  上一节讲的 SNI 字段是明文，代理（甚至链路上的其他设备）
  照样能看到域名，HTTPS 保护的是内容，不是“你访问了谁”这件事；
- **误区三：NAT 是一种安全机制。** NAT 的本意是解决 IPv4 地址
  不够用的问题，它顺带带来了一点“外部无法直接连入内网”的效果，
  容易被误认为是防火墙，但 NAT 本身不做任何访问控制判断——
  真正的安全边界应该是专门的防火墙规则，不能依赖 NAT 的副作用；
- **误区四：`X-Forwarded-For` 里最后一段一定是真实客户端 IP。**
  恰恰相反，正确读法是从右往左数、只信任你架构里可控代理追加的
  那几段，最左边才是最初的客户端（前提是没被伪造），细节见
  “X-Forwarded-For 传真实 IP”一节，别记反顺序。

# 排错对照

> “客户端 IP 不对”“连接超时但代理日志显示正常”这类问题，
> 按下面的表格定位环节。

| 现象 | 大概率原因 | 怎么确认 |
| --- | --- | --- |
| 后端日志里所有请求都显示同一个 IP | 前面有代理/LB/NAT，代码在直接读 `peer_addr()` | 检查后端是否应该读 `X-Forwarded-For` 而不是传输层地址 |
| `X-Forwarded-For` 能被随意伪造成任意值 | 后端没有区分“可信代理写的值”和“客户端自己带的值” | 检查最外层代理是否配置成“覆盖”而不是“追加”这个头 |
| 两个都在内网的设备互相连不上，但都能各自连公网服务 | 双方都在 NAT 后面，没有人能直接连入对方 | 参考“NAT 之后为什么难被连入”一节，需要中转或打洞方案 |
| SNI 路由把请求转发到了错误的后端 | 客户端请求里的 SNI 和实际想访问的 Host 不一致（少见但可能，比如手工构造的请求） | 对比 TLS 握手里的 SNI 字段和 HTTP 请求头里的 `Host` 字段是否一致 |
| 反向代理正常转发，但后端偶尔收不到数据 | 代理和后端之间的连接被中间设备（如云厂商 NAT 网关）空闲超时清掉 | 参考 [《TCP》](tcp.md) keepalive 一节，检查代理到后端这一段的空闲超时配置 |

# 动手实验

1. 运行 `cargo run -p network-proxy-nat`，观察日志里“前端 peer”
   和“代理连后端时用的本地地址”是否是两个不同的地址；
2. 把示例改成同时接受两个客户端连接，观察日志里两条转发链路
   是否互不干扰；
3. 试着往示例的转发逻辑里加一行：往转发给后端的数据前面拼一段
   “模拟 `X-Forwarded-For` 前缀”，体会应用层怎么把真实来源信息
   “捎带”过去；
4. 用 `curl -H "X-Forwarded-For: 127.0.0.1"` 访问一个你能控制的
   本地测试接口，观察如果后端天真地信任这个头会发生什么。

# 三句话带走

1. 正向代理替客户端说话（服务端不知道真实客户端），反向代理
   替服务端接客（客户端不知道背后还有别的机器）。
2. NAT 是网络层的地址转换，代理是应用层/传输层的主动转发，
   机制不同，但都会让“服务端看到的对端地址”偏离真实客户端。
3. `X-Forwarded-For` 能补回真实客户端 IP，但它只是普通请求头，
   能被伪造，只应信任自己架构里可控代理写入的那一段。

# 附：本课生词表

- **正向代理（forward proxy）** —— 代表客户端向外发起请求的
  中间节点，服务端不知道真实客户端。
- **反向代理（reverse proxy）** —— 代表服务端接收请求的中间
  节点，客户端不知道背后的真实服务器。
- **NAT（网络地址转换）** —— 网络层把私有地址转换成公网地址
  （或反向转换）的机制，常见于路由器/网关。
- **`X-Forwarded-For`** —— 用于在代理转发时携带真实客户端 IP
  的 HTTP 请求头，多级代理时以逗号分隔累加。
- **信任边界** —— 只应信任自己架构里可控、经过身份确认的组件
  写入的数据，不能直接信任客户端自带的声明。
- **`tokio::io::copy`** —— 把一个异步读端的数据搬运到一个异步
  写端的工具函数，反向代理双向转发的核心搬运工具。
- **SNI（服务器名称指示）** —— TLS 握手第一个包里明文携带的
  目标域名字段，代理可据此路由而不需要先解密。
- **TLS passthrough** —— 代理只看 SNI 转发连接、不解密也不持有
  证书私钥的转发方式。
- **NAT 打洞（hole punching）** —— 借助公网服务器牵线，让两个
  都在 NAT 后面的设备各自主动发包、绕开中转直接互通的技术。
- **对称型 NAT** —— 一种给同一内网地址访问不同公网目标分配不同
  端口的 NAT 类型，打洞技术在这类 NAT 上通常不生效。
