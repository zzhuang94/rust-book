# CPU 比内存快百倍

> 代码：`code/os-cpu-memory/`　运行：`cargo run -p os-cpu-memory`  
> （代码演示两件事：顺序访问数组 vs 跳着访问数组的耗时差异，  
> 以及「伪共享」拖慢多线程计数器的实测对比）

> 前置：[《程序如何跑起来》](computer-basics.md) 提到过 CPU 内部有个  
> 「记录下一条指令在哪」的寄存器。这一课把 CPU 内部到内存之间的  
> 整条链路拆开看：寄存器、多核、缓存、内存、闪存——每一层都比  
> 上一层慢一个数量级，这个「越往下越慢」的直觉会在后面反复用到。

你写 `let x = a + b;` 这一行代码时，`a`、`b`、`x` 具体“住”在哪里？  
答案不是一个简单的“内存”，而是一条长长的链路：寄存器 → 缓存  
（L1→L2→L3）→ 内存 → （如果要落盘）存储。链路上每一级都比  
下一级快、贵、小。这一课把这条链路逐级拆开。

# 寄存器是CPU的手

> 寄存器是这条链路里最快、也最小的一级，快到和 CPU 本身是同一个东西。

- 寄存器是 CPU **内部** 的存储单元，不需要经过任何总线去“外面”找，  
  访问速度和 CPU 主频同步（一个时钟周期左右，纳秒级都算慢的）；
- 数量极少（现代 CPU 通常几十个通用寄存器）、每个只能存几十字节  
  （64 位 CPU 上一个寄存器存 8 字节）；
- 其中一个特殊的寄存器叫 **程序计数器（PC）**，记录「下一条指令在哪」——  
  上一课已经提过；其余的通用寄存器存放「正在参与运算的数」，比如  
  `a + b` 这行代码，`a` 和 `b` 会先被从内存搬进两个寄存器，  
  CPU 的加法电路对这两个寄存器做运算，结果再放进第三个寄存器。

**记住这句话，它会在这系列课程后面反复被引用**：一段代码「执行到  
哪一步了」，本质上就是此刻这几十个寄存器的值——上一课讲进程/程序，  
后面讲线程切换、讲协程状态机，本质都在回答同一个问题：  
「这几十个寄存器的值，此刻应该属于哪一份执行流」。

编译器决定哪个变量该放进哪个寄存器（这个决策叫 **寄存器分配**），  
你写的源代码里几乎从不会直接点名某个寄存器——Rust 和 Go 都是如此，  
这一层细节完全交给编译器。极少数场景（比如写操作系统内核、写  
密码学库）会用 `asm!`/内联汇编手动摆弄寄存器，业务代码永远不需要。

# 主频撞上了功耗墙

> 「CPU 主频 3GHz」这句话到底是什么意思？为什么厂商不再靠“把主频  
> 拉得更高”来提升性能了？

主频（clock frequency）指 CPU 内部一个统一的“心跳”节奏，  
比如 3GHz = 每秒心跳 30 亿次，每次心跳推进流水线一点工作  
（简化地说，一次心跳约等于执行完一条简单指令的一部分）。

一个直觉：主频越高，CPU 干活越快。所以整个 1990 到 2000 年代，  
芯片厂商拼命拉高主频。但这条路在 2005 年前后基本走到了头——  
主频越高，芯片发热越多（功耗和频率大致是超线性关系），  
超过某个点，散热跟不上，芯片会因为过热而损坏或被迫降频。  
这个瓶颈业内叫 **功耗墙**。

厂商的应对方案不是“继续硬拉主频”，而是转向 **多核**：一颗芯片  
里塞两个、四个、几十个独立的执行单元，让它们并排工作，  
而不是让单个执行单元跑得更快。这就是为什么你的笔记本从  
“单核 3GHz”变成了“8 核 3.5GHz”，而不是“单核 20GHz”——  
后者物理上做不到（或者做到了也散热不了）。

# 物理核与超线程

> 单核提不了频，厂商在一颗 CPU 里塞多个核心；同时还有一个“半价”技巧——超线程。

- **物理核心（core）**：真正独立的执行单元，有自己完整的一套寄存器、  
  运算电路。4 个物理核心 = 真的能同时执行 4 条独立的指令流，  
  这是唯一意义上的「真并行」来源；
- **超线程 / SMT（Simultaneous Multi-Threading）**：一个物理核心  
  伪装成 2 个「逻辑核心」。原理是：CPU 执行一条指令流时，  
  经常要「等」（比如等数据从内存搬过来），这段空闲时间如果切换去  
  跑另一条指令流的工作，就不浪费。**但两条逻辑核心仍然共享同一套  
  运算电路**，所以收益远不是翻倍，业内经验大概是提升 20%~30%；
- 所以「8 核 16 线程」这句话里，「16」是逻辑核心数，  
  它的真并行能力更接近 8，不是 16。

怎么在自己机器上查看这两个数字：

```powershell
# Windows PowerShell：NumberOfCores 是物理核，NumberOfLogicalProcessors 是逻辑核
(Get-CimInstance Win32_Processor) | Select NumberOfCores,NumberOfLogicalProcessors
```

```bash
# Linux
lscpu | grep -E "^CPU\(s\)|Core\(s\) per socket|Thread\(s\) per core"
```

Rust 里能拿到的是逻辑核数（和超线程之后的数字对齐）：

```rust
let n = std::thread::available_parallelism()?.get();
println!("逻辑核数: {n}");
```

对照 Go：`runtime.NumCPU()` 返回的也是逻辑核数，  
含义完全一样。[《Tokio 运行时》](../async/tokio.md) 的多线程运行时  
默认开的工作线程数、[《Rust 多线程与并发》](../concurrency/threads.md)  
「CPU 密集切块并行」那节按核数切块，用的都是这同一个数字。

# 缓存为什么存在

> CPU 快、内存慢，中间的差距大到需要专门发明一整层东西来填。

一个直观数字：CPU 访问自己内部的寄存器，大约 1 个时钟周期；  
访问内存（RAM），大约 100 个时钟周期以上。**差了两个数量级**。  
如果 CPU 每次算东西都要老老实实等内存 100 个周期，  
现代 CPU 的大部分算力都会被“干等”浪费掉。

工程上的解决方案是在 CPU 和内存之间垫几层 **缓存（Cache）**，  
越靠近 CPU 越小越快，越靠近内存越大越慢：

| 层级 | 典型大小 | 典型延迟 | 备注 |
| --- | --- | --- | --- |
| L1 缓存 | 每核 32~64 KB | ~1ns（几个周期） | 分指令缓存和数据缓存两块 |
| L2 缓存 | 每核 256KB~1MB | ~4ns（十几个周期） | 每个核心独占 |
| L3 缓存 | 全芯片共享 8~32MB | ~10~20ns（几十个周期） | 所有核心共享，多核之间数据交换靠它 |
| 内存（RAM） | 几 GB~几百 GB | ~100ns 以上 | 全体核心共享，走内存总线 |

缓存能生效的前提是一个经验规律，叫 **局部性原理**：

- **时间局部性**：刚访问过的数据，很可能马上还会再访问一次  
  （比如循环变量、刚调用过的函数）；
- **空间局部性**：访问了地址 X，很可能马上还会访问 X 附近的地址  
  （比如遍历数组时，下一个元素紧挨着上一个）。

CPU 利用这两条规律，把「刚用过的」和「用过的附近的」数据都顺手  
搬进缓存，下次大概率能在缓存里直接找到（叫 **缓存命中**），  
而不用再跑一趟慢速的内存（**缓存未命中**，代价是白等上百周期）。

# 缓存行是搬运单位

> 缓存不是一个字节一个字节搬运的，而是以固定大小的「缓存行」为单位。  
> 这个事实会直接影响你代码的性能，也是后面理解并发问题的关键。

缓存和内存之间搬运数据，最小单位是一条 **缓存行（cache line）**，  
现代 CPU 上通常是 **64 字节**。也就是说，哪怕你只想读 1 个字节，  
CPU 也会把这个字节所在的整整 64 字节一次性搬进缓存——这正是  
「空间局部性」在硬件里的具体实现：既然大概率马上要用附近的数据，  
不如一次搬够。

这解释了一个常见的性能现象：**顺序遍历数组比跳着访问快得多**，  
即使总的读取字节数一样。

```rust
// 顺序访问：一个 cache line（64 字节 = 16 个 i32）一次搬进来，
// 接下来 15 次访问全部命中缓存，几乎零成本
let mut sum: i64 = 0;
for i in 0..data.len() {
    sum += data[i] as i64;
}

// 跳着访问（这里跳 64 个元素，跨度超过一个 cache line）：
// 几乎每次访问都要重新搬一条新的 cache line，缓存形同虚设
let mut sum2: i64 = 0;
let mut i = 0;
while i < data.len() {
    sum2 += data[i] as i64;
    i += 64;
}
```

真实跑一下（本章代码里有完整版本），在几百万元素的数组上，  
顺序版本往往比跳跃版本快 3~10 倍甚至更多——**代码逻辑完全没变，  
只是访问顺序变了，性能天差地别**。这也是为什么“数组比链表快”  
这句话在现代硬件上基本成立：数组的元素紧挨着放，天然对缓存行友好；  
链表的每个节点可能散落在内存的任意角落，几乎每跳一个节点就要  
重新搬一条缓存行。

# 多核抢同一缓存行

> 单核用缓存行是纯收益，多核用缓存行会出现一个反直觉的坏现象——伪共享。

一颗 CPU 上，每个核心有自己独立的 L1/L2 缓存，但 L3 和内存是  
**全部核心共享** 的。当两个核心同时频繁 **写** 同一条缓存行  
（哪怕它们各自写的是这条缓存行里不同的字节），会发生什么？

- 核心 A 把这条缓存行改了，为了保证核心 B 看到的不是过期数据，  
  硬件必须让核心 B 手里那份缓存行失效；
- 核心 B 下次要用时，发现自己的缓存行失效了，得重新从核心 A 那里  
  或者从更上层的缓存把这条缓存行搬过来；
- 如果 A、B 交替着写，这条缓存行就在两个核心之间来回“搬家”，  
  专业说法叫 **cache line 弹跳（cache line bouncing）**。

这个现象有个专门名字，叫 **伪共享（False Sharing）**——“伪”是因为  
两个线程逻辑上根本没有共享同一个变量，只是碰巧这两个变量的内存地址  
落在了同一条 64 字节的缓存行里，硬件层面把它们当成了“同一份数据”  
来回同步。

```rust
// 坏例子：两个线程各自累加自己的计数器，逻辑上互不相干，
// 但两个 u64 紧挨着放在同一个 struct 里，大概率落进同一条缓存行
struct BadCounters {
    a: std::sync::atomic::AtomicU64, // 线程1只写这个
    b: std::sync::atomic::AtomicU64, // 线程2只写这个 —— 但和 a 挤在一条缓存行里
}

// 修复：用对齐属性把每个字段强行摆到独立的缓存行上
#[repr(align(64))]
struct GoodCounter(std::sync::atomic::AtomicU64);

struct GoodCounters {
    a: GoodCounter, // 独占一条 64 字节缓存行
    b: GoodCounter, // 独占另一条
}
```

本章代码用这两种结构分别跑「两个线程各自累加 1000 万次」，  
你会实测到 `GoodCounters` 版本明显快过 `BadCounters` 版本——  
两份代码的“逻辑”完全一样，纯粹是内存布局的差异导致了性能差距。  
Go 里同样存在这个问题，标准库 `sync/atomic` 之上如果你自己拼  
struct 把多个热点计数器放在一起，一样会撞上伪共享；解决方式也一样，  
补齐字段之间的间隔（padding）到 64 字节。

> 🔬 底层视角：[《把读做成无锁》](../http/arcswap.md) 提到  
> `RwLock` 的读锁计数器在多核间「弹跳」，说的正是这一节的现象——  
> 高并发下成千上万个读者原子地改同一个计数器，这个计数器所在的  
> 缓存行在各个核心之间来回搬家，就是伪共享/缓存行竞争的真实案例。  
> ArcSwap 之所以能规避它，核心就是把「读」变成了完全无锁的指针读取，  
> 读者之间根本不用碰同一个可写的缓存行。

# 缓存一致性协议

> 多核各有自己的缓存，怎么保证大家看到的是同一份“真相”？靠硬件  
> 自动维护的一套协议，业务代码完全不用手写，但要知道它存在、它有代价。

多核 CPU 硬件内置了 **缓存一致性协议**（常见的实现叫 MESI，  
对应缓存行的四种状态：Modified/Exclusive/Shared/Invalid）。  
简化理解成一条规则：**任何时刻，一条缓存行如果被某个核心改了，  
硬件会自动通知所有持有这条缓存行副本的其他核心「你手里的是旧的」**。

这套协议是 [《共享状态：Arc / RwLock》](../async/shared-state.md)  
里「原子指令」「内存序（Ordering）」这些概念存在的物理地基——  
Rust 的 `fetch_add`、`compare_exchange` 之所以能保证多核看到  
一致的结果，靠的就是这层硬件协议在背后干活；`Ordering::Relaxed`  
和 `Ordering::SeqCst` 的区别，本质是在告诉硬件「这次同步要做到  
多严格」，做得越严格，协议要多花的协调成本就越高。

一句话记住：**硬件保证了“大家最终会看到一致的数据”，但没有免费  
午餐——协调这件事本身要花时间，写得越频繁、核心越多，代价越大**。  
这也是为什么无锁数据结构的性能优势往往体现在「读多写少」的场景：  
读操作不触发协议里最贵的那部分工作。

# 内存本身如何组织

> 内存条（DRAM）比缓存慢一个数量级，它自己内部也有值得知道的特性。

内存（DRAM，Dynamic RAM）用极小的电容存储每一个比特：有电荷代表 1，  
没电荷代表 0。这个设计带来两个后果：

- **需要不断刷新**：电容会漏电，所以内存芯片要每隔几十毫秒把  
  所有内容重新写一遍（这就是“Dynamic”的含义），这个刷新过程会  
  和正常的读写请求抢时间，是内存延迟高于缓存的原因之一；
- **断电即丢**：电容没有外部供电就会放光，所以内存的内容在断电  
  或进程退出后立刻消失——这一点在上一课「内存与存储的分工」  
  已经提过，这里补上物理原因。

内存的访问要经过 **内存控制器**，由它统一调度所有核心发来的  
读写请求、决定刷新的时机。这也解释了为什么「内存带宽」是有限的  
共享资源：所有核心的内存请求最终都要排队经过同一个内存控制器，  
核心数越多，抢内存带宽越激烈——这是多核程序里「加线程不一定线性  
提速」的一个硬件原因（另一个原因是任务本身能不能拆开并行，  
留给 [《Rust 多线程与并发》](../concurrency/threads.md) 细讲）。

# 闪存与内存的分野

> 手机、SSD 里用的“闪存（Flash）”，和内存条用的技术完全不同，  
> 这决定了它们各自适合干什么活。

| | 内存（DRAM） | 闪存（NAND Flash，SSD 的核心） |
| --- | --- | --- |
| 存储原理 | 电容电荷，需要持续供电和刷新 | 浮栅晶体管锁住的电荷，断电也保持 |
| 断电后 | 内容全部丢失 | 内容保留（这就是“非易失性”） |
| 读延迟 | ~100 纳秒 | ~几十到上百微秒（慢 100~1000 倍） |
| 写的特点 | 随便写、不限次数 | 擦写有寿命限制（几千到上万次），需要「磨损均衡」 |
| 典型角色 | CPU 直接可寻址、跑程序的地方 | 长期存放文件、数据库，通过操作系统的文件系统访问 |

这张表解释了一个业务代码里最基础的直觉：**为什么要区分「内存里的  
变量」和「写进数据库/文件的数据」**——前者活在 DRAM 上，进程退出  
就消失；后者最终落在闪存（或者传统机械硬盘）上，断电、重启、  
换进程都不会丢。闪存本身还有很多有趣的细节（页/块的擦写单位、  
垃圾回收、写入放大），这些和「业务代码怎么落盘」的直接关系更大，  
留给下一课 [《磁盘与 IO 直觉》](disk-io.md) 专门展开。

# Go 与 Rust 的内存布局观

> 两种语言在“要不要让你控制内存布局”这件事上，选了不同的默认值。

| | Rust | Go |
| --- | --- | --- |
| struct 字段默认排列 | 编译器可能重排以省空间；可用 `#[repr(C)]`/`#[repr(align(N))]` 精确控制 | 编译器基本按声明顺序排列，也有对齐规则，但语言层面暴露的控制手段更少 |
| 数组/切片 | `Vec<T>`、`[T; N]` 保证连续内存，天然对缓存行友好 | slice 底层同样是连续数组，效果一致 |
| 手动控制缓存行对齐 | `#[repr(align(64))]` 是语言原生特性 | 需要手动 padding 字段凑够 64 字节，没有语言级属性 |
| 垃圾回收对缓存的影响 | 无 GC，内存布局在编译期基本确定，缓存行为更可预测 | GC 扫描/移动对象会带来额外的内存访问，间接增加缓存压力 |

这不是说 Go 就一定慢——大多数业务代码根本摸不到这个层级的优化，  
GC 带来的便利性在绝大多数场景下物超所值。但当你遇到「同一份逻辑，  
换种写法性能差好几倍」这种诡异现象时，缓存行、伪共享、局部性这几个  
词，是排查这类问题时该往这个方向想的信号。

2. 跑本章代码里的 `BadCounters` vs `GoodCounters` 伪共享对比，  
   分别用 2 个线程、4 个线程重复实验，观察线程越多差距是否越明显；
3. 在你的机器上用 `lscpu`（Linux）或 CPU-Z（Windows）查看  
   L1/L2/L3 缓存的实际大小，和本课表格里的典型值做对比；
4. 试着把 `#[repr(align(64))]` 去掉重新跑一次伪共享实验，  
   用 `std::mem::size_of` 和 `std::mem::align_of` 打印两种  
   结构体的实际大小，直观看到「加了对齐属性后结构体变大了」这个代价——  
   性能和内存占用永远是一笔要一起算的账。

----

# 三句话带走

1. **CPU 到内存是一条越往下越慢的链路**：寄存器（1 周期）→ L1/L2/L3  
   缓存（几到几十周期）→ 内存（100+ 周期）→ 闪存/磁盘（慢上万倍）。  
   缓存能生效靠「局部性原理」，顺序访问远快于跳跃访问。
2. **缓存以 64 字节的「缓存行」为搬运单位**，多核频繁写同一条缓存行  
   会发生「弹跳」（cache line bouncing），逻辑上不相关的变量挤在同一条  
   缓存行里会造成「伪共享」——这是很多「代码逻辑没变、性能差几倍」  
   现象的真实病因。
3. **主频撞上功耗墙，催生了多核；一个物理核伪装两个逻辑核（超线程）  
   收益有限**；无锁数据结构（ArcSwap）、原子指令、内存序，  
   全部建立在硬件的缓存一致性协议之上，读多写少的场景才是它们的主场。

# 附：本课生词表

- **寄存器** —— CPU 内部的超高速存储单元，几十个、每个几十字节，  
  「代码执行到哪了」本质就是此刻这些寄存器的值。
- **程序计数器（PC）** —— 记录「下一条指令在哪」的特殊寄存器。
- **主频/功耗墙** —— CPU 心跳频率；2005 年前后物理散热限制了主频继续拉高，催生多核。
- **物理核心 / 超线程（SMT）** —— 真独立执行单元 / 一个物理核伪装两个逻辑核，收益约 20%~30%。
- **局部性原理** —— 刚用过的数据、附近的数据大概率马上还会用到，是缓存生效的前提。
- **缓存行（cache line）** —— 缓存与内存之间搬运数据的最小单位，通常 64 字节。
- **伪共享（False Sharing）** —— 逻辑上不相关的变量恰好落在同一条缓存行，多线程写它们时互相拖慢。
- **cache line 弹跳** —— 多核频繁写同一条缓存行，这条缓存行在核心间来回搬家。
- **缓存一致性协议（MESI）** —— 硬件自动维护多核缓存数据一致的协议，是原子指令/内存序的物理地基。
- **DRAM** —— 内存条的存储技术，靠电容电荷存数据，需要持续刷新，断电即丢。
- **NAND Flash（闪存）** —— SSD 的存储技术，断电不丢，但擦写次数有寿命限制。
- **内存控制器** —— 统一调度所有核心的内存请求、控制刷新的硬件部件，是内存带宽成为共享瓶颈的原因。
