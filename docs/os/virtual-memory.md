# 虚拟地址不是真地址

> 代码：`code/os-virtual-memory/`　运行：`cargo run -p os-virtual-memory`  
> （代码分配一块很大但几乎不写入的内存，对比「虚拟内存占用」和  
> 「真正驻留的物理内存」两个数字的巨大差异，帮你直观理解本课主题）

> 前置：[《程序与线程》](process-thread.md) 说过「每个进程有自己的  
> 虚拟地址空间，以为自己独占整台机器的内存」。这一课把这句话背后的  
> 机制彻底拆开：什么是页表、缺页是怎么回事、内存不够用时会发生什么——  
> 也是这个 `docs/os/` 系列的最后一课，会把前面几课串起来。

你写 `let v: Vec<u8> = Vec::with_capacity(1_000_000_000);`  
（申请约 1GB 内存），这行代码几乎瞬间返回，即使你的机器物理内存  
只剩 500MB。这是怎么做到的？答案就是这一课的主题：**虚拟内存**——  
操作系统给每个进程编织的一层「地址幻觉」。

----

# 地址不等于真实位置

> 你代码里读写的「内存地址」，从来不是内存芯片上的真实物理位置，  
> 中间隔着一层翻译。

- **虚拟地址（virtual address）**：你的代码（包括 Rust/Go 运行时）  
  看到、使用的地址，每个进程有自己独立的一套，从 0 开始编号，  
  理论上限往往是几十上百 TB（64 位地址空间），远超机器实际装的  
  物理内存；
- **物理地址（physical address）**：内存芯片上真正的存储位置，  
  全机器只有一套，容量等于你插的内存条大小。

进程 A 的虚拟地址 `0x1000` 和进程 B 的虚拟地址 `0x1000`，  
可能被翻译到完全不同的两块物理内存，谁也看不见谁——这正是  
[《程序与线程》](process-thread.md) 讲过的进程隔离效果的具体实现。  
这层「虚拟地址 → 物理地址」的翻译，谁来做、怎么做，是这一课的核心。

----

# 页表是一本翻译词典

> 翻译工作不是逐字节做的，而是按固定大小的「页」为单位，  
> 查一本叫「页表」的词典。

内存被划分成固定大小的块，叫 **页（page）**，常见大小是 **4KB**  
（部分场景也用更大的「大页」，后面提一句）。每个进程有一份自己的  
**页表（page table）**——这本词典记录着「这个进程的第几号虚拟页，  
对应物理内存的第几号物理页」。

```
进程的虚拟地址空间（一串连续编号的虚拟页）
虚拟页 0 ──┐
虚拟页 1 ──┤   （查页表）
虚拟页 2 ──┼──────────────→ 物理页 87
虚拟页 3 ──┘                物理页 12（不连续也没关系）
   ...
```

注意图里的一个关键事实：**虚拟地址空间里连续的几个页，翻译到物理  
内存后完全可以是不连续的**——这件事对你的代码毫无影响（你读写的  
永远是「连续」的虚拟地址），但给了操作系统极大的自由：物理内存  
哪里有空位就往哪里塞，不需要为了「凑出一块连续物理内存」而费心。

真正做「查词典、把虚拟地址翻译成物理地址」这件事的，  
不是操作系统软件本身，而是 CPU 里一个专门的硬件电路，叫  
**内存管理单元（MMU，Memory Management Unit）**——软件（操作系统）  
负责维护页表这本词典的内容，硬件（MMU）负责在每一次内存访问时  
飞快地查这本词典。这个分工是速度的关键：如果每次内存访问都要软件  
介入翻译，会慢到无法接受。

# 翻译也要缓存：TLB

> 查页表这个动作本身也要访问内存，为了不让它变成新的性能瓶颈，  
> 硬件又垫了一层缓存——这是「缓存」思想的又一次应用。

回顾 [《CPU 与内存》](cpu-memory.md) 的核心思想：**访问内存慢，  
所以要缓存常用的东西**。页表本身也存在内存里，如果每次内存访问都要  
先去内存里查一遍页表再去内存里取真正的数据，等于把访问成本翻倍。  
硬件的解决方案是给「虚拟地址→物理地址」的翻译结果本身也设一层  
高速缓存，叫 **TLB（Translation Lookaside Buffer）**——它和  
L1/L2 缓存一样，是 CPU 内部的高速存储，只是缓存的不是数据本身，  
而是「这个虚拟页翻译到哪个物理页」这条映射关系。

- **TLB 命中**：直接拿到翻译结果，几乎零额外延迟；
- **TLB 未命中**：要真的去内存里走一遍查页表的完整流程  
  （可能还要查好几级，现代系统页表通常是多级结构），代价明显更高，  
  这个过程叫 **页表遍历（page table walk）**。

这解释了「大页（Huge Page）」这个优化技巧存在的原因：把页从 4KB  
放大到 2MB 甚至更大，同样大小的内存需要的页数变少，  
TLB 能覆盖的内存总量随之变大，TLB 未命中的概率就降低了——  
这是数据库、JVM 等对内存访问性能极度敏感的软件常用的调优手段，  
业务 Rust/Go 代码一般不需要手动关心。

# 缺页异常与惰性分配

> 回到开头的问题：为什么申请 1GB 内存几乎瞬间返回，  
> 即使物理内存不够？答案是操作系统在「装死」——先不真的给你物理内存。

当你的代码申请一大块内存（比如 `Vec::with_capacity`），  
操作系统往往并不会立刻真的分配等量的物理内存，而是先记录一句话：  
「这个进程的这一段虚拟地址范围，将来如果被访问，请给它物理内存」——  
这个策略叫 **惰性分配（lazy allocation）** 或 **按需分页  
（demand paging）**。

真正的物理内存分配，发生在你 **第一次实际读写** 某个虚拟页的时候：

1. CPU 访问这个虚拟地址，MMU 查页表，发现这一页还没有对应的  
   物理页（页表里标记着「无效」或者「暂缺」）；
2. CPU 触发一个叫 **缺页异常（page fault）** 的信号，暂停当前执行，  
   把控制权交给操作系统内核；
3. 内核检查这次访问合不合法（是不是访问了根本没申请过的内存——  
   这种情况就是经典的「段错误 segmentation fault」）；
4. 如果合法，内核这才真正找一块空闲物理页，更新页表，  
   把这个虚拟页和这块物理页关联起来；
5. 内核把控制权还给 CPU，**同一条指令重新执行一次**——  
   这次 MMU 查页表就能找到有效的映射了，访问顺利完成。

这一整套过程对你的代码完全透明——你不会看到任何“异常”，  
它安静地发生在你第一次真正读写那块内存的瞬间。这也解释了开头的  
实验现象：`Vec::with_capacity(1_000_000_000)` 只是预留了虚拟地址  
范围，几乎不耗时；只有当你真的往这块内存里写数据时，  
才会一页一页地触发缺页异常、真正消耗物理内存。

> 🔬 底层视角：这正是「虚拟内存占用」和「物理内存占用（常见叫  
> RSS，Resident Set Size）」这两个数字为什么经常差很多的原因。  
> Linux 上 `/proc/<pid>/status` 里的 `VmSize`（虚拟内存总量）  
> 和 `VmRSS`（真正驻留的物理内存）就是这两个数字，  
> 任务管理器/`top` 显示的「内存占用」通常对应后者。

# 内存不够时的置换

> 物理内存终究是有限的，当大家都要用、物理内存真的不够分时，  
> 操作系统还有一招——把暂时不用的页挪去磁盘。

如果物理内存快用完，操作系统可以把一些「最近没怎么被访问」的  
物理页的内容，写到磁盘上一块专门的区域（Linux 叫 swap 分区/文件，  
Windows 叫页面文件），然后把这块物理页腾出来给别人用——  
这个过程叫 **页面置换（page swapping）**。等这一页的数据将来又  
被访问时，再触发一次缺页异常，把内容从磁盘读回物理内存。

代价是显而易见的：回顾 [《磁盘慢是常态》](disk-io.md) 那张延迟表，  
内存访问是百纳秒级，磁盘访问是几十微秒到几毫秒级——**一旦频繁发生  
页面置换，一次内存访问的实际延迟可能暴涨几百到几万倍**。  
如果系统内存长期严重不足，会陷入「不停地把这一页换出去、  
又立刻要把另一页换回来」的恶性循环，专业名字叫 **颠簸  
（thrashing）**——表现出来就是整台机器卡到几乎无法响应，  
CPU 大量时间花在搬运页面上，而不是真正的计算。这也是为什么  
生产环境的服务器通常建议 **关闭 swap 或严格限制其大小**：  
宁可让一个内存不够的服务直接失败（下一节的 OOM），  
也不要让它拖累整台机器陷入颠簸。

# OOM：内存实在不够了

> 当置换也无法解决问题（比如没开 swap，或者 swap 也用满了），  
> 操作系统的最终手段是杀死一个进程——这就是 OOM。

**OOM（Out Of Memory）** 指系统内存耗尽、无法满足新的内存请求的  
状态。Linux 上有一个专门的机制叫 **OOM Killer**：当内存实在不够，  
内核会按一套评分规则（大致是「占用内存越多、越不重要的进程分越高」）  
选出一个「牺牲品」进程，直接把它杀掉，腾出内存让系统能继续运转。  
被 OOM Killer 杀掉的进程，在容器环境里常见的退出码是 **137**  
（128 + 信号 9，也就是 `SIGKILL`），这是排查「容器莫名其妙重启」  
问题时最值得先确认的一个数字。

一个业务上常见的误解需要澄清：**「虚拟内存充足」不代表「物理内存  
充足」**。Linux 默认允许一定程度的 **内存超卖  
（overcommit）**——允许所有进程申请的虚拟内存总量超过实际物理  
内存加 swap 的总量，赌的是「大家不会同时把申请的内存全部真正用满」。  
这个赌注在大多数场景下是对的（因为惰性分配，很多申请的内存从不会  
被真正写入），但一旦赌输（某个进程突然真的开始大量写入之前只是  
「预留」的内存），就可能触发 OOM Killer，即使几分钟前 `free -h`  
看起来内存还挺充裕。

容器化部署里更常见的场景是 **cgroup 内存限制**：即使宿主机物理  
内存很充裕，一个容器被限制只能用 512MB，一旦这个容器内的进程  
实际驻留内存（RSS）超过限制，容器运行时会直接杀掉里面的进程——  
这层限制和「宿主机整体 OOM」是两件独立的事，容器里的 Rust/Go  
服务如果频繁被杀，第一步该查的正是这个限制值，而不是急着怀疑代码  
本身有内存泄漏（当然也不能排除，需要结合  
[《tracing 结构化日志》](../engineering/tracing.md) 之类的  
可观测手段一起排查）。

# Go 与 Rust 在虚拟内存上的行为差异

| | Rust | Go |
| --- | --- | --- |
| 分配大块内存（如 `Vec::with_capacity`） | 底层走 `mmap`，同样是先预留虚拟地址、惰性分配物理页 | 底层同样走 `mmap`/`brk`，机制一致 |
| 线程栈 | 默认约 8MB，同样是虚拟预留，真正用到的部分才占物理内存 | goroutine 栈初始约 2KB，按需增长，虚拟内存压力天生比 OS 线程小很多 |
| GC 对物理内存的影响 | 无 GC，内存释放（`drop`）后立刻可能归还给分配器，行为更可预测 | GC 扫描/整理堆内存，可能导致更多页被「触碰」，间接影响 RSS 的实际大小 |
| 常见 OOM 诱因 | 忘记释放的 `Vec`/`Box` 持续增长（较少见，借用检查器减少了很多此类错误） | 未预期的 goroutine 泄漏（持续创建、从不退出），更容易被忽视 |

这张表最值得记住的一行是「线程栈」——[《程序与线程》](process-thread.md)  
提过 OS 线程默认 8MB 栈「是虚拟内存预留量，不是马上占用的物理内存」，  
现在你已经知道这句话的完整机制：8MB 只是虚拟地址空间里划出的一段  
范围，真正的物理内存要等到线程的代码真的往栈深处写数据时，  
才会一页一页地通过缺页异常分配出来——绝大多数线程的栈用不到  
零头，这也是为什么「开几千个 OS 线程」在虚拟地址空间层面通常不是  
问题（64 位地址空间够大），真正的瓶颈在别处（内核调度压力、  
上下文切换成本，见 [《调度决定谁在跑》](scheduling.md)）。

# 动手实验

1. 跑本章代码，分配一块 1GB 的内存但完全不写入，用  
   `/proc/self/status`（Linux）里的 `VmSize`/`VmRSS`，  
   或者任务管理器/`top`（对应显示「虚拟内存」和「物理内存」两栏）  
   观察两者的巨大差异；
2. 修改代码，改成真正把这 1GB 完整写一遍（比如全部填成某个字节值），  
   重新观察 `VmRSS` 是否随之涨到接近 1GB，体感缺页异常发生的时机；
3. Linux 下用 `cat /proc/vmstat | grep pgfault` 观察系统累计的  
   缺页异常次数，跑一次「大量分配+写入内存」的程序前后各看一次，  
   验证数字确实会明显增长；
4. 如果条件允许，在一个内存受限的容器（`docker run -m 128m`）里  
   跑一个会不断增长内存占用的小程序，观察它被杀死时的退出码是否是  
   137，用 `docker inspect`/日志确认是被 OOM Killer 处理的。

# 三句话带走

1. **虚拟地址不是物理地址**：每个进程有自己独立的虚拟地址空间，  
   靠硬件 MMU 查页表翻译成物理地址，翻译结果本身还有一层专门的  
   缓存（TLB）——这是「进程隔离」在硬件层面的真实实现。
2. **内存分配默认是「惰性」的**：申请内存时只预留虚拟地址范围，  
   真正的物理内存分配发生在第一次实际读写时（缺页异常），  
   这解释了「虚拟内存占用」和「物理内存占用（RSS）」为什么经常  
   差很多。
3. **物理内存不够时，操作系统先尝试把冷页换到磁盘（swap，代价是  
   暴涨的延迟，严重时颠簸），最终手段是 OOM Killer 杀进程**——  
   容器环境里的 cgroup 内存限制是独立于宿主机整体内存状况的  
   另一层天花板，排查频繁重启时要优先确认这个数字。

# 附：本课生词表

- **虚拟地址 / 物理地址** —— 进程看到的地址 / 内存芯片上真实的存储位置，  
  中间靠硬件翻译。
- **页（page）** —— 虚拟内存管理的最小单位，常见大小 4KB。
- **页表（page table）** —— 记录「虚拟页对应哪个物理页」的数据结构，每个进程一份。
- **MMU（内存管理单元）** —— CPU 内部负责查页表、做地址翻译的硬件电路。
- **TLB（Translation Lookaside Buffer）** —— 缓存「虚拟页→物理页」翻译结果的高速硬件缓存。
- **大页（Huge Page）** —— 把页从 4KB 放大（如 2MB），减少页数、降低 TLB 未命中率的优化手段。
- **缺页异常（page fault）** —— 访问一个还没映射物理内存的虚拟页时触发的信号，  
  由内核负责真正分配物理页。
- **惰性分配 / 按需分页** —— 申请内存时先预留虚拟地址，真正读写时才分配物理内存。
- **RSS（Resident Set Size）** —— 一个进程真正驻留在物理内存里的大小，  
  区别于虚拟内存总量（VmSize）。
- **页面置换（swapping）/ 颠簸（thrashing）** —— 物理内存不足时把冷页换到磁盘 /  
  换页过于频繁导致系统卡死的恶性循环。
- **OOM（Out Of Memory）/ OOM Killer** —— 内存耗尽的状态 / Linux 内核按评分规则  
  杀死进程腾出内存的机制，被杀进程常见退出码 137。
- **内存超卖（overcommit）** —— 允许所有进程申请的虚拟内存总量超过实际物理内存+swap 总量。
- **cgroup 内存限制** —— 容器环境下独立于宿主机的内存天花板，超限即被杀，  
  与宿主机整体 OOM 是两件独立的事。
