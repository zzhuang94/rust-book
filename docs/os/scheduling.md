# 调度决定谁在跑

> 代码：`code/os-scheduling/`　运行：`cargo run -p os-scheduling`  
> （代码起「远多于核数」的线程各自空转计数，配合  
> `vmstat 1`/任务管理器观察上下文切换次数随线程数上涨）

> 前置：[《程序与线程》](process-thread.md) 说过创建一个线程之后，  
> 谁来决定「这个线程什么时候真正占用 CPU」。这一课就是答案——  
> 操作系统内核里的 **调度器**。理解了调度，才能理解「阻塞」  
> 为什么是异步编程里的头号大忌，也才能理解「单核也能同时听歌和编译」  
> 这种日常体验背后的机制。

你的机器可能只有 8 个核心，但同时「跑着」的进程和线程动辄成百上千——  
浏览器、IDE、后台服务、你写的程序……核心数远远不够分。  
是谁在决定「这一瞬间，哪个线程真正占着哪个核心」？答案是操作系统  
内核里的一个组件：**调度器（scheduler）**。这一课把调度器的核心机制  
拆开看。

----

# 线程只有三种状态

> 理解调度器怎么工作，先要认清一件事：一个线程在任意时刻，  
> 只能处于三种状态之一。

```
         被调度器选中                    等待的事发生了
 就绪 ────────────────→ 运行 ─┐    阻塞 ──────────────────→ 就绪
  ↑                          │      ↑
  └──── 时间片用完/被抢占 ─────┘      └── 运行中发起了要等待的操作
```

- **运行（running）**：正占着某个核心，CPU 在执行这个线程的指令；
- **就绪（ready）**：这个线程随时可以跑，万事俱备，只是暂时没有  
  空闲的核心分给它，在排队；
- **阻塞（blocked）**：这个线程在等某件还没发生的事——网络数据  
  还没到、磁盘还没读完、锁还没被释放、`sleep` 的时间还没到——  
  **阻塞状态的线程不占核心、也不参与排队**，事情真正发生后，  
  内核会把它从「阻塞」改回「就绪」，重新有资格被调度。

任意时刻，一台机器上的绝大多数线程往往都在「阻塞」或「就绪」，  
只有等于核心数的那一小部分线程真正处于「运行」。调度器的全部工作，  
就是不断决定：**核心空出来的这一刻，把它分给「就绪」队列里的哪个线程**。

----

# 阻塞的确切含义

> 这是整个 async 编程铁律「绝不在异步任务里阻塞线程」的物理地基，  
> 值得花一整节讲透。

上一课 [《用户态摸不到硬件》](user-kernel.md) 说过，发起一个系统调用  
要陷入内核态。如果这个系统调用请求的资源还没准备好（比如读一个  
还没数据的 socket），内核不会让你的线程傻等着空转 CPU，  
而是做一件更聪明的事：

1. 把这个线程标记成「阻塞」状态；
2. 把它登记到「等某个事件」的队列里（比如“等这个 socket 有数据”）；
3. **立刻把这个核心让给别的就绪线程**——核心闲不下来，  
   但你这个线程确确实实什么都没干，也不占核心；
4. 等事件真正发生（数据到了），内核把这个线程从阻塞队列挪回就绪队列，  
   等下一次被调度器选中，才会真正恢复执行。

**关键的一句话：线程阻塞期间，什么都做不了**。如果这个线程身上还  
背着别的活——比如 Tokio 的一个工作线程，本来要靠这条线程去推进  
几百个异步任务的进度——**这几百个任务会在这段阻塞期间全部停摆**，  
不是变慢，是完全停止前进，直到这个线程被唤醒。这就是为什么  
「在 async 代码里调用一个阻塞的系统调用」是异步编程里最容易犯、  
后果也最严重的错误：你以为只是「这一处稍微等一下」，实际上把  
同一个线程上排队的所有其他任务一起冻住了。[《async 基础》](../async/basics.md)  
的铁律、`spawn_blocking` 这个「逃生舱」的存在，都建立在这一节的  
事实之上。

# 时间片与抢占

> 阻塞是线程「主动」让出核心，但线程也可能「不主动」——比如陷入死循环。  
> 调度器有一套机制防止任何一个线程永远霸占核心。

调度器给每个正在「运行」的线程分配一小段时间，叫 **时间片**  
（time slice，通常几毫秒到几十毫秒）。时间片用完，即使这个线程  
还想继续跑，调度器也会强制把它切回「就绪」状态，把核心让给别人——  
这个「强制换人」的动作叫 **抢占（preemption）**。

抢占是怎么在硬件层面实现的？靠一个 **定时器中断（timer interrupt）**：  
硬件时钟每隔固定时间（比如 1 毫秒）就给 CPU 发一个信号，  
CPU 收到信号后暂停当前正在跑的代码，跳去执行内核里的调度器代码，  
调度器检查「这个线程时间片是不是用完了」，决定要不要换人，  
再把控制权交还（可能是原来的线程，也可能是新选中的线程）。

**这就是为什么单核机器也能“同时”听音乐和编译代码**：单个核心  
每隔几毫秒就在两个任务之间切换一次，切换的频率快到人的感知完全  
察觉不出「其实是轮流的」。**并发的本源，就是这个毫秒级的轮转**——  
Rust/Go 程序员天天说的“并发”，最朴素的物理实现就是这件事。

# 上下文切换的具体成本

> 无论是「阻塞让出」还是「被抢占」，核心从线程 A 换到线程 B  
> 都要经历同一套动作，这套动作本身要花时间。

回顾 [《CPU 与内存》](cpu-memory.md) 讲过的核心事实：  
「代码执行到哪了」本质就是此刻寄存器的值。换人时要做的事：

1. 把线程 A 此刻全部的寄存器值保存到内存里 A 专属的一块区域  
   （这份「快照」上一课 [《程序与线程》](process-thread.md) 叫它  
   「寄存器上下文」）；
2. 内核更新一些记账信息（这个核心现在归谁、调度队列怎么调整）；
3. 把线程 B 上次保存的寄存器上下文加载回来，从 B 上次的断点  
   继续执行。

这一整套动作的直接耗时是 **1~10 微秒**——听起来很短，  
但真正昂贵的是 **间接成本**：核心的 L1/L2 缓存里存的全是线程 A  
刚用过的数据，线程 B 上来之后，这些缓存对它来说全是「未命中」，  
要重新从内存慢慢加载——这就是 [《CPU 与内存》](cpu-memory.md)  
「局部性原理」的反面：**上下文切换会强行打断局部性，让缓存变冷**。  
线程数量一旦远超核心数，操作系统会被迫更频繁地换人，  
CPU 大量时间被花在「换人」本身而不是「真正干活」上——  
这正是 [《程序与线程》](process-thread.md) 那张成本表里，  
「线程数暴涨」被列为一大成本的原因。

> 🔬 底层视角：上下文切换分「自愿」和「非自愿」两种——自愿是  
> 线程自己调用了会阻塞的系统调用主动让出；非自愿是被时间片用完  
> 强制抢占。Linux 上用 `vmstat 1` 能看到 `cs` 这一列（context  
> switches per second，每秒切换次数），线程数远超核心数时  
> 压测，这个数字会明显飙升，是「线程模型扛不住高并发」  
> 最直接的量化证据。

# 调度器怎么决定选谁

> 「就绪队列里那么多线程，核心空出来了该选谁」这道题，  
> 现代操作系统有一套相对公平的算法，业务代码不用手动干预，但要知道它存在。

Linux 现代版本用的调度算法叫 **CFS（Completely Fair Scheduler）**，  
核心思想很朴素：**尽量让每个线程获得的 CPU 时间“公平”**——  
调度器持续追踪每个就绪线程「已经用掉了多少 CPU 时间」，  
优先把核心让给「用得最少、最该轮到」的那个。这也解释了为什么  
「优先级（priority/nice 值）」能生效：优先级高的线程，  
调度器会倾向于让它更频繁、时间片更长地被选中，但即使优先级很低，  
只要不是显式设成「永不调度」，最终也会轮到，不会被彻底饿死。

业务代码几乎不需要手动调整线程优先级——Rust 标准库和 Tokio  
都没有直接暴露这个能力（需要时可以通过操作系统提供的  
`libc::setpriority` 之类的接口，但这已经是很少见的场景），  
调度器的默认公平策略对绝大多数业务服务已经足够好。

# 多核下的调度更复杂

> 有多个核心时，调度器还要多解决一个问题：把线程分给哪个核心。

多核场景下，调度器除了决定「选谁」，还要决定「分给哪个核心」，  
额外要考虑：

- **负载均衡**：尽量让每个核心手里的就绪线程数量差不多，  
  不能一个核心排长队、另一个核心闲着；
- **缓存亲和性（cache affinity）**：一个线程如果总在同一个核心上跑，  
  它的数据更可能还留在那个核心的 L1/L2 缓存里，切到另一个核心  
  会让这些缓存全部作废——调度器会「尽量」让线程留在原来的核心上，  
  除非负载均衡的收益更大。

这也是为什么 [《Rust 多线程与并发》](../concurrency/threads.md)  
「CPU 密集切块并行」那一节，建议开的线程数「等于逻辑核数左右」——  
线程数刚好等于核心数时，调度器几乎不需要在线程之间来回切换，  
每个核心专心跑一个线程，缓存亲和性最好；线程数一旦远超核心数，  
调度器就要频繁换人，前面几节讲的所有切换成本全部叠加上来。

# Go 与 Rust：两层调度的关系

> Go 和 Rust 都在“操作系统的调度”之上，可能还叠了一层“自己的调度”。

- Go 程序的 goroutine 数量通常远超 OS 线程数——Go 运行时自己  
  维护一个 **用户态调度器**，把成千上万个 goroutine 映射到  
  `GOMAXPROCS` 个（默认等于逻辑核数）OS 线程上运行，  
  这层调度完全在用户态发生，不需要每次都陷入内核。  
  完整机制见 [《Go 与 GMP 调度》](../concurrency/go-gmp.md)；
- Rust 标准库的 `std::thread` **没有** 这层用户态调度——  
  一个 `std::thread` 就直接对应一个真实的 OS 线程，本课讲的调度  
  机制（时间片、抢占、就绪/运行/阻塞）对它来说是唯一的一层；
- Rust 如果要类似 Go 那样「大量任务映射到少量线程」，  
  需要显式引入 Tokio——Tokio 自己也维护一层用户态调度，  
  区别是它是 **协作式** 的（任务只在 `.await` 处主动让出），  
  而这一课讲的 OS 调度器对线程是 **抢占式** 的（时间片到了强制换人）。  
  这条差异正是下一课 [《状态机与协程》](coroutine-state.md) 要  
  展开的核心问题：一个在异步任务里写死循环、从不 `.await`，  
  会彻底霸占底下的 OS 线程，Tokio 调度器拿它没有办法——  
  这是「协作式」调度天生的软肋，OS 线程的「抢占式」调度不会有  
  这个问题。

一张表总结这一课涉及的两层调度：

| | OS 内核调度线程 | 用户态调度任务（Go 运行时 / Tokio） |
| --- | --- | --- |
| 调度方式 | 抢占式（时间片到了强制换人） | Go：准抢占；Tokio：协作式（`.await` 处让出） |
| 死循环会怎样 | 到时间片会被强制换下去，不影响别的线程 | Tokio：霸占线程，其他任务停摆；Go：运行时有一定检测手段，但仍应避免 |
| 切换成本 | 1~10 微秒 + 缓存变冷（本课重点） | 纳秒级，用户态完成，不进内核 |

# 三句话带走

1. **线程只有三种状态：运行、就绪、阻塞** ——阻塞的线程不占核心也不  
   排队，事件发生才被内核挪回就绪；「阻塞」的确切含义是「这个线程  
   身上背的所有活全部停摆」，这是 async 铁律的物理原因。
2. **调度器靠定时器中断实现抢占**，时间片用完强制换人，  
   这保证了单核也能「同时」处理多个任务（并发的本源就是毫秒级轮转），  
   也保证了死循环线程不会永远饿死别人。
3. **上下文切换有直接成本（1~10 微秒）和间接成本（缓存变冷）**，  
   线程数远超核心数时这些成本会大量叠加；Go/Tokio 在 OS 调度之上  
   叠加了一层更轻的用户态调度，纳秒级切换、不进内核，  
   区别是 Tokio 的协作式调度怕死循环，OS 的抢占式调度不怕。

# 附：本课生词表

- **调度器（scheduler）** —— 操作系统内核里决定「核心空出来分给谁」的组件。
- **就绪 / 运行 / 阻塞** —— 线程的三种状态，阻塞状态不占核心、不参与排队。
- **时间片（time slice）** —— 调度器分配给一个运行中线程的一小段时间，用完强制换人。
- **抢占（preemption）** —— 时间片用完，调度器强制把当前线程切回就绪状态的动作。
- **定时器中断** —— 硬件时钟定期打断 CPU 执行、跳去运行调度器代码的机制，是抢占的物理实现。
- **上下文切换** —— 保存/恢复寄存器上下文，把核心从一个线程换给另一个线程；直接成本微秒级，间接成本是缓存变冷。
- **自愿 / 非自愿上下文切换** —— 线程主动阻塞让出 / 被时间片用完强制抢占。
- **CFS（Completely Fair Scheduler）** —— Linux 现代默认调度算法，核心思想是让每个线程获得的 CPU 时间尽量公平。
- **缓存亲和性** —— 调度器倾向于让线程留在同一个核心上跑，以保留缓存里的热数据。
- **协作式 / 抢占式调度** —— Tokio 任务只在 `.await` 处主动让出（协作式）；OS 对线程强制按时间片换人（抢占式）。
