# 阻塞的确切含义

> 代码：`code/os-blocking-io/`　运行：`cargo run -p os-blocking-io`

前置：[《调度与时间片》](scheduling.md)（线程状态：运行/就绪/阻塞）、  
[《一切皆文件描述符》](file-fd.md)（fd 是本课“读一个 fd”的操作对象）。  
本课要把 [《async 基础》](../async/basics.md) 那条铁律「绝不在 async 里阻塞线程」  
背后的操作系统事实，一次讲透：阻塞到底在阻塞什么、非阻塞长什么样、  
epoll 这类机制怎么解决问题，以及 Tokio 到底是怎么搭出来的。

# 阻塞的确切含义

> 复习并加深：「阻塞」不是一个模糊的形容词，它是内核对一个线程  
> 做的一件具体的事。

一个线程调用 `read(fd, buf)` 读一个暂时没有数据的 socket 时，  
内核做的事情精确来说是：

1. 发现这个 fd 现在没数据可读，无法立即完成这次系统调用；
2. 把这个线程标记成**阻塞态**，从运行队列里摘掉；
3. 把这个线程占用的 CPU 核心让给别的线程去用；
4. 数据到达时（比如网卡收到包），内核把这个线程改回**就绪态**，  
   等下一次被调度器选中，才会真正从 `read` 调用里返回。

```rust
let mut buf = [0u8; 1024];
let n = stream.read(&mut buf)?; // 没数据就"卡"在这一行——这一行背后是上面 1-4 步
```

**关键点、也是最容易被忽略的一点**：从「阻塞」到「恢复」之间，  
这个线程**什么都做不了**，包括它自己身上背着的所有其他任务。  
如果这是一个只服务一个连接的线程，问题不大；如果这个线程还  
背着另外 999 个任务的执行进度（后面会讲 Tokio 的工作线程正是  
这种情况），这 999 个任务会跟着一起被冻住。

🔬 底层视角：阻塞态的线程在 Linux 下会出现在 `ps` 的 `S`（sleeping）  
状态列，`top` 里 CPU 占用是 0——它确实没有在消耗任何 CPU，  
纯粹在等一个事件，这也是它和「死循环」的本质区别（死循环占核心、  
阻塞态不占）。

# 非阻塞长什么样

> 换一种问法：不要卡住我，没数据就立刻告诉我「现在还没有」。

把 fd 设成**非阻塞模式**后，同样的读操作在没数据时会立刻返回一个  
特殊错误，线程完全不会被挂起：

```rust
use std::io::ErrorKind;

stream.set_nonblocking(true)?;
match stream.read(&mut buf) {
    Ok(0) => { /* 对端关闭 */ }
    Ok(n) => { /* 真读到了 n 字节 */ }
    Err(e) if e.kind() == ErrorKind::WouldBlock => {
        // 现在没数据，线程完全没有被卡住，可以马上去做别的事
    }
    Err(e) => return Err(e.into()),
}
```

Go 里对应的底层 flag 是 `O_NONBLOCK`，Unix 上对应的错误码是  
`EWOULDBLOCK`/`EAGAIN`（历史原因，同一个错误两个名字）。

**非阻塞独自解决不了问题**：如果只是把 `read` 换成非阻塞，  
代码要么写成「读一次没数据就立刻重试」的忙轮询（**busy polling**）——  
白白烧 CPU；要么自己猜一个休眠间隔再重试——猜快了还是烧 CPU，  
猜慢了数据到了也要等下一轮才能发现，延迟变差。

这正是「非阻塞」这一个技术点自己回答不了的问题：  
**我什么时候该再试一次？** ——答案是下一节的多路复用。

----

# 多路复用一次问一堆

> 与其一个个猜「这个 fd 现在有数据吗」，不如让内核一次性告诉你  
> 「这一万个 fd 里，哪些已经就绪了」。

**IO 多路复用（I/O multiplexing）** 就是干这件事的系统调用族：  
你把一堆 fd 交给内核托管，调用一次，内核阻塞（这次阻塞的是  
「等这堆 fd 里任意一个就绪」，效率上和阻塞单个 fd 完全不同）  
直到至少一个有动静，然后一次性把就绪列表还给你。

不同操作系统给的具体实现不同，但解决的是同一个问题：

| 系统 | 机制 | 一句话直觉 |
| --- | --- | --- |
| Linux | **epoll** | 内核维护一个「关注列表」，就绪的自己进一个「就绪队列」，你去取现成的 |
| macOS/BSD | **kqueue** | 和 epoll 同一代思路，接口形状不同 |
| Windows | **IOCP** | 换了个方向：不是「问谁就绪了」，是「操作完成后回调通知你」（完成端口模型） |

早期还有一种更原始的多路复用叫 `select`/`poll`：每次调用都要把  
整个 fd 列表重新传给内核扫一遍，fd 数量一大就很慢（O(n)）；  
epoll/kqueue 把「注册关注列表」和「取就绪结果」拆成两步，  
后者只需要 O(就绪数量)，这是它们能撑住海量连接的关键差别。

🔬 底层视角：epoll 的核心心智模型可以简化成——

```
epoll_create()               创建一个"监控本"
epoll_ctl(ADD, fd, 关心什么)  把一堆 fd 登记进监控本（一次性成本）
loop {
    epoll_wait()              阻塞直到监控本里有 fd 就绪，返回就绪列表
    for 就绪的 fd {
        read/write(fd)         只处理真正有数据的，不用挨个去问
    }
}
```

**一个线程，通过一次 `epoll_wait`，就能同时看住上万个连接**——  
这是 C10K 问题（[《调度与时间片》](scheduling.md) 提到过）  
的标准解法，也是 Tokio 和 Go netpoller 的地基。

# IO 模型三代对照

> 把前面三节串成一张进化表，看清楚每一代解决了上一代的什么问题。

| 代际 | 模型 | 一个线程能看住几个连接 | 缺点 |
| --- | --- | --- | --- |
| 第一代 | 阻塞 IO + 每连接一线程 | 1 | 线程数 = 连接数，几千并发线程成本就压垂了 |
| 第二代 | 非阻塞 + 多路复用（epoll/kqueue/IOCP） | 上万 | 代码要手写「状态机」，回调/轮询逻辑很难写、更难读 |
| 第三代 | 异步运行时（Tokio / Go netpoller） | 上万 | 需要一个运行时，写法上要理解 `.await`/goroutine |

第三代不是发明了新的操作系统机制，而是**把第二代包装成人能写的样子**：  
你写 `stream.read(&mut buf).await` 这种看起来顺序执行的代码，  
运行时在背后把它翻译成「把 fd 设成非阻塞 → 注册进 epoll →  
没就绪就把当前任务挂起，去处理别的任务 → epoll 通知就绪后  
用 `Waker` 把任务重新唤醒」。

这张表和 [《从零手写 HTTP》](../http/http-from-scratch.md) 的三步走  
完全对应：第 0/1 步是第一代，第 2 步的 tokio 任务版本就是第三代。

# Tokio 等于什么

> 拆开一个常被神化的等式：Tokio 底下没有魔法，就是这三样东西的组合。

**Tokio ≈ epoll/kqueue/IOCP 多路复用 + 一个操作系统线程池 + 一堆状态机**

- **多路复用**：Tokio 内部有一个「reactor」，负责调用  
  epoll_wait/kqueue/IOCP，拿到就绪列表；
- **线程池**：Tokio 的多线程运行时默认起「逻辑核数」个  
  `std::thread`，这些线程负责真正执行代码——它们本身就是  
  [《Rust 多线程与并发》](../concurrency/threads.md) 讲的那种线程，  
  没有任何特殊之处；
- **状态机**：每个 `async fn` 被编译器改写成一个状态机  
  （[《调度与时间片》](scheduling.md) 「状态机是什么」  
  一节详细讲过），reactor 收到就绪通知后，就去唤醒对应的状态机，  
  丢给线程池里某个空闲线程「推一下」（调用 `poll`）。

```rust
// 你写的（看起来顺序执行）：
let n = stream.read(&mut buf).await?;

// 运行时在背后大致做的（简化心智模型，非真实源码）：
// 1. 尝试非阻塞 read；WouldBlock 就把 fd 注册进 epoll，返回 Pending
// 2. 当前任务（状态机）被挂起，线程去处理下一个就绪的任务
// 3. epoll_wait 报告这个 fd 就绪 → 用 Waker 把任务标记"可以再推一下了"
// 4. 某个工作线程空出来时，捡起这个任务，从上次 Pending 的地方继续
```

Go 的 **netpoller** 走的是同一条路：goroutine 表面上写阻塞风格的  
`conn.Read`，Go 运行时在背后把每个 `net.Conn` 的 fd 设成非阻塞，  
用 epoll（Linux）/kqueue（macOS）托管，`Read` 卡住时挂起的是  
**goroutine**（用户态调度单元），不是操作系统线程——[《Go 并发实现  
（GMP）》](../concurrency/go-gmp.md) 讲的正是这台调度器。

两边的差异不在「用没用 epoll」（都用了），而在「暂停点长什么样」：  
Go 保留 goroutine 真实的栈和寄存器快照，随处能停；Rust 把暂停点  
编译成显式的状态机字段，只能在 `.await` 处停。这是  
[《调度与时间片》](scheduling.md) 「状态机是什么」一节  
讲过的取舍分野，这里只是指出双方在「多路复用」这一层其实用的是  
同一套操作系统机制。

# 阻塞在 async 里为什么是罪

> 把前面所有知识拼起来，回答「为什么绝不能在 async 里阻塞线程」。

Tokio 的一个工作线程，同一时刻可能背着成百上千个任务（状态机）  
排队等着被推进。如果某个任务里的代码调用了一个**真阻塞**的操作  
（比如误用了 `std::thread::sleep`，或者调用了一个内部走同步阻塞  
IO 的库函数），会发生什么：

- 这次调用会让**整个工作线程**被内核挂起（回到第一节的定义）；
- 这个线程背着的所有其他任务，包括和这次阻塞毫无关系的请求，  
  **全部陪着一起停摆**，直到这次阻塞结束；
- 症状是：平时响应很快的接口，偶尔集体卡顿一下——排查起来  
  很反直觉，因为报错日志往往指不出真正阻塞的那一行。

这就是为什么这条铁律排在 [《async 基础》](../async/basics.md) 最前面：  
它不是风格建议，是这一课讲的操作系统机制决定的硬约束。  
真需要做同步阻塞的事（重计算、调用同步库），正确做法是  
`spawn_blocking`，把它丢给专门的阻塞线程池，不占用处理 IO  
任务的工作线程。

# 动手实验

1. 运行 `cargo run -p os-blocking-io`，对照观察阻塞版和非阻塞版  
   读一个暂时没数据的 socket 时的返回结果差异；
2. 写一个最小的阻塞 TCP 服务端，只 `accept` 一个连接就 `read`，  
   在没有客户端连接时用 `ps`（Linux，看 `S` 状态列）或任务管理器  
   观察这个线程确实是「休眠」而不是「忙转」（CPU 占用 0）；
3. 把同一个 socket 换成非阻塞模式，在没数据时用一个死循环反复  
   `read`，观察 CPU 占用飙到接近 100%——亲眼看到「忙轮询」的代价；
4. 用 `strace -e trace=network,epoll_wait <你的 axum 程序>`  
   （仅 Linux）观察一个 tokio 服务真实调用 `epoll_wait` 的日志，  
   验证「Tokio 底下就是 epoll」这句话不是比喻。

# 三句话带走

1. **阻塞 = 线程被内核挂起、让出核心、期间它背的一切全部停摆**——  
   非阻塞让「没数据」立刻返回而不是卡住，但独自解决不了「该什么时候  
   再试」的问题。
2. **多路复用（epoll/kqueue/IOCP）一次系统调用返回一批就绪 fd**，  
   是「一个线程看住上万连接」的操作系统机制，Tokio 和 Go netpoller  
   都建在它上面。
3. **Tokio ≈ 多路复用 + 线程池 + 状态机**——`.await` 处没就绪就把  
   任务挂起还给线程去处理别人，就绪后被唤醒继续；在这套体系里  
   真阻塞一个工作线程，代价是背着的所有任务集体停摆。

# 附：本课生词表

- **阻塞（blocking）** —— 操作没法立即完成时，内核把线程挂起、  
  让出核心，直到条件满足才恢复。
- **非阻塞（non-blocking）** —— 操作没法立即完成时立刻返回  
  特定错误（`WouldBlock`/`EAGAIN`），不挂起线程。
- **忙轮询（busy polling）** —— 不停重试非阻塞操作，浪费 CPU  
  却不一定拿到更低延迟。
- **IO 多路复用** —— 一次系统调用查询一批 fd 的就绪状态，  
  是「一个线程看住多连接」的核心机制。
- **epoll / kqueue / IOCP** —— Linux / macOS-BSD / Windows 各自的  
  多路复用实现，接口不同、解决的问题相同。
- **reactor** —— 异步运行时里负责调用多路复用、把就绪事件翻译成  
  「唤醒某个任务」的组件。
- **`Waker`** —— Rust Future 生态里，reactor 用来通知「这个任务  
  值得再推一下」的句柄。
- **netpoller** —— Go 运行时内置的多路复用组件，让 goroutine  
  写出「看起来阻塞」但实际非阻塞的网络代码。
- **`spawn_blocking`** —— Tokio 提供的「把同步阻塞代码丢进专用  
  线程池」的桥，避免拖垮处理 IO 的工作线程。
