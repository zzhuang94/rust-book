# 程序如何跑起来

> 代码：`code/os-computer-basics/`　运行：`cargo run -p os-computer-basics`
> （本章代码是一个只打印几行「我是谁、我在哪」信息的最小程序，
> 配合正文用来对照「一个程序启动时发生了什么」）

> 这是 `docs/os/` 系列的第一课。如果你只写过业务代码（写 handler、
> 调接口、查数据库），从没关心过“程序是怎么跑起来的”，这一课就是为你写的。
> 宁可啰嗦，也要把每个词讲透——后面七课全部建立在这一课的直觉上。

你每天都在做一件事：写代码、编译（或者对 Go/Python 来说是直接跑）、
运行、看结果。这个过程被工具链藏得很好，好到你可能从没问过：
**我的代码到底是怎么“变成”电脑里正在跑的东西的？**

这一课不讲“背电脑组成原理教科书”，只讲一件事：**跑起来的程序，
到底是哪几个部件在合作，各自负责什么**。想清楚这个，后面「CPU 与内存」
「磁盘 IO」「进程与线程」这些课才有地基可以站。

# 一台电脑只干两件事

> 抛开所有细节，一台电脑（不管是你的笔记本还是云上的服务器）本质上只做两件事：
> **算** 和 **存**。

- **算**：把一个数变成另一个数——加减乘除、比较大小、判断条件。这件事只有
  一个部件在做：**CPU（中央处理器）**；
- **存**：把数据放在某个地方，需要的时候能再取出来。这件事分成两层——
  **内存（RAM）** 存“正在用”的数据，**存储（磁盘/SSD）** 存“暂时不用但不能丢”的数据。

你写的所有业务代码，最终都会被翻译成「从存的地方取数 → CPU 算一下 →
把结果放回存的地方」这样的循环，翻来覆去、亿万次重复。这个直觉最早由
数学家冯·诺依曼（John von Neumann）在 1945 年写清楚，后来的电脑几乎
全部照着这个结构造——所以这个结构现在叫 **冯·诺依曼架构**。

# 冯·诺依曼的核心洞察

> 在冯·诺依曼之前的机器，程序是硬接线接出来的，改程序要重新接线。
> 他的洞察只有一句话，却改变了一切：**程序本身也是数据**。

具体说：一段程序（一串指令）和它要处理的数据，**用同一种方式存放在
同一个内存里**。CPU 不区分“这块内存是指令”还是“这块内存是数据”，
它只知道「去某个地址取一个数」——取出来当指令执行，还是当数据处理，
取决于 CPU 当前在干哪件事。

这带来一个直接后果，也是你每天都在享受的便利：**换程序不用换硬件，
只要把新程序（新的一串数据）装进内存就行**。你今天用浏览器、明天用
Rust 编译器，用的是同一块 CPU、同一块内存——区别只是内存里当前
装的是哪个程序。这在今天听起来是废话，但在 1945 年之前完全不是。

> 🔬 底层视角：这个「程序即数据」的设计也是安全漏洞的祖师爷之一——
> 如果一个 bug 让攻击者能把「数据」写到本该只放「指令」的地方，
> CPU 会毫不犹豫地把这些数据当指令执行，这就是「缓冲区溢出攻击」的
> 底层原理。Rust 编译器对内存越界的严格检查，某种意义上就是在
> 堵这个从 1945 年就存在的口子。

# 五个部件各管一件事

> 抛开教科书的名词堆砌，业务程序员只需要认清五个角色，
> 以及它们之间谁找谁办事。

| 部件 | 一句话职责 | 你在代码里对应看到的东西 |
| --- | --- | --- |
| CPU | 执行指令、做运算、做判断 | 你写的每一行逻辑代码 |
| 内存（RAM） | 临时存放正在跑的程序和数据，断电即丢 | 变量、栈、堆、`Vec`/`slice` |
| 存储（磁盘/SSD） | 长期存放数据，断电不丢 | 文件、数据库、日志 |
| 输入输出设备（网卡/键盘/显示器…） | 让电脑和外部世界交换数据 | socket、终端输出、鼠标事件 |
| 总线（Bus） | 连接以上几个部件的“数据公路” | 你看不到，但决定了各部件之间的速度上限 |

这五个角色里，业务程序员打交道最多的是前三个（CPU、内存、存储），
第四个（网络这类 IO 设备）在 [《分层是为了解耦》](../network/layers.md)
整章都在讲，第五个（总线）你几乎永远不用关心，但它解释了一个关键事实：
**部件本身多快没用，数据在部件之间搬运的速度才是真正的瓶颈**——
这句话在下一课「CPU 与内存」会具体展开成「为什么 CPU 比内存快 100 倍，
中间要垫几层缓存」。

# 程序在磁盘上是静止的

> 先分清两个总被混用的词：**程序** 和 **进程**。这一课只建立直觉，
> 完整定义留给下一课 [《进程与线程》](process-thread.md)。

你执行 `cargo build` 之后，磁盘上多出一个可执行文件（Linux/macOS 上
没有后缀，Windows 上是 `.exe`）。这个文件此刻是 **静止的**——它就是
一堆字节，躺在磁盘的某个扇区里，没有人在“执行”它，和一个 `.txt` 文件
在“存在方式”上没有本质区别。

这个静止的文件里装了什么？主要是两样东西：

- **指令**：CPU 能听懂的二进制编码，一条指令对应你源代码里的某个操作
  （加法、跳转、调用函数……）；
- **只读数据**：编译期就能确定的常量，比如你代码里写的字符串字面量。

**这个静止的文件，专业术语叫“程序”（program）。它变成“正在跑的东西”
的那一刻，才叫“进程”（process）**——这一步具体怎么发生，是下一节的重点。

# 运行一次经历了什么

> 你在终端敲下 `cargo run -p os-computer-basics`，回车到看到输出，
> 中间发生了一整套协作。拆开看，每一步都不神秘。

按时间顺序拆解（以 Linux 为例，Windows 机制类似只是叫法不同）：

1. **shell 收到你的命令**：你敲的 `cargo run` 实际上是 cargo 先编译、
   再帮你执行编译出来的那个可执行文件；
2. **操作系统创建一个新进程**：内核在内存里给这个新进程分配一份
   **独立的地址空间**（下一课细讲），这是这个程序此刻唯一的“家”；
3. **加载器（loader）把磁盘上的程序搬进内存**：静止在磁盘上的那些
   指令和数据，被复制（准确说是按需映射）进这个新地址空间；
4. **CPU 被指向程序的入口地址**：CPU 的「下一条指令在哪」这个寄存器
   被设置成程序的第一条指令的位置，CPU 开始从这里逐条取指令执行；
5. **执行你写的 `main` 函数之前，运行时先做点准备工作**：Rust 的
   标准库要初始化栈保护、处理命令行参数等，这些代码也在这个可执行
   文件里，只是不是你直接写的；
6. **你的业务逻辑开始跑**：`main` 函数被调用，你写的 `println!` 之类
   的代码逐条执行；
7. **进程结束，资源被回收**：`main` 返回后，操作系统回收这个进程占用的
   内存、关闭它打开的文件和网络连接，这个进程从“活着”变回“不存在”。

用一张图总结这整个过程：

```
磁盘上的可执行文件（静止的字节）
        │  ① 操作系统创建进程 + 分配地址空间
        │  ② 加载器把文件内容装进这个地址空间
        ▼
内存里的进程（活的、有地址空间的、CPU 正在执行它）
        │  ③ CPU 从入口地址开始逐条取指令执行
        │  ④ 运行时初始化 → 调用你的 main
        ▼
你写的业务代码开始跑，直到 main 返回
        │  ⑤ 操作系统回收这个进程的全部资源
        ▼
进程消失，电脑恢复到运行之前的状态
```

对照 Go：`go run main.go` 会先在临时目录编译出一个二进制、再执行它，
后半段（③④⑤）和 Rust 完全一样——毕竟这是操作系统的工作，
和你用什么语言写业务逻辑没有关系。区别只在编译这一步：Go 的编译
出于设计目标，通常比 Rust 快很多，但产出的东西性质相同，
都是「一个静止的可执行文件，等着被操作系统装载执行」。

# 业务代码在整机里的位置

> 你写的每一行 Rust/Go 代码，其实叠在一座很高的楼上，
> 楼底下是硬件，楼顶上才是你。

分层来看，从硬件到你的业务代码，中间垫了这么几层：

```
┌─────────────────────────────────────┐
│  你的业务代码（handler、SQL、业务规则）  │  ← 你每天写的东西
├─────────────────────────────────────┤
│  语言运行时（Rust std / Go runtime）    │  ← 内存分配、GC（Go）、goroutine 调度（Go）
├─────────────────────────────────────┤
│  操作系统内核（进程、线程、文件、网络）    │  ← 本系列课程的主角
├─────────────────────────────────────┤
│  硬件（CPU、内存、磁盘、网卡）           │  ← 真正“算”和“存”发生的地方
└─────────────────────────────────────┘
```

每一层的存在都是为了让上面一层 **不用关心** 下面一层的细节：

- 你写 `HashMap::new()` 时不用关心内存具体从哪来——Rust 运行时帮你
  向操作系统申请；
- 操作系统帮你申请内存时不用关心物理内存具体在哪个芯片的哪个位置——
  硬件（内存控制器）帮它翻译；
- 你写 `TcpStream::connect(...)` 时不用关心网卡驱动怎么把字节送上
  网线——操作系统内核的网络栈帮你做完了（详见
  [《分层是为了解耦》](../network/layers.md)）。

这也是为什么本系列课程值得学：**你现在只需要往下看一层（操作系统），
就能解释很多之前“背下来但不知道为什么”的现象**——比如为什么一个
线程要占 8MB 内存、为什么读磁盘比读内存慢一万倍。再往下（硬件、
芯片设计）就不是业务程序员的日常必修课了，可以先放一放。

# 内存与存储的分工

> 「内存」和「存储」这两个词业务代码里天天说，但很多人说不清楚
> 它俩到底谁管什么。这一课先建立直觉，细节留给专门的课。

一句话区分：**内存快、贵、断电就丢；存储慢、便宜、断电不丢**。

| | 内存（RAM） | 存储（磁盘/SSD） |
| --- | --- | --- |
| 速度 | 纳秒级 | 微秒~毫秒级（慢 1000~100000 倍） |
| 断电后 | 数据全部消失 | 数据保留 |
| 典型容量 | 单机几十 GB | 单机几百 GB 到几十 TB |
| 业务代码里对应 | 变量、`Vec`、进程的堆栈 | 数据库文件、日志文件、配置文件 |
| 一句话 | 只服务“正在跑的这一刻” | 服务“明天、重启后、换一台机器后” |

这张对比表解释了一个你天天做但可能没细想过的动作：**为什么要把数据
“存到数据库”**？因为你程序里的变量活在内存里，进程一退出（哪怕是
正常重启）内存就被操作系统回收、数据全部消失；只有写进磁盘（数据库
落盘、写日志文件）的数据，才能在进程消失之后继续存在。这个「落盘」
动作具体怎么发生、有多贵，是专门一课
[《磁盘与 IO 直觉》](disk-io.md) 的主题。
内存内部还要再细分几层（CPU 里的寄存器、缓存、和真正的内存条），
这是下一课 [《CPU 与内存》](cpu-memory.md) 的主题。

# 一条指令的极简旅程

> CPU 具体怎么“执行一条指令”？这里先给最粗的直觉，
> 详细的寄存器视角在下一课展开。

CPU 内部有一个不停循环的动作，叫 **取指-译码-执行**（fetch-decode-execute）：

1. **取指（Fetch）**：CPU 根据「下一条指令在哪」这个内部记录，
   去内存里把那条指令的二进制编码取过来；
2. **译码（Decode）**：CPU 内部的电路把这串二进制翻译成「要做什么」
   （加法？跳转？读内存？）；
3. **执行（Execute）**：真正做这件事，可能会读写内存、可能会改变
   「下一条指令在哪」这个记录（这就是 `if`/循环/函数调用的底层实现）；
4. 回到第 1 步，取下一条指令，无限循环，直到进程被结束。

你写的一行 Rust 代码，编译后往往对应几条到几十条这样的指令。
`if`/`for`/`match` 这些控制流关键字，本质上都是在操纵第 3 步里
「下一条指令在哪」这个记录——这也是为什么 CPU 记录“执行到哪了”这
件事如此重要，它会在下一课、以及后面讲「线程切换」「协程状态机」的
课里反复出现。

> 🔬 底层视角：这个「下一条指令在哪」的记录，专业名字叫
> **程序计数器（Program Counter，PC）**，是 CPU 内部一个专门的
> 寄存器。下一课会把它和其他寄存器放在一起细讲。

# 用 strace 亲眼看一次

> 不用改代码，用一个系统自带的工具，就能看到「程序运行」这件事
> 背后操作系统到底做了多少工作。

Linux 上的 `strace` 能记录一个程序运行期间发起的每一次「找操作系统
帮忙」的请求（这类请求叫 **系统调用**，下一课的姐妹课
[《用户态与内核态》](user-kernel.md) 会专门讲）。跑一次：

```bash
strace -c ./target/debug/os-computer-basics
```

输出的末尾会有一张统计表，大致是这样（数字因机器而异）：

```
% time     seconds  usecs/call     calls    syscall
------ ----------- ----------- --------- ----------------
 35.00    0.000123          12        10 mmap
 20.00    0.000070           7        10 munmap
 15.00    0.000052          52         1 write
 10.00    0.000035           3        11 openat
  ...
```

哪怕你的代码只有一句 `println!("hello")`，操作系统在背后已经悄悄
做了几十次「帮忙」：给这个进程分配内存页（`mmap`）、打开动态链接库
（`openat`）、真正把字符串送到终端（`write`）……这些工作你一次都没
写在代码里，全部是运行时和操作系统替你完成的。这就是「业务代码在
整机里的位置」这一节说的「每一层帮你藏掉了下一层的细节」的直接证据。

Windows 上没有 `strace`，但有功能类似的工具，比如
[Process Monitor](https://learn.microsoft.com/sysinternals/downloads/procmon)，
效果同理：能看到一个程序运行期间访问了哪些文件、注册表、网络。

# Go 与 Rust：同一件事的两种走法

> 「程序如何跑起来」这件事，Go 和 Rust 的差异集中在「谁负责运行时初始化」，
> 骨架完全一样。

| 环节 | Rust | Go |
| --- | --- | --- |
| 编译产物 | 一个静态或动态链接的可执行文件 | 一个默认静态链接的可执行文件 |
| 启动时的运行时初始化 | 栈保护、`panic` 处理钩子等，很轻 | 更重：初始化 GMP 调度器、GC、多个后台 goroutine |
| `main` 之前有没有额外线程 | 一般没有 | 有——垂圾回收、调度器的后台工作 |
| 进程结束方式 | `main` 返回或 `process::exit` | `main` 返回或 `os.Exit` |

这也是本系列课程之后会反复用到的对照方式：Go 的运行时替你做了更多
「隐形工作」（垂圾回收、协程调度），代价是这些工作本身要占用 CPU
和内存；Rust 把这些工作尽量挪到编译期或者交给你显式调用，
换来的是更少的「隐形开销」，但你需要更了解底层机制才能写对——
这正是这一整个 `docs/os/` 系列存在的理由。

# 常见误区先澄清

> 「程序」「软件」「进程」「应用」这几个词业务场景里经常混用，
> 这一节把最容易搞混的两个误区单独拎出来讲清楚。

**误区一：“打开一个程序”和“运行一个进程”是一回事。** 不完全对——
你打开三次同一个 App（或者三次运行同一个 `.exe`），操作系统会创建
**三个独立的进程**，它们共享同一份磁盘上的程序文件，但运行起来之后
互不相干，一个卡死不会拖累另外两个。这也是为什么任务管理器里，
同一个程序名字后面常常跟着好几个不同的 PID——它们是同一个「程序」
派生出的多个不同「进程」。

**误区二：程序运行期间，如果我改了磁盘上的那个可执行文件，
运行中的进程会不会突然“变样”？** 答案是不会。这个进程早在启动时
就已经把代码内容装载（或者按需映射）进了自己的地址空间，之后
磁盘上那份文件再怎么改，都不会影响这个已经在跑的进程——它读写的
是内存里那份「装载好的副本」，不是磁盘上的实时内容。这也解释了
一个热更新场景里常见的操作：Linux 上滚动升级服务时，可以先把新的
可执行文件写到磁盘的同一个路径，正在跑的旧进程完全不受影响，
继续用它启动时装载的那份旧代码执行，直到你重启这个进程，
它才会重新走一遍「加载」流程，用上磁盘上的新内容
（Windows 上情况略有不同，运行中的可执行文件往往会被系统锁定，
不允许直接覆盖写，这也是 Windows 装完更新经常要求重启的原因之一）。

# 动态链接库是怎么回事

> 上面用 `strace` 看到过程序运行时会 `openat` 一些 `.so` 文件，
> 这一节把这件事背后的「静态链接」和「动态链接」讲透。

回顾「程序在磁盘上是静止的」那一节：可执行文件里装了指令和只读数据。
但你的代码几乎不可能什么都自己写——`println!` 背后要用到操作系统
提供的输出能力，这些「别人写好的代码」要怎么进到你的程序里？
有两种做法：

- **静态链接（static linking）**：编译时就把用到的库代码直接
  **拷贝一份** 塞进你的可执行文件里。产物是一个「胖」文件，
  运行时不需要再找任何外部文件，哪里都能跑，缺点是占磁盘空间、
  多个程序各自都拷贝一份同样的库代码，白白浪费；
- **动态链接（dynamic linking）**：可执行文件里只留一句「我需要
  `libc.so` 这个库，运行时帮我找到它」，真正的库代码是一个独立的
  `.so`（Linux）/`.dll`（Windows）文件，多个程序可以在运行时共享
  同一份库文件，省磁盘、省内存（同一份库代码在物理内存里只驻留
  一份，被多个进程共同映射），代价是这个库文件必须真实存在于
  目标机器上，找不到就是上面 `strace` 输出里那几十次 `openat`
  失败或成功的直接来源。

```bash
# Linux 上查看一个可执行文件依赖了哪些动态链接库
ldd ./target/debug/os-computer-basics

# Windows 上等价的工具（需要安装 Sysinternals）
# Listdlls.exe 或者用 Dependencies 这类第三方工具
```

对照 Go/Rust：Go 编译产物**默认是静态链接**（几乎不依赖系统上的
任何动态库，这也是为什么一个 Go 二进制文件复制到另一台同架构的
Linux 机器上，大概率直接能跑）；Rust 编译产物**默认动态链接
C 标准库（`glibc`）**，如果目标机器上的 `glibc` 版本差异太大，
可能会遇到 `.so` 版本不兼容的问题——常见的解决办法是换成
`x86_64-unknown-linux-musl` 这个编译目标，产出完全静态链接的
二进制文件，这在打包进最小化的 Docker 镜像（比如 `scratch` 基础
镜像）时特别常用，因为镜像里连 `glibc` 都不存在。

# 程序起不来时怎么查

> 把前面几节串起来，变成一张排错清单——「程序运行不起来」
> 这句抱怨，几乎总能对应到「运行一次经历了什么」里的某一步失败了。

| 你看到的错误 | 大致原因 | 对应哪一步 |
| --- | --- | --- |
| `command not found` / `'xxx' 不是内部或外部命令` | 磁盘上根本没有这个文件，或者不在 `PATH` 里 | 第 0 步：shell 还没找到要执行的文件 |
| `Permission denied` | 文件存在，但没有可执行权限 | 操作系统拒绝创建这个进程 |
| `exec format error` | 二进制格式和当前 CPU 架构不匹配（比如把 x86 编译的程序拿到 ARM 芯片的机器上跑） | 第 4 步：CPU 根本读不懂这些指令编码 |
| `error while loading shared libraries: libxxx.so: cannot open shared object file` | 动态链接依赖的库文件找不到 | 第 3 步：加载器找不到需要一并装载的动态库 |
| 程序启动但立刻 `panic`/崩溃退出，退出码非 0 | 到达了 `main` 函数，但业务逻辑本身出了问题 | 第 6 步之后，已经不是这一课的范畴，是你的代码逻辑 |

这张表的用法：**先分清是“连进程都没建立起来”还是“进程建立起来了
但代码逻辑出错”**——前几行属于本课讲的「运行一次经历了什么」的
第 0~4 步，往往是环境、权限、依赖问题；最后一行已经是业务代码
本身的问题，排查方向完全不同。养成看到报错先问「这卡在哪一步」的
习惯，能省下大量瞎猜的时间。

# 容器启动也是这一课

> 你可能天天在用 `docker run`，这个动作在更底层，
> 走的还是这一课讲的「运行一次经历了什么」那套流程，只是多垫了一层。

`docker run my-image` 大致发生了什么，可以对照本课的七步来理解：

1. Docker 先确保镜像内容（本质是一份打好包的文件系统，包含你的
   可执行文件和它依赖的库）已经在本机磁盘上；
2. Docker 调用操作系统提供的隔离机制（**namespace**、**cgroup**，
   这部分是容器和「直接跑一个进程」最大的区别，专门一课
   [《容器与 cgroup》](cgroup-container.md) 会展开），
   为将要创建的进程围出一个「看起来像独立系统」的沙盒；
3. 在这个沙盒里，操作系统还是老老实实走一遍本课的第 2~7 步：
   创建进程、分配地址空间、加载器装载镜像里的可执行文件、
   CPU 指向入口地址、运行你的业务代码；
4. 容器里的这个进程退出后，Docker 认为「这个容器结束了」——
   **一个容器本质上就是一个（或几个）被特殊隔离包裹起来的进程**，
   不是什么全新的运行方式。

理解了这一点，很多容器相关的日常疑惑会豁然开朗：为什么
`docker exec` 能让你「进入」一个容器（本质是在同一个 namespace
里再起一个进程）；为什么容器启动比裸机跑一个程序慢一点点
（多了创建 namespace/cgroup 这一步）；为什么容器「退出」的瞬间，
容器里所有进程都会被清理干净（因为它们本质是一批进程，
和普通进程退出没有本质区别，只是清理动作由容器运行时代劳了）。

# 动手实验

1. 在你的机器上跑一个最简单的 `fn main() { println!("hello"); }`，
   Linux/macOS 下用 `strace -c` 或 `ltrace -c` 观察它发起了哪些系统调用；
   Windows 下用 Process Monitor 观察它访问了哪些文件；
2. 用文件管理器或 `ls -lh` 看看你编译出的可执行文件有多大，
   再用 `file ./target/debug/os-computer-basics`（Linux/macOS）看看
   这个文件被操作系统认成什么类型；
3. 同一份代码分别用 `cargo build`（debug）和 `cargo build --release`
   编译，对比两个产物的体积，感受“优化”对静止在磁盘上的字节数量的影响；
4. 打开任务管理器（Windows）或 `htop`/`top`（Linux/macOS），
   跑一个会一直循环的小程序，观察它在“进程列表”里出现，
   再把它杀掉，观察它从列表消失——这就是“进程从活着变回不存在”的
   直接体感。

# 三句话带走

1. **一台电脑只做两件事：算（CPU）和存（内存+存储）**；冯·诺依曼的
   核心洞察是「程序和数据用同一种方式存在同一个内存里」，这让「换程序」
   变成了「换内存里的内容」，不需要换硬件。
2. **程序是磁盘上静止的字节，进程是它被操作系统装进内存、CPU 开始逐条
   执行之后的“活的实例”**——操作系统负责创建地址空间、装载代码、
   把 CPU 指向入口，这一整套动作你写的每一行业务代码都在“借用”。
3. **业务代码只是叠在很高的楼顶上的一层**：往下依次是语言运行时、
   操作系统内核、硬件，每一层的存在都是为了让上一层不用关心下一层的
   细节——这也是接下来七课要逐层往下拆开看的原因。

# 附：本课生词表

- **冯·诺依曼架构** —— 程序和数据用同一种方式存放在同一个内存里，
  CPU 按「取指-译码-执行」循环逐条处理的机器结构，现代绝大多数电脑都这样造。
- **CPU（中央处理器）** —— 真正做运算和判断的部件，一次只能执行一条指令流。
- **内存（RAM）** —— 临时存放正在跑的程序和数据的地方，速度快、断电即丢。
- **存储（磁盘/SSD）** —— 长期存放数据的地方，速度慢、断电不丢。
- **总线（Bus）** —— 连接 CPU、内存、存储等部件的数据通道，决定了搬运数据的速度上限。
- **程序（program）** —— 磁盘上静止的可执行文件，本身不是“正在跑的东西”。
- **进程（process）** —— 程序被操作系统装进内存、开始执行后的“活实例”，
  完整定义见 [《进程与线程》](process-thread.md)。
- **加载器（loader）** —— 操作系统里负责把可执行文件的内容搬进新进程地址空间的部件。
- **程序计数器（PC）** —— CPU 内部记录「下一条指令在哪」的寄存器，
  下一课 [《CPU 与内存》](cpu-memory.md) 会展开。
- **系统调用（syscall）** —— 程序向操作系统内核“求助”的标准通道，
  详见 [《用户态与内核态》](user-kernel.md)。
- **取指-译码-执行** —— CPU 内部不停循环的三个动作，是“CPU 在执行指令”这句话的具体展开。
- **静态链接 / 动态链接** —— 把依赖的库代码编译时拷贝进可执行文件 /
  运行时才去找共享的 `.so`/`.dll` 文件，前者产物大但独立、后者省空间但要求目标机器有对应库。
- **namespace / cgroup** —— Linux 提供容器隔离的两大机制，一个容器本质上是被
  它们包裹起来的一批普通进程，详见 [《容器与 cgroup》](cgroup-container.md)。
