以及“系统调用太多会变慢”这句话背后的具体数字。

# CPU 有两种运行模式

> 不是所有代码都被同等信任——CPU 硬件本身就把「谁能干什么」分成了两档。

现代 CPU 在硬件层面支持至少两种运行模式（专业叫「特权级」，
Intel 的实现里叫 ring 0~3，实践中主要用到两档）：

- **内核态（kernel mode）**：能执行任何指令，能直接访问所有硬件
  （内存管理单元、磁盘控制器、网卡……），操作系统内核的代码跑在这一档；
- **用户态（user mode）**：受限制的模式，不能直接访问硬件，
  不能执行某些危险指令（比如直接修改内存映射关系）。**你写的
  业务代码、Rust/Go 运行时，全部跑在用户态**。

为什么要这么分？想象一下如果没有这层限制：你的业务代码里一个手滑的
bug，理论上就能直接改写磁盘任意扇区的数据，或者让网卡疯狂发包——
这在多任务、多用户共享一台机器的世界里是不可接受的。**用户态这层
限制，本质是操作系统对「不受信任的应用代码」设的护栏**：你只能
在自己的沙盒（进程的地址空间）里折腾，任何需要碰硬件、碰其他进程
资源的操作，必须经过一个专门的、受控的入口——这就是下一节的主角。

# 系统调用是唯一的门

> 用户态代码想要“碰”硬件或者“碰”操作系统管理的资源，
> 只有一条合法的路：发起一次系统调用（syscall）。

**系统调用** 是用户态程序请求内核态代码帮忙做事的标准机制。
你天天在用但可能没意识到是系统调用的例子：

| 你写的代码 | 底层对应的系统调用（Linux 举例） |
| --- | --- |
| 打开一个文件 | `open` |
| 读/写文件或 socket | `read` / `write` |
| 建立 TCP 连接 | `socket` + `connect` |
| 申请一块内存 | `mmap` / `brk`（`Vec` 扩容背后就在用它） |
| 创建一个线程 | `clone` |
| 睡眠 | `nanosleep` |
| 查询当前时间 | `clock_gettime` |

发起一次系统调用，CPU 要经历一次 **模式切换**：从用户态陷入内核态，
内核代码检查这次请求合不合法、真正去操作硬件或者内核维护的数据结构，
处理完毕后再切回用户态，把结果交还给你的代码。这个「陷入-处理-切回」
的过程，专业术语叫 **trap**（有的 CPU 架构上对应的指令叫
`syscall`/`int 0x80`/`svc`，名字不同，机制一样）。

```
用户态：你的代码在跑
   │  调用 write()（Rust/Go 标准库把这一步封装好了）
   ▼
CPU 执行 syscall 指令，陷入内核态（trap）
   │  内核检查参数、真正操作硬件/内核数据结构
   ▼
处理完毕，CPU 切回用户态，返回值交给你的代码
   │
用户态：你的代码继续跑
```

# 一次系统调用不是免费的

> 陷入内核态、切回用户态，这个动作本身要花时间——
> 比一次普通函数调用贵得多，但比线程切换便宜得多。

一次系统调用的开销，大致来自这几部分：

- **模式切换本身**：CPU 要保存/恢复一些状态，检查权限，
  这部分是几十到几百纳秒级；
- **内核态代码真正做的事**：比如 `read` 要真的去检查数据在不在
  页缓存里（回顾 [《磁盘慢是常态》](disk-io.md)）、`write` 要真的
  把数据复制进内核缓冲区，这部分耗时因系统调用种类而异；
- **如果这次调用会阻塞**（比如读一个还没数据的 socket），
  当前线程会被挂起、让出 CPU 给别人——这部分的代价属于
  下一课 [《调度决定跑谁》](scheduling.md) 的范畴，往往比模式
  切换本身贵得多。

一个粗略的数量级对比，帮你建立直觉：

| 操作 | 大致耗时 |
| --- | --- |
| 一次普通函数调用（用户态内部） | 几纳秒 |
| 一次不阻塞的系统调用（如 `clock_gettime`） | 几十到几百纳秒 |
| 一次涉及内核态实际工作的系统调用（如 `write` 到页缓存） | 几百纳秒到几微秒 |
| 一次导致线程被挂起的系统调用（如读空 socket） | 挂起期间不消耗 CPU，但恢复时要经历一次上下文切换（1~10 微秒） |

**结论很直白：系统调用比函数调用贵一到两个数量级，但只要不阻塞，
单次开销仍然是微秒以下**——真正的性能杀手不是「用了系统调用」，
而是「用了太多次」，下一节展开。

# 为什么调用次数才是关键

> 单次系统调用不贵，但业务代码里很容易不知不觉发起成千上万次，
> 累计起来就很可观。

一个常见的反面例子：逐字节或逐小块地写文件/写 socket。

```rust
// 坏例子：循环里每次都直接调用系统级的 write，
// 假设要写 10000 条小数据，就是 10000 次系统调用
for line in &lines {
    file.write_all(line.as_bytes())?; // 如果 file 没有缓冲，这里每次都陷入内核态
}
```

哪怕每次系统调用只要 1 微秒，10000 次就是 10 毫秒——
对一个要求毫秒级响应的接口，这可能就是响应时间的大头。
标准的解法是 **减少系统调用的次数**，把多次小的操作合并成
少数几次大的操作：

```rust
use std::io::{BufWriter, Write};

// 好例子：BufWriter 先把数据攒在用户态自己的内存缓冲区里，
// 缓冲区攒满（或显式 flush）才真正发起一次 write 系统调用
let mut writer = BufWriter::new(file);
for line in &lines {
    writer.write_all(line.as_bytes())?; // 大多数时候只是内存拷贝，没有陷入内核态
}
writer.flush()?; // 这里才真正可能触发系统调用，把攒的数据一次性交给内核
```

对照 Go：`bufio.NewWriter(f)` 是完全相同的思路——攒够一批，
再统一发起一次底层的 `Write` 系统调用。这个模式在读的方向上同理，
`BufReader`/`bufio.NewReader` 会一次性向内核多要一些数据放进
用户态缓冲区，后续多次小的 `read` 调用直接从这个缓冲区取，
不必每次都重新陷入内核态。

**这解释了一个常被忽略的性能提示**：直接用 `File`（不带缓冲）
逐行读写，往往比用 `BufReader`/`BufWriter` 慢一个数量级以上——
差距几乎完全来自「省下了多少次系统调用」，不是别的什么魔法。

# 用 strace 数系统调用次数

> [《程序如何跑起来》](computer-basics.md) 已经用过 `strace -c`，
> 这里专门用它验证「攒批 vs 不攒批」对系统调用次数的直接影响。

```bash
strace -c -e trace=write ./target/debug/os-user-kernel --no-buffer
strace -c -e trace=write ./target/debug/os-user-kernel --buffered
```

`-e trace=write` 只统计 `write` 这一种系统调用，输出会清楚地显示
两次运行里 `calls`（调用次数）这一列的巨大差异——不带缓冲的版本
调用次数直接对应你循环了多少次，带缓冲的版本调用次数只取决于
「数据总量 ÷ 缓冲区大小」，和循环次数基本无关。这组对比是本课
最直观的证据：**同一份逻辑，仅仅调整“系统调用发起的频率”，
性能就能差出一个数量级**。

# 与后面课程的对接

> 这一课的直觉会在网络编程和异步编程里反复被用到，先埋几个钩子。

- [《socket 就是文件描述符》](../network/socket.md) 提到
  “读写 socket 用的函数名和读写文件很像（`read`/`write`）”——
  这不是巧合，它们在系统调用这一层几乎是同一套接口，
  这一课把这句话的“为什么”补全了；
- **非阻塞 IO + epoll 的存在，本质是在优化“系统调用怎么发起”这件事**：
  与其每个连接都发起一次可能阻塞的 `read`，不如用一次 `epoll_wait`
  系统调用同时问“这一万个连接里哪些准备好了”，用一次系统调用的
  开销换来对一万个连接状态的批量确认——[《调度决定跑谁》](scheduling.md)
  和后续 async 章节会继续这条线；
- Go 的 goroutine 在执行可能阻塞的系统调用（比如读文件）时，
  Go 运行时会把这个系统调用交给一个专门的 OS 线程去承担，
  避免拖住整个 GMP 调度器——这也是「系统调用开销」在语言运行时
  设计里留下的直接印记，细节见
  [《Go 与 GMP 调度》](../concurrency/go-gmp.md)。

# Go 与 Rust 的系统调用位置

| | Rust | Go |
| --- | --- | --- |
| 谁发起真正的系统调用 | 标准库函数内部直接调用（如 `std::fs::File::write`） | Go 运行时封装了一层，部分场景经过 netpoller 转成非阻塞 |
| 你能不能绕过标准库直接发起系统调用 | 可以，`libc` crate 或裸 `asm!` | 可以，`syscall` 包，但官方不建议业务代码直接用 |
| 缓冲 IO 的默认行为 | 默认不带缓冲，需要显式用 `BufWriter`/`BufReader` | `os.File` 默认不带缓冲，同样需要显式用 `bufio` |
| 减少系统调用的常见手段 | 缓冲、批量写、`writev`（一次调用写多个缓冲区） | 缓冲、批量写，`net.Buffers` 对应 `writev` |

两边在这件事上的设计哲学一致：**语言标准库不会替你偷偷加缓冲**，
要不要减少系统调用次数、要不要用缓冲，是你需要主动决定的事——
这也是这一课「宁可啰嗦讲清楚原理」的原因：知道了原理，
`BufWriter`/`bufio.Writer` 这类 API 才不再是“照抄的套路”，
而是“我知道我在省什么”的主动选择。

# 系统调用不等于阻塞

> “发起了系统调用”和“这次调用会阻塞”是两件经常被混为一谈、
> 但完全不同的事——上一课「一次系统调用不是免费的」已经埋了这个
> 区分，这一节展开讲清楚。

系统调用本身只是「陷入内核态、让内核帮你做点事、再切回用户态」，
这个动作不管做的事情耗时长短，都要经历。**是否阻塞，取决于内核
处理这件事时，数据/资源是不是「已经准备好了」**：

- `clock_gettime`（查当前时间）：内核里的数据随时都是准备好的，
  查完立刻返回，属于不阻塞的系统调用，虽然也陷入了内核态；
- `read` 一个已经有数据在页缓存里的文件：内核发现数据现成，
  复制一下就返回，几乎不阻塞；
- `read` 一个还没有数据到达的网络 socket：内核发现「你要的数据
  还没到」，如果这个 socket 是「阻塞模式」（默认模式），
  内核会把你的线程挂起，直到数据真正到达才唤醒你——**这才是
  「阻塞」的准确含义：线程被挂起、让出核心，不是「系统调用慢」**。

同一个系统调用（比如 `read`），阻塞与否甚至取决于你给 socket
设置的模式：设成 **非阻塞模式（non-blocking）**，内核发现数据还
没到时，不会挂起你的线程，而是立刻返回一个「现在还没数据，
你等会再问」的错误码（Linux 上是 `EWOULDBLOCK`/`EAGAIN`），
把「要不要等」的决定权交还给你的代码——这正是 epoll、Rust 的
`mio`/tokio 底层非阻塞 IO 的基础，[《调度决定谁在跑》](scheduling.md)
「阻塞的确切含义」一节会把这条线继续接上「阻塞会冻住整个线程上
排队的其他任务」这个后果。

# vDSO 不用陷入内核

> 系统调用的开销大头之一是「陷入-切回」这个模式切换本身，
> 有没有办法干脆绕开它？对少数特别高频的调用，答案是有。

前面提到 `clock_gettime` 这类「内核数据随时准备好」的调用，
即使不阻塞，也仍然要经历一次完整的陷入内核态再切回的过程，
而这类调用在很多程序里被调用得极其频繁（日志打时间戳、
tracing 记录时间点、性能分析工具）。为了省下这部分开销，
Linux 提供了一个巧妙的机制，叫 **vDSO（virtual Dynamic Shared
Object）**：内核把少数几个「只读数据、不需要权限检查」的调用
（典型如 `clock_gettime`、`gettimeofday`）对应的代码，
**直接映射进每个进程自己的用户态地址空间**——这样调用它们时，
根本不需要陷入内核态，就是一次普通的用户态函数调用，纳秒级，
比走一次完整的系统调用快一个数量级以上。

```bash
# Linux 上能看到 vDSO 作为一个特殊的“共享库”出现在进程的内存映射里
cat /proc/self/maps | grep vdso
```

业务代码完全不需要手动关心 vDSO 的存在——Rust 的
`std::time::Instant::now()`、Go 的 `time.Now()`，
在支持 vDSO 的平台上，标准库/运行时会自动走这条快速路径。
提这个机制的意义在于：**“系统调用”这个词描述的是「用户态请求
内核帮忙」这件事的语义，不代表它在硬件层面一定要发生一次完整的
陷入-切回**——极少数被高频调用又不涉及敏感操作的调用，
可以被特殊优化到几乎零开销，这也是为什么“频繁查询当前时间”
这件事在现代系统上通常不值得你专门去优化。

# 日志库拖慢了整个服务

> 一个真实常见的性能故事——加了一个日志库之后，QPS 掉了一半，
> 排查后发现问题出在这一课讲的机制上。

场景：一个 HTTP 服务给每个请求都打几条日志，日志库的实现细节里，
每条日志调用一次不带缓冲的 `write` 直接写到文件（或者更糟，
每条日志都单独 `open` 一次日志文件再 `write` 再 `close`）。
压测时 QPS 远低于预期，`strace -c` 一跑，发现 `write`/`open`
系统调用的次数和请求数、日志条数成正比地暴涨，累计耗时占了
总耗时的一大部分。

这正是「为什么调用次数才是关键」那一节讲的现象在真实业务场景里的
翻版。修复思路完全一致：

1. 确认日志库有没有开启缓冲写入（大多数成熟日志库默认会有，
   但一些「自己随手写的」日志代码容易漏掉这一步）；
2. 确认日志文件是不是打开一次、长期持有文件句柄，而不是每条
   日志都重新 `open`/`close`；
3. 如果对日志的实时性要求不高（大多数业务日志确实如此），
   考虑异步写日志——业务线程只把日志内容塞进一个内存队列，
   由专门的一个线程/任务负责批量落盘，业务线程完全不用等
   任何系统调用完成。

这也是为什么生产级的日志库（`tracing-appender`、Go 生态的
`zap`/`lumberjack`）几乎都自带缓冲和/或异步写入选项——这不是
过度设计，而是这一课讲的「系统调用次数才是关键」这条规律，
在日志这个「几乎每个请求都要触发好多次」的场景里体现得格外明显。

# 三句话带走

1. **CPU 硬件层面区分用户态和内核态**：业务代码永远跑在用户态，
   不能直接碰硬件；系统调用是用户态请求内核态帮忙的唯一合法通道，
   发起时 CPU 要「陷入」再「切回」，这个模式切换本身有开销。
2. **单次系统调用的开销是几十纳秒到几微秒（不阻塞的情况），比函数调用贵，
   但远比线程切换和阻塞挂起便宜**——真正的性能杀手是「调用次数太多」，
   不是「用了系统调用」本身。
3. **减少系统调用次数的标准解法是用户态先攒一批再统一提交**——
   `BufWriter`/`bufio.Writer` 就是这个思路的封装；同样的直觉再往上
   一层，就是 epoll/非阻塞 IO「用一次系统调用批量查询一万个连接」
   的设计动机。

# 附：本课生词表

- **内核态 / 用户态** —— CPU 硬件支持的两种运行模式，内核态能直接访问硬件，
  用户态受限、业务代码永远跑在这里。
- **系统调用（syscall）** —— 用户态程序请求内核态代码帮忙做事的标准机制，
  是跨越两态唯一合法的门。
- **trap（陷入）** —— 发起系统调用时 CPU 从用户态陷入内核态的硬件动作。
- **模式切换** —— 陷入内核态、处理完再切回用户态的整个过程，有固定的时间开销。
- **`BufWriter`（Rust）/ `bufio.Writer`（Go）** —— 在用户态先攒一批数据，
  减少真正发起系统调用的次数。
- **`writev`（Rust 对应 `write_vectored`，Go 对应 `net.Buffers`）** ——
  一次系统调用写多个不连续的缓冲区，进一步减少调用次数。
- **epoll / 非阻塞 IO（提及）** —— 用一次系统调用批量查询大量连接状态，
  是系统调用开销优化思路的延伸，详见后续调度和异步章节。
- **阻塞 / 非阻塞模式** —— 数据未就绪时，内核挂起线程等待 / 立刻返回
  `EWOULDBLOCK`/`EAGAIN` 把决定权交还给调用者。
- **vDSO** —— 内核把少数高频、无需权限检查的调用（如 `clock_gettime`）
  直接映射进用户态地址空间，让它们无需陷入内核态即可执行。
