# 两种时钟不要混用

> 代码：`code/os-time-clock/`　运行：`cargo run -p os-time-clock`

前置：[《调度与时间片》](scheduling.md)。
本课讲操作系统提供的两种「时间」——它们回答的问题完全不同，
混用是超时/重试/限流类 bug 的常见根源。讲完直觉后，
链到 [《超时、重试与连接池》](../network/timeouts-retries.md)（超时怎么设、
怎么配合重试）和 [《超时、限流与任务组》](../async/task-control.md)
（tokio 里 `timeout`/`interval` 的具体用法）。

# 墙上时钟回答几点了

> 墙上时钟（wall clock）就是你手机上显示的那个时间——它的职责
> 是「现在几点几分」，不是「过了多久」。

**墙上时钟（wall clock，也叫 system clock/墙钟）** 对应的是
「现实世界的日期时间」，Rust 里是 `std::time::SystemTime`，
Go 里 `time.Now()` 默认返回的就是它。它有两个关键性质：

- 会被 **NTP（网络时间协议）** 定期校准——你的机器时钟天然会
  有漂移（晶振不完美），NTP 客户端定期跟时间服务器对表，
  把本地时钟往前或往后修正一点；
- **可能跳变**：手动改系统时间、NTP 校准发现偏差过大、
  夏令时切换（部分地区）——都可能让「现在的时间」突然跳前或跳后，
  甚至倒退。

```rust
use std::time::SystemTime;
let now = SystemTime::now(); // 墙上时钟，回答"现在是哪个时间点"
```

```go
now := time.Now() // 同样是墙上时钟视角
fmt.Println(now.Format("2006-01-02 15:04:05"))
```

**墙上时钟的正确用途**：显示给人看的时间戳、写日志、
和外部系统比较「这个事件发生在哪个绝对时间点」。
**它不该被用来测量「过了多久」**——下一节讲为什么。

----

# 单调时钟只会前进

> 单调时钟不关心「现在几点」，只保证「这一刻比上一刻晚，
> 绝不倒退」——这正是测量时长需要的性质。

**单调时钟（monotonic clock）** 是操作系统提供的另一路时钟，
它的值没有现实世界的意义（不代表哪年哪月），**唯一保证是严格
不减**——不受 NTP 校准、时区、夏令时、手动改系统时间的影响。
Rust 里是 `std::time::Instant`，Go 里 `time.Now()` 返回的
`Time` 值内部其实带着一个隐藏的单调时钟读数，**用两个
`time.Time` 相减（`t2.Sub(t1)`）时，Go 会自动优先用这部分单调
读数**——这是 Go 一个不太广为人知但很贴心的设计。

```rust
use std::time::Instant;

let start = Instant::now();      // 单调时钟起点
do_some_work();
let elapsed = start.elapsed();   // 只会是非负数，不受系统时钟被改动影响
println!("耗时: {elapsed:?}");
```

```go
start := time.Now()
doSomeWork()
elapsed := time.Since(start) // 内部走单调读数相减，同样不受系统时钟改动影响
```

**为什么「测量时长」必须用单调时钟**：假设你用墙上时钟算超时——

```
t1 := 墙上时钟.Now()   // 假设此刻恰好触发了 NTP 向后校准 5 秒
... 实际只过了 100 毫秒 ...
t2 := 墙上时钟.Now()   // 因为校准，t2 可能比 t1 还早！
elapsed := t2 - t1     // 算出一个负数，或者一个荒谬的巨大值
```

一旦系统时钟发生跳变（哪怕只跳 1 秒），用墙上时钟算出的
「已经过去的时间」就可能是错的——**表现为超时逻辑莫名其妙提前
触发或者永远不触发**，这类 bug 极难复现（只在时钟被校准的那个
瞬间才会犯病），排查成本很高。

# NTP 回拨的故事

> 把上一节「墙上时钟可能跳变」这句抽象的话，换成一个真实会发生的
> 时间线，感受它到底有多容易在你毫无察觉的情况下发生。

**NTP（Network Time Protocol）** 客户端（Linux 上常见是 `chronyd`
或 `ntpd`）会定期联系时间服务器，比较「本机时钟」和「权威时间」
之间的偏差，然后校准本机时钟。问题在于**机器本地的晶振从来不是
完美的**，长期运行会累积出秒级甚至更大的偏差——虚拟机场景下，
因为宿主机 CPU 争抢、挂起恢复等因素，偏差出现得比物理机更频繁。

一次典型的回拨故事：

```
t=0    机器本地时钟因为长期漂移，比真实时间快了 2 秒
t=0    某个请求记下 startTime = 墙上时钟.Now()
t=0.1s NTP 客户端这时候完成了一次校准，发现偏差过大，
       决定"跳跃式"（step，而不是缓慢修正）把本地时钟往回调 2 秒
t=0.1s 这个请求处理完，记下 endTime = 墙上时钟.Now()
       此刻的墙上时钟，比 startTime 记录的那一刻还要早
elapsed = endTime - startTime  →  算出一个负数
```

如果这段超时/耗时统计代码用的是墙上时钟相减，**负数耗时**会
以各种意想不到的方式表现出来：Rust 里 `SystemTime` 相减返回的是
`Result<Duration, SystemTimeError>`，遇到这种情况会直接返回一个
`Err`（这是 Rust 标准库特意设计成「不能悄悄给你一个错误数字」，
逼着你处理这个分支）；如果代码图省事直接 `.unwrap()`，
程序会在这一刻**莫名其妙地 panic**——而且只在 NTP 恰好完成一次
回拨校准的那个瞬间才会发生，平时怎么测都测不出来，是这类 bug
出了名地难复现、难排查的根本原因。

**「跳跃式」和「渐进式」是 NTP 修正时钟的两种方式**，也值得记住：

| 方式 | 行为 | 常见实现 |
| --- | --- | --- |
| step（跳跃式） | 发现偏差直接一次性调整到位，时钟瞬间「跳」了一下 | 偏差过大时的兜底手段 |
| slew（渐进式） | 把偏差摊开，用略微加快或放慢时钟走动速度的方式，用一段时间慢慢追平，不会有瞬间跳变 | `chronyd`/`ntpd` 日常校准的默认方式，偏差较小时优先使用 |

生产环境通常会优先用渐进式修正，只有偏差大到一定程度（比如机器
刚从长时间挂起状态恢复、或者时钟因为某种故障漂得特别离谱）才会
触发跳跃式修正——这也是为什么「墙上时钟会跳变」这件事平时几乎
感觉不到，一旦发生，因为足够罕见，代码里对应的分支往往完全没有
被测试覆盖过。**闰秒（leap second）**也是同一类问题的另一个来源：
为了让基于地球自转的时间和原子钟保持同步，历史上会不定期在某个
UTC 时刻插入一秒，如果系统对闰秒的处理方式是「直接跳一秒」而不是
把这一秒摊开处理，效果和上面的回拨故事完全一样。

# 两种时钟的分工一张表

> 记住这张表，之后写「要不要用 Instant/SystemTime」不用再犹豫。

| 问题 | 该用哪种时钟 | Rust | Go |
| --- | --- | --- | --- |
| 现在是哪年哪月哪天几点 | 墙上时钟 | `SystemTime::now()` | `time.Now()`（取其时间戳部分） |
| 这段代码跑了多久 | 单调时钟 | `Instant::now()` + `.elapsed()` | `time.Now()` + `time.Since()` |
| 距离超时还剩多久 | 单调时钟 | `Instant` 做算术 | 同上 |
| 日志里打印的时间戳 | 墙上时钟 | `SystemTime` 转格式化字符串 | `time.Now().Format(...)` |
| 两台不同机器的时间比较 | 都不完全可靠 | 需要更严谨的分布式时钟方案（如向量时钟），本课不展开 | 同左 |

最后一行值得单独提一句：**即便两台机器都各自正确使用了单调时钟，
它们之间也没法直接比较**——单调时钟只在单台机器内部有意义，
不同机器的单调时钟起点、走速都完全无关。跨机器比较时间只能靠
墙上时钟 + NTP，而 NTP 本身就有误差，这是分布式系统「时间」
话题天生比单机复杂的根源，本课到此点一句，不深入。

# 为什么超时必须用 Instant

> 把前面的故事总结成一条可以直接写进代码规范里的硬规则。

**任何「测量经过了多久」「还剩多久超时」的代码，都必须使用单调
时钟，不能使用墙上时钟相减**——这不是风格偏好，是上一节那个
回拨故事决定的正确性问题。落到 Rust/Go 具体写法上：

```rust
// 反面教材：用 SystemTime 相减实现超时判断
use std::time::SystemTime;

let start = SystemTime::now();
loop {
    if SystemTime::now().duration_since(start).unwrap().as_secs() > 5 {
        break; // 一旦 NTP 在这期间回拨过，.unwrap() 可能直接 panic
    }
    do_work();
}

// 正确写法：全程只用 Instant，不掺一滴 SystemTime
use std::time::Instant;

let start = Instant::now();
loop {
    if start.elapsed().as_secs() > 5 {
        break; // Instant 的差值永远非负，没有需要处理的错误分支
    }
    do_work();
}
```

Go 里的坑更隐蔽一些，因为 `time.Now()` 表面上返回的是同一个
`time.Time` 类型，内部却悄悄带着单调时钟读数——`t2.Sub(t1)`
在两个 `Time` 都携带这个隐藏读数时会优先使用它，效果上是安全的；
但**一旦这个 `Time` 值经过序列化再反序列化**（比如存进数据库、
编码成 JSON 再解码回来），Go 官方文档明确说明**单调读数会被
丢弃**，之后的 `Sub` 就退化成了纯粹的墙上时钟相减——这是 Go
开发者社区里一个流传很广、却仍然经常被踩中的细节：

```go
start := time.Now()               // 带隐藏单调读数，安全
saved, _ := json.Marshal(start)
var restored time.Time
json.Unmarshal(saved, &restored)  // 单调读数已丢失！
elapsed := time.Since(restored)   // 现在退化成了纯墙上时钟相减，
                                    // 可能因为回拨算出负值
```

**工程含义可以浓缩成一句检查规则**：写代码或做代码评审时，
只要看到「拿两个时间点相减来算耗时或判断超时」，先确认这两个
时间点都来自 `Instant::now()`（Rust）或者「未经序列化往返的
`time.Now()`」（Go）；如果其中一个时间点来自数据库、来自网络请求
里的字段、来自任何形式的持久化，那它已经不是可信的单调时钟读数，
这类场景下的「超时」判断要么改用相对值（存一个 `Duration`/秒数而
不是绝对时刻），要么明确接受「这是墙上时钟语义，允许有一定误差」。

# 定时器怎么实现的

> `sleep(3秒)`/超时倒计时，操作系统内部大致是怎么做到「精确地
> 在未来某个时刻叫醒你」的。

- 内核维护着一份**定时器列表**，每个定时器记着「到期的单调时钟
  时刻」和「到期后要做什么（通常是把某个阻塞的线程唤醒）」；
- 硬件层面有一个**时钟中断**，以固定频率（比如每毫秒）触发，
  内核在这个中断处理里检查有没有定时器到期，到期就执行对应动作；
- 大量定时器同时存在时（想象一个网络服务同时有几万个连接各自的
  超时倒计时），逐个检查太慢，常见实现用
  **分层定时轮（hashed/hierarchical timing wheel）** 这类数据结构，
  把「插入定时器」「到期取出」都优化到接近 O(1)——Tokio 的定时器
  内部就是这类结构的一种实现。

```rust
use tokio::time::{sleep, Duration};

sleep(Duration::from_secs(3)).await; // 底层：注册一个定时器，任务挂起，到期被唤醒
```

```go
time.Sleep(3 * time.Second) // Go runtime 内部也是类似的定时器堆/时间轮结构
```

🔬 底层视角：`sleep` 在阻塞线程的世界里（[《阻塞的确切含义》](blocking-io.md)
讲过），是线程主动告诉内核「N 秒后叫醒我」然后进入阻塞态；
在 async 世界里，是任务把自己注册进运行时的定时器结构后返回
`Pending`，运行时的定时器到期后调用 `Waker` 唤醒——**两种实现
的操作系统底座都是同一套「内核定时器 + 时钟中断」**，只是「谁被
挂起」不同（一个是 OS 线程，一个是用户态任务），这正呼应
[《阻塞的确切含义》](blocking-io.md) 讲过的暂停单元差异。

# 定时器精度不是绝对的

> 「3 秒后执行」经常不是精确的 3.000 秒——这不是 bug，
> 是几层因素叠加的正常现象。

- 时钟中断本身有固定粒度（比如 1 毫秒），定时器最快也只能按这个
  粒度对齐，不可能有纳秒级精度；
- 到期后能不能**立刻**执行，还要看调度器：如果核心正忙着跑别的
  高优先级任务，被唤醒的线程/任务要排队等到轮到它；
- 虚拟化/容器环境下（[《容器不是轻量 VM》](cgroup-container.md)
  会展开），CPU 限流会让「本该马上执行」的代码被迫等待更久，
  定时器的实际延迟会比裸机上明显放大。

**工程含义**：不要依赖定时器给出「刚好精确」的时间点，
超时/重试逻辑应该按「大致到了就行」设计（比如给 20% 左右的
容忍余量），把定时器当成「至少等这么久」而不是「精确等这么久」的
承诺去用。

# 动手实验

1. 运行 `cargo run -p os-time-clock`，对比程序打印的
   `SystemTime` 和 `Instant` 差值，理解两者在数值上完全不是
   一回事（一个是绝对时间戳，一个是相对某个任意起点的计数）；
2. 手动把系统时间往回调 1 分钟（虚拟机/测试环境操作，别在生产
   机器上试），观察用 `SystemTime` 计算的「耗时」变成负数，
   而用 `Instant` 计算的耗时完全不受影响；
3. 写一个 `tokio::time::sleep(Duration::from_millis(1)).await` 的
   循环，测量实际耗时和名义耗时的偏差，感受「定时器精度不是
   绝对」这句话在你机器上的具体数字；
4. 阅读 [《超时、重试与连接池》](../network/timeouts-retries.md)
   的超时设置代码，确认它底层用的是单调时钟语义（`Duration`/
   `Instant` 而不是两个 `SystemTime` 相减）；
5. 写一段用 `SystemTime::now().duration_since(start).unwrap()`
   算耗时的代码，手动把系统时间往回调几秒后立刻触发这段代码，
   验证它确实会 panic；再把同样的逻辑换成 `Instant`，验证不受
   任何影响；
6. （Go）写一个 `time.Time` 经过 `json.Marshal`/`Unmarshal`
   往返后再 `time.Since` 的小实验，配合手动回调系统时间，
   验证「单调读数被序列化丢弃」这句话在你机器上的真实效果。

# 三句话带走

1. **墙上时钟回答「现在几点」，会被 NTP 校准、可能跳变**——
   适合显示和记录日志，绝不能用来测量时长。
2. **单调时钟只保证严格不减、不受任何外部校准影响**——
   Rust 的 `Instant`、Go 的 `time.Since` 内部都是走单调读数，
   是测量耗时和实现超时的唯一正确选择。
3. **定时器靠内核定时器列表 + 时钟中断实现，精度有下限、
   到期不代表立即执行**——超时/重试逻辑要按「至少等这么久」
   设计，别依赖绝对精确的时间点。

# 附：本课生词表

- **墙上时钟（wall clock）** —— 表示现实世界日期时间的时钟，
  会被 NTP 校准，可能跳变。
- **单调时钟（monotonic clock）** —— 只保证严格不减、
  不受外部校准影响的时钟，用于测量时长。
- **NTP（网络时间协议）** —— 让机器本地时钟对齐权威时间源的协议，
  校准时可能让墙上时钟发生跳变。
- **`Instant`（Rust）/单调读数（Go `time.Time` 内部）** ——
  各语言里代表单调时钟的类型/机制。
- **时钟中断** —— 硬件以固定频率触发的中断，内核借此检查定时器
  是否到期、驱动调度器的时间片轮转。
- **定时轮（timing wheel）** —— 高效管理海量定时器的数据结构，
  插入/到期检查接近 O(1)。
- **定时器精度** —— 定时器「到期」和「实际被执行」之间存在的、
  受粒度和调度共同决定的误差。
- **step（跳跃式修正）/slew（渐进式修正）** —— NTP 校准本地时钟
  的两种方式：前者瞬间调整到位（会造成时间跳变），后者摊开时间
  慢慢追平（不会跳变），偏差较小时优先用后者。
- **闰秒（leap second）** —— 为对齐地球自转和原子钟不定期插入的
  一秒，处理不当时效果和时钟回拨一样会造成耗时计算异常。
- **单调读数丢失** —— Go 的 `time.Time` 经过序列化/反序列化后，
  内部携带的单调时钟读数会被丢弃，`Sub` 退化为墙上时钟相减。
