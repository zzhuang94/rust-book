# 程序不是进程

> 代码：`code/os-process-thread/`　运行：`cargo run -p os-process-thread`
> （代码演示 `std::process::Command` 起一个子进程、以及
> `std::thread::spawn` 起几个线程，打印各自的 PID/线程 ID 做对照）

> 前置：[《程序如何跑起来》](computer-basics.md) 已经区分过
> “程序（静止的文件）”和“进程（活的实例）”。这一课把进程讲透，
> 再引出进程内部的“线程”，最后给一张贯穿全系列最重要的成本表。
> 完整的线程编程实操，留给后续 [《Rust 多线程与并发》](../concurrency/threads.md)。

“进程”和“线程”这两个词你可能天天挂在嘴边——“这个服务占了多少个
进程”“数据库连接池开了多少个线程”，但很多人说不清楚它们到底是什么、
差别在哪、为什么贵。这一课把这两个概念从零建立起来。

----

# 进程是运行的实例

> 上一课已经埋过这个区分，这里正式定义。

- **程序（program）**：磁盘上一个静止的可执行文件，本身不占用
  CPU、不占用运行时的内存，和一个 `.txt` 文件的存在方式没有本质区别；
- **进程（process）**：程序被操作系统启动之后的 **运行实例**——
  它拥有一份独立的内存空间、打开的资源（文件、网络连接）、
  至少一个正在被 CPU 执行的线程。

关键的一点：**同一个程序可以同时起很多个进程，互不相干**。
你打开三个浏览器窗口（准确说是三个浏览器进程，现代浏览器一个
标签页往往就是一个进程），它们共享同一份磁盘上的程序文件，
但运行起来之后，各自有各自独立的内存、各自的状态——一个窗口崩了，
不会牵连另外两个。这个「互不相干」的隔离性，正是下一节的主题。

----

# 进程有自己的地址空间

> 进程最重要的特征是隔离，隔离靠的是每个进程独占一份「地址空间」。

每个进程都被操作系统分配了一份 **虚拟地址空间**——你可以先把它
想象成「这个进程独占整台机器的全部内存」的一个幻觉：进程里的代码
访问地址 `0x1000`，拿到的永远是「这个进程自己的」`0x1000` 位置的数据，
和另一个进程的 `0x1000` 完全是两块不同的物理内存，谁也看不见谁。

这个幻觉具体怎么在硬件层面实现（页表、地址翻译），是专门一课
[《虚拟内存》](virtual-memory.md) 的主题，这里先记住效果：
**进程 A 崩了、内存写坏了，进程 B 毫发无伤；进程 A 的代码里根本
没有语法能直接读写进程 B 的一个变量**。这是操作系统给你的一份
「安全保证」——你写业务代码时几乎从没意识到它的存在，
恰恰是因为它一直在默默生效，从没让你踩过坑。

**代价**：既然进程之间互相看不见，**跨进程共享数据就必须走专门的
通道**，业务代码里天天用但很少细想的一件事，正是下面这一节。

----

# 跨进程通信要走 IPC

> 进程隔离是优点，但两个进程真要交换数据时，隔离就变成了要绕过的障碍——
> 这条绕过的路统称 IPC（Inter-Process Communication）。

| IPC 方式 | 一句话 | 常见场景 |
| --- | --- | --- |
| 管道（pipe） | 一个进程写、另一个进程读的单向字节流 | shell 里的 `\|` |
| 共享内存 | 操作系统让两个进程的地址空间「重叠」出一块共同区域 | 高性能场景，速度最快但要自己管同步 |
| 消息队列 | 操作系统维护的一个消息缓冲区 | 传统本机 IPC，较少直接用 |
| 网络 socket | 通过网络协议栈交换数据，本机和跨机都能用 | HTTP 调用、Redis 客户端、gRPC |
| 文件 | 一个进程写文件，另一个进程读同一个文件 | 日志采集、批处理数据交换 |

业务代码里最常打交道的其实是最后两种。[《接入 Redis》](../http/redis.md)
把共享状态放进 Redis，本质就是「用网络 socket 做跨进程共享」——
你的多个服务实例（每个都是独立进程，甚至在不同机器上）通过
连接同一个 Redis 进程来共享数据，Redis 承担了「大家都能看见的
那块内存」的角色，只是访问方式从「直接读内存」变成了「发一条
网络请求」。这也解释了为什么“进程内共享变量”和“进程间共享数据”
在成本上有天壤之别：前者是纯内存操作，后者要走一整套网络协议栈
（详见 [《分层是为了解耦》](../network/layers.md)）。

# 创建进程是重体力活

> 进程隔离带来的另一面代价：创建一个新进程，操作系统要做不少准备工作。

创建一个新进程，操作系统至少要做：

1. 分配一份全新的虚拟地址空间（一套页表，见
   [《虚拟内存》](virtual-memory.md)）；
2. 把程序文件的内容装载进这份地址空间（上一课「运行一次经历了什么」）；
3. 建立这个进程要用到的资源表（打开的文件、网络连接的初始状态）；
4. 至少创建一个线程，把 CPU 指向程序入口。

这一整套动作的耗时是 **毫秒级**——听起来很快，但对比一下
下一节马上要讲的「创建一个线程」（微秒级），慢了近千倍。
Unix 系统上创建进程的经典系统调用叫 `fork`（复制当前进程）
配 `exec`（把复制出来的进程替换成另一个程序），Windows 上是
一步到位的 `CreateProcess`。业务代码几乎不会直接调用这些系统调用，
但你敲的每一条 shell 命令、每一次「启动一个子进程去跑外部工具」，
背后都在走这条路。

> 🔬 底层视角：`fork` 复制出的新进程，一开始并不会真的把父进程
> 的全部内存复制一份物理副本——现代操作系统用一种叫
> **写时复制（Copy-on-Write，COW）** 的技巧：父子进程先共享同一份
> 物理内存页（都标记成只读），只有当某一方真的要 **写** 某一页时，
> 才会真正复制那一页给写的那一方。这让 `fork` 本身的耗时和「进程
> 内存有多大」基本脱钩——这也是为什么很多脚本语言、Redis 等
> 生成快照喜欢用 `fork` 的原因：几乎瞬间就能拿到一份「此刻内存的
> 完整副本」，代价只在真正发生写操作时才逐页显现。

Rust 里启动一个子进程：

```rust
use std::process::Command;

let output = Command::new("echo")
    .arg("hello from child process")
    .output()?;
println!("子进程输出: {}", String::from_utf8_lossy(&output.stdout));
```

对照 Go：

```go
out, _ := exec.Command("echo", "hello from child process").Output()
fmt.Println("子进程输出:", string(out))
```

两边都是「一步到位」的封装，背后都对应上面 1~4 步的完整流程——
这也是为什么“起一个子进程去跑个外部命令”这种操作，
即使命令本身跑得很快，也总能感觉到一点点“慢启动”的延迟。

# 线程是进程内的执行流

> 一个进程至少有一个线程，也可以开很多个——它们的关键设计是「共享」。

**线程（thread）** 是进程内的一条执行流。一个进程创建时自带一个
线程（通常叫“主线程”/main 线程），代码里可以再开更多线程。
和进程最大的区别只有一句话：**同一个进程内的所有线程，共享同一份
地址空间**——你的全局变量、堆上分配的数据，每个线程都能直接读写，
不需要任何 IPC。

正因为共享地址空间，创建线程比创建进程轻得多：不需要分配新的
虚拟地址空间、不需要重新装载程序文件，只需要给这条新的执行流
准备三样「独有」的东西——下一节细说。

----

# 线程独有的三样东西

> 共享是主旋律，但每个线程仍然需要一点点「私有财产」，才能算是独立的一条执行流。

- **栈（stack）**：存放这个线程的局部变量、函数调用记录。
  每个线程有自己独立的一份，Linux 上默认大小约 8MB（是虚拟内存
  预留量，不是马上占用的物理内存，具体机制见
  [《虚拟内存》](virtual-memory.md)）；
- **寄存器上下文**：这一课的姐妹课 [《CPU 与内存》](cpu-memory.md)
  说过「代码执行到哪了」本质是寄存器的值——每个线程要有自己的
  一份寄存器「快照」，这样操作系统才能在多个线程之间来回切换，
  每次都能准确恢复到「这个线程上次执行到哪了」；
- **线程 ID**：操作系统用来标识和管理这条执行流的编号。

除了这三样，**其余全部共享**：全局变量、堆内存、打开的文件描述符、
网络连接——这也是下一节「福与祸」的根源。

----

# 共享是福也是祸

> 线程共享地址空间这一条设计，同时带来了最大的便利和最大的麻烦。

- **福**：线程间共享数据几乎零成本——直接读写同一个变量就行，
  不需要走 IPC、不需要网络往返，这也是「多线程」比「多进程」
  更适合处理「同一份数据的并行计算」的原因；
- **祸**：两个线程同时读写同一个变量，如果没有协调，就是
  **数据竞争（data race）**——谁先谁后不确定，结果可能是错的、
  甚至每次跑都不一样，调试起来极其痛苦。

Go 和 Rust 应对这个「祸」的方式很不一样：Go 靠约定和一个专门的
检测工具（`go run -race`）在运行时抓数据竞争，但语言本身不强制你
加锁；**Rust 把这个检查做进了编译器**——`Send`/`Sync` 这两个标记
让编译器在编译期就能判断「这个类型能不能安全地跨线程共享/传递」，
写不满足条件的代码，编译直接不通过，不需要等到运行时、
甚至不需要专门跑一次测试才发现问题。共享数据具体怎么正确地上锁、
用原子操作，是 [《共享状态：Arc / RwLock》](../async/shared-state.md)
和 [《Rust 多线程与并发》](../concurrency/threads.md) 的主题，
这一课只需要建立「共享是零成本的福，也是竞态的祸根」这个直觉。

# 一张贯穿全系列的成本表

> 这是本课最重要的一张表——进程、线程创建和切换的成本量级，
> 之后 [《调度决定跑谁》](scheduling.md) 和
> [《状态机与协程》](coroutine-state.md) 会在这张表基础上再加一列。

| 成本项 | 进程 | 线程 |
| --- | --- | --- |
| 独占内存 | 一整套地址空间 | 栈，默认约 8MB（虚拟内存预留量） |
| 创建耗时 | 毫秒级 | 微秒级 |
| 切换耗时 | 最贵（连地址空间的映射都要换） | 1~10 微秒 + 缓存变冷 |
| 崩溃影响 | 只死自己 | 整个进程陪葬（同进程其他线程一起没了） |
| 通信方式 | IPC（管道/socket/共享内存/Redis…） | 直接共享内存 + 锁/原子操作/通道 |
| 谁负责创建/调度 | 操作系统内核 | 操作系统内核 |

这张表能直接回答很多业务上的架构选择：**为什么微服务用「多进程
（甚至多机器）+ 网络调用」而不是「一个进程开一堆线程」**——
因为进程级别的隔离能让一个服务崩溃不牵连其他服务，代价是通信要走
网络；**为什么同一个服务内部处理并发请求用线程（或者更轻的用户态
任务）而不是每个请求起一个新进程**——因为进程创建和切换的成本
比线程高出一到两个数量级，扛不住高并发。

----

# 线程贵在哪：具体到数字

> 「线程比进程轻」不代表「线程免费」，这一节把线程本身的成本
> 拆得更细，为下一课的调度铺垫。

| 成本项 | 数量级 | 说明 |
| --- | --- | --- |
| 栈内存 | 默认 ~8MB 虚拟内存/线程 | 一万个线程约需 80GB 虚拟地址空间，实际物理内存按用到的部分算，但也可观 |
| 创建/销毁 | 微秒~几十微秒 | 要向内核登记、分配栈 |
| 上下文切换 | 1~10 微秒 + 缓存变冷 | 详见 [《调度决定跑谁》](scheduling.md) |
| 内核调度压力 | 线程越多越重 | 调度器要管理的对象随线程数线性增长 |

一句直白的结论：**「每来一个请求就开一个 OS 线程」这种最朴素的
并发模型，到几千并发连接就会开始吃力**——这正是网络编程里
著名的「C10K 问题」的根源，也是
[《从零手写 HTTP》](../http/http-from-scratch.md) 从「每连接一线程」
进化到「用户态任务（tokio task）」的直接动机。「用户态任务」
（Go 的 goroutine、Rust 的 async 任务）比线程还要轻一个数量级，
它们既不是进程也不是线程，属于一种新的并发单元——这一整套「为什么
需要它、它到底是什么」正是下一课 [《状态机与协程》](coroutine-state.md)
要展开的内容。

# Go 与 Rust 的进程/线程 API 对照

| 动作 | Rust | Go |
| --- | --- | --- |
| 起子进程跑外部命令 | `std::process::Command::new(...)` | `os/exec.Command(...)` |
| 起一个线程 | `std::thread::spawn(closure)` | `go func(){...}()`（起的是 goroutine，不是 OS 线程） |
| 等待/回收 | `handle.join()` | `sync.WaitGroup.Wait()` |
| 查看当前进程 PID | `std::process::id()` | `os.Getpid()` |
| 查看当前线程 ID | `std::thread::current().id()`（不透明句柄） | Go 语言层面不暴露真实 OS 线程 ID，只能拿 `runtime.NumGoroutine()` |

这张表里最值得注意的一行是「起一个线程」——**Go 的 `go` 关键字
起的从来不是 OS 线程，而是 goroutine**（一种用户态任务），
Go 运行时自己在背后管理一批 OS 线程（数量大致等于逻辑核数），
把成千上万个 goroutine 调度到这一小批线程上运行。这个机制叫
**GMP 调度模型**，完整细节在
[《Go 与 GMP 调度》](../concurrency/go-gmp.md)；Rust 的
`thread::spawn` 才是这张表里唯一「货真价实起一个 OS 线程」的一行，
Rust 如果要用类似 goroutine 的用户态任务，要显式引入 tokio，
写 `tokio::spawn`——这个选择权在 Rust 里是显式的，
在 Go 里是默认隐藏的。

# 进程死亡的两种中间态

> “杀掉一个进程”听起来是瞬间完成的，但父子进程之间还牵扯着一点
> 收尾的礼节，没做好会留下两种奇怪的“幽灵”——这里先点出概念，
> 完整机制、信号种类、优雅关闭的代码套路是专门一课
> [《信号与进程生命周期》](signals-lifecycle.md) 的主题。

一个进程结束时，它的退出状态需要 **父进程** 调用 `wait`/`waitpid`
去“认领”，这个认领动作叫 **回收（reap）**。没做好这一步，
会出现两种反直觉的状态：**僵尸进程（zombie）**——子进程已经死了，
但父进程没认领，会一直挂在进程列表里（`ps` 的 `STAT` 列显示 `Z`）；
**孤儿进程（orphan）**——父进程先退出，子进程还活着，会被操作系统
过继给 PID 1（`init`/`systemd`）接手。这也是容器里
「`docker stop` 清不干净进程」「`kill -9` 之后还有残留」这类怪现象
的根源：容器的主进程（PID 1）如果没有正确转发信号、正确回收子进程，
这两种“幽灵”就会在容器里堆积——很多容器实践会引入 `tini`、
`dumb-init` 这类极简 init 进程来专门处理这件事。

Rust/Go 里只要显式调用了回收接口，业务代码基本不会踩这个坑：

```rust
let mut child = std::process::Command::new("sleep").arg("1").spawn()?;
child.wait()?; // 内部调用 waitpid，正确回收退出状态，避免留下僵尸
```

```go
cmd := exec.Command("sleep", "1")
cmd.Start()
cmd.Wait() // 同样处理了回收
```

信号具体有哪些种类、`SIGTERM` 和 `SIGKILL` 的区别、优雅关闭的完整
骨架代码，留给 [《信号与进程生命周期》](signals-lifecycle.md) 展开——
这一课只需要记住：**`kill` 不是让进程瞬间消失，而是递交一份「请你
处理」的通知，进程死亡的收尾工作，操作系统和你的代码要共同完成**。

# 连接池要限制线程数

> 前面几节讲了线程的成本，这一节把它落到一个你大概率天天在用、
> 却没细想过“为什么这么设计”的业务组件上——连接池。

数据库连接池、HTTP 客户端连接池几乎都有一个「最大连接数/最大线程数」
的配置项，初学者常见的直觉是「设得越大越安全，免得请求排队」。
但结合这一课的成本表看，这个直觉是错的：

- 每个数据库连接背后往往对应数据库服务端的一个进程或线程
  （比如 PostgreSQL 默认「一个连接一个进程」），连接数开得过大，
  压力会直接转移到数据库那一端，可能先把数据库服务器的资源
  （内存、上下文切换开销）拖垮，而不是让你的服务变快；
- 你的服务这一端，如果用「每个请求占一个线程」的模型去驱动这些
  连接，线程数上去了，本课「线程贵在哪」那张表里的每一项成本
  （栈内存、创建/切换开销、内核调度压力）都会跟着涨；
- **真正的瓶颈往往不是“连接不够用”，而是下游服务（数据库、
  另一个微服务）本身处理并发请求的能力有上限**——连接池的最大
  数量设置，本质是在帮你的服务「主动限流」，避免因为你这一端
  无限制地开线程/开连接，把下游直接压垮。

一个直觉性的经验公式（不是精确公式，帮助建立数量级感觉）：
连接池大小大致应该对齐「下游服务能舒服处理的并发数」，而不是
「你这边理论上想同时发出多少请求」——连接池满了之后，多出来的
请求应该排队等待或者快速失败，而不是无限制地继续开新连接/新线程。
这也是为什么连接池几乎都会同时提供「排队超时」这个配置：
排队太久还拿不到连接，宁可给调用方一个明确的失败，也不要让请求
无限堆积，最终在某个更糟糕的时刻（比如内存暴涨）集中爆发。

# 进程线程数排错清单

> 把这一课的几张成本表变成排查现场问题时的具体操作。

| 现象 | 优先怀疑 | 排查手段 |
| --- | --- | --- |
| `ps aux` 里出现大量 `STAT` 为 `Z` 的进程 | 僵尸进程堆积，父进程没有正确回收 | 找到父进程 PID（`ps -o ppid=`），检查它的回收逻辑 |
| 一台机器上线程数远超预期，内存却没见涨多少 | 线程栈默认按虚拟内存预留，真正占用的物理内存不多 | 结合 [《虚拟内存》](virtual-memory.md) 用 `VmRSS` 而不是 `VmSize` 判断真实占用 |
| 容器 `docker stop` 要等很久才真正退出 | 主进程没有正确处理 `SIGTERM` | 检查容器 `ENTRYPOINT` 是否直接是业务程序，还是套了一层不转发信号的 shell 脚本 |
| 并发压测时响应时间陡增，CPU 却没跑满 | 线程数远超核心数，上下文切换开销主导 | 参考下一课 [《调度决定谁在跑》](scheduling.md) 的排错清单 |
| 服务偶尔"卡死"不响应，但没崩溃 | 某个持有锁的线程阻塞住了，其他线程排队等锁 | 用调试器/`pstack`（Linux）看各线程的调用栈，找到卡住的那一个 |

# 动手实验

1. 跑本章代码，打印主进程的 PID 和它起的子进程的 PID，
   在任务管理器（Windows）或 `ps -ef`（Linux/macOS）里找到这两个
   进程，观察它们是父子关系；
2. 用本章代码起 4 个线程，各自打印 `thread::current().id()`，
   在 Linux 上跑 `ps -eLf | grep os-process-thread` 观察这个进程下
   实际有多少个「轻量级进程（LWP，Linux 对线程的底层实现）」；
3. 分别计时「创建并销毁 1000 个线程」和「创建并销毁 100 个子进程」，
   感受两者耗时差一到两个数量级的直接体感；
4. 故意让一个子线程 panic（比如 `panic!("boom")`），观察主线程能否
   通过 `join()` 的返回值 `Result` 感知到，主进程是否因此退出；
   再对照「让一个子进程异常退出」，观察父进程是否受影响——
   直接验证「线程崩溃陪葬整个进程，进程崩溃只死自己」这条结论。

# 三句话带走

1. **程序是磁盘上静止的文件，进程是它跑起来的独立实例**——每个进程
   有自己独立的地址空间，天然隔离；跨进程共享数据必须走 IPC
   （管道/共享内存/网络 socket，Redis 就是一种）。
2. **线程是进程内的执行流，和进程最大的区别是「共享地址空间」**——
   独有的只有栈、寄存器上下文、线程 ID；共享带来零成本通信的福，
   也带来数据竞争的祸，Rust 用 `Send`/`Sync` 把这层检查做进了编译器。
3. **成本递减：进程（毫秒级创建，最贵）> 线程（微秒级创建，共享内存但仍要进内核）
   > 用户态任务（下一课揭晓，最便宜）**——这条成本阶梯直接决定了架构
   选择：强隔离用多进程，CPU 并行用多线程，海量 IO 并发用用户态任务。

# 附：本课生词表

- **程序（program）/ 进程（process）** —— 静止的可执行文件 / 它跑起来后的独立实例。
- **虚拟地址空间** —— 每个进程独占的「私有内存幻觉」，是进程隔离的直接实现，详见
  [《虚拟内存》](virtual-memory.md)。
- **IPC（进程间通信）** —— 管道、共享内存、消息队列、网络 socket、文件，
  跨进程共享数据必须走的通道。
- **`fork` / `exec`** —— Unix 创建进程的经典系统调用组合：复制当前进程 / 替换成另一个程序。
- **写时复制（Copy-on-Write，COW）** —— `fork` 出的父子进程先共享物理内存页，
  只有真正写入时才复制，让 `fork` 本身很快。
- **线程（thread）** —— 进程内的一条执行流，共享进程的地址空间，独有栈+寄存器上下文+线程 ID。
- **数据竞争（data race）** —— 多线程无同步地读写同一数据，结果不确定；
  Rust 靠 `Send`/`Sync` 在编译期拦截。
- **C10K 问题** —— 「每连接一个 OS 线程」模型撑到几千并发就吃力的经典问题。
- **goroutine** —— Go 的用户态并发单元，不是 OS 线程，由 GMP 调度模型管理，
  详见 [《Go 与 GMP 调度》](../concurrency/go-gmp.md)。
- **用户态任务** —— 比 OS 线程更轻的并发单元（goroutine、tokio task），
  下一课 [《状态机与协程》](coroutine-state.md) 展开。
- **僵尸进程（zombie）/ 孤儿进程（orphan）** —— 子死父未收尸 / 父死子还活着，
  完整机制见 [《信号与进程生命周期》](signals-lifecycle.md)。
- **回收（reap）** —— 父进程调用 `wait`/`waitpid` 认领子进程退出状态的动作。
- **连接池限流** —— 连接池的最大连接数本质是在保护下游服务，而不是单纯为了不排队。
