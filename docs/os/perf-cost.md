# 性能数量级要记住

> 代码：`code/os-perf-cost/`　运行：`cargo run -p os-perf-cost`

前置：[《CPU 与存储层次》](cpu-memory.md)（上下文切换、缓存行）、
[《阻塞的确切含义》](blocking-io.md)、
[《同步原语的操作系统直觉》](sync-primitives.md)（futex、锁竞争）。
这一课不讲新机制，只做一件事：**把前面几课提到的各种操作，
按耗时数量级排成一张表**，配合几个能在自己机器上直接用的排查工具。
建这张表的目的很实际：遇到性能问题时，能一眼判断「这个耗时正常吗、
大概是哪一层出的问题」，而不是凭感觉瞎猜。

----

# 上下文切换的成本

> 复习 [《CPU 与存储层次》](cpu-memory.md) 的结论，加上一个
> 容易被忽略的隐藏成本。

把 CPU 核心从线程 A 换给线程 B，直接成本（保存/恢复寄存器、
内核记账）大约 **1~10 微秒**。但真正让「线程数暴涨」变慢的，
往往是没写在这个数字里的**间接成本**：

- 核心的 L1/L2 缓存里装的是线程 A 刚用过的数据，切到线程 B 后
  这些缓存**大概率全部作废**，B 要重新从更慢的内存里加载
  （「缓存变冷」）——这部分耗时可能比切换本身还大；
- 线程越多，调度器要维护的运行队列越大，选下一个该跑谁的决策
  本身也变慢；
- 如果切换还牵涉切换到不同的进程（而不是同进程的线程），
  还要多付「切换虚拟地址空间映射」的代价，比同进程线程间
  切换更贵。

**这就是为什么“线程数超过核数很多”这件事本身就是一种性能损耗**：
不是因为线程本身有多贵，是因为核心在忙着换人、缓存反复变冷，
真正花在「干活」上的时间占比反而下降了。

# 缓存未命中的成本

> CPU 和内存之间的速度差，是本课几乎所有数字的物理背景。

一个粗略但足够有用的直觉数量级（不同硬件会有出入，记数量级即可）：

| 访问目标 | 大致耗时 |
| --- | --- |
| CPU 寄存器 | 不到 1 纳秒 |
| L1 缓存 | ~1 纳秒 |
| L2 缓存 | ~3~10 纳秒 |
| L3 缓存 | ~10~20 纳秒 |
| 主内存（缓存全部未命中） | ~100 纳秒 |
| SSD 随机读 | ~10~100 微秒 |
| 机械盘随机读 | ~1~10 毫秒 |
| 同机房网络往返 | ~0.5~1 毫秒 |
| 跨地域网络往返 | ~几十~上百毫秒 |

看这张表最重要的收获不是记住具体数字，而是记住**每一行和上一行
之间大约差 1~2 个数量级**——寄存器到主内存差了大约 100 倍，
主内存到网络又差了上万倍。这就是为什么
[《CPU 与存储层次》](cpu-memory.md) 反复强调「缓存行」、
[《ArcSwap 无锁读》](../http/arcswap.md) 要专门优化「缓存行弹跳」——
省下的是这张表里最贵的那几行之间的差距。

🔬 底层视角：这张表业界俗称「每个程序员都应该知道的延迟数字」，
每隔几年硬件进步会让绝对数字略有变化，但**相对比例这十几年
惊人地稳定**——记住比例关系，比记住某一年的具体纳秒数更耐用。

# 系统调用的成本

> 每次「求助内核」都不是免费的——这解释了为什么前面几课反复
> 强调「减少系统调用次数」。

调用一次系统调用（比如 `read`/`write`/`epoll_wait`），CPU 要从
**用户态**切换到**内核态**再切换回来——这个切换本身（俗称
「模式切换」）就有固定开销，大致 **零点几到几微秒**，具体数字
受硬件安全特性影响（比如为了防范 Meltdown/Spectre 类漏洞而
加的隔离措施，会让这个开销更高一些）。

这解释了几个实际的工程选择：

- 为什么 [《socket 就是文件描述符》](../network/socket.md) 提到的
  「批量读写」比「一个字节一个字节读」快得多——不是数据量变了，
  是系统调用次数少了；
- 为什么 [《阻塞的确切含义》](blocking-io.md) 里 epoll 要把
  「查询就绪状态」批量化成一次调用返回一堆结果，而不是每个 fd
  单独问一次；
- 为什么 Rust/Go 的 IO 库默认都带缓冲（`BufReader`/`bufio.Reader`）——
  攒够一批数据再触发一次系统调用，用一点内存换掉大量系统调用次数。

# 锁竞争的成本

> 复习 [《同步原语的操作系统直觉》](sync-primitives.md) 的 futex 结论，
> 换成具体数字感受一下差距。

| 场景 | 大致耗时 |
| --- | --- |
| 无竞争的原子操作 | ~几纳秒 |
| 无竞争的 mutex 加解锁（futex 快路径，不进内核） | ~十几~几十纳秒 |
| 有竞争、需要陷入内核挂起/唤醒的 mutex | ~微秒级（相当于一次上下文切换） |
| 高竞争下缓存行反复弹跳的原子量 | 可能比无竞争时慢一到两个数量级 |

这张表印证了上一课的结论：**「锁」这个词本身不贵，「等待」才贵**。
一把几乎不冲突的锁，代价接近一次原子操作；一把被几十个线程同时
疯抢的锁，代价可能暴涨到接近一次上下文切换，因为每次真正的竞争
基本都要走「陷入内核」这条路。排查「加了锁之后变慢很多」的问题，
第一步永远是看**竞争程度**，而不是纠结「用哪种锁」。

# 一张总表按数量级排序

> 把前面四节揉成一张总表，量级从小到大排开，方便随手对照。

| 数量级 | 典型操作 |
| --- | --- |
| ~1 纳秒 | 寄存器/L1 缓存访问、无竞争原子操作 |
| ~10~100 纳秒 | L2/L3 缓存访问、无竞争 mutex 加解锁、Rust async 任务切换 |
| ~100 纳秒~1 微秒 | 主内存访问、一次系统调用的模式切换 |
| ~1~10 微秒 | 一次线程上下文切换、有竞争的锁 |
| ~10~100 微秒 | SSD 随机读、goroutine 创建 |
| ~毫秒级 | 一次线程创建、一次进程创建、机械盘寻道、同机房网络往返 |
| ~几十~上百毫秒 | 跨地域网络往返 |

这张表和 [《CPU 与存储层次》](cpu-memory.md) 那张
「三种并发单元」对照表是一回事的两种切法：那张表按「进程/线程/
用户态任务」切，这张表按「操作类型」切——两张表拼起来，就是
「什么值得用什么单元去做」这件事的完整数量级依据。

# 数字要对上号才算懂

> 光记数量级表容易变成「背了但用不上」，配几个真实会发生的场景，
> 把表格里的行和实际症状对上号。

**案例一：日志同步写，拖慢了整条请求链路。** 一个接口平时耗时
1~2 毫秒，某天开始偶尔飙到几十毫秒，代码没有任何改动。排查后
发现是磁盘偶尔出现短暂拥堵，而这个接口在关键路径上用的是**同步
写日志**（每条请求都直接 `write` 一次到日志文件，不带缓冲）。
对照数量级表：一次系统调用本身只要零点几微秒，但如果这次
`write` 恰好撞上磁盘 IO 排队（机械盘寻道 ~1~10 毫秒、SSD 偶发
抖动也能到几百微秒），这次「顺手记个日志」的开销，比它要记录的
那次业务逻辑本身还贵出一两个数量级。修复方式是把日志写入换成
带缓冲、异步刷盘的实现（比如攒够一批或者定时 flush），
把「记日志」的成本从「请求路径上的同步系统调用」搬到「后台批量
处理」，量级差着好几档，效果立竿见影。

**案例二：逐字节读文件，慢得不成比例。** 一段读取大文件逐行处理
的代码，用 `read()` 一次读一个字节，处理一个几十 MB 的文件却要
跑几十秒。原因很直白：每读 1 字节就发生一次系统调用（模式切换，
~零点几到几微秒），文件有几千万字节，系统调用次数就是几千万次——
仅切换开销累加起来就是几十秒，还没算真正的数据搬运。换成带缓冲的
读取（`BufReader`，内部攒够一个缓冲区大小，比如 8KB，才真正触发
一次系统调用），系统调用次数直接砍到几千次，耗时从几十秒降到
毫秒级——这正是「批量读写省的是系统调用次数」这句话在真实代码里
的样子。

**案例三：加锁之后反而变慢，但「锁」不是真凶。** 一段代码在
低并发时表现良好，并发数上来后，加了锁保护的那部分代码耗时暴涨。
第一反应容易是「锁太贵，去掉」，但按上一节「锁本身不贵、等待才贵」
的结论，正确的下一步是先测**竞争程度**：用 `perf stat`/自制计数器
统计「等锁的时间占比」，发现是因为临界区里恰好还包了一次网络调用
（等下游服务响应），导致每个线程持锁的时间被下游延迟拉得很长，
其他线程只能排更久的队。修复方式不是去掉锁，而是把网络调用挪到
临界区外面，让临界区回归到「只做几条指令的内存操作」——量级对上号
之后，问题出在「持锁时间」而不是「锁本身」，方向完全不同。

三个案例的共同结构：**症状（变慢）本身不会告诉你原因是哪一层**，
必须靠数量级表 + 工具去缩小范围，缩小范围之后，修复方案往往
是把某个操作从「贵的量级」搬到「便宜的量级」，而不是单纯删代码
或者加代码。

# 先量再优化

> 本课最重要的一条工作方法：**任何性能优化，第一步永远是测量，
> 不是猜**——这条原则比本课任何一张表格都更值得记住。

「先量再优化」不是一句正确却空洞的口号，它对应一套具体可执行的
工作流：

```
① 先确认「慢」是真实存在的、可复现的问题
   —— 而不是凭印象觉得"这里应该会慢"
② 用工具定位耗时在哪一层
   —— 系统调用多？锁竞争高？缓存未命中多？网络往返次数多？
      （本课前四节的表格，就是回答"哪一层"用的参照系）
③ 只针对定位出的那一层做改动，改完立刻重新测量
   —— 不要一次性改十个地方，改完发现变快了却说不清是哪个改动起了作用
④ 用同样的测量方法对比改动前后的数字，确认真的变快了、
   变快的幅度是否值得这次改动引入的复杂度
```

**为什么这个顺序不能反过来**：性能问题的直觉判断经常是错的——
开发者往往会凭经验怀疑「这个算法复杂度不够优」，但真实瓶颈可能是
一次不起眼的日志同步写（案例一）或者一次可以合并的系统调用
（案例二）。凭直觉优化的风险是：花大量时间重写了一个「看起来慢」
但其实占总耗时比例很小的部分，总耗时几乎没有变化，真正的瓶颈
仍然原地不动——这类无效优化在工程实践里比想象中更常见。

这也是为什么「过早优化是万恶之源」这句流传很广的话，准确的下半句
应该是「在测量之前做的优化，大概率是无效优化」——不是说优化本身
有害，是说**没测量支撑的优化方向，命中真正瓶颈的概率并不高**。
一个务实的判断标准：如果你说不出「这次改动大概能省下百分之多少的
耗时、这个数字是怎么估出来的」，大概率还没到可以动手改代码的
阶段，应该先回到第②步，把测量工具跑起来。

# 用工具粗看而不是靠猜

> 光记数字没用，遇到问题要能上手用工具验证——这里给最低成本的
> 几件。

**Linux：**

- `vmstat 1`：每秒刷新一次系统概况，重点看 `cs`（上下文切换
  次数/秒）、`us`/`sy`（用户态/内核态 CPU 占比——`sy` 异常高
  往往意味着大量系统调用或锁竞争）；
- `top`/`htop`：看单个进程的 CPU 占比、线程数；`htop` 能展开
  看到每个线程；
- `perf stat <命令>`：跑一遍目标程序，报告缓存命中率、
  上下文切换次数等硬件计数器数据，是「猜」和「量」之间的分界线；
- `strace -c <命令>`：统计程序调用了哪些系统调用、各调用多少次、
  各花了多久——排查「怀疑系统调用太频繁」的第一件武器。

**Windows：**

- **任务管理器 - 性能页**：CPU/内存总览，「详细信息」标签页
  能加「线程数」「上下文切换」等列；
- **资源监视器**（`resmon`）：比任务管理器更细的每进程 CPU/句柄
  （Windows 里的 fd）统计；
- **Process Explorer**（Sysinternals 出品）：能看到每个线程的
  调用栈、上下文切换计数，是 Windows 上最接近 `perf`/`strace`
  的免费工具。

🔬 底层视角：这些工具报告的都是**内核维护的统计计数器**，
不是猜测——比如 `vmstat` 的 `cs` 列直接来自内核调度器每次真实
切换时的自增计数。工具本身几乎不改变系统行为（开销很低），
可以放心用在怀疑有问题的生产环境上做初步排查。

# 动手实验

1. 运行 `cargo run -p os-perf-cost`，程序会分别测量「无竞争原子
   操作」「无竞争 mutex」「高竞争 mutex」「一次系统调用」的
   实测耗时，和本课表格里的数量级对照；
2. Linux：跑一个 CPU 密集的小程序，同时开一个终端跑 `vmstat 1`，
   观察 `us` 占比接近 100%、`cs` 很低；再跑一个开了大量线程互相
   抢锁的版本，观察 `cs` 飙升；
3. Windows：打开任务管理器「详细信息」页，右键列头加上
   「上下文切换（增量）」列，对比空闲程序和高并发服务器程序的
   数值差异；
4. （可选，Linux）用 `strace -c` 跑一遍 [《从零手写 HTTP》](../http/http-from-scratch.md)
   的第 0 步（纯阻塞单线程版）压测，数一下 `read`/`write` 系统调用
   的总次数，直观感受「批量读写」能省掉多少次调用；
5. 自己写一个「逐字节 `read`」和一个「`BufReader` 包一层」的对照
   实验，读同一个几十 MB 的文件，用 `time` 命令分别计时，
   把结果和「案例二」里的数量级推算对上号；
6. 挑一段你自己项目里「感觉可能慢」的代码，先不要改，
   用本课的工具量一遍，写下你的猜测和实测结果的差异——
   多数人第一次做这个实验都会发现猜测和实测不完全一致。

# 三句话带走

1. **每差一层存储/每差一种切换，耗时大约差 1~2 个数量级**——
   寄存器→缓存→内存→磁盘→网络，记比例关系比记具体纳秒数更耐用。
2. **系统调用和上下文切换都不是免费的**（微秒级 + 缓存变冷）——
   批量化（缓冲 IO、epoll 批量查询）省的正是这部分开销；
   **锁本身不贵，等待才贵**，竞争程度决定实际代价。
3. **遇到性能问题先用工具量，别凭感觉猜**：Linux 用
   `vmstat`/`perf stat`/`strace`，Windows 用任务管理器/
   Process Explorer，看的是内核真实的统计计数器。

# 附：本课生词表

- **模式切换（用户态 ↔ 内核态）** —— 每次系统调用要经历的
  CPU 特权级切换，本身有固定开销。
- **缓存变冷** —— 上下文切换后，新线程发现核心的缓存里是别人
  用过的数据，需要重新从内存加载。
- **延迟数字表（latency numbers）** —— 业界总结的「各类操作大致
  耗时数量级」参考表，相对比例比绝对数字更稳定。
- **`vmstat`** —— Linux 系统级概况工具，`cs` 列是每秒上下文
  切换次数。
- **`perf stat`** —— Linux 性能计数器工具，报告缓存命中率、
  切换次数等硬件级统计。
- **`strace -c`** —— 统计一个程序调用了哪些系统调用、
  各多少次、各耗时多久。
- **Process Explorer** —— Windows 上的进程/线程深度观察工具，
  Sysinternals 套件出品。
- **资源监视器（resmon）** —— Windows 内置的、比任务管理器更细的
  每进程资源统计工具。
- **先量再优化** —— 性能优化的工作方法：先用工具定位耗时在哪一层，
  再针对性改动并重新测量，而不是凭直觉猜测瓶颈位置。
- **无效优化** —— 花时间重写了「看起来慢」但实际占比很小的部分，
  总耗时几乎没有变化，真正瓶颈仍未被触及的常见误区。
