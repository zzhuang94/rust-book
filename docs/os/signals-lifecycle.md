# 信号与进程生命周期

> 代码：`code/os-signals-lifecycle/`　运行：`cargo run -p os-signals-lifecycle`

前置：[《进程与线程》](process-thread.md)（进程是运行实例）。  
本课讲清「信号」这个操作系统概念的直觉，以及进程从「活着」到  
「彻底消失」之间会经过的几种特殊状态（僵尸、孤儿）。  
[《中间件与优雅退出》](../http/middleware-shutdown.md) 讲的是「axum 服务  
怎么接住 Ctrl-C 再收尾」；[《生产任务生命周期》](../async/service-lifecycle.md)  
讲的是「多个 tokio 任务怎么被统一喊停」——本课是这两课共同的  
操作系统地基，讲完之后回那两课看具体写法。

----

# 信号是什么

> 一句话：信号是操作系统发给进程的一份「插队通知」，  
> 不需要进程主动去查询。

正常情况下，进程按自己的代码顺序往下跑；**信号（signal）**  
是内核（或者另一个进程）随时可以「塞」给目标进程的一个通知，  
进程会被打断去处理它（除非提前声明「不处理」或「忽略」）。  
概念上很像——但绝不等同于——「软件层面的硬件中断」：  
一个正在执行的东西，被外部事件打断去做别的事。

信号只是一个小整数编号 + 一个名字，没有携带任何数据（这一点  
和后面 [《生产任务生命周期》](../async/service-lifecycle.md) 里  
`CancellationToken` 这种能传递上下文的取消信号不同——**信号是  
操作系统级的、极简的「发生了某件事」通知**，程序自己决定收到后  
干什么）。

# 常用信号速查表

> 不需要记住全部信号，记住这几个日常会遇到的就够用。

| 信号 | 触发场景 | 默认行为 | 能否被程序捕获/忽略 |
| --- | --- | --- | --- |
| `SIGINT` | 终端按下 Ctrl-C | 终止进程 | 能 |
| `SIGTERM` | `kill <pid>`（默认）、容器/编排系统请求停止 | 终止进程 | 能 |
| `SIGKILL` | `kill -9 <pid>` | 立即终止 | **不能**，内核强制执行 |
| `SIGHUP` | 终端关闭、传统上用于「重新加载配置」 | 终止进程 | 能 |
| `SIGCHLD` | 子进程状态变化（结束/停止） | 忽略 | 能 |
| `SIGPIPE` | 往一个已经没人读的管道/socket 写数据 | 终止进程 | 能 |

**`SIGTERM` 和 `SIGKILL` 的区别是本课最重要的一对概念**：  
`SIGTERM` 是「请你自己收拾好再走」的礼貌请求，程序可以捕获它、  
做完清理工作再真正退出；`SIGKILL` 是「立刻死」，程序**完全没有  
机会**执行任何代码，内核直接把它从进程表里摘掉——这就是为什么  
「优雅退出」的实现，靠的永远是捕获 `SIGTERM`（或 Ctrl-C 触发的  
`SIGINT`），而不可能靠捕获 `SIGKILL`（根本捕获不到）。

Windows 没有完全对等的信号机制，但 Ctrl-C 事件、  
服务控制管理器发出的「停止服务」请求，在效果上分别对应  
`SIGINT`/`SIGTERM` 这两种「礼貌请求」的角色。

# 优雅退出的套路

> 把「收到停止请求」到「进程真正退出」之间，标准做法拆成清楚的  
> 几步——这也是 [《中间件与优雅退出》](../http/middleware-shutdown.md)  
> 和 [《生产任务生命周期》](../async/service-lifecycle.md) 具体代码要  
> 实现的骨架。

```
① 收到 SIGTERM/Ctrl-C（SIGINT）
② 停止接受新请求/新任务（关掉监听 socket，或者不再从队列取新活）
③ 等在途的请求/任务做完（给一个合理的超时，别无限等）
④ 释放资源（关数据库连接池、落盘缓存、flush 日志）
⑤ 进程退出
```

- 第 ③ 步的「超时」很关键：如果卡住的请求永远等不完，  
  进程会一直不退出——生产环境的编排系统（如容器运行时）  
  通常会给一个固定的宽限期，超时后直接发 `SIGKILL` 强制杀掉，  
  所以「自己设的超时」应该短于「外部强杀的宽限期」，  
  否则清理工作根本来不及做完；
- Rust 里接住这些信号的标准做法是 `tokio::signal::ctrl_c()`  
  （Ctrl-C）和 `tokio::signal::unix::signal(SignalKind::terminate())`  
  （`SIGTERM`，仅 Unix）；
- Go 里对应 `signal.Notify(ch, syscall.SIGTERM, os.Interrupt)`，  
  思路完全一样：注册一个通道，信号到达时往通道塞一个值，  
  主逻辑用 `select` 收它。

```rust
// Rust：等 Ctrl-C 或 SIGTERM 任意一个先到
use tokio::signal;

async fn wait_for_shutdown_signal() {
    let ctrl_c = signal::ctrl_c();
    #[cfg(unix)]
    let mut term = signal::unix::signal(signal::unix::SignalKind::terminate())
        .expect("安装 SIGTERM 处理器失败");
    #[cfg(unix)]
    tokio::select! {
        _ = ctrl_c => {}
        _ = term.recv() => {}
    }
    #[cfg(not(unix))]
    { ctrl_c.await.ok(); }
}
```

```go
// Go：signal.Notify 是同样的心智
ch := make(chan os.Signal, 1)
signal.Notify(ch, os.Interrupt, syscall.SIGTERM)
<-ch // 阻塞直到收到任意一个
```

具体怎么把这个「收到信号」的事件，广播给正在跑的多个任务、  
让它们各自安全收尾，是 [《生产任务生命周期》](../async/service-lifecycle.md)  
`CancellationToken`/`select` 那几节的内容，本课只负责讲清  
「信号从哪来、意味着什么」这一层。

# 僵尸进程是什么

> 一个已经死了、却还占着进程表一个位置的「幽灵」。

进程结束时，它的大部分资源（内存、fd）会被内核立即回收，  
但**它的退出状态码**（成功还是失败、失败的具体原因）需要留一份，  
供父进程查询。这个「只剩一个退出状态、其他资源都已释放」的  
残留进程条目，就叫 **僵尸进程（zombie process）**。

```
子进程结束 → 内核把它变成僵尸（进程表里还留一条，状态=Z）
父进程调用 wait()/waitpid() → 内核把僵尸彻底清除、把退出状态交给父进程
```

**僵尸进程只有在父进程「忘记」调用 `wait`/`waitpid` 时才会长期  
积压**——正常情况下，僵尸态只存在极短的一瞬间。如果一个长期  
运行的父进程反复 fork 子进程却从不 `wait`，僵尸会越攒越多，  
每个僵尸占用进程表里的一个条目，达到系统进程数上限后新进程会  
创建失败——这是排查「明明没多少活跃进程，`fork` 却失败」这类  
问题时该想到的可能性。

Rust 的 `std::process::Command::spawn()` 返回的 `Child`，  
在其 `Drop` 里**不会**自动帮你 `wait`（需要显式调用  
`child.wait()`，或者用 `Child::wait()`/`try_wait()`）——  
这是一个容易踩的坑：如果只是 `spawn` 之后就把 `Child` 扔掉，  
子进程结束后会短暂变成僵尸，直到某个时机被回收（进程退出时  
内核会做最终清理，但长期运行的父进程里最好显式 `wait`）。

# 孤儿进程归谁管

> 僵尸是「子进程死了、父进程还在但没收尸」；  
> 孤儿是反过来——「父进程先死了、子进程还活着」。

如果父进程在子进程还在运行时就退出了，这个子进程就变成了  
**孤儿进程（orphan process）**——Unix 系统的解法很干脆：  
内核会把孤儿进程「过继」给 **PID 为 1 的 init 进程**  
（现代 Linux 通常是 `systemd`，容器场景下常常是容器自己指定的  
一个极简 init）。收养它的新父进程负责在它最终退出时调用  
`wait`，避免它变成永久僵尸。

这个机制解释了一个容器里的常见坑：如果容器里的主进程（PID 1）  
自己没有正确实现「收养并 wait 孤儿」的逻辑（大多数普通应用程序  
根本没考虑过这件事，因为正常情况下应用不需要关心），容器里跑  
起来的子进程如果意外变成孤儿，可能一直堆积成僵尸——这也是很多  
容器镜像会引入一个极简 init 进程（如 `tini`、`dumb-init`）作为  
PID 1 的原因，本课后面 [《容器不是轻量 VM》](cgroup-container.md)  
会再提到这一点。

# 一张进程终态对照表

> 把「僵尸」「孤儿」和正常退出放在一起看清区别。

| 情形 | 谁还活着 | 谁负责收尾 |
| --- | --- | --- |
| 正常退出 | 都活着，父进程按流程 `wait` | 父进程 |
| 僵尸进程 | 只有父进程活着，子进程已死但没被 `wait` | 父进程（迟迟没做） |
| 孤儿进程 | 只有子进程活着，父进程已死 | 被过继给 init（PID 1） |

# 动手实验

1. 运行 `cargo run -p os-signals-lifecycle`，观察程序捕获  
   Ctrl-C 后打印收尾日志再退出，对比直接 `kill -9 <pid>` 时  
   完全没有任何收尾输出；
2. （Linux/macOS）另开终端 `kill -TERM <pid>` 和  
   `kill -KILL <pid>` 分别发给同一个测试进程，对照它们的  
   反应差异；
3. （Linux）写一个 fork 子进程后父进程不调用 `wait` 就继续跑的  
   最小示例，用 `ps aux | grep Z` 观察子进程结束后短暂出现的  
   僵尸状态（`STAT` 列里的 `Z`）；
4. 查阅你项目里用到的容器基础镜像的 `ENTRYPOINT`，  
   看它有没有用 `tini`/`dumb-init` 之类的极简 init 包一层——  
   如果没有，想一想你的主进程代码是否可能产生孤儿子进程。

# 三句话带走

1. **信号是内核发给进程的极简通知**，`SIGTERM`/`SIGINT` 可以被  
   程序捕获后自行收尾，`SIGKILL` 完全无法捕获——这决定了「优雅  
   退出」只能靠捕获前两者实现。
2. **优雅退出的标准骨架**：收到信号 → 停止接受新活 → 限时等在途  
   任务收尾 → 释放资源 → 退出；超时时长要短于外部强杀的宽限期。
3. **僵尸进程 = 子死父未收尸；孤儿进程 = 父死子还活着**——  
   Unix 用「过继给 PID 1」解决孤儿问题，容器场景常需要一个极简  
   init 进程来正确处理这两种情形。

# 附：本课生词表

- **信号（signal）** —— 内核发给进程的极简异步通知，  
  只是一个编号，不携带数据。
- **`SIGTERM`** —— 礼貌的「请停止」请求，可被程序捕获后自行收尾。
- **`SIGKILL`** —— 强制立即终止，进程无法捕获也无法忽略。
- **`SIGINT`** —— 终端 Ctrl-C 触发的信号，语义上和 `SIGTERM` 相近。
- **优雅退出（graceful shutdown）** —— 收到停止信号后，  
  先完成在途工作、释放资源，再真正退出的流程。
- **僵尸进程（zombie）** —— 子进程已结束但父进程尚未 `wait`，  
  只在进程表留一条「退出状态」记录的残留态。
- **孤儿进程（orphan）** —— 父进程先退出、子进程仍在运行，  
  会被过继给 PID 1（init）。
- **`wait`/`waitpid`** —— 父进程查询子进程退出状态、  
  同时让内核彻底清除该子进程条目的系统调用。
- **PID 1 / init** —— 系统里第一个启动的进程，负责收养孤儿进程；  
  容器里常用 `tini`/`dumb-init` 扮演这个角色。
