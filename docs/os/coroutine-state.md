答案的核心词只有一个：**状态机**。

# 生活里的状态机

> 不着急上代码，先用两个生活例子建立直觉。

**红绿灯** 就是一台状态机：它任何时刻处于
{红、黄、绿} 这三个状态中的 **一个**；收到「计时器到点」这个
**事件**，就按预先定好的规则 **转移** 到下一个状态（红→绿→黄→红）。
红绿灯不需要一个人 24 小时盯着它、记着「现在该变成什么颜色」——
状态本身就是那个「记忆」。

**自动售货机** 同理：{待机 → 已投币 → 出货中 → 待机}，
投币、按下按钮、出货完成，是驱动它在这几个状态之间转移的事件。
你往里投一枚硬币，售货机不需要“记住你之前做了什么”——它现在
处于「已投币」这个状态，这个状态本身就包含了「已经知道你投了钱」
这个信息。

抽象出来，一台状态机由三样东西组成：

1. **一组可能的状态**（红绿灯的三种颜色）；
2. **当前处于哪个状态**（此刻是红灯）；
3. **一张「收到什么事件就转移到哪个状态」的规则表**。

它最可贵的地方在于：**「进行到哪一步了」不再藏在某个人的脑子里、
不再需要专门的人力去盯着，而是变成了一个普通的、可以拿在手里的值**——
红绿灯的当前颜色可以用一个字符串或者一个数字表示，可以被读取、
被写入日志、被另一个系统查询。这句话看起来平淡，但下一节会揭示
它在程序设计里的巨大威力。

# 把可暂停的过程改写成状态机

> 上一节的比喻要落到代码上：程序里一个「中途要等待」的函数，
> 可以被改写成状态机，改写之后，「暂停」这件事的成本会发生质变。

先看一个中途要等待的函数，用最朴素的顺序写法：

```
fn 打招呼() {
    做准备工作();
    等定时器 3 秒();      // ← 在这一行"暂停"，等 3 秒
    打印结果();
}
```

这段代码「暂停」的那一刻，谁在负责“记住这件事进行到哪了”？
答案是：**这个函数所在的那条执行流的寄存器和栈**——「下一条要执行
的指令在哪」「局部变量的值是什么」全部散落在寄存器和栈帧里。
如果要让这条执行流暂停、把 CPU 让给别人干别的事，
必须有人把这些寄存器和栈的内容 **原样保存下来**，等恢复时再
**原样加载回去**——这正是 [《调度决定谁在跑》](scheduling.md)
讲的「上下文切换」，微秒级，还伴随着缓存变冷。

现在换一种写法，把「执行到哪了」显式地变成 **一个普通的值**：

```
enum 打招呼状态 {
    还没开始,
    正在等定时器 { 已经等了多久: u64 },   // ← 暂停点，变成了一个状态
    已完成,
}

// 每被"推一下"（后面会讲这个"推"就叫 poll），执行一小步：
fn 推进(当前状态: &mut 打招呼状态) {
    match 当前状态 {
        还没开始 => {
            做准备工作();
            *当前状态 = 正在等定时器 { 已经等了多久: 0 };
        }
        正在等定时器 { 已经等了多久 } => {
            if *已经等了多久 >= 3 {
                打印结果();
                *当前状态 = 已完成;
            }
            // 还没到 3 秒，什么都不做，直接返回——这就是"暂停"
        }
        已完成 => { /* 没有更多事情要做 */ }
    }
}
```

这个改写带来三条决定性的好处，是这一课最重要的结论：

- **暂停 = 函数返回，恢复 = 再调用一次函数**——都是普通的函数调用，
  纳秒级，**不需要保存/恢复任何寄存器**（因为「执行到哪了」这个信息
  已经变成了 `当前状态` 这个普通变量的值，本来就一直好好地
  待在内存里，从没「消失」过，无需专门去「保存」）；
- 状态机是 **普通数据**（这里是一个枚举值，占几十字节）：
  可以放进数组、可以在多个线程之间传递、可以被任何一个线程
  捡起来「推一下」——这为后面「工作窃取（work-stealing）」这类
  调度技巧打开了大门；
- 一个线程可以轮流「推」**成千上万台** 这样的状态机——
  这就是「用户态任务」能用少量线程扛住海量并发的全部秘密。

# 两条实现路线：栈快照 vs 编译期改写

> 「把可暂停的过程变成状态机」这件事，Go 和 Rust 选了两条完全不同的路，
> 分野从这里开始。

**Go 的路：goroutine 保留真实的栈和寄存器快照。** 你写 Go 代码时，
完全不需要标注哪个函数是「可能会暂停的」，写法和普通函数一模一样，
可以在任意一行暂停（比如调用一个会阻塞的操作）。Go 运行时在背后
维护每个 goroutine 自己的一份栈（初始很小，约 2KB，可按需增长）
和寄存器快照，暂停时由运行时的调度器负责把这些内容保存下来，
恢复时加载回去——**本质上和 OS 线程的上下文切换是同一套思路，
只是这套「保存/恢复」的动作被挪到了用户态**，不需要陷入内核，
所以比 OS 线程的上下文切换快得多，但仍然是「保存一整份执行现场」，
不是本课主推的「状态变成数据」那种改写。

**Rust 的路：编译器自动把 `async fn` 改写成状态机。**
你写的 `async fn` 里每一个 `.await`，编译器都会把它当成一个
状态边界，跨越这个边界还要用到的局部变量，会被编译器自动收进
生成的枚举类型的字段里——这正是上一节手写的那个「打招呼状态」
枚举，只是真实场景里编译器帮你写、而且状态可能嵌套得很复杂。
生成出来的这个状态机类型实现了一个叫 `Future` 的 trait，
它的核心方法叫 `poll`——对应上一节手写代码里的「推一下」；
`Future` 和 `poll` 的完整细节留给
[《async 基础》](../async/basics.md)，这一课只需要记住：
**Rust 异步任务本质就是一个编译器生成的状态机枚举**，
不需要保存/恢复寄存器和栈，因为「执行到哪了」已经是这个枚举值
本身的一部分。

两条路线对比：

| | Go（goroutine） | Rust（async 任务） |
| --- | --- | --- |
| 暂停点能否随处发生 | 能，任意一行都可能暂停 | 只能在 `.await` 处（编译器强制标注） |
| 谁负责「记住执行到哪了」 | 运行时保存真实的栈+寄存器快照 | 编译器生成的状态机枚举本身就是这个记录 |
| 初始内存占用 | 每个 goroutine 约 2KB 起，按需增长 | 几十到几百字节（状态机枚举的大小），不需要独立栈 |
| 需不需要在源码里标注 | 不需要，写法和普通函数一样 | 需要，函数要标 `async`，暂停点要写 `.await` |
| 编译器/运行时的角色 | 运行时更重（要管理栈增长、调度） | 编译器更重（要在编译期做状态机改写），运行时（Tokio）更薄 |

这条分野解释了很多后续课程会反复提到的现象：Go 的 `async` 味道
「隐形」，你甚至不会意识到 goroutine 在暂停恢复；Rust 的
`async`/`.await` 是「显式」的，你必须在函数签名和调用处主动标注——
这不是 Rust 更啰嗦，而是 Rust 选择把「暂停点在哪」这个信息
交给编译器在编译期就搞清楚，换来的是运行时不需要维护独立栈、
状态机可以做得更小、切换可以做到真正的零额外内存分配。

# 用户态任务补全成本阶梯

> 现在可以把 [《程序与线程》](process-thread.md) 那张成本表补完整了。

| | 进程 | OS 线程 | 用户态任务（goroutine / tokio task） |
| --- | --- | --- | --- |
| 独有内存 | 整套地址空间 | 栈，默认约 8MB | goroutine 约 2KB 起；tokio 任务是状态机枚举，几十到几百字节 |
| 创建耗时 | 毫秒级 | 微秒级 | 纳秒级 |
| 切换耗时 | 最贵（连地址空间映射都要换） | 1~10 微秒 + 缓存变冷（陷入内核） | 纳秒级，用户态完成，不陷入内核 |
| 谁负责调度 | OS 内核 | OS 内核（抢占式） | Go 运行时（准抢占）/ Tokio（协作式） |
| 崩溃影响 | 只死自己 | 整个进程陪葬 | Go：进程陪葬；Tokio：包成错误，进程不死 |
| 一个线程/核心能扛多少个 | 受内存和内核调度压力限制，通常几十到几百 | 同上 | 单线程轻松扛几万到几十万个 |

这张表现在完整地回答了「为什么需要用户态任务」：**当并发量远远
超过核心数（比如一万个网络连接），进程和线程的创建/切换成本会
成为瓶颈，而状态机式的用户态任务几乎不受这个瓶颈限制**——
这正是 [《IO 模型三代》](../network/layers.md) 里从「每连接一线程」
进化到「用户态任务」的核心动机，也是
[《从零手写 HTTP》](../http/http-from-scratch.md) 三步走要走到最后一步的理由。

# 别急着学 async，先记住这个模型

> 这一课只铺地基，故意没有展开 `Future`/`Waker`/`poll` 的具体代码，
> 是希望你先把「状态机」这个心智模型吃透，再去看真实的 async 代码时，
> 每一行都能对上号。

带着这一课的心智模型，去读 [《async 基础》](../async/basics.md) 时，
可以预先建立这几个对应关系（届时会展开成完整代码）：

- 你写的 `async fn xxx() { ... }` ≈ 这一课手写的那个「状态」枚举
  + 「推进」函数，只是编译器帮你自动生成；
- `.await` ≈ 这一课「正在等定时器」那个状态分支——「如果还没准备好，
  就原地返回，不往下走」；
- `poll` ≈ 这一课的「推进」函数，每次被调用，尝试往前推进一小步；
- `Waker` ≈ 一个「事情准备好了，请再来推我一下」的回调通知机制，
  解决的问题是「不该傻等着反复瞎猜『现在能推进了吗』，而是等真正
  准备好了主动通知」；
- `tokio::spawn` ≈ 把一个状态机交给运行时的调度器，等着被反复推进
  直到跑完。

Go 那边对应去读 [《Go 与 GMP 调度》](../concurrency/go-gmp.md)，
会看到「goroutine 保留真实栈快照」这条路线具体怎么落地成
G（goroutine）/M（machine，OS 线程）/P（processor，调度上下文）
三个字母的模型，以及 Go 运行时怎么在这三者之间搬运工作。

# 协程不是更轻的线程

> “协程/异步任务无非是更省资源的线程”——这句话说对了「更省资源」，
> 但漏掉了一个决定性的行为差异，这个差异会直接影响你怎么写代码。

线程（无论是 OS 线程还是概念上的“轻量级线程”）由 **调度器强制
抢占**——[《调度决定谁在跑》](scheduling.md) 讲过，时间片用完，
不管线程愿不愿意，都会被换下去，这保证了「一个线程写死循环，
不会永远卡住整台机器」。**Rust 的 async 任务完全不是这样**：
Tokio 的调度是 **协作式** 的，一个任务只有在代码主动执行到
`.await` 时，才会给调度器一个「可以考虑换别的任务跑」的机会。
如果你在一个 async 任务里写了一个不含任何 `.await` 的死循环，
或者调用了一个耗时很长又不 `.await` 的同步函数，**这个任务会
一直霸占底下的 OS 线程，Tokio 调度器对此完全无能为力**——不是
“不想管”，而是协作式调度天生没有「强制抢占」这个手段。

Go 的 goroutine 情况稍好一些但也不是绝对安全：Go 运行时实现了
**协作式为主、辅以一定程度的信号抢占（sysmon 定期检测长时间未
让出的 goroutine，尝试插入抢占点）**，但这套机制仍然主要依赖
函数调用、循环回边等「主动检查点」，一个真正精心构造的、
不含任何函数调用的纯计算死循环，同样可能长时间不被抢占。
**一句话总结这一节最重要的结论：协程更轻，但“轻”是用“调度器
管不住死循环”换来的，业务代码里避免写死循环/避免长时间不让出，
在协程世界里比在线程世界里更重要，不是可选的代码风格问题**。

# 死循环卡住全部请求

> 把上一节的结论落地成一个具体会发生的生产事故，理解它之后，
> 再看 [《async 基础》](../async/basics.md) 的铁律会更有说服力。

假设一个 Tokio 服务的默认配置是 4 个工作线程，同时驱动着几千个
并发连接对应的异步任务。某个请求触发了一段业务逻辑，这段逻辑里
不小心调用了一个耗时较长的同步函数（比如一次没有用异步版本的
正则匹配、一次同步的图片压缩、甚至只是不小心写成了
`std::thread::sleep` 而不是 `tokio::time::sleep`），且这段代码里
没有任何 `.await`。

结果是：负责跑这个任务的那个 Tokio 工作线程，被这一个请求
**整整占用**，直到这段同步逻辑跑完。这段时间里，**恰好被调度到
同一个工作线程上的其他所有任务，全部停止前进**——不是变慢，
是彻底停摆，哪怕它们本身逻辑很简单、原本几毫秒就能跑完。
从外部观察，现象是「服务大部分请求都很快，但偶尔有一小撮请求
莫名其妙卡了很久」，卡住的时长恰好和那个「元凶」同步调用的耗时
对得上——这是排查 Tokio 服务「间歇性延迟毛刺」问题时，
第一个该怀疑的方向。

标准修复方式是把这段耗时的同步逻辑挪到专门的线程池上跑：

```rust
// 坏例子：耗时的同步计算直接写在 async 函数里，没有任何 .await
async fn handle_request() {
    let result = expensive_sync_computation(); // 霸占工作线程直到算完
    send_response(result).await;
}

// 好例子：用 spawn_blocking 把耗时同步计算挪到专门的阻塞线程池
async fn handle_request() {
    let result = tokio::task::spawn_blocking(|| expensive_sync_computation())
        .await
        .expect("阻塞任务 panic");
    send_response(result).await;
}
```

`spawn_blocking` 背后维护着一个独立的、可以适度扩容的线程池
（本质就是这一整个 `docs/os/` 系列讲的「货真价实的 OS 线程」），
把耗时的同步工作丢给它们去扛「阻塞」这件事，Tokio 的核心工作
线程则继续专心推进其他真正轻量的异步任务——这也是为什么这个
方法叫 `spawn_blocking`（生成一个可以安心阻塞的任务），
而不是让你在原来的异步任务里硬扛。

# 该用线程还是该用协程

> 把这一课和前两课的结论汇总成一张决策表，业务代码里遇到
> “这段逻辑该开线程还是该丢给 async 任务”的选择时可以直接对照。

| 场景 | 建议 | 理由 |
| --- | --- | --- |
| CPU 密集型计算（图像处理、复杂计算、加密解密） | OS 线程（数量约等于核心数） | 这类工作本来就要真正占用 CPU，用协程包一层不会变快，还多一层调度开销 |
| 会调用同步阻塞 API 且逻辑较重（同步文件 IO、调用没有异步版本的库） | OS 线程，或 `spawn_blocking` | 协作式调度扛不住真正的阻塞/长耗时同步代码 |
| 海量网络连接、每个连接大部分时间在等 IO（HTTP 服务、代理、聊天服务） | 用户态异步任务（tokio task） | 这正是异步任务的主场：几万个任务大多数时候都在“等”，纳秒级切换成本可以忽略 |
| 简单的、快速返回的业务逻辑，穿插在异步请求处理流程里 | 直接写在 async 函数里，别加多余的线程 | 没必要为了“听起来更安全”给不耗时的逻辑加线程开销 |

判断的核心问题永远是这一课反复强调的那句话：**这段代码会不会
长时间占着 CPU 不撒手，或者会不会调用真正阻塞的系统调用**——
会，就该用线程（或者 `spawn_blocking`）；不会（大部分时间在等
网络/等其他任务），协程才是真正划算的选择。

# 异步任务卡住排错清单

> 遇到「Tokio 服务偶尔卡顿、部分请求延迟异常」时，按这个顺序排查。

| 现象 | 优先怀疑 | 排查手段 |
| --- | --- | --- |
| 延迟毛刺和某类请求（比如带某个参数）强相关 | 那类请求触发了不该在 async 里跑的阻塞逻辑 | 审查该请求路径上有没有同步 IO、`std::thread::sleep`、重计算 |
| `tokio-console` 显示某个任务 poll 耗时异常长 | 找到了具体是哪个任务在霸占线程 | 用 `tokio-console` 或手动加耗时日志定位 |
| 增加 Tokio 工作线程数之后毛刺缓解但没消失 | 阻塞逻辑仍然存在，只是稀释到更多线程上 | 治标不治本，还是要找到并挪走真正阻塞的调用 |
| CPU 使用率很低但请求排队严重 | 大部分任务在等 IO，正常现象；但如果排队时间远超预期，检查下游服务延迟 | 结合 [《调度决定谁在跑》](scheduling.md) 的排错清单一起看 |

# 三句话带走

1. **状态机 = 一组状态 + 当前状态 + 转移规则**，把「一个可暂停的过程
   进行到哪了」从「藏在寄存器和栈里」变成「一个普通的、可以拿在手里
   的数据」——这个改写是用户态任务能做到纳秒级切换的全部秘密。
2. **Go 用 goroutine 保留真实栈和寄存器快照**（暂停恢复靠用户态版的
   「上下文切换」，可以随处暂停）；**Rust 用编译器把 `async fn`
   自动改写成状态机枚举**（暂停恢复靠一次普通函数调用，
   只能在 `.await` 处暂停）——两条路线殊途同归，都比 OS 线程轻。
3. **用户态任务补全了进程/线程/任务的成本阶梯**：创建和切换成本
   一路从毫秒级（进程）到微秒级（线程）再到纳秒级（用户态任务），
   这正是「单线程扛几万并发连接」在物理上成立的原因，
   完整细节留给 [《async 基础》](../async/basics.md) 和
   [《Go 与 GMP 调度》](../concurrency/go-gmp.md)。

# 附：本课生词表

- **状态机（state machine）** —— 「一组状态 + 当前状态 + 转移规则」
  组成的模型，把「进行到哪了」变成一个普通的值。
- **暂停 = 函数返回，恢复 = 再调用一次** —— 状态机式暂停/恢复的核心机制，
  不需要保存/恢复寄存器和栈。
- **用户态任务** —— goroutine、tokio task 等比 OS 线程更轻的并发单元，
  切换发生在用户态，不陷入内核。
- **goroutine** —— Go 的用户态任务，保留真实栈（约 2KB 起）和寄存器快照，
  可在任意一行暂停，运行时负责调度，详见 [《Go 与 GMP 调度》](../concurrency/go-gmp.md)。
- **`async fn` / `Future`** —— Rust 编译器把带 `.await` 的函数自动改写成
  状态机枚举，该枚举实现 `Future` trait，详见 [《async 基础》](../async/basics.md)。
- **`.await`** —— Rust 异步代码里的暂停点，编译器在这里切分状态边界。
- **`poll`** —— 推进一个 `Future` 状态机前进一步的方法，对应本课「推进」函数。
- **`Waker`** —— 通知运行时「这个任务现在值得再推一下」的回调机制，
  避免运行时靠瞎猜反复轮询。
- **GMP 模型** —— Go 运行时用 Goroutine/Machine（OS 线程）/Processor（调度上下文）
  三个角色实现用户态调度的模型。
- **协作式调度的软肋** —— 任务只在 `.await`/让出点才可能被换下去，
  死循环或长耗时同步代码会一直霸占底下的线程，调度器强制不了。
- **`spawn_blocking`** —— 把耗时同步代码丢给专门的阻塞线程池执行，
  让 Tokio 核心工作线程继续推进其他异步任务的逃生舱。
- **`tokio-console`** —— 观测 Tokio 任务调度状态（哪个任务 poll 耗时异常）的诊断工具。
