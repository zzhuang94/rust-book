# 磁盘慢是常态

> 前置：[《CPU 与内存》](cpu-memory.md) 讲了内存和缓存有多快。  
> 这一课讲“慢”的那一头——磁盘和 SSD。业务代码里的“落盘”“持久化”  
> 到底在硬件层面发生了什么，为什么有的写操作快得像没发生、有的写  
> 操作慢得让人怀疑网络挂了，这一课给你一套完整的直觉。

写业务代码时你天天说“存到数据库”“落盘”“持久化”，但这些操作  
背后到底经过了多少层、每一层慢在哪，大多数人说不清楚。这一课把  
「一次写磁盘」从业务代码到物理磁盘的完整路径拆开，讲清楚  
**延迟**（等多久）和 **吞吐**（一秒能搬多少）这两个常被混用的指标，  
以及贯穿这一课的核心词：**页缓存** 和 **fsync**。

# 机械硬盘的直觉

> 机械硬盘（HDD）里真的有一片会转动的磁盘和一根会移动的磁头，  
> 这决定了它的延迟主要花在“物理移动”上。

HDD 内部结构很直白：一张（或几张叠起来的）涂了磁性材料的圆盘  
高速旋转（常见 5400~7200 转/分钟），一根磁头悬在盘片上方，  
靠磁头的位置和盘片旋转到的位置共同定位到某个数据块，读写数据。

读写一个数据块要等两件物理动作完成：

- **寻道（seek）**：磁头移动到目标数据所在的磁道，几毫秒级；
- **旋转等待（rotational latency）**：等盘片转到目标数据经过磁头  
  下方，平均要等半圈，7200 转的盘平均约 4 毫秒。

两者加起来，HDD 随机读写一次的延迟大约在 **5~10 毫秒**——  
听起来很短，但比内存的“百纳秒级”慢了 **五万倍以上**。这也是为什么  
「随机读写一大堆小文件」在 HDD 上特别慢：每次都要重新寻道、  
重新等旋转；而 **顺序** 读写（磁头几乎不用移动，盘片转到哪就读到哪）  
可以快得多，这条“顺序远快于随机”的规律和上一课「顺序访问数组比  
跳跃访问快」是同一个道理的另一层体现（只是这里慢的原因是物理机械  
运动，不是缓存行）。

# SSD 没有会动的零件

> SSD（固态硬盘）用闪存芯片存数据，没有磁头和转盘，  
> 快了一个数量级，但仍然远慢于内存。

上一课「闪存与内存的分野」提过 SSD 用的是 NAND Flash，靠晶体管  
锁住电荷来存数据，没有任何机械运动部件。少了寻道和旋转等待这两大  
延迟来源，SSD 的随机读写延迟降到了 **几十到上百微秒**，比 HDD 快  
了 50~100 倍，但依然比内存慢 100~1000 倍。

SSD 内部也有自己的“坑”，业务程序员至少要知道存在：

- **只能按“页”读写、按“块”擦除**：典型页大小 4KB，块通常是  
  几十到上百个页拼起来。SSD 不能像 HDD 一样直接覆盖写某个字节，  
  必须先擦除整个块再写，这个机制叫 **异地更新（out-of-place update）**；
- **写入放大**：因为擦除粒度比写入粒度大得多，实际写入闪存芯片的  
  数据量往往比你请求写入的数据量更大，这个倍数叫 **写入放大系数**；
- **磨损均衡（wear leveling）**：每个存储单元的擦写次数有寿命上限  
  （几千到上万次），SSD 内部的控制器会自动把写操作分散到不同物理  
  位置，让所有单元磨损得尽量均匀，业务代码完全不用关心这一层，  
  SSD 控制器全部替你做完了。

对业务程序员的直接影响是一句经验规律：**SSD 上顺序写依然比随机写快，  
但差距比 HDD 小得多**——因为没有寻道和旋转的物理惩罚，  
只剩下页/块管理带来的间接成本。

# 延迟与吞吐是两件事

> 这两个词经常被混着用，但回答的是完全不同的问题，  
> 一张表把常见存储介质的两个指标都摆出来。

- **延迟（latency）**：发起一次操作到拿到结果，等了多久——  
  回答“这一次要等多久”；
- **吞吐（throughput）**：单位时间内能搬运多少数据——  
  回答“一秒钟能干多少活”。

一个类比：延迟是“从下单到外卖送到你手上要多久”，吞吐是“外卖平台  
一小时能送多少单”。**低延迟不等于高吞吐，高吞吐不等于低延迟**——  
一辆车跑得快（低延迟）不代表马路上能同时跑很多车（高吞吐），  
反过来一条八车道的路（高吞吐）也不代表你这一辆车能跑得特别快。

| 介质 | 随机读延迟（数量级） | 顺序吞吐（数量级） |
| --- | --- | --- |
| 内存（RAM） | ~100 纳秒 | 几十 GB/s |
| NVMe SSD | ~几十微秒 | 几 GB/s |
| SATA SSD | ~几十到上百微秒 | 几百 MB/s |
| 机械硬盘（HDD） | ~5~10 毫秒 | 几十到上百 MB/s |

这张表解释了两个业务上常见的现象：**为什么数据库把热点数据缓存在  
内存里能带来几个数量级的提速**（内存延迟比 SSD 低 100~1000 倍）；  
**为什么批量导入数据时“攒一批再写”比“来一条写一条”快得多**  
（前者利用了顺序写的高吞吐，后者每次都要单独承受一次随机写的延迟）。  
[《接入 Redis》](../http/redis.md) 把热数据整个放进内存，  
本质就是在用这张表最上面一行的速度，换掉最下面几行的速度。

# write 系统调用没有真的写盘

> 你调用一次 `write`，函数返回得很快——但这不代表数据已经躺在  
> 磁盘上了。这是本课最容易被误解的一个事实。

业务代码里写文件常见的样子：

```rust
use std::fs::File;
use std::io::Write;

let mut file = File::create("/tmp/example.log")?;
file.write_all(b"hello disk\n")?;
```

这个 `write_all` 调用几乎是瞬间返回的——但它并没有真的把数据  
送到磁盘的存储介质上。真实发生的事情是：

1. 你的数据先被复制进操作系统内核维护的一块内存区域，  
   叫 **页缓存（Page Cache）**；
2. `write` 调用一旦把数据复制进页缓存，就认为「写完了」，  
   直接返回给你的程序——这块被写过、还没同步到磁盘的内存页，  
   叫 **脏页（dirty page）**；
3. 操作系统在后台，按自己的节奏（定时、脏页太多、内存不够用等条件）  
   把脏页异步刷到真正的磁盘上，这个动作叫 **write-back**。

也就是说，**从你的角度看“写完了”，从磁盘的角度看“还完全不知道这件事”**。  
中间这段时间差，如果恰好断电或者内核崩溃，这些还没刷盘的脏页  
就会永久丢失——这是数据库、消息队列这类“不能丢数据”的系统  
必须专门处理的问题，下面 `fsync` 那一节就是答案。

> 🔬 底层视角：页缓存不只用来缓冲写，**读也一样过一遍它**——  
> 第一次读一个文件时，操作系统把文件内容从磁盘搬进页缓存，  
> 之后同一份数据再被读，直接从内存返回，根本不碰磁盘。这也是  
> 为什么“同一个文件第二次读比第一次快得多”——不是磁盘变快了，  
> 是压根没有再访问磁盘。

# 亲眼看页缓存

> 不用写代码，用系统自带的命令就能看到页缓存正在占用的内存。

Linux 上：

```bash
free -h
#               total        used        free      shared  buff/cache   available
# Mem:           16Gi        2.1Gi       8.3Gi       123Mi        5.6Gi        13Gi
```

`buff/cache` 那一列就是页缓存（以及一些内核缓冲区）占用的内存——  
数值往往会随着你读写的文件越来越多而涨上去。**这块内存不是“浪费”，  
而是操作系统主动拿闲置内存换缓存命中率**：如果某个应用程序真的需要  
更多内存，操作系统会优先回收页缓存腾出空间给它，所以这块内存  
「看起来占用很高」但其实是可以随时让出来的。

Windows 上没有完全对应的单一命令，但任务管理器「性能」页的  
「已缓存」数值反映的是类似的机制。

清空/绕过页缓存做对照实验（Linux）：

```bash
# 清空页缓存（需要 root，仅用于实验环境，生产机器别乱跑）
sync; echo 3 > /proc/sys/vm/drop_caches

# 对比同一个大文件「第一次读」和「第二次读」的耗时
time cat bigfile.bin > /dev/null   # 第一次：真的要读磁盘
time cat bigfile.bin > /dev/null   # 第二次：命中页缓存，快得多
```

# fsync 才是真正的落盘

> 想要「数据真的到磁盘上了，断电也不会丢」这个保证，  
> 必须显式调用一个专门的系统调用——`fsync`。

`fsync`（Windows 上对应 `FlushFileBuffers`）的作用是：**阻塞等待，  
直到这个文件对应的所有脏页都被真正写进磁盘的存储介质**。调用返回后，  
你才能有把握地说“这份数据即使现在断电也不会丢”。

Rust 对应 API：

```rust
use std::fs::File;
use std::io::Write;

let mut file = File::create("/tmp/important.log")?;
file.write_all(b"这条数据不能丢\n")?;
file.sync_all()?; // 阻塞直到数据和元数据真正落盘
```

对照 Go：

```go
f, _ := os.Create("/tmp/important.log")
f.Write([]byte("这条数据不能丢\n"))
f.Sync() // 等价于 fsync
```

`fsync` 的代价是显而易见的：它要老老实实等一次真正的磁盘写入完成，  
延迟直接对齐上面表格里那一行——SSD 上几十到上百微秒，HDD 上  
几毫秒到十几毫秒（很多 HDD 还要多转一圈确认写入，实际更慢）。  
**如果你的服务每处理一个请求就调一次 `fsync`，吞吐会被死死限制在  
「磁盘每秒能完成多少次同步写」这个数字上**，这正是数据库设计里  
「WAL（写前日志）批量提交」「多个事务共享一次 fsync（group commit）」  
这类优化技巧存在的原因——它们本质都是在减少 `fsync` 被调用的次数，  
用「稍微多等一点点合并写」换「显著提升的吞吐」。

# 业务代码里的落盘取舍

> “数据要不要立刻落盘”不是一道非黑即白的题，  
> 而是在“丢数据的风险”和“性能”之间选一个点。

常见的几档选择，从最快最危险到最慢最安全：

| 策略 | 崩溃/断电时的后果 | 性能 |
| --- | --- | --- |
| 只 `write`，从不 `fsync` | 最近若干秒的写可能全部丢失 | 最快，几乎不受磁盘延迟影响 |
| 定时 `fsync`（比如每秒一次） | 最多丢最近这一个周期内的写 | 较快，均摊了 `fsync` 的代价 |
| 每次关键写操作都 `fsync` | 基本不丢（除非磁盘本身损坏） | 最慢，吞吐被磁盘同步写次数限制 |
| 多副本 + 每副本各自 `fsync`（分布式共识） | 单机崩溃完全不丢 | 更慢，还要加上网络往返的延迟 |

- 日志文件、埋点数据：往往能接受“进程崩溃丢最近几秒”，  
  选第一档或第二档，图吞吐；
- 订单、支付这类不能丢的业务数据：数据库内部的 WAL 机制会在  
  提交事务时做类似 `fsync` 的保证，你作为业务代码的作者，  
  只需要正确使用数据库的事务边界，这层取舍已经被数据库开发者  
  想清楚并封装好了；
- **一个常见误区**：以为“调用了数据库的 `INSERT`/`COMMIT` 就等于  
  数据已经安全落盘”——这要看数据库和它的配置（比如 MySQL 的  
  `innodb_flush_log_at_trx_commit`），有些配置为了性能会放宽这层  
  保证，业务上如果对“绝不丢数据”有硬性要求，需要确认数据库这一层  
  的真实行为，而不是想象它。

# Go 与 Rust 的落盘 API 对照

| 动作 | Rust | Go |
| --- | --- | --- |
| 打开/创建文件 | `File::create` / `File::open` | `os.Create` / `os.Open` |
| 写数据（经过页缓存） | `file.write_all(buf)` | `f.Write(buf)` |
| 强制落盘 | `file.sync_all()` | `f.Sync()` |
| 只同步数据不同步元数据（更快） | `file.sync_data()` | 无直接对应，一般用 `Sync()` |
| 带缓冲的写（减少系统调用次数） | `BufWriter::new(file)` | `bufio.NewWriter(f)` |

两边的语义几乎一一对应——毕竟 `fsync` 是操作系统提供的能力，  
语言只是薄薄地包了一层。**真正决定性能的从来不是语言，  
是你调不调 `fsync`、多久调一次**。

# 动手实验

1. 跑本章代码里的顺序写 vs 随机写对比（在同一块磁盘上写等量数据），  
   记录耗时倍数；如果你的机器是 SSD，差距会比 HDD 小很多——  
   这本身就是一次“亲手验证 SSD 没有寻道惩罚”的实验；
2. 跑本章代码里的「只 write」vs「write 后 fsync」对比，  
   记录 `fsync` 版本慢了多少倍；把每次写的数据量从 1 条变成攒够  
   100 条再一次性 `fsync`，观察吞吐的提升——这就是「批量提交」  
   优化的原理；
3. 用 `free -h`（Linux）观察写完一个几百 MB 的大文件之后，  
   `buff/cache` 的数值变化，理解页缓存是「写了但可能还没落盘」的  
   物理载体；
4. 找一个你熟悉的数据库（比如 SQLite、MySQL），查一下它默认的  
   落盘策略（WAL 模式、`fsync` 相关配置），对照本课的“落盘取舍表”，  
   判断它默认站在哪一档。

# 三句话带走

1. **HDD 慢在物理移动（寻道+旋转，毫秒级），SSD 没有机械部件但仍受  
   页/块管理限制（微秒级）** ——两者都比内存慢 100~50000 倍不等，  
   顺序读写始终快于随机读写。
2. **`write` 只是把数据复制进内核的页缓存就返回，不代表已经落盘**；  
   真正保证“断电不丢”要显式调用 `fsync`（Rust `sync_all`/Go  
   `Sync`），代价是要老实等一次真实的磁盘同步写。
3. **落盘策略是「丢数据风险」和「性能」之间的权衡**，从「只 write  
   不 fsync」到「每次关键写都 fsync」再到「多副本共识」，代价递增、  
   安全性递增；数据库的事务/WAL 机制帮你把这层权衡封装好了，  
   但配置不同结论可能不同，别凭想象假设它绝对安全。

# 附：本课生词表

- **HDD（机械硬盘）** —— 靠旋转盘片和移动磁头读写数据，延迟主要来自  
  寻道和旋转等待，毫秒级。
- **SSD（固态硬盘）** —— 用闪存芯片存数据，没有机械部件，微秒级延迟，  
  但受页/块管理和磨损均衡影响。
- **寻道（seek）/ 旋转等待** —— HDD 延迟的两大来源，顺序读写能规避它们。
- **写入放大** —— SSD 实际写入闪存的数据量超过请求写入量的倍数，  
  源于擦除粒度大于写入粒度。
- **磨损均衡（wear leveling）** —— SSD 控制器自动把写操作分散到不同  
  物理单元，延长整体寿命。
- **延迟（latency）** —— 一次操作从发起到完成要等多久。
- **吞吐（throughput）** —— 单位时间能处理的数据量/操作次数。
- **页缓存（Page Cache）** —— 操作系统内核用内存缓存文件读写内容的机制，  
  读写文件默认都会经过它。
- **脏页（dirty page）** —— 已经被写过、还没同步到磁盘的页缓存内容。
- **write-back** —— 操作系统按自己的节奏，把脏页异步刷到磁盘的过程。
- **`fsync`（Rust `sync_all` / Go `Sync`）** —— 阻塞等待直到数据真正  
  写进磁盘存储介质的系统调用，是「保证不丢数据」的关键操作。
- **WAL（写前日志）/ group commit** —— 数据库常用的技巧：先顺序写一份  
  日志再异步整理数据、多个事务合并成一次 `fsync`，用少量延迟换吞吐。
