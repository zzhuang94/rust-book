# 数据竞争到底是什么

> 先把「为什么需要同步原语」这个问题的根源钉死。

两个线程**共享同一块内存**（回顾 [《进程与线程》](process-thread.md)：  
同进程线程共享地址空间），同时不加保护地读写同一个变量，  
就会发生 **数据竞争（data race）**。它不是「有小概率出错」，  
而是**结果完全不确定**，取决于两个线程的指令在硬件上被交错执行的  
具体顺序——同一份代码，这次跑对、下次跑错，还可能在开发机上  
永远跑不出问题、只在生产的高并发下偶发。

最经典的例子是「读-改-写」不是一个原子步骤：

```
线程 A：读 counter（值=5）→ 加 1（算出 6）→ 写回 counter
线程 B：读 counter（值=5）→ 加 1（算出 6）→ 写回 counter
```

如果 A、B 的「读」几乎同时发生，两边都读到 5，最终 counter 只被  
写成 6 一次——本该 `+2` 变成 7，实际却变成 6，**丢了一次更新**。  
这三个动作（读、加、写）在 CPU 指令层面天生就是分开的三步，  
没有任何东西天然保证它们「整体不被打断」——这正是同步原语  
存在的全部理由：**人为地把某一段操作，变成对其他线程「不可分割」  
的整体**。

----

# mutex 一句话

> 互斥锁（mutex）解决的问题就是上一节的「不可分割」。

**mutex = 一把只有一个人能拿到的钥匙。** 拿到钥匙（lock）的线程才能  
进临界区（critical section，即被保护的那段代码/数据）；没拿到的  
线程要么排队等待（进入阻塞态，回顾  
[《阻塞的确切含义》](blocking-io.md)），要么（自旋锁版本）原地  
空转反复尝试。释放（unlock）钥匙后，排队里的某一个线程被唤醒，  
拿到钥匙继续。

```
线程 A: lock() → 【临界区：读改写 counter】 → unlock()
线程 B:            lock()（A 没释放前一直等）→ 【临界区】→ unlock()
```

同一时刻**只有一个线程**能处于临界区内，「读-改-写」三步因此重新  
变成了对外表现为「不可分割」的一整块——数据竞争被消除。

具体的 API 写法（`std::sync::Mutex`、`tokio::sync::Mutex`、  
它们的区别、`Arc<Mutex<T>>` 怎么在多线程间共享）见  
[《Rust 多线程与并发》](../concurrency/threads.md) 与  
[《共享状态：Arc / RwLock》](../async/shared-state.md)，本课到此为止，  
只关心「锁在操作系统层面到底在做什么」——下面几节继续往下钻。

# 条件变量解决什么

> mutex 只回答「谁能进临界区」，没回答「该等到什么条件才进去」——  
> 这是条件变量（condition variable）补的那一块。

想象一个「生产者-消费者」队列：消费者线程想拿数据，但队列现在  
是空的。用 mutex 单独实现，只能写成一个丑陋的忙轮询：

```
反复地：lock() → 看队列空不空 → unlock() → 睡一会 → 再来一次
```

这既浪费 CPU（一直在问），又有延迟（数据到了也要等下一轮才发现）——  
和上一课「非阻塞独自解决不了问题」是完全同构的困境。**条件变量**  
就是这里的解法：它让线程能够「在持锁检查条件不满足时，原地睡眠  
并**自动释放锁**，被通知后醒来并**自动重新拿锁**」，整个过程零忙等：

```
消费者：lock() → 队列空 → wait(条件变量)（睡眠 + 自动放锁）
                              ...（某个时刻）...
生产者：lock() → 放入数据 → notify(条件变量) → unlock()
消费者：（被叫醒，自动重新 lock）→ 队列不空了 → 取数据 → unlock()
```

Rust 标准库对应 `std::sync::Condvar`，Go 里对应  
`sync.Cond`——但**实际业务代码几乎不会直接用条件变量**：  
Rust 更常用 `std::sync::mpsc`/`tokio::sync::mpsc` 通道，  
Go 更常用 `channel`，它们的内部实现往往就是「条件变量 + 锁」  
这一层原语的封装。知道条件变量存在，是为了理解通道「消费者在空  
队列上等待时不占 CPU」这件事，到底是怎么做到的。

# 原子操作的硬件底座

> 比 mutex 更轻的另一条路：不用锁，靠 CPU 提供的一条特殊指令，  
> 把「读改写」焊死成硬件层面的一步。

现代 CPU 提供专门的 **原子指令**（如 x86 的 `LOCK` 前缀指令、  
`CMPXCHG`），可以让「读一个值、判断/修改、写回」这几步，  
在硬件层面表现为**不可被其他核心插队打断**的一个整体——  
不需要「排队等锁」这套机制，直接靠硬件保证。

```rust
use std::sync::atomic::{AtomicU64, Ordering};

let counter = AtomicU64::new(0);
counter.fetch_add(1, Ordering::Relaxed); // 硬件层面的一条不可分割的"加1"
```

**原子操作不是没有代价**，只是代价的形式和 mutex 不同：

- 无竞争时：原子指令通常只比普通指令慢一点点（不用陷入内核）；
- 有竞争时（很多核心同时对同一个原子变量下手）：这个变量所在的  
  **缓存行**（[《进程与线程》](process-thread.md) 讲过的  
  64 字节单位）要在各核心的缓存之间来回「搬家」，这就是  
  [《ArcSwap 无锁读》](../http/arcswap.md) 提到的「缓存行弹跳  
  （cache line bouncing）」——竞争越激烈，这个开销越接近甚至  
  超过一把普通的锁。

一句话总结原子操作的位置：它是**比 mutex 更底层的积木**——  
mutex 内部往往就是用一个原子变量实现「有没有人持有锁」这个状态；  
业务代码里能用简单的 `fetch_add`/`compare_exchange` 表达的场景  
（计数器、标志位），优先用原子量；涉及多个字段要一起保护、  
或者操作逻辑复杂，用 mutex 更清晰、不容易写错。

# futex 一句话

> Linux 上 mutex 真正落地时用的机制，一句话讲清它聪明在哪。

**futex（fast userspace mutex）** 的核心思路：**没有竞争时，  
加锁/解锁全程留在用户态**（一次原子指令判断「这把锁现在没人拿」，  
直接成功，完全不进内核，代价接近一次原子操作）；**只有真的发生  
竞争**（线程 A 想拿锁，却发现线程 B 已经拿着）时，才通过  
`futex` 这个系统调用陷入内核，把线程 A 挂到一个等待队列上  
（进入阻塞态），等 B 释放锁时内核负责唤醒它。

```
无竞争（99% 的情况）：用户态原子操作，全程不进内核，极快
有竞争（少数情况）  ：futex 系统调用 → 内核挂起等待线程 → 释放时内核唤醒
```

这解释了一个常见的性能直觉：**「几乎不冲突的锁」几乎免费，  
「频繁冲突的锁」才真正昂贵**——昂贵的部分不是「锁」这个抽象概念，  
而是竞争发生时那次陷入内核的上下文切换成本（下一课  
[《性能数量级要记住》](perf-cost.md) 会给出具体数字）。  
Rust/Go 标准库的锁在 Linux 上基本都是靠 futex 实现的，  
这也是为什么「Mutex 在无竞争路径下很快」是一个可以放心依赖的  
工程常识。

# 自旋锁与互斥锁的取舍

> 除了「排队睡眠」，还有一种更简单粗暴的等法：原地空转。

**自旋锁（spinlock）**：拿不到锁时不睡眠，在一个死循环里反复  
检查锁是否释放了，一旦释放立刻抓到手——**代价是占着 CPU 空转**，  
但也省掉了「挂起-唤醒」的上下文切换开销。

| | 互斥锁（mutex，含 futex 语义） | 自旋锁 |
| --- | --- | --- |
| 等锁时 | 让出核心（阻塞态），可能有上下文切换成本 | 占着核心空转，零切换成本 |
| 适合场景 | 临界区较长、可能等较久 | 临界区极短、预期很快能拿到 |
| 典型使用者 | 应用层业务代码 | 操作系统内核内部、极高性能库的热路径 |

日常业务代码几乎不需要手写自旋锁，标准库/常用 crate 的默认  
mutex 已经是恰当的选择；知道它存在，是为了看懂「为什么某些  
底层库/内核代码会用一种看起来很浪费 CPU 的等待方式」——  
在临界区只有几条指令、预期几十纳秒就能拿到锁的场景下，  
自旋反而比「挂起再唤醒」更快。

1. 运行 `cargo run -p os-sync-primitives`，对比程序里「无保护读改写」

   观察 `context-switches` 指标随并发线程数上升的变化趋势。

# 三句话带走

1. **数据竞争 = 多线程无保护地读写同一内存，结果不确定**——  
   根源是「读-改-写」这类操作天生不是硬件层面的一个原子步骤。
2. **mutex 用「排队等钥匙」保证临界区独占；条件变量补上「该等到  
   什么条件才继续」；原子指令是比锁更底层的硬件积木**——  
   竞争激烈时，原子量的缓存行弹跳可能比锁还贵。
3. **Linux 上的锁靠 futex 实现：无竞争全程用户态几乎免费，  
   只有真竞争才陷入内核**——「几乎不冲突的锁很便宜」是可以放心  
   依赖的工程常识；具体 Rust API 见  
   [《Rust 多线程与并发》](../concurrency/threads.md) 与  
   [《共享状态：Arc / RwLock》](../async/shared-state.md)。

# 附：本课生词表

- **数据竞争（data race）** —— 多线程无同步地读写同一内存，  
  结果由指令交错顺序决定，不确定。
- **临界区（critical section）** —— 需要被同步原语保护、  
  不允许多线程同时进入的一段代码/数据访问。
- **mutex（互斥锁）** —— 保证临界区任意时刻只有一个线程能进入  
  的同步原语，等锁的线程通常进入阻塞态。
- **条件变量（condition variable）** —— 让线程能「持锁检查条件、  
  不满足就睡眠并自动放锁，被唤醒后自动重新拿锁」的原语。
- **原子操作（atomic operation）** —— CPU 提供的、硬件层面  
  不可被打断的读改写指令，代价通常低于 mutex（无竞争时）。
- **缓存行弹跳（cache line bouncing）** —— 多核频繁争抢同一个  
  原子变量所在的缓存行，导致该行在核心间反复搬家，竞争大时很贵。
- **futex（fast userspace mutex）** —— Linux 锁的实现机制：  
  无竞争留在用户态、有竞争才陷入内核挂起/唤醒线程。
- **自旋锁（spinlock）** —— 拿不到锁时原地空转重试而不是睡眠，  
  适合临界区极短、预期很快拿到锁的场景。
