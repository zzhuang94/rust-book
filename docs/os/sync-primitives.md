# 数据竞争到底是什么

> 先把「为什么需要同步原语」这个问题的根源钉死。

两个线程**共享同一块内存**（回顾 [《进程与线程》](process-thread.md)：
同进程线程共享地址空间），同时不加保护地读写同一个变量，
就会发生 **数据竞争（data race）**。它不是「有小概率出错」，
而是**结果完全不确定**，取决于两个线程的指令在硬件上被交错执行的
具体顺序——同一份代码，这次跑对、下次跑错，还可能在开发机上
永远跑不出问题、只在生产的高并发下偶发。

最经典的例子是「读-改-写」不是一个原子步骤：

```
线程 A：读 counter（值=5）→ 加 1（算出 6）→ 写回 counter
线程 B：读 counter（值=5）→ 加 1（算出 6）→ 写回 counter
```

如果 A、B 的「读」几乎同时发生，两边都读到 5，最终 counter 只被
写成 6 一次——本该 `+2` 变成 7，实际却变成 6，**丢了一次更新**。
这三个动作（读、加、写）在 CPU 指令层面天生就是分开的三步，
没有任何东西天然保证它们「整体不被打断」——这正是同步原语
存在的全部理由：**人为地把某一段操作，变成对其他线程「不可分割」
的整体**。

----

# mutex 一句话

> 互斥锁（mutex）解决的问题就是上一节的「不可分割」。

**mutex = 一把只有一个人能拿到的钥匙。** 拿到钥匙（lock）的线程才能
进临界区（critical section，即被保护的那段代码/数据）；没拿到的
线程要么排队等待（进入阻塞态，回顾
[《阻塞的确切含义》](blocking-io.md)），要么（自旋锁版本）原地
空转反复尝试。释放（unlock）钥匙后，排队里的某一个线程被唤醒，
拿到钥匙继续。

```
线程 A: lock() → 【临界区：读改写 counter】 → unlock()
线程 B:            lock()（A 没释放前一直等）→ 【临界区】→ unlock()
```

同一时刻**只有一个线程**能处于临界区内，「读-改-写」三步因此重新
变成了对外表现为「不可分割」的一整块——数据竞争被消除。

具体的 API 写法（`std::sync::Mutex`、`tokio::sync::Mutex`、
它们的区别、`Arc<Mutex<T>>` 怎么在多线程间共享）见
[《Rust 多线程与并发》](../concurrency/threads.md) 与
[《共享状态：Arc / RwLock》](../async/shared-state.md)，本课到此为止，
只关心「锁在操作系统层面到底在做什么」——下面几节继续往下钻。

# 条件变量解决什么

> mutex 只回答「谁能进临界区」，没回答「该等到什么条件才进去」——
> 这是条件变量（condition variable）补的那一块。

想象一个「生产者-消费者」队列：消费者线程想拿数据，但队列现在
是空的。用 mutex 单独实现，只能写成一个丑陋的忙轮询：

```
反复地：lock() → 看队列空不空 → unlock() → 睡一会 → 再来一次
```

这既浪费 CPU（一直在问），又有延迟（数据到了也要等下一轮才发现）——
和上一课「非阻塞独自解决不了问题」是完全同构的困境。**条件变量**
就是这里的解法：它让线程能够「在持锁检查条件不满足时，原地睡眠
并**自动释放锁**，被通知后醒来并**自动重新拿锁**」，整个过程零忙等：

```
消费者：lock() → 队列空 → wait(条件变量)（睡眠 + 自动放锁）
                              ...（某个时刻）...
生产者：lock() → 放入数据 → notify(条件变量) → unlock()
消费者：（被叫醒，自动重新 lock）→ 队列不空了 → 取数据 → unlock()
```

Rust 标准库对应 `std::sync::Condvar`，Go 里对应
`sync.Cond`——但**实际业务代码几乎不会直接用条件变量**：
Rust 更常用 `std::sync::mpsc`/`tokio::sync::mpsc` 通道，
Go 更常用 `channel`，它们的内部实现往往就是「条件变量 + 锁」
这一层原语的封装。知道条件变量存在，是为了理解通道「消费者在空
队列上等待时不占 CPU」这件事，到底是怎么做到的。

# 原子操作的硬件底座

> 比 mutex 更轻的另一条路：不用锁，靠 CPU 提供的一条特殊指令，
> 把「读改写」焊死成硬件层面的一步。

现代 CPU 提供专门的 **原子指令**（如 x86 的 `LOCK` 前缀指令、
`CMPXCHG`），可以让「读一个值、判断/修改、写回」这几步，
在硬件层面表现为**不可被其他核心插队打断**的一个整体——
不需要「排队等锁」这套机制，直接靠硬件保证。

```rust
use std::sync::atomic::{AtomicU64, Ordering};

let counter = AtomicU64::new(0);
counter.fetch_add(1, Ordering::Relaxed); // 硬件层面的一条不可分割的"加1"
```

**原子操作不是没有代价**，只是代价的形式和 mutex 不同：

- 无竞争时：原子指令通常只比普通指令慢一点点（不用陷入内核）；
- 有竞争时（很多核心同时对同一个原子变量下手）：这个变量所在的
  **缓存行**（[《进程与线程》](process-thread.md) 讲过的
  64 字节单位）要在各核心的缓存之间来回「搬家」，这就是
  [《ArcSwap 无锁读》](../http/arcswap.md) 提到的「缓存行弹跳
  （cache line bouncing）」——竞争越激烈，这个开销越接近甚至
  超过一把普通的锁。

一句话总结原子操作的位置：它是**比 mutex 更底层的积木**——
mutex 内部往往就是用一个原子变量实现「有没有人持有锁」这个状态；
业务代码里能用简单的 `fetch_add`/`compare_exchange` 表达的场景
（计数器、标志位），优先用原子量；涉及多个字段要一起保护、
或者操作逻辑复杂，用 mutex 更清晰、不容易写错。

# futex 一句话

> Linux 上 mutex 真正落地时用的机制，一句话讲清它聪明在哪。

**futex（fast userspace mutex）** 的核心思路：**没有竞争时，
加锁/解锁全程留在用户态**（一次原子指令判断「这把锁现在没人拿」，
直接成功，完全不进内核，代价接近一次原子操作）；**只有真的发生
竞争**（线程 A 想拿锁，却发现线程 B 已经拿着）时，才通过
`futex` 这个系统调用陷入内核，把线程 A 挂到一个等待队列上
（进入阻塞态），等 B 释放锁时内核负责唤醒它。

```
无竞争（99% 的情况）：用户态原子操作，全程不进内核，极快
有竞争（少数情况）  ：futex 系统调用 → 内核挂起等待线程 → 释放时内核唤醒
```

这解释了一个常见的性能直觉：**「几乎不冲突的锁」几乎免费，
「频繁冲突的锁」才真正昂贵**——昂贵的部分不是「锁」这个抽象概念，
而是竞争发生时那次陷入内核的上下文切换成本（下一课
[《性能数量级要记住》](perf-cost.md) 会给出具体数字）。
Rust/Go 标准库的锁在 Linux 上基本都是靠 futex 实现的，
这也是为什么「Mutex 在无竞争路径下很快」是一个可以放心依赖的
工程常识。

# 自旋锁与互斥锁的取舍

> 除了「排队睡眠」，还有一种更简单粗暴的等法：原地空转。

**自旋锁（spinlock）**：拿不到锁时不睡眠，在一个死循环里反复
检查锁是否释放了，一旦释放立刻抓到手——**代价是占着 CPU 空转**，
但也省掉了「挂起-唤醒」的上下文切换开销。

| | 互斥锁（mutex，含 futex 语义） | 自旋锁 |
| --- | --- | --- |
| 等锁时 | 让出核心（阻塞态），可能有上下文切换成本 | 占着核心空转，零切换成本 |
| 适合场景 | 临界区较长、可能等较久 | 临界区极短、预期很快能拿到 |
| 典型使用者 | 应用层业务代码 | 操作系统内核内部、极高性能库的热路径 |

日常业务代码几乎不需要手写自旋锁，标准库/常用 crate 的默认
mutex 已经是恰当的选择；知道它存在，是为了看懂「为什么某些
底层库/内核代码会用一种看起来很浪费 CPU 的等待方式」——
在临界区只有几条指令、预期几十纳秒就能拿到锁的场景下，
自旋反而比「挂起再唤醒」更快。

# 死锁常见写法

> 锁能消除数据竞争，但引入了一种新的、更难排查的病：
> **死锁（deadlock）**——几个线程互相等对方手里的锁，谁都动不了。

死锁不是「代码写错了逻辑」这种能一眼看出的错误，它往往是几段
**各自看起来都正确**的代码，凑在一起运行时才会暴露的时序问题。
下面是三种最常见的写法，几乎覆盖了日常开发里 90% 的死锁场景。

----

# 锁顺序反了就死锁

两个线程各自需要拿两把锁才能完成工作，但**拿锁的顺序相反**：

```
线程 A：lock(锁X) → 想要 lock(锁Y) （被 B 占着，等）
线程 B：lock(锁Y) → 想要 lock(锁X) （被 A 占着，等）
```

A 等 B 手里的 Y，B 等 A 手里的 X，谁都不会主动放手——这就是
**循环等待（circular wait）**，是死锁四个必要条件里最容易在代码
里被打破、也是最值得记住的一条。

```rust
// 反面教材：两个函数以相反顺序获取同一对锁
fn transfer_a_to_b(a: &Mutex<i32>, b: &Mutex<i32>) {
    let _la = a.lock().unwrap();
    let _lb = b.lock().unwrap(); // 若另一线程正相反地先锁 b 再锁 a，就可能死锁
}
fn transfer_b_to_a(a: &Mutex<i32>, b: &Mutex<i32>) {
    let _lb = b.lock().unwrap();
    let _la = a.lock().unwrap();
}
```

**修复方式只有一个原则**：给所有锁定义一个**全局固定的获取顺序**
（比如按账户 ID 从小到大排序后依次加锁），所有代码路径都严格遵守
这个顺序，循环等待就永远不可能出现——这也是为什么很多资金转账类
系统的代码会先比较两个账户 ID 大小再决定加锁顺序，而不是按参数
传入的顺序。

# 同线程重复加锁会自锁

```rust
let m = Mutex::new(0);
let _g1 = m.lock().unwrap();
let _g2 = m.lock().unwrap(); // 💥 同一线程再次 lock 同一把非重入锁，直接卡死
```

Rust/Go 标准库的 mutex 都是**非重入（non-reentrant）**的：
一把锁不知道「这次加锁请求和上次是不是同一个线程发出的」，
它只知道「现在有人占着」。同一个线程如果在持锁期间，
沿着调用栈往深处走，又不小心调用了另一个也要 `lock()` 同一把锁
的函数，就会等待一个**永远不会被释放**的锁（因为释放它的代码
在自己下面，永远执行不到）——这在函数调用层次较深、锁的持有范围
不够清晰的代码里很容易无意间发生。

**修复方式**：让持锁的代码块尽量短小、职责单一，避免在持锁期间
调用「不确定内部会不会也去锁同一个对象」的函数；如果确实需要
「同一线程可重复加锁」的语义，改用专门的**可重入锁
（reentrant lock）**，但这类锁在 Rust/Go 生态里并不常见，
更惯用的做法始终是重新设计代码结构，避免嵌套加锁本身。

# 持锁跨越 await 也会死锁

```rust
// 反面教材：持有同步锁的守卫跨越了一个 .await 点
async fn bad(data: &tokio::sync::Mutex<Vec<i32>>) {
    let mut guard = data.lock().await;
    some_async_io_call().await; // ⚠️ 这次 IO 可能要等很久，锁一直没释放
    guard.push(1);
}
```

如果这次 `.await` 里的操作本身又依赖另一个任务去拿同一把锁
（比如等一个需要先获取这把锁才能完成的响应），就会形成和写法一
同样的循环等待，只是等待的双方从「操作系统线程」变成了
「异步任务」。[《共享状态：Arc / RwLock》](../async/shared-state.md)
里专门有一节复现这种异步死锁，本课只提醒一句原则：
**尽量缩短持锁的代码块，锁guard 在跨越 `.await` 之前就应该被
`drop` 掉**，需要在异步代码里长时间持有的共享状态，往往更适合用
`RwLock` 或者干脆把访问收敛到一个专门的任务后面用消息传递。

🔬 底层视角：Linux 下怀疑死锁时，可以用 `gdb -p <pid>` 附加上去后
执行 `thread apply all bt`，把所有线程当前的调用栈一次性打印出来——
如果看到几个线程都停在 `pthread_mutex_lock`（或者 Rust/Go 对应的
锁实现）里，且它们各自等待的锁能在另一个线程的调用栈里找到「正被
持有」的痕迹，基本就实锤了死锁，而不需要靠猜。

# RwLock 读写偏向直觉

> mutex 是「唯一一个人能进」，`RwLock`（读写锁）是「很多读者可以
> 同时进，但只要有一个写者，其他人都要让路」——这一节讲它比 mutex
> 更微妙的一面。

`RwLock` 的基本规则很直观：

```
多个读锁（read）可以同时持有 —— 大家都只是看，互不干扰
一个写锁（write）独占 —— 写的时候必须确保没有任何人（读或写）在同时访问
读锁和写锁不能同时存在
```

在「读多写少」的场景下，`RwLock` 比 mutex 明显更有优势——十个线程
同时读，用 mutex 得排成一队，用 `RwLock` 可以十个一起进去。
但「读写偏向（read/write bias）」这个词描述的是一个容易被忽略的
风险：**如果读请求源源不断地到来，写请求可能被一直往后排，
迟迟拿不到锁**——这叫 **写者饥饿（writer starvation）**。

```
时间线：读1进 读2进 读3进 （写者到达，排队）读4进 读5进 …
         如果读请求一直连续不断地到达，写者可能永远排在队尾
```

不同实现对这个问题的取舍不同，这是使用 `RwLock` 前必须弄清楚的一件事：

| 实现 | 策略 | 后果 |
| --- | --- | --- |
| Rust `std::sync::RwLock` | **未声明具体策略**，文档明确说明「不保证避免写者饥饿，具体行为依赖底层平台实现」 | 不同操作系统/不同版本表现可能不一致，不能依赖某种特定行为 |
| `parking_lot::RwLock` | 默认对写者友好：新来的读请求如果发现已经有写者在排队，会主动让写者先拿到锁 | 更适合「写请求也不能被无限拖延」的场景，是很多生产代码的实际选择 |
| Go `sync.RWMutex` | 官方文档说明会做类似的保护：如果一个写锁在等待，后续新来的读锁请求会被排在这个写锁之后，避免写者被无限拖延 | 相当于内置了「写者优先排队」的保护 |

**工程含义**：如果你的场景是「写请求虽然少，但一旦发生就必须尽快
拿到锁完成（比如配置热更新）」，且用的是 Rust `std::sync::RwLock`，
最好不要假设它会自动照顾写者——评估切换到
`parking_lot::RwLock`，或者干脆把「写」这件事收敛成低频、
在流量低谷执行，规避掉写者饥饿在你的具体场景里造成的实际影响。

另一个常被忽略的直觉：`RwLock` 的读锁虽然「可以多人同时拿」，
但**拿读锁本身也不是零成本的**——它至少需要一次原子操作去给
「当前读者计数」加一，高并发大量短读的场景下，这个计数器本身可能
成为新的竞争热点（回顾上面「原子操作的硬件底座」一节的缓存行
弹跳），这也是为什么「读多写少」不等于「一定该无脑上 RwLock」——
如果每次读的操作本身极短、竞争极高，有时一把简单的 mutex 反而
更快，具体该选哪个，最终还是要回到
[《性能数量级要记住》](perf-cost.md) 那句话：**先量，再决定**。

# 动手实验

1. 运行 `cargo run -p os-sync-primitives`，对比程序里「无保护读改写」
   复现数据竞争、和「加锁保护」得到正确结果的两组输出；
2. 回到 [《Rust 多线程与并发》](../concurrency/threads.md) 的
   `concurrency-threads` 示例，把 `Mutex<u64>` 换成拆开的
   `load`/`store`（去掉原子性），多跑几遍观察结果小于期望值——
   这是数据竞争在你自己机器上的真实复现；
3. 用 `htop`/任务管理器观察：给一个自旋等待的死循环和一个正常阻塞
   等待的线程分别截图对比 CPU 占用曲线的差异；
4. （Linux）用 `perf stat` 跑一遍高竞争的 mutex 基准测试，
   观察 `context-switches` 指标随并发线程数上升的变化趋势；
5. 写一个「锁顺序不一致」的最小死锁复现：两个线程按相反顺序锁
   两把 `Mutex`，运行几次，观察程序卡死不再有任何输出，
   再（Linux）用 `gdb -p <pid>` 附加上去执行
   `thread apply all bt`，在两个线程的调用栈里分别找到它们卡在
   哪一次 `lock()` 上；
6. 用 `RwLock` 写一个「大量短读线程 + 一个写线程」的小基准，
   分别测量 Rust `std::sync::RwLock` 和 `parking_lot::RwLock`
   在这个场景下，写线程平均等待多久才能拿到锁，感受两者
   「写者是否被优待」的实际差异。

# 三句话带走

1. **数据竞争 = 多线程无保护地读写同一内存，结果不确定**——
   根源是「读-改-写」这类操作天生不是硬件层面的一个原子步骤。
2. **mutex 用「排队等钥匙」保证临界区独占；条件变量补上「该等到
   什么条件才继续」；原子指令是比锁更底层的硬件积木**——
   竞争激烈时，原子量的缓存行弹跳可能比锁还贵。
3. **Linux 上的锁靠 futex 实现：无竞争全程用户态几乎免费，
   只有真竞争才陷入内核**——「几乎不冲突的锁很便宜」是可以放心
   依赖的工程常识；具体 Rust API 见
   [《Rust 多线程与并发》](../concurrency/threads.md) 与
   [《共享状态：Arc / RwLock》](../async/shared-state.md)。

# 附：本课生词表

- **数据竞争（data race）** —— 多线程无同步地读写同一内存，
  结果由指令交错顺序决定，不确定。
- **临界区（critical section）** —— 需要被同步原语保护、
  不允许多线程同时进入的一段代码/数据访问。
- **mutex（互斥锁）** —— 保证临界区任意时刻只有一个线程能进入
  的同步原语，等锁的线程通常进入阻塞态。
- **条件变量（condition variable）** —— 让线程能「持锁检查条件、
  不满足就睡眠并自动放锁，被唤醒后自动重新拿锁」的原语。
- **原子操作（atomic operation）** —— CPU 提供的、硬件层面
  不可被打断的读改写指令，代价通常低于 mutex（无竞争时）。
- **缓存行弹跳（cache line bouncing）** —— 多核频繁争抢同一个
  原子变量所在的缓存行，导致该行在核心间反复搬家，竞争大时很贵。
- **futex（fast userspace mutex）** —— Linux 锁的实现机制：
  无竞争留在用户态、有竞争才陷入内核挂起/唤醒线程。
- **自旋锁（spinlock）** —— 拿不到锁时原地空转重试而不是睡眠，
  适合临界区极短、预期很快拿到锁的场景。
- **死锁（deadlock）** —— 多个线程互相等待对方持有的锁，
  形成循环等待，谁都无法继续执行。
- **循环等待（circular wait）** —— 死锁发生的核心条件：
  锁的获取顺序在不同线程间形成了一个环。
- **非重入锁（non-reentrant lock）** —— 同一线程重复获取同一把
  已持有的锁会卡死，Rust/Go 标准库的 mutex 都是这种。
- **读写锁（`RwLock`/`sync.RWMutex`）** —— 允许多个读者同时持有、
  写者独占的锁，适合读多写少的场景。
- **写者饥饿（writer starvation）** —— 读请求持续不断到达，
  导致写请求一直排不到队、迟迟拿不到锁的现象。
- **`parking_lot`** —— 社区常用的高性能同步原语 crate，
  其 `RwLock` 对写者更友好，行为比标准库更可预期。
