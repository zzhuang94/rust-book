# 容器不是轻量 VM

> 代码：`code/os-cgroup-container/`　运行：`cargo run -p os-cgroup-container`

前置：[《进程与线程》](process-thread.md)（进程隔离）、  
[《信号与进程生命周期》](signals-lifecycle.md)（孤儿进程、PID 1）。  
本课讲清「容器」到底是什么操作系统机制拼出来的、以及为什么这些  
机制会直接影响业务代码看到的 CPU 核数、内存上限、被杀掉的时机。  
本课只讲通用的 Linux/容器/编排概念，不涉及任何具体公司的集群配置。

----

# 容器到底是什么

> 先打破一个常见的误解：容器**不是**一台缩小版的虚拟机。

**虚拟机（VM）** 靠 hypervisor 模拟出一整套虚拟硬件，  
每个 VM 里跑着**完整独立的操作系统内核**——重、启动慢（秒级到  
分钟级），但隔离非常彻底。

**容器**完全是另一条路：容器里跑的进程，**和宿主机上的其他进程  
共享同一个操作系统内核**。所谓「容器」，本质上就是宿主机上的  
一个**普通进程**，只是内核用两套机制把它的「视野」和「资源用量」  
都限制/隔离了起来：

```
              宿主机（一个内核）
┌──────────────┬──────────────┬──────────────┐
│   容器 A      │   容器 B      │  宿主机自己的进程 │
│（普通进程 +   │（普通进程 +   │                  │
│  namespace + │  namespace + │                  │
│  cgroup）    │  cgroup）    │                  │
└──────────────┴──────────────┴──────────────┘
```

这两套机制分别是：

- **namespace（命名空间）**：负责「让容器看到一个被裁剪过的  
  世界」——看不见宿主机上的其他进程、有自己独立的一套网络接口；
- **cgroup（控制组）**：负责「限制容器能用多少资源」——  
  最多用多少 CPU、多少内存。

**这就是为什么容器启动是毫秒到秒级**（不需要启动一个新内核，  
只是给一个普通进程套上这两层限制），**也是为什么容器的隔离性弱于  
虚拟机**（内核是共享的，内核层面的漏洞可能影响到所有容器）。

----

# namespace 让容器看不见外面

> namespace 的直觉：不是真的把资源分割开，是给进程发一副  
> 「特制眼镜」，让它看到的世界比实际小。

Linux 提供好几种 namespace，各自负责裁剪一类「视野」：

| namespace | 裁剪的是什么 | 容器里看到的效果 |
| --- | --- | --- |
| PID | 进程编号空间 | 容器里的第一个进程 PID 是 1，看不到宿主机上其他进程 |
| NET | 网络接口、路由表、端口 | 容器有自己的一套网卡/IP，端口占用互不影响 |
| MNT | 文件系统挂载点 | 容器看到自己的根文件系统，看不到宿主机的其他目录 |
| UTS | 主机名、域名 | 容器可以有自己的 hostname，不影响宿主机 |
| IPC | 进程间通信资源（共享内存、消息队列） | 容器间的 IPC 资源互不可见 |
| USER | 用户/用户组 ID | 容器内的 root 可以映射成宿主机上的一个普通用户 |

**核心直觉**：namespace 不创造任何新资源，它只是给「查询/枚举  
系统资源」这件事加了一层过滤。容器里执行 `ps aux` 只会看到  
PID namespace 允许它看到的那几个进程——不是宿主机上真的只有  
这几个进程在跑，是这个容器被限定只能「看见」它们。

🔬 底层视角：这解释了一个新手常有的疑问——「容器里 PID 1 是我的  
应用进程，它的父进程是谁？」PID namespace 让容器内部看到的  
「进程树根」是它自己，但从宿主机视角，这个进程仍然是宿主机上  
某个真实进程的子进程（通常是容器运行时守护进程）。这也是  
[《信号与进程生命周期》](signals-lifecycle.md) 提到的「容器主进程  
最好是个能正确处理孤儿/信号转发的极简 init」这条建议的完整背景：  
容器内部看不到宿主机的 init，容器自己的 PID 1 就要扛起这份责任。

----

# cgroup 给资源记账和封顶

> namespace 管「看不见什么」，cgroup 管「最多能用多少」——  
> 两者配合才是完整的容器。

**cgroup（control group）** 是内核的一套记账 + 限制机制，  
按「控制器（controller）」分门类：`cpu`、`memory`、`io`、  
`pids` 等各管一类资源。容器运行时创建容器时，会把容器里的  
进程放进一组 cgroup 限制里，之后这个进程（以及它 fork 出的  
所有子进程）用了多少 CPU、多少内存，内核全程记账，  
一旦碰到设定的上限就按规则处理（限流或者直接杀掉，  
下面两节分别展开）。

目前主流是 **cgroup v2**（统一层级，各控制器挂在同一套目录树下），  
早期是 **cgroup v1**（各控制器各自一套独立层级，配置更繁琐）——  
业务代码通常不需要直接操作 cgroup 文件系统，容器运行时和编排  
系统会替你翻译成命令行参数或者配置文件里的 `resources.limits` 之类  
字段，本课只需要知道它存在、以及它对下面两节现象的因果关系。

----

# CPU 限制怎么影响业务

> 容器里「CPU 限制」不是把某几个核心专门划给你，  
> 通常是一种**限流（throttling）**，理解这一点能避免很多误判。

cgroup 的 CPU 限制常见配置方式是「配额/周期」——比如「每 100  
毫秒的周期里，最多允许用 50 毫秒的 CPU 时间」，这大致相当于  
「0.5 个核心」。**注意这和「独占半个核心」完全不同**：

- 容器进程可以在某个瞬间用满好几个核心（多线程并发跑），  
  只要在每个周期窗口内**总的**CPU 时间不超过配额；
- 一旦某个周期内用超了配额，内核会把这个容器的进程**限流**——  
  强制它在这个周期剩下的时间里不能再获得 CPU，直到下一个周期  
  重新开始。这个现象叫 **CPU throttling**，症状是「延迟突然  
  一阵一阵变差，过一会又恢复正常」，很容易被误判成「代码变慢了」，  
  实际是被周期性地限流打断。

一个容易踩的坑：**容器内看到的 CPU 核数，往往是宿主机的物理  
核数，不是分配给这个容器的配额！** Rust 的  
`std::thread::available_parallelism()`、Go 的  
`runtime.NumCPU()`，在很多环境下默认读的是宿主机的  
`/proc/cpuinfo`，如果代码据此决定「开多少个工作线程」，  
可能会在只分配了 0.5 核的容器里，开出几十个线程互相抢那点  
CPU 配额，上下文切换（[《性能数量级要记住》](perf-cost.md)  
讲过成本）反而把本就不多的 CPU 时间消耗在切换上。生产实践中  
常见的应对方式是让程序读取 cgroup 里实际的配额来决定并发度，  
或者由编排系统通过环境变量显式告诉程序该用多少线程。

# 内存限制怎么影响业务

> 内存限制更狠：一旦超限，不是「限流」，通常是**直接被杀掉**。

cgroup 的内存限制设一个上限字节数，容器进程（含它所有子进程）  
的内存使用一旦触顶：

- 内核首先尝试回收一些可回收的内存（比如页缓存）；
- 如果实在腾不出空间，触发 **OOM Killer（Out-Of-Memory  
  Killer）**——内核选中一个「最该死」的进程，直接发  
  `SIGKILL`（回顾 [《信号与进程生命周期》](signals-lifecycle.md)：  
  这个信号**完全无法被捕获**，进程没有任何机会做清理）。

这解释了容器场景一个典型的、让人摸不着头脑的故障现象：  
**应用日志里什么异常都没有，进程突然就没了**——因为它是被  
`SIGKILL` 秒杀的，压根没机会写下最后一行日志。排查这类问题，  
第一反应应该是查容器编排系统/内核日志里有没有  
「OOMKilled」/`oom-kill` 记录，而不是继续翻应用自己的日志。

一个和 Rust/Go 运行时相关的细节：这两个语言的内存分配器/GC  
默认往往参考的是**宿主机总内存**而不是 cgroup 限制，可能导致  
「运行时觉得内存还很宽裕、迟迟不主动释放」而实际已经顶到了  
容器限制——这也是为什么很多语言运行时近些年都在增加  
「感知 cgroup 限制」的能力（比如读取容器内存上限来调整 GC  
触发阈值），业务侧则常见的应对方式是把容器内存上限设得比  
应用实际峰值用量留出足够余量。

# VM 与容器一张对照表

> 把本课的结论收进一张表。

| | 虚拟机 | 容器 |
| --- | --- | --- |
| 内核 | 每个 VM 独立一份 | 和宿主机共享同一份 |
| 启动速度 | 秒级到分钟级 | 毫秒到秒级 |
| 隔离机制 | hypervisor 模拟硬件 | namespace（视野）+ cgroup（限额） |
| 隔离强度 | 强（内核级漏洞影响范围小） | 弱于 VM（共享内核） |
| CPU 超限行为 | 通常按虚拟核心硬分配 | 常见是周期性限流（throttling） |
| 内存超限行为 | 受虚拟机自身内存管理约束 | 常见是 OOM Killer 直接 `SIGKILL` |
| 典型单机密度 | 十数个 | 成百上千个 |

# 动手实验

1. 运行 `cargo run -p os-cgroup-container`，程序会读取并打印  
   当前环境里能探测到的 CPU 配额与内存上限信息（如果在容器里跑），  
   或者在裸机上打印「未检测到 cgroup 限制」；
2. （需要 Docker）执行  
   `docker run --rm --cpus=0.5 alpine sh -c "nproc"`，  
   观察容器内 `nproc` 报告的核数和 `--cpus` 的关系，  
   验证「看到的核数不等于实际配额」这句话；
3. （需要 Docker）执行  
   `docker run --rm --memory=50m alpine sh -c "yes | tr \\n x | head -c 200000000 > /dev/null"`，  
   观察进程被容器运行时报告为 `OOMKilled` 退出，  
   而不是应用层面的报错；
4. 查一下你日常使用的容器/编排配置里，业务容器有没有单独设置  
   `ENTRYPOINT` 为 `tini`/`dumb-init`，回顾  
   [《信号与进程生命周期》](signals-lifecycle.md) 理解它在这里的作用。

# 三句话带走

1. **容器不是轻量虚拟机**：它是宿主机上一个共享同一个内核的  
   普通进程，靠 **namespace 裁剪视野 + cgroup 限制资源用量**  
   实现隔离感，因此启动快、隔离弱于真正的 VM。
2. **CPU 限制常见表现是周期性限流**（throttling），容器内看到的  
   核数往往是宿主机核数而不是配额，据此决定并发线程数容易踩坑。
3. **内存限制超限常见表现是 OOM Killer 直发 `SIGKILL`**——  
   进程没有任何清理机会、日志戛然而止，排查要先看编排系统/内核的  
   OOM 记录，而不是应用自身日志。

# 附：本课生词表

- **namespace（命名空间）** —— 内核用来裁剪进程「能看到什么」的  
  机制（PID/NET/MNT/UTS/IPC/USER 等各管一类）。
- **cgroup（控制组）** —— 内核用来记账并限制进程组资源用量的机制  
  （CPU/内存/IO/PID 数等）。
- **cgroup v1 / v2** —— cgroup 的两代实现，v2 统一层级、  
  是当前主流。
- **CPU throttling（CPU 限流）** —— 某周期内 CPU 用量超过配额后，  
  内核强制该周期内不再给它调度时间的现象。
- **配额/周期（quota/period）** —— cgroup CPU 限制的常见配置方式，  
  「每 period 毫秒最多用 quota 毫秒」。
- **OOM Killer** —— 内存触顶且无法回收时，内核选中并  
  `SIGKILL` 一个进程的机制。
- **`OOMKilled`** —— 容器被 OOM Killer 杀掉后，  
  编排系统/运行时上报的退出原因标记。
- **`tini`/`dumb-init`** —— 常用作容器 PID 1 的极简 init 程序，  
  负责正确转发信号、回收孤儿/僵尸进程。
