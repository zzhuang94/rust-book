# 为什么学这篇

> 预备课，无配套代码。后续「线程、阻塞、调度、上下文切换、并行」这些词全部来自操作系统。这一篇一次讲清，之后任何一章看到它们卡住了，回来查。

后面几章的核心论断都建立在 OS 常识上，比如：

- [《async 基础》](../async/basics.md) 的铁律「绝不在 async 里阻塞线程」——什么叫「阻塞」？「线程」又是什么？
- [《Tokio 运行时》](../async/tokio.md) 的「多线程运行时、工作窃取、协作式调度」——调度是谁在调度谁？
- [《共享状态：Arc / RwLock》](../async/shared-state.md) 的「数据竞争、原子指令、缓存行弹跳」——这些是硬件层面的什么现象？
- [《从零手写 HTTP》](../http/http-from-scratch.md) 的「每连接一个线程撑不住 C10K」——线程到底贵在哪？

不懂这些，学起来就是在背结论；懂了，每条结论都是显然的。

----

# CPU 一次一件事

> CPU 是一台「取指令 → 执行 → 再取下一条」的循环机器。对我们来说只需要三个概念。

- **寄存器**：CPU 内部的一小撮超高速存储（每个几十字节），保存「正在算的数」和「下一条指令的位置」。**一段代码「执行到哪了」，  
  本质就是这几十个寄存器的当前值**——记住这句，后面讲切换和协程都靠它；
- **时钟频率**：如 3GHz = 每秒约 30 亿个节拍，每拍推进一点工作。频率提升在 2005 年前后撞上功耗墙，  
  **单核变快的时代结束了** ——这是「多核」存在的历史原因；
- **一个核心一次只能执行一条指令流**。你机器上同时「运行」的几百个程序，在单个核心上其实是快速轮流（见后面「调度」）。

----

# 多核与超线程

> 单核提不了频，厂商就在一颗 CPU 里塞多个核心。

- **物理核心（core）**：真正独立的执行单元，4 核 = 真的能同时跑 4 条指令流；
- **超线程 / SMT（逻辑核心）**：一个物理核心伪装成 2 个「逻辑核心」，趁一条指令流卡住（等内存）时跑另一条。  
  有收益但远不是翻倍，所以「8 核 16 线程」的机器，真并行能力更接近 8；
- 查看核数：Windows 任务管理器-性能页（或 PowerShell `(Get-CimInstance Win32_Processor).NumberOfCores`），  
  Linux `nproc` 或 `lscpu`。Rust 里 `std::thread::available_parallelism()` 拿到的就是逻辑核数—— [《Tokio 运行时》](../async/tokio.md) 多线程运行时默认开的工作线程数正是它。

----

# 缓存与缓存行

> CPU 比内存快百倍以上，中间垫了几层缓存（L1/L2/L3，越靠近 CPU 越小越快）。两个后面会用到的事实。

- 缓存以 **缓存行（cache line，通常 64 字节）** 为单位搬运。多个核心频繁写 **同一个缓存行** 时，  
  这行数据要在核心之间来回「搬家」，性能骤降——这就是 [《ArcSwap 无锁读》](../http/arcswap.md) 说的「读锁计数器在多核间弹跳（cache line bouncing）」的物理原型；
- 多核各有自己的缓存，改了同一变量要靠 **缓存一致性协议** 互相通知。 [《共享状态：Arc / RwLock》](../async/shared-state.md) 的原子指令（`fetch_add`）和内存序（`Ordering`），  
  就是在这层硬件现实之上，给你「多核看到一致结果」的保证。

----

# 进程是运行实例

> 先分清「程序」和「进程」——一个是静态文件，一个是跑起来的实例。

- **程序**：磁盘上那个可执行文件（你 `cargo build` 出来的东西），是静态的；
- **进程**：程序被启动后的 **运行实例** = 一份独立的内存空间 + 打开的资源（文件、socket）+ 至少一个线程。  
  同一个程序可以起 N 个进程，互不相干。

----

# 进程隔离且昂贵

> 进程有两大特征，决定了它的用法：隔离，以及昂贵。

- **隔离**：每个进程有自己的 **虚拟地址空间** ——它以为自己独占整台机器的内存，OS 在背后做翻译。好处是进程 A 崩了、  
  内存写坏了，进程 B 毫发无伤；进程 A 根本 **没有语法** 能直接读写进程 B 的变量；
- 所以 **跨进程共享数据必须走专门通道**（IPC）：管道、socket、共享内存文件…… [《接入 Redis》](../http/redis.md) 把数据放 Redis，  
  本质就是「用网络 socket 跨进程共享」——Redis 是另一个进程（甚至另一台机器），你的多个服务实例（多个进程）通过它共享状态；
- **昂贵**：创建进程要复制/建立整套地址空间和资源表，毫秒级。所以「每个请求 fork 一个进程」的上古 CGI 模型早被淘汰。

----

# 线程共享地址空间

> 线程 = 进程内的一条执行流。它和进程最大的区别：同进程的线程 **共享** 同一份内存。这一条设计同时是福和祸。

一个进程至少有一个线程（main 线程），可以再开很多个。关键设计：

- **同进程的所有线程共享同一个地址空间** ——你的全局变量、堆上的数据，每个线程都能直接读写；
- 每个线程 **独有** 的只有三样：自己的 **栈**（局部变量住这里，Linux 默认 8MB）、自己的 **寄存器上下文**（「我执行到哪了」）、线程 ID。

这一条设计同时是福和祸：

- 福：线程间共享数据零成本——直接读同一个变量就行（对比进程要走 IPC）；
- 祸：两个线程同时读写同一个变量 → **数据竞争**（谁先谁后不确定，结果随机）→ 于是需要锁、原子操作—— [《共享状态：Arc / RwLock》](../async/shared-state.md) 整章就是在解决这个祸。  
  Rust 的特殊之处在于，把竞争检查做进了编译器（`Send`/`Sync`）。

----

# 线程贵在哪

> 这是 C10K 问题的根源。线程不是免费的，「每连接一个 OS 线程」到几千并发就吃力。

| 成本项 | 数量级 | 说明 |
| --- | --- | --- |
| 栈内存 | 默认 ~8MB 虚拟内存/线程 | 一万个线程 = 80GB 虚拟地址空间，物理内存按实际使用算也很可观 |
| 创建/销毁 | 微秒~几十微秒 | 要进内核登记 |
| 上下文切换 | 1~10 微秒 + 缓存变冷 | 见后面「上下文切换的成本」 |
| 内核调度压力 | 线程越多越重 | 调度器要管的对象线性增长 |

所以「每连接一个 OS 线程」到几千并发就开始吃力——这就是 [《从零手写 HTTP》](../http/http-from-scratch.md) 从「每连接一线程」进化到「tokio 任务」的动机。

----

# 调度决定跑谁

> 核心只有 8 个，线程有几百个，谁来分配？——OS 内核里的 **调度器**。先看线程的三种状态。

```
         被调度选中                   等待的事发生了
 就绪 ────────────→ 运行 ─┐    阻塞 ──────────────→ 就绪
  ↑                      │      ↑
  └──── 时间片用完/被抢占 ─┘      └── 运行中发起了等待（读磁盘/等锁/sleep）
```

- **运行**：正占着某个核心执行；
- **就绪**：万事俱备，排队等核心；
- **阻塞**：在等某件事（磁盘数据到达、网络包、锁被释放、sleep 到点）， **不占核心、也不参与排队** ——事情发生后，由内核把它改回「就绪」。

----

# 阻塞的确切含义

> 这是 async 铁律的物理地基，务必理解透。

「阻塞」的确切含义：线程发起一个要等待的系统调用（如读一个还没数据的 socket），内核把它标成阻塞态、把核心让给别人。  
**线程阻塞期间什么都做不了** ——如果这个线程身上还背着别的活（比如 Tokio 工作线程背着几百个任务），那些活 **全部停摆**。

这就是「阻塞工作线程」为什么是异步代码的头号大罪：你以为只是这一处等一下，实际上把同一个线程上排队的成百上千个任务全冻住了。

----

# 时间片与抢占

> 单核也能「同时」放音乐 + 编译代码，靠的就是毫秒级轮转。

- 调度器给每个「运行」线程一小段 **时间片**（毫秒级），到点强制换人（**抢占**），保证没人饿死；
- 所以单核机器也能「同时」放音乐 + 编译代码——毫秒级轮转快到人无法察觉。 **并发的本源就是这个轮转**；
- 呼应 [《Tokio 运行时》](../async/tokio.md)：OS 对线程是 **抢占式** 的（到点强制切），  
  而 Tokio 对任务是 **协作式** 的（只在 `.await` 处让出）——所以 async 里的死循环会霸占线程，  
  而 Go/OS 的世界里不会有这个问题。

----

# 上下文切换的成本

> 把核心从线程 A 换给线程 B，这个「换人」动作本身是有成本的。

把核心从线程 A 换给线程 B，要做的事：

1. 把 A 的全部寄存器（「执行到哪了」）保存到内存；
2. 切换内存映射等内核记账；
3. 把 B 上次保存的寄存器加载回来，从 B 的断点继续。

直接成本 1~10 微秒， **间接成本更大**：核心的缓存里全是 A 的数据，B 上来全部 miss，要重新从内存慢慢加载（「缓存变冷」）。  
线程数暴涨时，CPU 大量时间花在换人而不是干活上。

**用户态的「切换」为什么便宜**：goroutine 和 Rust 异步任务的切换不进内核——Go 是用户态调度器换栈，  
Rust 更极端：任务是编译好的 **状态机**，「切换」= 一次普通函数返回（Pending）+ 换个状态机调用 poll，  
纳秒级、没有内核参与。这就是三种并发单元成本差异的来源。

# 状态机是什么

> 这是理解 Rust 异步的核心词。上一节说「任务是编译好的状态机」——状态机到底是什么？

先看生活例子：

- **红绿灯** 就是一台状态机：它任何时刻处于 {红、黄、绿} 中的 **一个状态**；收到「计时器到点」这个 **事件** 就按规则 **转移** 到下一个状态。
- **自动售货机** 同理：{待机 → 已投币 → 出货中}，投币、按键、出货完成是驱动转移的事件。

抽象出来，状态机 = 三样东西：**一组可能的状态 + 当前处于哪个状态 + 「收到什么事件就转移到哪个状态」的规则表**。  
它的可贵之处在于：「进行到哪一步了」不再藏在某个执行流的脑子里，而是变成了 **一个普通的、可以拿在手里的值**。

**程序里的妙用：把「可暂停的过程」改写成状态机。** 一个中途要等待的函数，传统上暂停时需要有人保存它的寄存器和栈（这正是线程上下文切换干的事、  
也是它贵的原因）。换成状态机思路，「执行到哪了」被显式地存成一个值：

```
原始写法（暂停时需要保存执行现场）：
    fn 打招呼() {
        做准备工作;
        等定时器 3 秒;      ← 在这里暂停，寄存器和栈得有人保管
        打印结果;
    }

状态机改写（"执行到哪"变成一个普通的枚举值）：
    enum 打招呼状态 {
        还没开始,
        正在等定时器 { 保存的中间数据 },   ← 暂停点 = 一个状态
        已完成,
    }
    每被"推一下"（调用一次 poll）：
        match 当前状态 {
            还没开始       => { 做准备工作; 状态改成 正在等定时器 }
            正在等定时器   => { 定时器到了吗？没到→返回"还没好"；到了→打印结果，状态改成 已完成 }
            已完成         => 返回结果
        }
```

这个改写带来三条决定性的好处：

- **暂停 = 函数返回，恢复 = 再调一次函数** ——都是纳秒级的普通调用，不需要内核保存/恢复任何东西（对比上一节微秒级的上下文切换）；
- 状态机是 **普通数据**（一个枚举值，几十字节）：可以放进数组、在线程之间传递、由任何一个线程捡起来「推一下」——这就是 [《Tokio 运行时》](../async/tokio.md) 「工作窃取」能把任务搬来搬去的原因；
- 一个线程可以轮流推 **成千上万台** 状态机——这就是异步运行时「少量线程扛海量任务」的全部秘密。

**和后面的对接**：[《async 基础》](../async/basics.md) 的 `async fn` 就是编译器 **自动** 帮你做这个改写——你写顺序风格的代码，  
每个 `.await` 成为一个状态边界，跨越暂停点还要用的变量成为枚举的字段；`poll` 就是「推一下」，`Waker` 负责决定「什么时候值得再推」。  
而 Go 选了另一条路：goroutine 不做改写、保留真实的栈和寄存器快照（所以 Go 函数不用标 async、  
随处可停），代价是每个 goroutine 至少 2KB——两条路线的取舍就此分野。

# 并发不是并行

> 两个常被混用的词，配着核心数辨析一遍。

- **并发（concurrency）**：多个任务都「在进行中」，靠快速轮流也算。 **单核也能并发**；
- **并行（parallelism）**：多个任务 **同一物理瞬间** 各占一个核心真同时执行。 **并行需要多核**；
- 一句话：并发是「结构」（程序组织成多个可交替推进的单元），并行是「资源」（真有多个核）。有并发结构的程序放到多核上，才能吃到并行。

对应后面：[《async 基础》](../async/basics.md) 的 `join!` 是并发、不保证并行（同一任务内交替）；  
[《Tokio 运行时》](../async/tokio.md) 的 `spawn` 把任务交给多线程运行时，才可能真并行。

# 三种并发单元

> **全书最重要的一张表。** 进程、线程、用户态任务的对比，串起了后面所有架构选择。

| | 进程 | OS 线程 | 用户态任务（goroutine / tokio task） |
| --- | --- | --- | --- |
| 地址空间 | 独立（隔离） | 共享进程的 | 共享进程的 |
| 独有内存 | 整套地址空间 | 栈 ~8MB | goroutine 栈 ~2KB 起；tokio 任务=状态机，几百字节 |
| 创建成本 | 毫秒级 | 微秒级 | 纳秒级 |
| 切换成本 | 最贵（换地址空间） | 1~10 微秒 + 缓存变冷（进内核） | 纳秒级（用户态，不进内核） |
| 谁调度 | OS 内核 | OS 内核（抢占式） | Go 运行时（准抢占）/ Tokio（协作式） |
| 崩溃影响 | 只死自己 | 整个进程陪葬 | Go: 进程陪葬；Tokio: 包成 JoinError，进程不死 |
| 通信方式 | IPC（socket/管道/Redis…） | 共享内存 + 锁 / 通道 | 同左 |
| 适合场景 | 强隔离、多机扩展 | CPU 密集并行、数量可控 | 海量 IO 并发（连接数 ≫ 核数） |

这张表串起了整个课程的架构选择：[《共享状态：Arc / RwLock》](../async/shared-state.md) 在「共享内存 + 锁」格子里，  
[《Tokio 运行时》](../async/tokio.md) 的通道在「通信」格子里， [《从零手写 HTTP》](../http/http-from-scratch.md) 三步走就是从「OS 线程」列迁到「用户态任务」列，  
[《接入 Redis》](../http/redis.md) 在「进程 IPC」格子里。

# IO 模型三代

> 异步运行时的地基。一个网络服务处理一万个连接，读数据的方式进化了三代。

- **(1) 阻塞 IO + 每连接一线程**：`read()` 没数据就阻塞该线程。简单直观，但线程成本封顶了并发数；
- **(2) IO 多路复用**：一个线程问内核「这一万个 socket 哪些有数据了？」——Linux 的 **epoll**（macOS kqueue / Windows IOCP）一次系统调用返回就绪列表，  
  线程只处理就绪的几个。 **一个线程看住一万个连接**；
- **(3) 异步运行时**（Tokio）：把 (2) 包装成人能写的形态——你写顺序风格的 `stream.read(&mut buf).await`，  
  运行时把「socket 没就绪」翻译成任务挂起 + 把 socket 注册进 epoll，就绪后用 `Waker` 叫醒任务。

一句话：**Tokio 底下就是 epoll + 一个线程池 + 一堆状态机**。Go 的 netpoller 同理——两者殊途同归，  
都是「多路复用打底，用轻量并发单元还你顺序编程的体验」。这三代正好对应 [《从零手写 HTTP》](../http/http-from-scratch.md) 的三步走。

# 动手观察

> 不用写代码，用系统工具亲眼看看上面这些概念。

1. **看核数**：任务管理器-性能-CPU（右下角「内核/逻辑处理器」），或 Linux `lscpu`；
2. **看进程里的线程**：跑起 axum 服务，Windows 用任务管理器-详细信息加「线程数」列（或 Process Explorer）；  
   Linux `ps -eLf | grep http-server | wc -l`——你会看到 Tokio 起的工作线程数 ≈ 逻辑核数（+ 若干辅助线程）；
3. **看上下文切换**：Linux `vmstat 1` 的 `cs` 列（每秒切换次数）；跑「每连接一线程」版再压测，看它飙升；
4. **对比三步的线程数**：分别跑 [《从零手写 HTTP》](../http/http-from-scratch.md) 的 step1/step2/step3，  
   压 50 个并发连接，观察线程数：step1 恒为 1，step2 随连接数涨，step3 恒为核数左右——三种模型的差别在数字上一目了然。

# 三句话带走

1. **核心一次只能跑一条指令流**；多核是真并行的唯一来源；「程序都在同时跑」是调度器毫秒级轮转造出的幻觉——并发是结构、并行是资源。
2. **进程隔离而昂贵（跨进程要 IPC，如 Redis）；线程共享地址空间** ——共享是零成本通信的福，也是数据竞争的祸根；  
   线程的栈和上下文切换让它贵（C10K 的根源）。
3. **「阻塞」= 线程被内核挂起、期间它背着的一切全部停摆** ——这是 async 铁律的物理含义；异步运行时 = epoll 多路复用 + 少量线程 + 纳秒级切换的用户态任务（状态机）。

| 术语 | 一句话解释 | 在后面哪里出现 |
| --- | --- | --- |
| 核心 core | CPU 里独立的执行单元，一次一条指令流 | Tokio 工作线程数 |
| 逻辑核心/超线程 | 一个物理核伪装成两个，收益有限 | `available_parallelism` |
| 缓存行 cache line | 缓存搬运的最小单位（64B），多核争写同一行会「弹跳」 | ArcSwap 读锁记账开销 |
| 进程 process | 程序的运行实例：独立地址空间 + 资源 + 线程 | 跨进程共享 |
| 虚拟地址空间 | 进程的「私人内存幻觉」，隔离的来源 | 进程隔离 |
| IPC | 跨进程通信：管道/socket/共享内存 | Redis 即网络 IPC |
| 线程 thread | 进程内的执行流；共享地址空间，独有栈 + 寄存器上下文 | 多线程、共享状态 |
| 栈 stack | 线程独有的局部变量区，OS 线程默认 ~8MB | 线程成本 |
| 调度器 scheduler | 内核中决定「哪个线程上哪个核」的组件 | 对照 GMP |
| 时间片/抢占 | 运行一小段强制换人；OS 抢占式 vs Tokio 协作式 | 霸占线程实验 |
| 阻塞 blocked | 线程在等事件，被内核挂起、不占核心 | async 铁律 |
| 上下文切换 | 保存/恢复寄存器换线程，微秒级 + 缓存变冷 | 上下文切换的成本 |
| 状态机 state machine | 「一组状态 + 当前状态 + 转移规则」；把「执行到哪」变成普通值，暂停/恢复零成本 | Future 的本质 |
| 数据竞争 data race | 多线程无同步地读写同一数据，结果不确定 | 共享状态 |
| epoll/IOCP | 「这堆 socket 哪些就绪了」的批量查询，多路复用 | Tokio 的地基 |
| C10K | 一万并发连接问题：线程模型的天花板 | 手写 HTTP 三步走 |
