# 掀开 goroutine

> 预备课，无配套代码。前置：[《协程与状态机》](../os/coroutine-state.md)（线程/调度/阻塞/epoll）。

你天天在写 `go func(){...}()`，这一篇把它掀开：goroutine 到底是什么、GMP 怎么调度、  
阻塞时运行时在背后干了什么。懂了 Go 的实现，Rust/Tokio 的每个设计取舍都有了参照系——全书里所有「对照 Go」的论断（自动补线程、  
带抢占、netpoller…），这一篇给出机制层面的答案。

----

# goroutine 是什么

> 一句话：**goroutine 是 Go 运行时在用户态实现的轻量线程** —— [《协程与状态机》](../os/coroutine-state.md) 那张三种并发单元表里「用户态任务」那一列的有栈流派。

它和 OS 线程的关系：**不是** 一比一包装，而是「多对多」复用——成千上万个 goroutine 跑在少数几个 OS 线程上，  
由 Go 自己的调度器（不是内核）分配。这套设计要回答三个问题，也就是本篇的主线：

- 轻在哪？（栈）
- 谁来调度、怎么调度？（GMP、抢占）
- 阻塞了怎么办？（与 OS 关联最深的部分）

----

# 轻量的秘密

> OS 线程栈默认 ~8MB 是 **预先划死** 的；goroutine 反其道而行——起步 2KB，按需长大。

- 创建时只给 **~2KB** 栈（所以百万 goroutine 只是 GB 级内存）；
- 每个 Go 函数的开头都被编译器插了几条指令（序言）：**检查剩余栈够不够本函数用**，不够就触发扩容；
- 扩容方式是「搬家」：分配一块两倍大的新栈，把旧栈内容 **整体拷贝** 过去，然后修正所有指向旧栈的指针。

----

# 栈能搬家与 Pin

> 「栈能搬家」是个了不起的能力，顺便解释 Rust 的 `Pin` 从哪来。

把栈拷去新地址，栈里所有指向栈内的指针就全错了——Go 敢这么干，是因为 **运行时精确知道每个指针在哪**（GC 本来就需要这份信息），搬完统一修正。

而 Rust 没有运行时和 GC， **没有能力搬家后修指针** ——所以 Rust 的异步任务干脆不用栈（编译成状态机，  
见 [《协程与状态机》](../os/coroutine-state.md)），而状态机内部可能有自引用（一个字段指向另一个字段），一旦整个状态机被挪动同样会悬空。  
这就是你在 [《Tokio 运行时》](../async/tokio.md) 见到 **`Box::pin`** 的根本原因：  
**Pin = 向编译器承诺「这个东西从此不挪窝」**。同一个问题（「指向自己内部的指针怎么办」），Go 用运行时修正解决，  
Rust 用「钉住不许动」解决——两种哲学的又一次分野。

----

# GMP 三个角色

> Go 调度器的三个角色，是理解一切的基础。

| 字母 | 全称 | 是什么 | 数量 |
| --- | --- | --- | --- |
| **G** | goroutine | 一个并发单元：小栈 + 寄存器快照 + 状态 | 想开多少开多少 |
| **M** | machine | 一个真 OS 线程（内核眼里只有它） | 按需创建，可达上千 |
| **P** | processor | 「逻辑处理器」：调度的席位 + 本地运行队列 | = `GOMAXPROCS`，默认为核数 |

规则一句话：**M 必须先抢到一个 P，才有资格执行 G**。P 的数量固定为核数，所以 **同时真正在跑 Go 代码的线程永远不超过核数** ——多出来的 M 都在干别的（睡觉、  
陷在系统调用里）。

----

# 为什么要有 P

> 早期 Go 只有 G 和 M，所有 M 抢一个全局队列，锁争用严重。P 这个中间层就是为了解决它。

- 每个 P 有自己的 **本地运行队列**（装最多 256 个 G）：M 从自己 P 的队列取 G， **不加锁** ——这是调度快的关键；
- P 还挂着内存分配缓存（mcache）等每核资源，同样免锁；
- 全局队列退化为「溢出区」，偶尔才碰。

```
     [P0 本地队列: G G G]      [P1 本地队列: G G]      （P 数 = 核数）
          │                        │
        [M0]──执行中的 G         [M1]──执行中的 G       （持 P 的 M 在干活）
                                              [M2 睡眠] [M3 陷在系统调用里]（没 P 的 M）
     [全局队列: G G ...]  ← 溢出区，大家偶尔来取
```

----

# 工作窃取

> 某个 P 的本地队列空了怎么办？按顺序找活，实在没有就去别人那偷一半。

找活顺序：本地队列 → 全局队列 → **去别的 P 偷一半** → netpoller 里捞就绪的 G。「偷一半」保证负载自动均衡。

对照 Tokio：[《Tokio 运行时》](../async/tokio.md) 多线程运行时也是「每工作线程一个本地队列 + 工作窃取」—— **这套结构就是从 GMP 一脉相承的**（准确说都源自更早的 work-stealing 调度理论）。  
差别不在队列结构，在下面：抢占与阻塞处理。

----

# 抢占怎么实现

> Go 怎么打断一个不讲理的 goroutine（比如纯计算死循环）？手段进化过两代。

- **协作式检查点（老办法）**：编译器在每个函数序言里插的「栈检查」顺便充当调度检查点——运行时想抢占谁，就把它的栈警戒线设成无穷大，  
  下次函数调用时检查必然「失败」，控制权交回调度器。 **漏洞**：一个不调用任何函数的纯计算循环（`for { x++ }`）永远不会路过检查点，  
  1.13 之前真能卡死调度；
- **基于信号的异步抢占（Go 1.14+）**：运行时里有个特殊线程 **sysmon**（不需要 P、独立于 GMP 转），  
  每隔一会巡逻一遍，发现某个 G 连续跑了 **10ms** 还不让位，就向它所在的 M 发一个 **SIGURG 信号** ——信号处理函数在内核的配合下强行打断该 G（把它的现场保存成快照、  
  压回队列），M 转去跑别人。

和调度那节对上了：**内核抢占线程靠硬件时钟中断，Go 抢占 goroutine 靠 OS 信号** ——都是「借外力打断」。  
而 Tokio 是纯库、两样外力都借不到，所以只能协作式（那个死循环霸占线程的实验，在 Go 1.14+ 里不成立，  
在 Tokio 里永远成立）。

----

# 阻塞系统调用补线程

> Go「会自动补线程救你」的真相就在这。Go 把阻塞分三类，第一类是阻塞系统调用（读文件、某些 cgo）。

G 发起一个会阻塞的系统调用时， **内核挂起的是整个 M**（内核不认识 goroutine）。Go 的对策：

1. 进系统调用前先打标记；
2. sysmon 巡逻发现「这个 M 陷进去有一会了」→ 把 P 从这个 M 身上 **剥离**；
3. P 带着队列里其他 G **改嫁** 给另一个 M（空闲的，没有就 **新建一个 OS 线程**）——其他 G 照跑不误；
4. 原 M 从系统调用回来后，两手空空（没 P 了）：把手里的 G 放回队列，自己进入空闲池。

这就是「自动补线程」的真相：**代价是 M 的数量会膨胀** ——一千个 goroutine 同时陷在慢系统调用里，  
就真的有约一千个 OS 线程（每个 8MB 栈、内核调度压力，账单一分不少）。Go 只是把这件事做成了 **自动**，  
没有做成 **免费**。

对照 Rust：Tokio 不巡逻、不自动剥离——它要求 **你** 把阻塞操作显式交给 `spawn_blocking` 的专用线程池。  
手动 vs 自动，但底层账单一样：阻塞一定占用一个真线程。

----

# 网络 IO 靠 netpoller

> 网络读写 **不走** 上面那条昂贵路径。Go 把所有网络 socket 设成非阻塞，配一个 netpoller（Linux 上就是 epoll）。

1. G 读一个没数据的 socket → 运行时把 G **停车**（gopark：改成等待态、从 M 上摘下来，纯用户态操作，  
   M 和 P 毫发无损，转身跑别人）；
2. socket 登记进 epoll，关联着这个 G；
3. 数据到了 → 内核在 epoll 里报告 → 运行时把 G 改回可运行、塞进队列，等 P 捡起来从断点继续。

把这条链和 [《从零手写 HTTP》](../http/http-from-scratch.md) 的异步版并排放—— **一模一样**：

| | Go | Tokio |
| --- | --- | --- |
| socket 模式 | 非阻塞 | 非阻塞 |
| 没数据时 | gopark 停车 goroutine | poll 返回 Pending 挂起任务 |
| 谁盯着 socket | netpoller（epoll） | reactor（epoll） |
| 就绪后 | 把 G 标记可运行、入队 | 调 Waker、任务入队 |

所以「Go 写同步风格代码也能扛高并发」没有魔法：**netpoller 在背后把你的『同步』read 偷偷改写成了异步**。  
Rust 只是把这层不再隐藏、用 `.await` 明说。

----

# channel 锁不进内核

> 第三类阻塞：在 channel 上等、在锁上排队。它连内核都不进，纯用户态。

G 在 channel 上等数据、在 `sync.Mutex` 上排队时，走的也是 gopark 停车—— **纯用户态**，  
把 G 挂到 channel/锁自己的等待队列里，M 和 P 继续跑别人；对端 send/unlock 时直接把停着的 G 拎回运行队列。  
全程不碰内核，这也是「channel 通信很便宜」的原因。

**三种阻塞总结表**（本篇最值得记的一张）：

| 阻塞类型 | 谁被挂起 | 走不走内核 | 线程有损耗吗 |
| --- | --- | --- | --- |
| 阻塞系统调用（文件/cgo） | 整个 M | 走（内核挂起线程） | 有：P 改嫁、可能新建 M |
| 网络 IO | 只有 G | 就绪通知走 epoll | 无：M/P 照常干活 |
| channel/锁/sleep | 只有 G | 完全不走 | 无 |

----

# channel 内部

> `make(chan T, n)` 在堆上分配一个 hchan 结构，掀开看看它长什么样。

核心是四样：

- 一个 **环形缓冲区**（容量 n，无缓冲 channel 就没有它）；
- **sendq**：排队等着发的 goroutine 链表（缓冲满了的发送者停在这）；
- **recvq**：排队等着收的 goroutine 链表（缓冲空了的接收者停在这）；
- 一把小 **互斥锁** 保护以上所有（是的，channel 内部有锁——「用通信代替共享内存」的通信本身是用锁 + 队列实现的）。

两个有味道的细节：

- **直递优化**：send 时如果 recvq 里正停着一个等待的接收者，数据 **直接从发送者拷到接收者的栈上**，绕过缓冲区——少一次拷贝；
- **无缓冲 channel** = 只有直递路径：收发双方必须「面对面交接」，一方先到就停车等另一方——这就是「无缓冲 channel 即同步点」的实现层含义。

对照 tokio mpsc：同样是「缓冲 + 两个等待队列」，只是队列里停的是任务（挂 Waker）而不是 goroutine（gopark）——你现在能看出它们是同一张图纸了。

# sync.Mutex 一瞥

> Go 的 Mutex 和现代锁设计一致，和 [《共享状态：Arc / RwLock》](../async/shared-state.md) 讲的对得上。

- **快路径**：一条 CAS 原子指令，用户态、纳秒级；
- **慢路径**：先自旋几圈（赌马上会释放），不行就通过信号量沉到内核等待（Linux 上最终是 futex，即阻塞态）——不过 Go 多垫了一层：  
  先 gopark 停 goroutine，真不行才涉及 M；
- **饥饿模式**（Go 1.9+）：某个等待者超过 1ms 没拿到锁，锁切换成严格 FIFO 交接，防饿死——对照 [《共享状态：Arc / RwLock》](../async/shared-state.md)：  
  tokio Mutex 天生 FIFO，std Mutex 不保证，Go 在两者之间做了自适应。

另外，运行时里还有几个常驻角色一笔带过：**sysmon**（巡逻员：发起抢占、剥离陷在系统调用里的 P、网络轮询兜底、  
GC 触发）； **GC worker**（垃圾回收也以 goroutine 形式并发跑，这是「goroutine 数量比你开的多」的原因之一）。

# Go vs Tokio 对照

> 把两套机制并排放，差别一目了然。

| 维度 | Go（GMP） | Rust（Tokio） |
| --- | --- | --- |
| 并发单元 | 有栈协程（2KB 可增长栈 + 寄存器快照） | 无栈状态机（几十字节~KB） |
| 「函数可暂停」怎么实现 | 运行时换栈，任何函数随处可停 | 编译器改写，只能停在 `.await` |
| 自引用/搬家问题 | 运行时搬栈修指针 | 不许搬：Pin 钉住 |
| 调度结构 | P 本地队列 + 全局队列 + 工作窃取 | 每线程本地队列 + 全局队列 + 工作窃取（同款） |
| 抢占 | 有：sysmon + 信号，10ms 强制打断 | 无：纯协作式，死循环会霸占线程 |
| 阻塞系统调用 | 自动：P 改嫁、补 M | 手动：spawn_blocking 专用线程池 |
| 网络 IO | netpoller（epoll），对用户隐藏 | reactor（epoll），以 `.await` 显式呈现 |
| 运行时 | 语言内置，不可拆卸 | 库，可插拔可裁剪 |
| 换来的代价/收益 | 心智极简；每单元 2KB 起、全家桶不可选 | 单元最轻、零强制开销；心智负担前移到编译期 |

一句话总结这张表：**Go 用一个厚运行时把复杂度全部吃掉，Rust 用编译器把复杂度提前结清**。两边的底层素材（线程、  
epoll、工作窃取、CAS）完全相同——差别只在「谁来付账、什么时候付」。

# 动手观察

> 在你熟悉的 Go 里做几个实验，把上面的机制看成数字。

1. **看调度器现场**：`GODEBUG=schedtrace=1000 ./你的程序`——每秒打一行 GMP 状态（P 数、  
   就绪 G 数、M 数、系统调用中的线程数），跑个高并发程序观察 M 怎么涨；
2. **验证「阻塞系统调用补线程」**：起 500 个 goroutine 各自打开文件做慢读（或调用会阻塞的 cgo），  
   另开终端 `ps -eLf | grep 程序名 | wc -l` 看 OS 线程数飙升；再换成 500 个 goroutine 做 `time.Sleep`（纯用户态停车），  
   线程数纹丝不动——三种阻塞的差别在数字上立现；
3. **验证抢占**：写一个 `for { x++ }` 的纯计算 goroutine + 一个每秒打印的 goroutine，  
   `GOMAXPROCS=1` 跑——Go 1.14+ 里打印照常（信号抢占生效）；对比 Tokio 版（current_thread 下会全卡死），  
   亲手摸到「抢占 vs 协作」的分界线；
4. **数 goroutine**：`runtime.NumGoroutine()` 在 main 一开始就打——比你想的多（GC worker 等系统 goroutine）。

# 三句话带走

1. **goroutine = 2KB 起步可增长的有栈协程**；栈能搬家是因为运行时知道所有指针（Rust 没这能力，  
   所以选了无栈状态机 + Pin）——两条路线在「可暂停函数怎么实现」上分了岔。
2. **GMP：M（真线程）必须持 P（席位，数量=核数）才能跑 G**；本地队列免锁 + 工作窃取（Tokio 同款结构）；  
   sysmon 用信号实现 10ms 抢占（Tokio 做不到的外力）。
3. **三种阻塞三种待遇**：阻塞系统调用 → P 改嫁补 M（自动但不免费）；网络 IO → netpoller/epoll 停车 goroutine（和 Tokio 的 reactor 同一张图纸）；  
   channel/锁 → 纯用户态停车。「Go 写同步代码扛高并发」的全部魔法就是 netpoller 在背后做异步。
