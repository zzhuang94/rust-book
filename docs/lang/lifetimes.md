# 生命周期防悬垂

> 代码：[`code/lang-lifetimes/`](../../code/lang-lifetimes/)　  
> 运行：在 `code/` 目录下执行 `cargo run -p lang-lifetimes`

> 前置：[《所有权与借用》](ownership.md)（必读）。  
> 上一章管「谁拥有、谁在借、能不能同时写」。  
> 这一章管另一件事：**引用能不能活得比它指向的数据还久**。  
> 能，就是悬垂引用——C 里最危险的 bug 之一；Rust 用生命周期在编译期掐死它。

----

# 悬垂引用长什么样

> 「指向已释放内存的引用」= 悬垂引用。Rust **直接拒绝** 这种程序。

```rust
fn main() {
    let r;
    {
        let x = 5;
        r = &x;          // ❌ error[E0597]: `x` does not live long enough
    }                    // x 在这里被 drop
    // println!("{r}");  // 若允许，r 指向已释放的栈槽
}
```

这段在 **Go 里合法**：逃逸分析发现 `x` 被外部引用，会把 `x` 挪到堆上，GC 兜底。  
Rust 既不偷偷挪，也不靠 GC—— **引用活得比数据久 = 编译错误**。

> 🔬 **生命周期是记账，不是控制杆。**  
> 它不延长、不缩短任何值的真实存活时间；只用来查账：  
> 这个引用借自谁？那个「谁」还活着吗？

----

# 'a 是说明书

> 当函数 **返回引用**，且编译器分不清「到底借自哪个参数」时，  
> 你要用 `'a` 这类标注写一份说明书。  
> **关键认知：`'a` 不让引用活得更久，它只描述约束。**

经典例子——返回较长的那个字符串切片：

```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

读法：

1. `<'a>`：本函数引入一个生命周期参数，名叫 `'a`；  
2. `x: &'a str`：`x` 的有效期至少是 `'a`；`y` 同理；  
3. `-> &'a str`：返回的引用有效期也是 `'a`——  
   也就是说， **它不会比 `x`、`y` 里较短命的那个活得更久**。

编译器拿着说明书，去每个调用点查账：

```rust
let s1 = String::from("长长的字符串");
let result;
{
    let s2 = String::from("短");
    result = longest(&s1, &s2);  // 说明书：result ≤ min(s1, s2) 的寿命
}                                 // s2 死了 → result 按说明书已过期
// println!("{result}");          // ❌ E0597
```

合法调用：保证两个参数都活过「使用返回值」的那个点：

```rust
let s1 = String::from("长长的字符串");
let s2 = String::from("短");
let result = longest(&s1, &s2);
println!("{result}");             // ✅ s1、s2 都还在
```

**为什么编译器不自己推断？**  
函数体可能返回 `x` 或 `y`，签名必须写出 **最保守的依赖**，  
这样调用方只看签名就能检查，不必翻函数体——这也是报错能精准的原因。

# 大多数时候不用写

> 前面那么多 `&str` 都没写 `'a`？因为有三条 **省略规则（elision）**，  
> 编译器帮你补了 90% 的场景。

三条规则（简化版）：

1. **每个引用参数** 各自获得一个独立的生命周期参数；  
2. 若 **只有一个** 引用参数，它的生命周期自动赋给所有 **返回的引用**；  
3. 若是方法且有 `&self` / `&mut self`，返回的引用默认借自 `self`。

所以这些 **不必手写**，和显式版完全等价：

```rust
// 省略
fn first_word(s: &str) -> &str { /* ... */ }

// 显式（等价）
fn first_word<'a>(s: &'a str) -> &'a str { /* ... */ }
```

方法同理：

```rust
impl Book {
    fn title_ref(&self) -> &str { &self.title }
    // 等价于 fn title_ref<'a>(&'a self) -> &'a str
}
```

**信号**：报错 `missing lifetime specifier`  
= 编译器说「说明书信息不够，请你补一下」——典型就是 `longest` 那种两个输入都可能当返回值。

# 结构体里存引用

> 结构体字段若是引用，必须声明生命周期——  
> 等于承诺：**我这个结构体活不过我借的那份数据**。

```rust
struct Excerpt<'a> {
    part: &'a str,
}

let novel = String::from("Call me Ishmael. Some years ago...");
let first = novel.split('.').next().unwrap();
let excerpt = Excerpt { part: first };
// excerpt 不能活过 novel
```

**给初学者的强烈建议**：  
让结构体 **持有所有权**（存 `String` / `Vec` / `Arc`），  
能避开 80% 的生命周期难题。本书几乎所有业务 struct 都这么做。  
真要零拷贝再上引用字段。

```rust
struct ExcerptOwned {
    part: String,    // 无需 <'a>
}
```

# 到处见到的 'static

> `'static` = 「可以活到程序结束」这种最长生命周期。

常见来源：

| 东西 | 为什么是 `'static` |
| --- | --- |
| 字符串字面量 `"abc"` | 直接编进二进制，进程结束才没 |
| 某些全局常量 | 同理 |
| 拥有所有权且不借外部短期数据的值 | 例如 `String` 本身可被要求 `'static`（任务拿走所有权后） |

```rust
let s: &'static str = "编进二进制的字面量";
```

和异步的关系（后面会反复撞上）：  
[《Tokio 运行时》](../async/tokio.md) 的 `spawn` 要求 Future 是 `'static`——  
后台任务 **不许借用** 任何「可能先死」的栈上数据。  
合法三选一：

1. **`move` 进去**（拿走所有权）；  
2. **`Arc` 共享所有权**；  
3. 只借用真正的 `'static` 数据（字面量等）。

这和 [《函数与闭包》](functions-closures.md) 的 `move` 是同一类问题，只是挂上了生命周期名字。

# 两个生命周期参数

> 有时返回值只跟 **其中一个** 参数走，另一个只是陪跑。  
> 若硬写成同一个 `'a`，会把陪跑的也绑进约束，调用处无故变严。

```rust
// 返回值只可能借自 first → 用两个生命周期说清楚
fn first_or_hint<'a, 'b>(first: &'a str, _hint: &'b str) -> &'a str {
    first
}
```

对照「错误地」写成同一个：

```rust
fn first_or_hint<'a>(first: &'a str, hint: &'a str) -> &'a str {
    first  // 即使不用 hint，说明书仍要求 hint 和 first 同寿
}
```

口诀：**返回值依赖谁，就在签名里跟谁绑同一字母；无关的参数用别的字母。**

进阶还有 `'a: 'b`（`'a` 活得至少和 `'b` 一样久）等子类型关系——  
日常写业务少见，读标准库签名时再查即可。

# 和 NLL 别混

> 上一章的 **NLL**：一个借用在「最后一次使用」后结束，好让后面的代码再借。  
> 本章的 **生命周期标注**：跨函数边界时，返回的引用和参数之间的约束说明书。

| | NLL | 生命周期参数 `'a` |
| --- | --- | --- |
| 管什么 | 函数 **内部** 借用何时结束 | 函数 **边界** 上引用从哪来、能活多久 |
| 你要不要写 | 不用，编译器自动 | 多数省略；分不清来源时手写 |
| 典型报错 | E0502 借用冲突 | E0597 活得不够久 / missing lifetime |

两者都叫「lifetime」相关，但解题思路不同：  
冲突 → 调顺序/缩作用域；悬垂 → 改返回所有权或补对说明书。

----

# Go 对照：逃逸 vs 拒绝

| 场景 | Go | Rust |
| --- | --- | --- |
| 返回局部变量指针 | 逃逸分析上堆 + GC | 编译拒绝（E0597） |
| 引用比数据活得久 | GC 保证数据还在 | 必须在类型上证明活得够久 |
| 结构体存别人的指针 | 很常见 | 要标 `'a`，初学建议改存所有权 |
| 后台任务捕获变量 | goroutine + GC | `move` / `Arc` / `'static` |

Go 的策略是「运行时兜底」；Rust 是「编译期证明」。  
所以你在 Rust 里会多写一点类型，少掉一整类线上崩溃。

# 报错怎么拆

> 生命周期报错读起来像天书，按下面拆：

1. **`does not live long enough`（E0597）**  
   引用还在用，数据已经要 drop。  
   改法：扩大数据的作用域；或别返回引用，改返回 `String`/`Vec`/`Arc`。

2. **`missing lifetime specifier`**  
   返回了引用，编译器不知道跟哪个参数走。  
   改法：给参数和返回值补上 `'a`（参考 `longest`）。

3. **`lifetime may not live long enough`**  
   说明书和函数体矛盾（比如声称返回借自 `x`，却返回了局部变量）。  
   改法：要么改实现去满足说明书，要么改说明书/改返回拥有的值。

处置优先级（和所有权章同一套精神）：

1. **能返回所有权就返回所有权**（最省心）；  
2. 确认数据作用域盖住所有使用点；  
3. 再手写 `'a` 把关系说清；  
4. 多方长期共享 → `Arc`（[《智能指针全家桶》](smart-pointers.md)）。

# 三句话带走

1. **生命周期标注是说明书，不是控制杆** ——描述引用借自谁、能活多久。  
2. **90% 场景靠省略规则**；出现 `missing lifetime specifier` 再手写。  
3. **初学让结构体持有所有权**；`'static` + `move`/`Arc` 是异步任务的标配思路。

# 附：本章生词表

- **悬垂引用（dangling reference）**：指向已释放内存的引用。  
- **生命周期（lifetime）**：引用有效期的编译期标记与约束。  
- **生命周期参数 `'a`**：签名上的占位符，用来把多个引用绑到同一有效期约束。  
- **省略规则（elision）**：编译器自动补生命周期标注的三条惯例。  
- **`'static`**：可活到程序结束的生命周期；字面量满足。  
- **E0597**：`does not live long enough`——数据活得不够久。  
- **逃逸分析**：Go 把「被外部引用的局部变量」挪到堆上的编译器技术。  
- **说明书 vs 控制杆**：`'a` 只描述约束，不改变实际 drop 时机。
